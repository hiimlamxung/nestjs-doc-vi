Resolvers

Resolver cung cấp các hướng dẫn để chuyển đổi một GraphQL operation (query, mutation hoặc subscription) thành dữ liệu. Chúng trả về cùng dạng dữ liệu mà chúng ta chỉ định trong schema — có thể đồng bộ hoặc dưới dạng promise resolve ra kết quả có dạng đó. Thông thường, bạn tạo một resolver map thủ công. Ngược lại, package @nestjs/graphql tạo resolver map tự động bằng cách sử dụng metadata được cung cấp bởi các decorator mà bạn dùng để annotate các class. Để minh họa quá trình sử dụng các tính năng của package để tạo GraphQL API, chúng ta sẽ tạo một API authors đơn giản.

Code first #

Trong cách tiếp cận code first, chúng ta không tuân theo quy trình thông thường là tạo GraphQL schema bằng cách viết GraphQL SDL thủ công. Thay vào đó, chúng ta sử dụng TypeScript decorator để tạo SDL từ các định nghĩa class TypeScript. Package @nestjs/graphql đọc metadata được định nghĩa thông qua các decorator và tự động tạo schema cho bạn.

Object types #

Hầu hết các định nghĩa trong một GraphQL schema là object type. Mỗi object type bạn định nghĩa nên đại diện cho một domain object mà client của ứng dụng có thể cần tương tác. Ví dụ, API mẫu của chúng ta cần có khả năng lấy danh sách author và các post của họ, vì vậy chúng ta nên định nghĩa type Author và type Post để hỗ trợ chức năng này.

Nếu chúng ta sử dụng cách tiếp cận schema first, chúng ta sẽ định nghĩa schema như vậy bằng SDL như sau:

GraphQL
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post!]!
}

Trong trường hợp này, sử dụng cách tiếp cận code first, chúng ta định nghĩa schema bằng TypeScript class và sử dụng TypeScript decorator để annotate các field của các class đó. Phiên bản tương đương của SDL trên trong cách tiếp cận code first là:

authors/models/author.model
import { Field, Int, ObjectType } from '@nestjs/graphql';
import { Post } from './post';

@ObjectType()
export class Author {
  @Field(type => Int)
  id: number;

  @Field({ nullable: true })
  firstName?: string;

  @Field({ nullable: true })
  lastName?: string;

  @Field(type => [Post])
  posts: Post[];
}
Gợi ý

Hệ thống metadata reflection của TypeScript có một số hạn chế khiến cho việc xác định class bao gồm những property nào hoặc nhận biết một property cho trước là optional hay required trở nên bất khả thi. Do những hạn chế này, chúng ta phải sử dụng decorator @Field() một cách tường minh trong các class định nghĩa schema để cung cấp metadata về GraphQL type và tính optional của mỗi field, hoặc sử dụng CLI plugin để tự động tạo chúng.

Object type Author, giống như bất kỳ class nào, được tạo thành từ một tập hợp các field, với mỗi field khai báo một type. Type của field tương ứng với một GraphQL type. GraphQL type của field có thể là một object type khác hoặc một scalar type. GraphQL scalar type là một kiểu nguyên thủy (như ID, String, Boolean hoặc Int) resolve thành một giá trị đơn.

Gợi ý

Ngoài các scalar type tích hợp sẵn của GraphQL, bạn có thể định nghĩa các custom scalar type (đọc thêm).

Định nghĩa object type Author ở trên sẽ khiến Nest tạo ra SDL mà chúng ta đã trình bày ở trên:

GraphQL
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post!]!
}

Decorator @Field() nhận một type function tùy chọn (ví dụ: type => Int), và tùy chọn một options object.

Type function là bắt buộc khi có khả năng gây nhập nhằng giữa TypeScript type system và GraphQL type system. Cụ thể: nó không bắt buộc cho kiểu string và boolean; nó bắt buộc cho number (phải được ánh xạ sang GraphQL Int hoặc Float). Type function chỉ đơn giản trả về GraphQL type mong muốn (như được trình bày trong các ví dụ khác nhau trong các chương này).

Options object có thể có bất kỳ cặp key/value nào sau đây:

  • nullable: để chỉ định liệu một field có nullable hay không (trong @nestjs/graphql, mỗi field mặc định là non-nullable); boolean
  • description: để thiết lập mô tả cho field; string
  • deprecationReason: để đánh dấu field là deprecated; string

Ví dụ:

TypeScript
@Field({ description: `Book title`, deprecationReason: 'Not useful in v2 schema' })
title: string;
Gợi ý

Bạn cũng có thể thêm description, hoặc deprecate toàn bộ object type: @ObjectType({ description: 'Author model' }).

Khi field là một mảng, chúng ta phải chỉ định thủ công kiểu mảng trong type function của decorator Field(), như dưới đây:

TypeScript
@Field(type => [Post])
posts: Post[];
Gợi ý

Sử dụng ký hiệu ngoặc vuông ([ ]), chúng ta có thể chỉ định độ sâu của mảng. Ví dụ, sử dụng [[Int]] sẽ biểu diễn một ma trận số nguyên.

Để khai báo rằng các phần tử của mảng (không phải chính mảng) là nullable, hãy thiết lập property nullable thành 'items' như dưới đây:

TypeScript
@Field(type => [Post], { nullable: 'items' })
posts: Post[];
Gợi ý

Nếu cả mảng lẫn các phần tử của nó đều nullable, hãy thiết lập nullable thành 'itemsAndList'.

Bây giờ object type Author đã được tạo, hãy định nghĩa object type Post.

posts/models/post.model
import { Field, Int, ObjectType } from '@nestjs/graphql';

@ObjectType()
export class Post {
  @Field(type => Int)
  id: number;

  @Field()
  title: string;

  @Field(type => Int, { nullable: true })
  votes?: number;
}

Object type Post sẽ tạo ra phần sau của GraphQL schema trong SDL:

GraphQL
type Post {
  id: Int!
  title: String!
  votes: Int
}

Code first resolver #

Tại thời điểm này, chúng ta đã định nghĩa các object (type definition) có thể tồn tại trong data graph, nhưng client chưa có cách nào để tương tác với các object đó. Để giải quyết điều đó, chúng ta cần tạo một resolver class. Trong phương pháp code first, một resolver class vừa định nghĩa các resolver function vừa tạo ra Query type. Điều này sẽ rõ ràng hơn khi chúng ta thực hiện ví dụ dưới đây:

authors/authors.resolver
@Resolver(() => Author)
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query(() => Author)
  async author(@Args('id', { type: () => Int }) id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField()
  async posts(@Parent() author: Author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}
Gợi ý

Tất cả các decorator (ví dụ: @Resolver, @ResolveField, @Args, v.v.) đều được export từ package @nestjs/graphql.

Bạn có thể định nghĩa nhiều resolver class. Nest sẽ kết hợp chúng lại tại thời điểm chạy. Xem phần module bên dưới để biết thêm về cách tổ chức code.

Lưu ý

Logic bên trong các class AuthorsService và PostsService có thể đơn giản hoặc phức tạp tùy ý. Điểm chính của ví dụ này là trình bày cách xây dựng resolver và cách chúng tương tác với các provider khác.

Trong ví dụ trên, chúng ta đã tạo AuthorsResolver, định nghĩa một query resolver function và một field resolver function. Để tạo resolver, chúng ta tạo một class với các resolver function làm method, và annotate class bằng decorator @Resolver().

Trong ví dụ này, chúng ta định nghĩa một query handler để lấy author object dựa trên id được gửi trong request. Để chỉ định rằng method là một query handler, hãy sử dụng decorator @Query().

Argument được truyền vào decorator @Resolver() là tùy chọn, nhưng phát huy tác dụng khi graph trở nên phức tạp. Nó được sử dụng để cung cấp parent object mà các field resolver function sử dụng khi duyệt xuống qua object graph.

Trong ví dụ của chúng ta, vì class bao gồm một field resolver function (cho property posts của object type Author), chúng ta phải cung cấp cho decorator @Resolver() một giá trị để chỉ ra class nào là parent type (tức là tên class ObjectType tương ứng) cho tất cả các field resolver được định nghĩa trong class này. Như rõ ràng trong ví dụ, khi viết field resolver function, cần truy cập parent object (object mà field đang được resolve thuộc về). Trong ví dụ này, chúng ta populate mảng posts của author bằng field resolver gọi service nhận id của author làm argument. Do đó, cần xác định parent object trong decorator @Resolver(). Lưu ý cách sử dụng tương ứng decorator tham số method @Parent() để trích xuất tham chiếu đến parent object đó trong field resolver.

Chúng ta có thể định nghĩa nhiều @Query() resolver function (cả trong class này lẫn trong bất kỳ resolver class nào khác), và chúng sẽ được tổng hợp vào một định nghĩa Query type duy nhất trong SDL được tạo ra cùng với các entry tương ứng trong resolver map. Điều này cho phép bạn định nghĩa các query gần với model và service mà chúng sử dụng, và giữ chúng được tổ chức tốt trong các module.

Gợi ý

Nest CLI cung cấp một generator (schematic) tự động tạo tất cả boilerplate code để giúp chúng ta tránh phải làm tất cả điều này, và làm cho trải nghiệm developer đơn giản hơn nhiều. Đọc thêm về tính năng này tại đây.

Query type names #

Trong các ví dụ trên, decorator @Query() tạo GraphQL schema query type name dựa trên tên method. Ví dụ, xét cấu trúc sau từ ví dụ trên:

TypeScript
@Query(() => Author)
async author(@Args('id', { type: () => Int }) id: number) {
  return this.authorsService.findOneById(id);
}

Điều này tạo ra entry sau cho author query trong schema (query type sử dụng cùng tên với tên method):

GraphQL
type Query {
  author(id: Int!): Author
}
Gợi ý

Tìm hiểu thêm về GraphQL query tại đây.

Theo quy ước, chúng ta thường muốn tách biệt các tên này; ví dụ, chúng ta muốn sử dụng tên như getAuthor() cho query handler method, nhưng vẫn sử dụng author cho query type name. Điều tương tự áp dụng cho field resolver. Chúng ta có thể dễ dàng làm điều này bằng cách truyền mapping name làm argument của các decorator @Query() và @ResolveField(), như dưới đây:

authors/authors.resolver
@Resolver(() => Author)
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query(() => Author, { name: 'author' })
  async getAuthor(@Args('id', { type: () => Int }) id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField('posts', () => [Post])
  async getPosts(@Parent() author: Author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}

Handler method getAuthor ở trên sẽ tạo ra phần sau của GraphQL schema trong SDL:

GraphQL
type Query {
  author(id: Int!): Author
}

Query decorator options #

Options object của decorator @Query() (nơi chúng ta truyền {name: 'author'} ở trên) chấp nhận một số cặp key/value:

  • name: tên của query; string
  • description: mô tả sẽ được sử dụng để tạo tài liệu GraphQL schema (ví dụ: trong GraphQL playground); string
  • deprecationReason: thiết lập metadata query để hiển thị query là deprecated (ví dụ: trong GraphQL playground); string
  • nullable: liệu query có thể trả về null data response hay không; boolean hoặc 'items' hoặc 'itemsAndList' (xem ở trên để biết chi tiết về 'items' và 'itemsAndList')

Args decorator options #

Sử dụng decorator @Args() để trích xuất argument từ request để sử dụng trong method handler. Điều này hoạt động theo cách rất tương tự với trích xuất argument tham số route REST.

Thông thường, decorator @Args() của bạn sẽ đơn giản và không cần một object argument, như đã thấy với method getAuthor() ở trên. Ví dụ, nếu kiểu của identifier là string, cấu trúc sau đây là đủ, và đơn giản trích xuất named field từ inbound GraphQL request để sử dụng làm method argument.

TypeScript
@Args('id') id: string

Trong trường hợp getAuthor(), kiểu number được sử dụng, điều này đặt ra thách thức. Kiểu number của TypeScript không cung cấp đủ thông tin về biểu diễn GraphQL mong đợi (ví dụ: Int vs. Float). Do đó, chúng ta phải truyền tường minh type reference. Chúng ta làm điều đó bằng cách truyền argument thứ hai vào decorator Args(), chứa argument options, như dưới đây:

TypeScript
@Query(() => Author, { name: 'author' })
async getAuthor(@Args('id', { type: () => Int }) id: number) {
  return this.authorsService.findOneById(id);
}

Options object cho phép chúng ta chỉ định các cặp key-value tùy chọn sau:

  • type: một function trả về GraphQL type
  • defaultValue: giá trị mặc định; any
  • description: metadata mô tả; string
  • deprecationReason: để deprecate một field và cung cấp metadata mô tả lý do; string
  • nullable: liệu field có nullable hay không

Các query handler method có thể nhận nhiều argument. Hãy tưởng tượng rằng chúng ta muốn lấy author dựa trên firstName và lastName. Trong trường hợp này, chúng ta có thể gọi @Args hai lần:

TypeScript
getAuthor(
  @Args('firstName', { nullable: true }) firstName?: string,
  @Args('lastName', { defaultValue: '' }) lastName?: string,
) {}
Gợi ý

Trong trường hợp firstName, là một GraphQL nullable field, không cần thêm các kiểu non-value null hoặc undefined vào kiểu của field này. Chỉ cần lưu ý, bạn sẽ cần type guard cho các kiểu non-value có thể có trong resolver, vì một GraphQL nullable field sẽ cho phép các kiểu đó đi qua resolver của bạn.

Dedicated arguments class #

Với các lệnh gọi @Args() inline, code như ví dụ trên sẽ trở nên cồng kềnh. Thay vào đó, bạn có thể tạo một class argument GetAuthorArgs chuyên dụng và truy cập nó trong handler method như sau:

TypeScript
@Args() args: GetAuthorArgs

Tạo class GetAuthorArgs sử dụng @ArgsType() như dưới đây:

authors/dto/get-author.args
import { MinLength } from 'class-validator';
import { Field, ArgsType } from '@nestjs/graphql';

@ArgsType()
class GetAuthorArgs {
  @Field({ nullable: true })
  firstName?: string;

  @Field({ defaultValue: '' })
  @MinLength(3)
  lastName: string;
}
Gợi ý

Một lần nữa, do các hạn chế của hệ thống metadata reflection của TypeScript, cần phải sử dụng decorator @Field để chỉ định thủ công type và tính optional, hoặc sử dụng CLI plugin. Ngoài ra, trong trường hợp firstName, là một GraphQL nullable field, không cần thêm các kiểu non-value null hoặc undefined vào kiểu của field này. Chỉ cần lưu ý, bạn sẽ cần type guard cho các kiểu non-value có thể có trong resolver, vì một GraphQL nullable field sẽ cho phép các kiểu đó đi qua resolver của bạn.

Điều này sẽ tạo ra phần sau của GraphQL schema trong SDL:

GraphQL
type Query {
  author(firstName: String, lastName: String = ''): Author
}
Gợi ý

Lưu ý rằng các argument class như GetAuthorArgs hoạt động rất tốt với ValidationPipe (đọc thêm).

Class inheritance #

Bạn có thể sử dụng kế thừa class TypeScript tiêu chuẩn để tạo các base class với các tính năng utility type generic (field và field property, validation, v.v.) có thể được mở rộng. Ví dụ, bạn có thể có một tập hợp các argument liên quan đến pagination luôn bao gồm các field tiêu chuẩn offset và limit, nhưng cũng có các field index khác dành riêng cho type. Bạn có thể thiết lập một hệ thống phân cấp class như dưới đây.

Base @ArgsType() class:

TypeScript
@ArgsType()
class PaginationArgs {
  @Field(() => Int)
  offset: number = 0;

  @Field(() => Int)
  limit: number = 10;
}

Sub-class dành riêng cho type của base @ArgsType() class:

TypeScript
@ArgsType()
class GetAuthorArgs extends PaginationArgs {
  @Field({ nullable: true })
  firstName?: string;

  @Field({ defaultValue: '' })
  @MinLength(3)
  lastName: string;
}

Cách tiếp cận tương tự có thể được áp dụng với các object @ObjectType(). Định nghĩa các property generic trên base class:

TypeScript
@ObjectType()
class Character {
  @Field(() => Int)
  id: number;

  @Field()
  name: string;
}

Thêm các property dành riêng cho type trên sub-class:

TypeScript
@ObjectType()
class Warrior extends Character {
  @Field()
  level: number;
}

Bạn cũng có thể sử dụng kế thừa với resolver. Bạn có thể đảm bảo type safety bằng cách kết hợp kế thừa và TypeScript generic. Ví dụ, để tạo một base class với query findAll generic, sử dụng cấu trúc như sau:

TypeScript
function BaseResolver<T extends Type<unknown>>(classRef: T): any {
  @Resolver({ isAbstract: true })
  abstract class BaseResolverHost {
    @Query(() => [classRef], { name: `findAll${classRef.name}` })
    async findAll(): Promise<T[]> {
      return [];
    }
  }
  return BaseResolverHost;
}

Lưu ý những điều sau:

  • Cần có return type tường minh (any ở trên); nếu không, TypeScript sẽ phàn nàn về việc sử dụng định nghĩa private class. Khuyến nghị: định nghĩa một interface thay vì sử dụng any.
  • Type được import từ package @nestjs/common
  • Property isAbstract: true chỉ ra rằng SDL (các câu lệnh Schema Definition Language) không nên được tạo cho class này. Lưu ý, bạn cũng có thể thiết lập property này cho các type khác để ngăn việc tạo SDL.

Đây là cách bạn có thể tạo một concrete subclass của BaseResolver:

TypeScript
@Resolver(() => Recipe)
export class RecipesResolver extends BaseResolver(Recipe) {
  constructor(private recipesService: RecipesService) {
    super();
  }
}

Cấu trúc này sẽ tạo ra SDL sau:

GraphQL
type Query {
  findAllRecipe: [Recipe!]!
}

Generics #

Chúng ta đã thấy một cách sử dụng generic ở trên. Tính năng mạnh mẽ này của TypeScript có thể được sử dụng để tạo các abstraction hữu ích. Ví dụ, đây là một triển khai pagination dựa trên cursor mẫu dựa trên tài liệu này:

TypeScript
import { Field, ObjectType, Int } from '@nestjs/graphql';
import { Type } from '@nestjs/common';

interface IEdgeType<T> {
  cursor: string;
  node: T;
}

export interface IPaginatedType<T> {
  edges: IEdgeType<T>[];
  nodes: T[];
  totalCount: number;
  hasNextPage: boolean;
}

export function Paginated<T>(classRef: Type<T>): Type<IPaginatedType<T>> {
  @ObjectType(`${classRef.name}Edge`)
  abstract class EdgeType {
    @Field(() => String)
    cursor: string;

    @Field(() => classRef)
    node: T;
  }

  @ObjectType({ isAbstract: true })
  abstract class PaginatedType implements IPaginatedType<T> {
    @Field(() => [EdgeType], { nullable: true })
    edges: EdgeType[];

    @Field(() => [classRef], { nullable: true })
    nodes: T[];

    @Field(() => Int)
    totalCount: number;

    @Field()
    hasNextPage: boolean;
  }
  return PaginatedType as Type<IPaginatedType<T>>;
}

Với base class ở trên đã được định nghĩa, giờ chúng ta có thể dễ dàng tạo các type chuyên biệt kế thừa hành vi này. Ví dụ:

TypeScript
@ObjectType()
class PaginatedAuthor extends Paginated(Author) {}

Schema first #

Như đã đề cập trong chương trước, trong cách tiếp cận schema first, chúng ta bắt đầu bằng cách định nghĩa thủ công các schema type trong SDL (đọc thêm). Xem xét các định nghĩa SDL type sau.

Gợi ý

Để tiện lợi trong chương này, chúng ta đã tổng hợp tất cả SDL vào một vị trí (ví dụ: một file .graphql, như dưới đây). Trong thực tế, bạn có thể thấy phù hợp khi tổ chức code theo kiểu modular. Ví dụ, có thể hữu ích khi tạo các file SDL riêng lẻ với các định nghĩa type đại diện cho từng domain entity, cùng với các service liên quan, resolver code và Nest module definition class, trong một thư mục chuyên dụng cho entity đó. Nest sẽ tổng hợp tất cả các định nghĩa schema type riêng lẻ tại thời điểm chạy.

GraphQL
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post]
}

type Post {
  id: Int!
  title: String!
  votes: Int
}

type Query {
  author(id: Int!): Author
}

Schema first resolver #

Schema ở trên expose một query duy nhất — author(id: Int!): Author.

Gợi ý

Tìm hiểu thêm về GraphQL query tại đây.

Bây giờ hãy tạo class AuthorsResolver để resolve các author query:

authors/authors.resolver
@Resolver('Author')
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query()
  async author(@Args('id') id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField()
  async posts(@Parent() author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}
Gợi ý

Tất cả các decorator (ví dụ: @Resolver, @ResolveField, @Args, v.v.) đều được export từ package @nestjs/graphql.

Lưu ý

Logic bên trong các class AuthorsService và PostsService có thể đơn giản hoặc phức tạp tùy ý. Điểm chính của ví dụ này là trình bày cách xây dựng resolver và cách chúng tương tác với các provider khác.

Decorator @Resolver() là bắt buộc. Nó nhận một string argument tùy chọn với tên của một class. Tên class này là bắt buộc bất cứ khi nào class chứa các decorator @ResolveField() để thông báo cho Nest rằng method được decorate liên kết với parent type (type Author trong ví dụ hiện tại). Thay vào đó, thay vì thiết lập @Resolver() ở đầu class, điều này có thể được thực hiện cho từng method:

TypeScript
@Resolver('Author')
@ResolveField()
async posts(@Parent() author) {
  const { id } = author;
  return this.postsService.findAll({ authorId: id });
}

Trong trường hợp này (decorator @Resolver() ở cấp method), nếu bạn có nhiều decorator @ResolveField() bên trong một class, bạn phải thêm @Resolver() cho tất cả chúng. Điều này không được coi là best practice (vì nó tạo thêm overhead).

Gợi ý

Bất kỳ class name argument nào được truyền vào @Resolver() không ảnh hưởng đến query (decorator @Query()) hoặc mutation (decorator @Mutation()).

Cảnh báo

Việc sử dụng decorator @Resolver ở cấp method không được hỗ trợ với cách tiếp cận code first.

Trong các ví dụ trên, decorator @Query() và @ResolveField() được liên kết với các GraphQL schema type dựa trên tên method. Ví dụ, xét cấu trúc sau từ ví dụ trên:

TypeScript
@Query()
async author(@Args('id') id: number) {
  return this.authorsService.findOneById(id);
}

Điều này tạo ra entry sau cho author query trong schema (query type sử dụng cùng tên với tên method):

GraphQL
type Query {
  author(id: Int!): Author
}

Theo quy ước, chúng ta thường muốn tách biệt các tên này, sử dụng các tên như getAuthor() hoặc getPosts() cho resolver method. Chúng ta có thể dễ dàng làm điều này bằng cách truyền mapping name làm argument cho decorator, như dưới đây:

authors/authors.resolver
@Resolver('Author')
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query('author')
  async getAuthor(@Args('id') id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField('posts')
  async getPosts(@Parent() author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}
Gợi ý

Nest CLI cung cấp một generator (schematic) tự động tạo tất cả boilerplate code để giúp chúng ta tránh phải làm tất cả điều này, và làm cho trải nghiệm developer đơn giản hơn nhiều. Đọc thêm về tính năng này tại đây.

Generating types #

Giả sử rằng chúng ta sử dụng cách tiếp cận schema first và đã bật tính năng tạo typing (với outputAs: 'class' như đã trình bày trong chương trước), sau khi bạn chạy ứng dụng, nó sẽ tạo ra file sau (tại vị trí bạn chỉ định trong method GraphQLModule.forRoot()). Ví dụ, trong src/graphql.ts:

graphql
export class Author {
  id: number;
  firstName?: string;
  lastName?: string;
  posts?: Post[];
}
export class Post {
  id: number;
  title: string;
  votes?: number;
}

export abstract class IQuery {
  abstract author(id: number): Author | Promise<Author>;
}

Bằng cách tạo class (thay vì kỹ thuật mặc định là tạo interface), bạn có thể sử dụng decorator validation khai báo kết hợp với cách tiếp cận schema first, đây là một kỹ thuật cực kỳ hữu ích (đọc thêm). Ví dụ, bạn có thể thêm decorator class-validator vào class CreatePostInput được tạo ra như dưới đây để enforce độ dài string tối thiểu và tối đa trên field title:

TypeScript
import { MinLength, MaxLength } from 'class-validator';

export class CreatePostInput {
  @MinLength(3)
  @MaxLength(50)
  title: string;
}
Lưu ý

Để bật auto-validation cho input (và parameter), hãy sử dụng ValidationPipe. Đọc thêm về validation tại đây và cụ thể hơn về pipe tại đây.

Tuy nhiên, nếu bạn thêm decorator trực tiếp vào file được tạo tự động, chúng sẽ bị ghi đè mỗi khi file được tạo lại. Thay vào đó, hãy tạo một file riêng và đơn giản extend class được tạo ra.

TypeScript
import { MinLength, MaxLength } from 'class-validator';
import { Post } from '../../graphql.ts';

export class CreatePostInput extends Post {
  @MinLength(3)
  @MaxLength(50)
  title: string;
}

GraphQL argument decorators #

Chúng ta có thể truy cập các argument chuẩn của GraphQL resolver bằng cách sử dụng các decorator chuyên dụng. Dưới đây là so sánh giữa các Nest decorator và các tham số Apollo thuần mà chúng đại diện.

Nest Decorator Apollo Parameter
@Root() và @Parent() root/parent
@Context(param?: string) context / context[param]
@Info(param?: string) info / info[param]
@Args(param?: string) args / args[param]

Các argument này có ý nghĩa sau:

  • root: một object chứa kết quả trả về từ resolver trên parent field, hoặc trong trường hợp field Query cấp cao nhất, là rootValue được truyền từ cấu hình server.
  • context: một object được chia sẻ bởi tất cả resolver trong một query cụ thể; thường được sử dụng để chứa state mỗi request.
  • info: một object chứa thông tin về trạng thái thực thi của query.
  • args: một object chứa các argument được truyền vào field trong query.

Module #

Khi chúng ta hoàn thành các bước trên, chúng ta đã khai báo tất cả thông tin cần thiết bởi GraphQLModule để tạo resolver map. GraphQLModule sử dụng reflection để introspect metadata được cung cấp thông qua các decorator, và biến đổi class thành resolver map chính xác một cách tự động.

Điều duy nhất khác bạn cần quan tâm là cung cấp (tức là liệt kê như một provider trong một module nào đó) các resolver class (AuthorsResolver), và import module (AuthorsModule) ở đâu đó, để Nest có thể sử dụng nó.

Ví dụ, chúng ta có thể làm điều này trong AuthorsModule, module cũng có thể cung cấp các service khác cần thiết trong context này. Hãy chắc chắn import AuthorsModule ở đâu đó (ví dụ: trong root module, hoặc một module khác được import bởi root module).

authors/authors.module
@Module({
  imports: [PostsModule],
  providers: [AuthorsService, AuthorsResolver],
})
export class AuthorsModule {}
Gợi ý

Sẽ hữu ích khi tổ chức code theo cái gọi là domain model (tương tự cách bạn tổ chức entry point trong REST API). Trong cách tiếp cận này, giữ các model (class ObjectType), resolver và service cùng nhau trong một Nest module đại diện cho domain model. Giữ tất cả các thành phần này trong một thư mục duy nhất cho mỗi module. Khi bạn làm điều này, và sử dụng Nest CLI để tạo từng phần tử, Nest sẽ tự động kết nối tất cả các phần này lại (định vị file trong các thư mục phù hợp, tạo entry trong mảng provider và imports, v.v.).
