Other features

Trong thế giới GraphQL, có rất nhiều cuộc tranh luận về việc xử lý các vấn đề như authentication, hoặc side-effects của các operation. Chúng ta nên xử lý mọi thứ bên trong business logic? Chúng ta nên sử dụng higher-order function để tăng cường các query và mutation với logic authorization? Hay chúng ta nên sử dụng schema directives? Không có một câu trả lời duy nhất phù hợp cho tất cả các câu hỏi này.

Nest giúp giải quyết các vấn đề này với các tính năng đa nền tảng của nó như guards và interceptors. Triết lý là giảm sự trùng lặp và cung cấp công cụ giúp tạo ra các ứng dụng có cấu trúc tốt, dễ đọc và nhất quán.

Overview #

Bạn có thể sử dụng các guards, interceptors, filters và pipes tiêu chuẩn theo cùng một cách với GraphQL như với bất kỳ ứng dụng RESTful nào. Ngoài ra, bạn có thể dễ dàng tạo các decorator riêng bằng cách tận dụng tính năng custom decorators. Hãy xem một ví dụ về GraphQL query handler.

TypeScript
@Query('author')
@UseGuards(AuthGuard)
async getAuthor(@Args('id', ParseIntPipe) id: number) {
  return this.authorsService.findOneById(id);
}

Như bạn có thể thấy, GraphQL hoạt động với cả guards và pipes theo cùng cách như các HTTP REST handler. Vì vậy, bạn có thể chuyển logic authentication của mình sang một guard; thậm chí bạn có thể tái sử dụng cùng một guard class cho cả REST và GraphQL API interface. Tương tự, interceptors hoạt động trên cả hai loại ứng dụng theo cùng một cách:

TypeScript
@Mutation()
@UseInterceptors(EventsInterceptor)
async upvotePost(@Args('postId') postId: number) {
  return this.postsService.upvoteById({ id: postId });
}

Execution context #

Vì GraphQL nhận một loại dữ liệu khác trong request đến, execution context mà cả guards và interceptors nhận được có phần khác biệt giữa GraphQL và REST. Các GraphQL resolver có một tập hợp argument riêng biệt: root, args, context, và info. Do đó, guards và interceptors phải chuyển đổi ExecutionContext generic thành một GqlExecutionContext. Điều này rất đơn giản:

TypeScript
import { CanActivate, ExecutionContext, Injectable } from '@nestjs/common';
import { GqlExecutionContext } from '@nestjs/graphql';

@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const ctx = GqlExecutionContext.create(context);
    return true;
  }
}

Object GraphQL context được trả về bởi GqlExecutionContext.create() expose một phương thức get cho mỗi argument của GraphQL resolver (ví dụ: getArgs(), getContext(), v.v.). Sau khi chuyển đổi, chúng ta có thể dễ dàng lấy bất kỳ GraphQL argument nào cho request hiện tại.

Exception filters #

Các exception filters tiêu chuẩn của Nest cũng tương thích với các ứng dụng GraphQL. Tương tự như ExecutionContext, các ứng dụng GraphQL nên chuyển đổi object ArgumentsHost thành object GqlArgumentsHost.

TypeScript
@Catch(HttpException)
export class HttpExceptionFilter implements GqlExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const gqlHost = GqlArgumentsHost.create(host);
    return exception;
  }
}
Gợi ý

Cả GqlExceptionFilter và GqlArgumentsHost đều được import từ package @nestjs/graphql.

Lưu ý rằng khác với trường hợp REST, bạn không sử dụng object response gốc để tạo response.

Custom decorators #

Như đã đề cập, tính năng custom decorators hoạt động như mong đợi với các GraphQL resolver.

TypeScript
export const User = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) =>
    GqlExecutionContext.create(ctx).getContext().user,
);

Sử dụng custom decorator @User() như sau:

TypeScript
@Mutation()
async upvotePost(
  @User() user: UserEntity,
  @Args('postId') postId: number,
) {}
Gợi ý

Trong ví dụ trên, chúng ta đã giả định rằng object user được gán vào context của ứng dụng GraphQL của bạn.

Execute enhancers at the field resolver level #

Trong ngữ cảnh GraphQL, Nest không chạy các enhancers (tên gọi chung cho interceptors, guards và filters) ở cấp field (xem issue này): chúng chỉ chạy cho method @Query()/@Mutation() cấp cao nhất. Bạn có thể yêu cầu Nest thực thi interceptors, guards hoặc filters cho các method được đánh dấu bằng @ResolveField() bằng cách thiết lập tùy chọn fieldResolverEnhancers trong GqlModuleOptions. Truyền vào một danh sách 'interceptors', 'guards', và/hoặc 'filters' tùy theo nhu cầu:

TypeScript
GraphQLModule.forRoot({
  fieldResolverEnhancers: ['interceptors']
}),
Cảnh báo

Việc bật enhancers cho field resolvers có thể gây ra vấn đề hiệu năng khi bạn trả về nhiều bản ghi và field resolver của bạn được thực thi hàng nghìn lần. Vì lý do này, khi bạn bật fieldResolverEnhancers, chúng tôi khuyên bạn nên bỏ qua việc thực thi các enhancers không thực sự cần thiết cho các field resolvers. Bạn có thể làm điều này bằng cách sử dụng hàm helper sau:

TypeScript
export function isResolvingGraphQLField(context: ExecutionContext): boolean {
  if (context.getType<GqlContextType>() === 'graphql') {
    const gqlContext = GqlExecutionContext.create(context);
    const info = gqlContext.getInfo();
    const parentType = info.parentType.name;
    return parentType !== 'Query' && parentType !== 'Mutation';
  }
  return false;
}

Creating a custom driver #

Nest cung cấp hai driver chính thức sẵn có: @nestjs/apollo và @nestjs/mercurius, cũng như một API cho phép các nhà phát triển xây dựng các custom driver mới. Với custom driver, bạn có thể tích hợp bất kỳ thư viện GraphQL nào hoặc mở rộng tích hợp hiện có, thêm các tính năng bổ sung lên trên.

Ví dụ, để tích hợp package express-graphql, bạn có thể tạo driver class sau:

TypeScript
import { AbstractGraphQLDriver, GqlModuleOptions } from '@nestjs/graphql';
import { graphqlHTTP } from 'express-graphql';

class ExpressGraphQLDriver extends AbstractGraphQLDriver {
  async start(options: GqlModuleOptions<any>): Promise<void> {
    options = await this.graphQlFactory.mergeWithSchema(options);

    const { httpAdapter } = this.httpAdapterHost;
    httpAdapter.use(
      '/graphql',
      graphqlHTTP({
        schema: options.schema,
        graphiql: true,
      }),
    );
  }

  async stop() {}
}

Và sau đó sử dụng nó như sau:

TypeScript
GraphQLModule.forRoot({
  driver: ExpressGraphQLDriver,
});
