Subscriptions

Ngoài việc lấy dữ liệu bằng query và thay đổi dữ liệu bằng mutation, đặc tả GraphQL còn hỗ trợ loại thao tác thứ ba, gọi là subscription. GraphQL subscriptions là cách đẩy dữ liệu từ server đến các client chọn lắng nghe các tin nhắn thời gian thực từ server. Subscriptions tương tự như queries ở chỗ chúng chỉ định một tập hợp các trường sẽ được gửi đến client, nhưng thay vì trả về ngay một câu trả lời duy nhất, một kênh được mở ra và kết quả được gửi đến client mỗi khi một sự kiện cụ thể xảy ra trên server.

Một trường hợp sử dụng phổ biến của subscriptions là thông báo cho phía client về các sự kiện cụ thể, ví dụ việc tạo một đối tượng mới, cập nhật các trường, v.v. (đọc thêm tại đây).

Bật subscriptions với Apollo driver #

Để bật subscriptions, đặt thuộc tính installSubscriptionHandlers thành true.

GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  installSubscriptionHandlers: true,
}),
Cảnh báo

Tùy chọn cấu hình installSubscriptionHandlers đã bị xóa khỏi phiên bản mới nhất của Apollo server và sẽ sớm bị deprecated trong package này. Mặc định, installSubscriptionHandlers sẽ fallback sử dụng subscriptions-transport-ws (đọc thêm) nhưng chúng tôi khuyến nghị mạnh mẽ sử dụng thư viện graphql-ws (đọc thêm) thay thế.

Để chuyển sang sử dụng package graphql-ws, hãy dùng cấu hình sau:

GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'graphql-ws': true
  },
}),
Gợi ý

Bạn cũng có thể sử dụng cả hai package (subscriptions-transport-ws và graphql-ws) cùng lúc, ví dụ để tương thích ngược.

Code first #

Để tạo một subscription sử dụng cách tiếp cận code first, chúng ta dùng decorator @Subscription() (được export từ package @nestjs/graphql) và class PubSub từ package graphql-subscriptions, cung cấp một publish/subscribe API đơn giản.

Subscription handler sau đây đảm nhận việc subscribe vào một sự kiện bằng cách gọi PubSub#asyncIterableIterator. Phương thức này nhận một đối số duy nhất, triggerName, tương ứng với tên topic sự kiện.

const pubSub = new PubSub();

@Resolver(() => Author)
export class AuthorResolver {
  // ...
  @Subscription(() => Comment)
  commentAdded() {
    return pubSub.asyncIterableIterator('commentAdded');
  }
}
Gợi ý

Tất cả các decorator được export từ package @nestjs/graphql, trong khi class PubSub được export từ package graphql-subscriptions.

Lưu ý

PubSub là một class cung cấp publish và subscribe API đơn giản. Đọc thêm về nó tại đây. Lưu ý rằng tài liệu Apollo cảnh báo rằng triển khai mặc định không phù hợp cho production (đọc thêm tại đây). Các ứng dụng production nên sử dụng triển khai PubSub được hỗ trợ bởi một store bên ngoài (đọc thêm tại đây).

Điều này sẽ tạo ra phần sau của GraphQL schema trong SDL:

type Subscription {
  commentAdded(): Comment!
}

Lưu ý rằng subscriptions, theo định nghĩa, trả về một đối tượng với một thuộc tính cấp cao nhất duy nhất có key là tên của subscription. Tên này được kế thừa từ tên của phương thức subscription handler (tức là commentAdded ở trên), hoặc được cung cấp rõ ràng bằng cách truyền một option với key name làm đối số thứ hai cho decorator @Subscription(), như được hiển thị bên dưới.

@Subscription(() => Comment, {
  name: 'commentAdded',
})
subscribeToCommentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}

Cấu trúc này tạo ra cùng một SDL như mẫu code trước đó, nhưng cho phép chúng ta tách rời tên phương thức khỏi subscription.

Publishing #

Bây giờ, để publish sự kiện, chúng ta sử dụng phương thức PubSub#publish. Phương thức này thường được sử dụng trong một mutation để kích hoạt cập nhật phía client khi một phần của object graph đã thay đổi. Ví dụ:

posts/posts.resolver
@Mutation(() => Comment)
async addComment(
  @Args('postId', { type: () => Int }) postId: number,
  @Args('comment', { type: () => Comment }) comment: CommentInput,
) {
  const newComment = this.commentsService.addComment({ id: postId, comment });
  pubSub.publish('commentAdded', { commentAdded: newComment });
  return newComment;
}

Phương thức PubSub#publish nhận một triggerName (hãy nghĩ về nó như một tên topic sự kiện) làm tham số đầu tiên, và một event payload làm tham số thứ hai. Như đã đề cập, subscription, theo định nghĩa, trả về một giá trị và giá trị đó có một hình dạng. Hãy xem lại SDL được tạo cho subscription commentAdded của chúng ta:

type Subscription {
  commentAdded(): Comment!
}

Điều này cho chúng ta biết rằng subscription phải trả về một đối tượng với tên thuộc tính cấp cao nhất là commentAdded có giá trị là một đối tượng Comment. Điểm quan trọng cần lưu ý là hình dạng của event payload được phát ra bởi phương thức PubSub#publish phải tương ứng với hình dạng của giá trị được mong đợi trả về từ subscription. Vì vậy, trong ví dụ trên, câu lệnh pubSub.publish('commentAdded', { commentAdded: newComment }) publish một sự kiện commentAdded với payload có hình dạng phù hợp. Nếu các hình dạng này không khớp, subscription của bạn sẽ thất bại trong giai đoạn GraphQL validation.

Lọc subscriptions #

Để lọc ra các sự kiện cụ thể, đặt thuộc tính filter thành một hàm lọc. Hàm này hoạt động tương tự như hàm được truyền cho filter của mảng. Nó nhận hai đối số: payload chứa event payload (như được gửi bởi event publisher), và variables nhận bất kỳ đối số nào được truyền vào trong quá trình subscription request. Nó trả về một boolean xác định liệu sự kiện này có nên được publish đến các client listener hay không.

@Subscription(() => Comment, {
  filter: (payload, variables) =>
    payload.commentAdded.title === variables.title,
})
commentAdded(@Args('title') title: string) {
  return pubSub.asyncIterableIterator('commentAdded');
}

Thay đổi subscription payloads #

Để thay đổi event payload đã publish, đặt thuộc tính resolve thành một hàm. Hàm này nhận event payload (như được gửi bởi event publisher) và trả về giá trị phù hợp.

@Subscription(() => Comment, {
  resolve: value => value,
})
commentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}
Lưu ý

Nếu bạn sử dụng tùy chọn resolve, bạn nên trả về payload chưa được bọc (ví dụ, với ví dụ của chúng ta, trả về trực tiếp đối tượng newComment, không phải đối tượng { commentAdded: newComment }).

Nếu bạn cần truy cập các provider được inject (ví dụ sử dụng một service bên ngoài để validate dữ liệu), hãy sử dụng cấu trúc sau.

@Subscription(() => Comment, {
  resolve(this: AuthorResolver, value) {
    // "this" refers to an instance of "AuthorResolver"
    return value;
  }
})
commentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}

Cấu trúc tương tự hoạt động với filters:

@Subscription(() => Comment, {
  filter(this: AuthorResolver, payload, variables) {
    // "this" refers to an instance of "AuthorResolver"
    return payload.commentAdded.title === variables.title;
  }
})
commentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}

PubSub #

Chúng ta đã khởi tạo một instance PubSub local ở trên. Cách tiếp cận được ưu tiên là định nghĩa PubSub như một provider và inject nó thông qua constructor (sử dụng decorator @Inject()). Điều này cho phép chúng ta tái sử dụng instance trên toàn bộ ứng dụng. Ví dụ, định nghĩa một provider như sau, sau đó inject 'PUB_SUB' ở nơi cần thiết.

{
  provide: 'PUB_SUB',
  useValue: new PubSub(),
}

Tùy chỉnh subscriptions server #

Để tùy chỉnh subscriptions server (ví dụ thay đổi path), sử dụng thuộc tính tùy chọn subscriptions.

GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'subscriptions-transport-ws': {
      path: '/graphql'
    },
  }
}),

Nếu bạn đang sử dụng package graphql-ws cho subscriptions, hãy thay thế key subscriptions-transport-ws bằng graphql-ws, như sau:

GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'graphql-ws': {
      path: '/graphql'
    },
  }
}),

Xác thực qua WebSockets #

Kiểm tra xem người dùng đã được xác thực hay chưa có thể được thực hiện bên trong hàm callback onConnect mà bạn có thể chỉ định trong tùy chọn subscriptions.

onConnect sẽ nhận connectionParams được truyền đến SubscriptionClient làm đối số đầu tiên (đọc thêm).

GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'subscriptions-transport-ws': {
      onConnect: (connectionParams) => {
        const authToken = connectionParams.authToken;
        if (!isValid(authToken)) {
          throw new Error('Token is not valid');
        }
        // extract user information from token
        const user = parseToken(authToken);
        // return user info to add them to the context later
        return { user };
      },
    }
  },
  context: ({ connection }) => {
    // connection.context will be equal to what was returned by the "onConnect" callback
  },
}),

authToken trong ví dụ này chỉ được client gửi một lần, khi kết nối lần đầu được thiết lập. Tất cả các subscriptions được thực hiện với kết nối này sẽ có cùng authToken, và do đó cùng thông tin user.

Lưu ý

Có một bug trong subscriptions-transport-ws cho phép các kết nối bỏ qua giai đoạn onConnect (đọc thêm). Bạn không nên giả định rằng onConnect đã được gọi khi người dùng bắt đầu một subscription, và luôn kiểm tra rằng context đã được điền đầy đủ.

Nếu bạn đang sử dụng package graphql-ws, chữ ký của callback onConnect sẽ hơi khác:

GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'graphql-ws': {
      onConnect: (context: Context<any>) => {
        const { connectionParams, extra } = context;
        // user validation will remain the same as in the example above
        // when using with graphql-ws, additional context value should be stored in the extra field
        extra.user = { user: {} };
      },
    },
  },
  context: ({ extra }) => {
    // you can now access your additional context value through the extra field
  },
});

PubSub #

Trong các ví dụ trên, chúng ta đã sử dụng PubSub emitter mặc định (mqemitter). Cách tiếp cận được ưu tiên (cho production) là sử dụng mqemitter-redis. Ngoài ra, có thể cung cấp một triển khai PubSub tùy chỉnh (đọc thêm tại đây).

GraphQLModule.forRoot<MercuriusDriverConfig>({
  driver: MercuriusDriver,
  subscription: {
    emitter: require('mqemitter-redis')({
      port: 6579,
      host: '127.0.0.1',
    }),
  },
});

Xác thực qua WebSockets #

Kiểm tra xem người dùng đã được xác thực hay chưa có thể được thực hiện bên trong hàm callback verifyClient mà bạn có thể chỉ định trong tùy chọn subscription.

verifyClient sẽ nhận đối tượng info làm đối số đầu tiên mà bạn có thể sử dụng để lấy các headers của request.

GraphQLModule.forRoot<MercuriusDriverConfig>({
  driver: MercuriusDriver,
  subscription: {
    verifyClient: (info, next) => {
      const authorization = info.req.headers?.authorization as string;
      if (!authorization?.startsWith('Bearer ')) {
        return next(false);
      }
      next(true);
    },
  }
}),
