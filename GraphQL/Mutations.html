Mutations

Hầu hết các cuộc thảo luận về GraphQL tập trung vào việc lấy dữ liệu, nhưng bất kỳ nền tảng dữ liệu hoàn chỉnh nào cũng cần một cách để chỉnh sửa dữ liệu phía server. Trong REST, bất kỳ request nào cũng có thể gây ra side-effects trên server, nhưng best practice gợi ý rằng chúng ta không nên chỉnh sửa dữ liệu trong các GET request. GraphQL cũng tương tự - về mặt kỹ thuật, bất kỳ query nào cũng có thể được triển khai để gây ra việc ghi dữ liệu. Tuy nhiên, giống như REST, nên tuân theo quy ước rằng bất kỳ operation nào gây ra việc ghi dữ liệu nên được gửi một cách rõ ràng thông qua một mutation (đọc thêm tại đây).

Tài liệu chính thức của Apollo sử dụng ví dụ mutation upvotePost(). Mutation này triển khai một phương thức để tăng giá trị thuộc tính votes của một bài viết. Để tạo một mutation tương đương trong Nest, chúng ta sẽ sử dụng decorator @Mutation().

Code first #

Hãy thêm một phương thức khác vào AuthorResolver đã sử dụng ở phần trước (xem resolvers).

author.resolver
@Mutation(() => Post)
async upvotePost(@Args({ name: 'postId', type: () => Int }) postId: number) {
  return this.postsService.upvoteById({ id: postId });
}
Gợi ý

Tất cả các decorators (ví dụ: @Resolver, @ResolveField, @Args, v.v.) đều được export từ package @nestjs/graphql.

Điều này sẽ tạo ra phần sau của GraphQL schema trong SDL:

schema.graphql
type Mutation {
  upvotePost(postId: Int!): Post
}

Phương thức upvotePost() nhận postId (Int) làm argument và trả về một entity Post đã được cập nhật. Vì những lý do đã giải thích trong phần resolvers, chúng ta phải đặt rõ ràng kiểu mong đợi.

Nếu mutation cần nhận một object làm argument, chúng ta có thể tạo một input type. Input type là một loại object type đặc biệt có thể được truyền vào làm argument (đọc thêm tại đây). Để khai báo một input type, sử dụng decorator @InputType().

upvote-post.input
import { InputType, Field } from '@nestjs/graphql';

@InputType()
export class UpvotePostInput {
  @Field()
  postId: number;
}
Gợi ý

Decorator @InputType() nhận một options object làm argument, vì vậy bạn có thể, ví dụ, chỉ định description của input type. Lưu ý rằng, do hạn chế của hệ thống metadata reflection trong TypeScript, bạn phải sử dụng decorator @Field để chỉ định kiểu thủ công, hoặc sử dụng CLI plugin.

Sau đó chúng ta có thể sử dụng type này trong resolver class:

author.resolver
@Mutation(() => Post)
async upvotePost(
  @Args('upvotePostData') upvotePostData: UpvotePostInput,
) {}

Schema first #

Hãy mở rộng AuthorResolver đã sử dụng ở phần trước (xem resolvers).

author.resolver
@Mutation()
async upvotePost(@Args('postId') postId: number) {
  return this.postsService.upvoteById({ id: postId });
}

Lưu ý rằng chúng ta đã giả định ở trên rằng business logic đã được chuyển sang PostsService (truy vấn bài viết và tăng thuộc tính votes của nó). Logic bên trong class PostsService có thể đơn giản hoặc phức tạp tùy ý. Điểm chính của ví dụ này là cho thấy cách các resolvers có thể tương tác với các providers khác.

Bước cuối cùng là thêm mutation của chúng ta vào định nghĩa types hiện có.

schema.graphql
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post]
}

type Post {
  id: Int!
  title: String
  votes: Int
}

type Query {
  author(id: Int!): Author
}

type Mutation {
  upvotePost(postId: Int!): Post
}

Mutation upvotePost(postId: Int!): Post giờ đã sẵn sàng để được gọi như một phần của GraphQL API của ứng dụng.
