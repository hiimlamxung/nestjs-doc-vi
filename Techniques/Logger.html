Logger

Nest đi kèm với một text-based logger tích hợp sẵn, được sử dụng trong quá trình khởi động ứng dụng và một số trường hợp khác như hiển thị các caught exception (tức là system logging). Chức năng này được cung cấp thông qua class Logger trong package @nestjs/common. Bạn có thể kiểm soát hoàn toàn hành vi của hệ thống logging, bao gồm bất kỳ điều nào sau đây:

  • Vô hiệu hóa hoàn toàn việc logging
  • Chỉ định mức độ chi tiết của log (ví dụ: hiển thị errors, warnings, debug info, v.v.)
  • Cấu hình format của log message (raw, json, có màu, v.v.)
  • Ghi đè timestamp trong logger mặc định (ví dụ: sử dụng tiêu chuẩn ISO8601 làm date format)
  • Ghi đè hoàn toàn logger mặc định
  • Tùy chỉnh logger mặc định bằng cách extend nó
  • Sử dụng dependency injection để đơn giản hóa việc composing và testing ứng dụng của bạn

Bạn cũng có thể sử dụng logger tích hợp sẵn, hoặc tạo custom implementation của riêng mình, để log các application-level event và message của riêng bạn.

Nếu ứng dụng của bạn yêu cầu tích hợp với các hệ thống logging bên ngoài, file-based logging tự động, hoặc chuyển tiếp log đến centralized logging service, bạn có thể implement giải pháp logging hoàn toàn tùy chỉnh bằng thư viện logging Node.js. Một lựa chọn phổ biến là Pino, được biết đến với hiệu suất cao và sự linh hoạt.

Basic customization #

Để vô hiệu hóa logging, đặt property logger thành false trong (optional) Nest application options object được truyền làm argument thứ hai cho method NestFactory.create().

const app = await NestFactory.create(AppModule, {
  logger: false,
});
await app.listen(process.env.PORT ?? 3000);

Để bật các log level cụ thể, đặt property logger thành một mảng các chuỗi chỉ định các log level cần hiển thị, như sau:

const app = await NestFactory.create(AppModule, {
  logger: ['error', 'warn'],
});
await app.listen(process.env.PORT ?? 3000);

Các giá trị trong mảng có thể là bất kỳ sự kết hợp nào của 'log', 'fatal', 'error', 'warn', 'debug', và 'verbose'.

💡 Hint

Các log level trong Nest có tính chất cascading (kế thừa). Điều này có nghĩa là việc cung cấp một log level cụ thể (như 'log') sẽ tự động bao gồm tất cả các level có mức độ nghiêm trọng cao hơn (ví dụ: 'warn', 'error', và 'fatal').

Để vô hiệu hóa colorized output, truyền object ConsoleLogger với property colors được đặt thành false làm giá trị của property logger.

const app = await NestFactory.create(AppModule, {
  logger: new ConsoleLogger({
    colors: false,
  }),
});

Để cấu hình prefix cho mỗi log message, truyền object ConsoleLogger với thuộc tính prefix được thiết lập:

const app = await NestFactory.create(AppModule, {
  logger: new ConsoleLogger({
    prefix: 'MyApp', // Default is "Nest"
  }),
});

JSON logging #

JSON logging là điều cần thiết cho việc observability ứng dụng hiện đại và tích hợp với các log management system. Để bật JSON logging trong ứng dụng NestJS, cấu hình object ConsoleLogger với property json được đặt thành true. Sau đó, cung cấp cấu hình logger này làm giá trị cho property logger khi tạo application instance.

const app = await NestFactory.create(AppModule, {
  logger: new ConsoleLogger({
    json: true,
  }),
});

Cấu hình này xuất log ở structured JSON format, giúp dễ dàng tích hợp với các hệ thống bên ngoài như log aggregator và cloud platform. Ví dụ, các nền tảng như AWS ECS (Elastic Container Service) hỗ trợ JSON log một cách native, cho phép các tính năng nâng cao như:

  • Log Filtering: Dễ dàng thu hẹp log dựa trên các field như log level, timestamp hoặc custom metadata.
  • Search and Analysis: Sử dụng các công cụ query để phân tích và theo dõi xu hướng trong hành vi ứng dụng.

Ngoài ra, nếu bạn đang sử dụng NestJS Mau, JSON logging sẽ đơn giản hóa quá trình xem log ở structured, well-organized format, điều này đặc biệt hữu ích cho debugging và performance monitoring.

📘 Note

Khi json được đặt thành true, ConsoleLogger sẽ tự động vô hiệu hóa text colorization bằng cách đặt property colors thành false. Điều này đảm bảo rằng output vẫn là valid JSON, không có các formatting artifact. Tuy nhiên, cho mục đích development, bạn có thể override hành vi này bằng cách đặt colors thành true một cách rõ ràng. Điều này thêm colorized JSON log, giúp các log entry dễ đọc hơn trong quá trình local debugging.

Khi JSON logging được bật, output sẽ trông như thế này:

{
  "level": "log",
  "pid": 19096,
  "timestamp": 1607370779834,
  "message": "Starting Nest application...",
  "context": "NestFactory"
}

Using the logger for application logging #

Chúng ta có thể kết hợp một số kỹ thuật trên để cung cấp hành vi và formatting nhất quán cho cả Nest system logging và application event/message logging của riêng chúng ta.

Một thực hành tốt là khởi tạo class Logger từ @nestjs/common trong mỗi service. Chúng ta có thể cung cấp tên service làm argument context trong constructor của Logger, như sau:

import { Logger, Injectable } from '@nestjs/common';

@Injectable()
export class MyService {
  private readonly logger = new Logger(MyService.name);

  doSomething() {
    this.logger.log('Doing something...');
  }
}

Trong default logger implementation, context được in trong dấu ngoặc vuông, như NestFactory trong ví dụ bên dưới:

[Nest] 19096   - 12/08/2019, 7:12:59 AM   [NestFactory] Starting Nest application...

Nếu chúng ta cung cấp một custom logger thông qua app.useLogger(), nó sẽ thực sự được Nest sử dụng internally. Điều đó có nghĩa là code của chúng ta vẫn implementation-agnostic, trong khi chúng ta có thể dễ dàng thay thế logger mặc định bằng cách gọi app.useLogger().

Bằng cách đó, nếu chúng ta làm theo các bước từ phần trước và gọi app.useLogger(app.get(MyLogger)), các lệnh gọi sau đến this.logger.log() từ MyService sẽ dẫn đến việc gọi method log từ instance MyLogger.

Điều này sẽ phù hợp với hầu hết các trường hợp. Nhưng nếu bạn cần nhiều tùy chỉnh hơn (như thêm và gọi các custom method), hãy chuyển sang phần tiếp theo.

Logs with timestamps #

Để bật timestamp logging cho mỗi logged message, bạn có thể sử dụng thiết lập tùy chọn timestamp: true khi tạo logger instance.

import { Logger, Injectable } from '@nestjs/common';

@Injectable()
class MyService {
  private readonly logger = new Logger(MyService.name, { timestamp: true });

  doSomething() {
    this.logger.log('Doing something with timestamp here ->');
  }
}

Điều này sẽ tạo ra output với format như sau:

[Nest] 19096   - 04/19/2024, 7:12:59 AM   [MyService] Doing something with timestamp here +5ms

Lưu ý +5ms ở cuối dòng. Với mỗi log statement, thời gian chênh lệch từ message trước được tính toán và hiển thị ở cuối dòng.

Custom implementation #

Bạn có thể cung cấp custom logger implementation để Nest sử dụng cho system logging bằng cách đặt giá trị của property logger thành một object implement interface LoggerService. Ví dụ, bạn có thể yêu cầu Nest sử dụng JavaScript console object toàn cục tích hợp sẵn (implement interface LoggerService), như sau:

const app = await NestFactory.create(AppModule, {
  logger: console,
});
await app.listen(process.env.PORT ?? 3000);

Việc implement custom logger của riêng bạn rất đơn giản. Chỉ cần implement từng method của interface LoggerService như bên dưới.

import { LoggerService, Injectable } from '@nestjs/common';

@Injectable()
export class MyLogger implements LoggerService {
  /**
   * Write a 'log' level log.
   */
  log(message: any, ...optionalParams: any[]) {}

  /**
   * Write a 'fatal' level log.
   */
  fatal(message: any, ...optionalParams: any[]) {}

  /**
   * Write an 'error' level log.
   */
  error(message: any, ...optionalParams: any[]) {}

  /**
   * Write a 'warn' level log.
   */
  warn(message: any, ...optionalParams: any[]) {}

  /**
   * Write a 'debug' level log.
   */
  debug?(message: any, ...optionalParams: any[]) {}

  /**
   * Write a 'verbose' level log.
   */
  verbose?(message: any, ...optionalParams: any[]) {}
}

Sau đó, bạn có thể cung cấp instance của MyLogger thông qua property logger của Nest application options object.

const app = await NestFactory.create(AppModule, {
  logger: new MyLogger(),
});
await app.listen(process.env.PORT ?? 3000);

Extend built-in logger #

Thay vì viết logger từ đầu, bạn có thể đáp ứng nhu cầu bằng cách extend class ConsoleLogger tích hợp sẵn và override hành vi được chọn của default implementation.

import { ConsoleLogger } from '@nestjs/common';

export class MyLogger extends ConsoleLogger {
  error(message: any, stack?: string, context?: string) {
    // add your tailored logic here
    super.error(...arguments);
  }
}

Bạn có thể sử dụng extended logger trong các feature module như mô tả trong phần Injecting a custom logger bên dưới.

Để yêu cầu Nest sử dụng extended logger cho system logging, truyền instance của nó thông qua property logger của application options (như trong phần Custom implementation ở trên), hoặc sử dụng method app.useLogger() như trong phần Dependency injection bên dưới.

Dependency injection #

Để có chức năng logging nâng cao hơn, bạn sẽ muốn tận dụng dependency injection. Ví dụ, bạn có thể muốn inject một ConfigService vào logger để tùy chỉnh nó, và đến lượt nó, inject custom logger vào các controller và/hoặc provider khác. Để bật dependency injection cho custom logger, tạo một class implement LoggerService và đăng ký class đó như một provider trong một module. Ví dụ, bạn có thể:

  1. Định nghĩa một class MyLogger extend ConsoleLogger tích hợp sẵn hoặc override hoàn toàn nó, như trong các phần trước. Hãy đảm bảo implement interface LoggerService.
  2. Tạo một LoggerModule như bên dưới, và cung cấp MyLogger từ module đó.
logger.module.ts
import { Module } from '@nestjs/common';
import { MyLogger } from './my-logger.service';

@Module({
  providers: [MyLogger],
  exports: [MyLogger],
})
export class LoggerModule {}

Với cấu trúc này, bạn đang cung cấp custom logger để sử dụng bởi bất kỳ module nào khác. Vì class MyLogger là một phần của module, nó có thể sử dụng dependency injection (ví dụ: để inject ConfigService). Cần thêm một kỹ thuật nữa để cung cấp custom logger này cho Nest sử dụng cho system logging (ví dụ: cho bootstrapping và error handling).

Vì việc khởi tạo ứng dụng (NestFactory.create()) xảy ra bên ngoài context của bất kỳ module nào, nó không tham gia vào giai đoạn Dependency Injection bình thường của quá trình initialization. Vì vậy, chúng ta phải đảm bảo rằng ít nhất một application module import LoggerModule để trigger Nest khởi tạo singleton instance của class MyLogger.

Sau đó, chúng ta có thể hướng dẫn Nest sử dụng cùng singleton instance của MyLogger với cấu trúc sau:

main.ts
const app = await NestFactory.create(AppModule, {
  bufferLogs: true,
});
app.useLogger(app.get(MyLogger));
await app.listen(process.env.PORT ?? 3000);
📘 Note

Trong ví dụ trên, chúng ta đặt bufferLogs thành true để đảm bảo rằng tất cả log sẽ được buffer cho đến khi custom logger được đính kèm (MyLogger trong trường hợp này) và quá trình khởi tạo ứng dụng hoàn tất hoặc thất bại. Nếu quá trình khởi tạo thất bại, Nest sẽ fallback về ConsoleLogger ban đầu để in bất kỳ error message nào. Ngoài ra, bạn có thể đặt autoFlushLogs thành false (mặc định là true) để flush log thủ công (bằng method Logger.flush()).

Ở đây chúng ta sử dụng method get() trên NestApplication instance để truy xuất singleton instance của object MyLogger. Kỹ thuật này về cơ bản là cách để "inject" một logger instance để Nest sử dụng. Lệnh gọi app.get() truy xuất singleton instance của MyLogger, và phụ thuộc vào việc instance đó đã được inject trước đó trong một module khác, như mô tả ở trên.

Bạn cũng có thể inject provider MyLogger này vào các feature class, từ đó đảm bảo hành vi logging nhất quán cho cả Nest system logging và application logging. Xem Using the logger for application logging và Injecting a custom logger bên dưới để biết thêm thông tin.

Injecting a custom logger #

Để bắt đầu, extend logger tích hợp sẵn với code như sau. Chúng ta cung cấp option scope làm configuration metadata cho class ConsoleLogger, chỉ định transient scope, để đảm bảo rằng chúng ta sẽ có instance duy nhất của MyLogger trong mỗi feature module. Trong ví dụ này, chúng ta không extend các method ConsoleLogger riêng lẻ (như log(), warn(), v.v.), mặc dù bạn có thể chọn làm vậy.

my-logger.service.ts
import { Injectable, Scope, ConsoleLogger } from '@nestjs/common';

@Injectable({ scope: Scope.TRANSIENT })
export class MyLogger extends ConsoleLogger {
  customLog() {
    this.log('Please feed the cat!');
  }
}

Tiếp theo, tạo một LoggerModule với cấu trúc như sau:

logger.module.ts
import { Module } from '@nestjs/common';
import { MyLogger } from './my-logger.service';

@Module({
  providers: [MyLogger],
  exports: [MyLogger],
})
export class LoggerModule {}

Tiếp theo, import LoggerModule vào feature module. Vì chúng ta đã extend default Logger, chúng ta có sự tiện lợi của method setContext. Vì vậy, chúng ta có thể bắt đầu sử dụng context-aware custom logger, như sau:

cats.service.ts
import { Injectable } from '@nestjs/common';
import { MyLogger } from './my-logger.service';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  constructor(private myLogger: MyLogger) {
    // Due to transient scope, CatsService has its own unique instance of MyLogger,
    // so setting context here will not affect other instances in other services
    this.myLogger.setContext('CatsService');
  }

  findAll(): Cat[] {
    // You can call all the default methods
    this.myLogger.warn('About to return cats!');
    // And your custom methods
    this.myLogger.customLog();
    return this.cats;
  }
}

Cuối cùng, hướng dẫn Nest sử dụng instance của custom logger trong file main.ts như bên dưới. Trong ví dụ này, chúng ta thực sự không tùy chỉnh hành vi logger (bằng cách extend các method Logger như log(), warn(), v.v.), vì vậy bước này không cần thiết. Nhưng nó sẽ cần thiết nếu bạn thêm custom logic vào các method đó và muốn Nest sử dụng cùng implementation.

main.ts
const app = await NestFactory.create(AppModule, {
  bufferLogs: true,
});
app.useLogger(new MyLogger());
await app.listen(process.env.PORT ?? 3000);
💡 Hint

Ngoài ra, thay vì đặt bufferLogs thành true, bạn có thể tạm thời disable logger bằng directive logger: false. Lưu ý rằng nếu bạn cung cấp logger: false cho NestFactory.create, sẽ không có gì được log cho đến khi bạn gọi useLogger, vì vậy bạn có thể bỏ lỡ một số initialization error quan trọng. Nếu bạn không phiền khi một số initial message sẽ được log bằng default logger, bạn chỉ cần bỏ qua option logger: false.

Use external logger #

Các ứng dụng production thường có các yêu cầu logging cụ thể, bao gồm advanced filtering, formatting, và centralized logging. Logger tích hợp của Nest được sử dụng để giám sát hành vi hệ thống Nest, và cũng có thể hữu ích cho basic formatted text logging trong feature module khi đang develop, nhưng các ứng dụng production thường tận dụng các module logging chuyên dụng như Winston. Giống như bất kỳ ứng dụng Node.js tiêu chuẩn nào, bạn có thể tận dụng tối đa các module như vậy trong Nest.
