Database

Nest là framework không phụ thuộc vào database cụ thể (database agnostic), cho phép bạn dễ dàng tích hợp với bất kỳ cơ sở dữ liệu SQL hay NoSQL nào. Bạn có nhiều lựa chọn tùy theo sở thích. Ở mức độ tổng quát nhất, kết nối Nest với một database chỉ đơn giản là việc nạp một Node.js driver phù hợp cho database đó, tương tự như cách bạn làm với Express hay Fastify.

Bạn cũng có thể trực tiếp sử dụng bất kỳ thư viện tích hợp database hoặc ORM nào cho Node.js, chẳng hạn như MikroORM (xem MikroORM recipe), Sequelize (xem Sequelize integration), Knex.js, TypeORM, và Prisma (xem Prisma recipe), để thao tác ở mức độ trừu tượng cao hơn.

Để thuận tiện, Nest cung cấp tích hợp chặt chẽ với TypeORM và Sequelize ngay từ đầu thông qua các package @nestjs/typeorm và @nestjs/sequelize tương ứng — đây là nội dung chúng ta sẽ đề cập trong chương này — và với Mongoose thông qua @nestjs/mongoose, được đề cập ở chương này. Các tích hợp này cung cấp thêm nhiều tính năng đặc thù của NestJS như model/repository injection, khả năng test, và cấu hình bất đồng bộ (async configuration) để việc truy cập database trở nên dễ dàng hơn.

TypeORM Integration #

Để tích hợp với các cơ sở dữ liệu SQL và NoSQL, Nest cung cấp package @nestjs/typeorm. TypeORM là Object Relational Mapper (ORM) trưởng thành nhất hiện có cho TypeScript. Vì được viết bằng TypeScript, nó tích hợp tốt với framework Nest.

Để bắt đầu sử dụng, chúng ta cần cài đặt các dependency cần thiết. Trong chương này, chúng ta sẽ minh họa với MySQL — một Relational DBMS phổ biến — nhưng TypeORM hỗ trợ nhiều cơ sở dữ liệu quan hệ như PostgreSQL, Oracle, Microsoft SQL Server, SQLite, và thậm chí cả NoSQL như MongoDB. Quy trình trong chương này sẽ áp dụng tương tự cho mọi database mà TypeORM hỗ trợ. Bạn chỉ cần cài đặt các thư viện client API tương ứng với database của mình.

npm install --save @nestjs/typeorm typeorm mysql2

Khi cài đặt hoàn tất, chúng ta có thể import TypeOrmModule vào root AppModule.

app.module.ts
TS JS 
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';

@Module({
  imports: [
    TypeOrmModule.forRoot({
      type: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      entities: [],
      synchronize: true,
    }),
  ],
})
export class AppModule {}
Cảnh báo Không nên đặt synchronize: true trong môi trường production — nếu làm vậy, bạn có thể mất dữ liệu production.

Phương thức forRoot() hỗ trợ tất cả các thuộc tính cấu hình mà constructor DataSource của package TypeORM cung cấp. Ngoài ra còn có một số thuộc tính cấu hình bổ sung được mô tả dưới đây.

Thuộc tínhMô tả
retryAttemptsSố lần thử kết nối lại đến database (mặc định: 10)
retryDelayThời gian chờ giữa các lần thử kết nối lại (ms) (mặc định: 3000)
autoLoadEntitiesNếu là true, các entity sẽ được nạp tự động (mặc định: false)
Tìm hiểu thêm về các tùy chọn data source tại đây.

Sau khi thiết lập xong, các object DataSource và EntityManager của TypeORM sẽ có thể được inject trong toàn bộ project (không cần import thêm module nào), ví dụ:

app.module.ts
TSJS
import { DataSource } from 'typeorm';

@Module({
  imports: [TypeOrmModule.forRoot(), UsersModule],
})
export class AppModule {
  constructor(private dataSource: DataSource) {}
}

Repository pattern #

TypeORM hỗ trợ repository design pattern, trong đó mỗi entity có repository riêng của nó. Các repository này có thể lấy được từ database data source.

Để tiếp tục ví dụ, chúng ta cần ít nhất một entity. Hãy định nghĩa entity User.

user.entity.ts
TSJS
import { Entity, Column, PrimaryGeneratedColumn } from 'typeorm';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  firstName: string;

  @Column()
  lastName: string;

  @Column({ default: true })
  isActive: boolean;
}
Tìm hiểu thêm về các entity trong tài liệu TypeORM.

File entity User nằm trong thư mục users — thư mục chứa tất cả các file liên quan đến UsersModule. Bạn có thể tự quyết định nơi lưu trữ các file model, tuy nhiên chúng tôi khuyến nghị tạo chúng gần với domain của chúng, trong thư mục module tương ứng.

Để bắt đầu sử dụng entity User, chúng ta cần cho TypeORM biết về nó bằng cách thêm vào mảng entities trong các tùy chọn của phương thức forRoot() (trừ khi bạn dùng một glob path tĩnh):

app.module.ts
TSJS
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './users/user.entity';

@Module({
  imports: [
    TypeOrmModule.forRoot({
      type: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      entities: [User],
      synchronize: true,
    }),
  ],
})
export class AppModule {}

Tiếp theo, hãy xem UsersModule:

users.module.ts
TSJS
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { UsersService } from './users.service';
import { UsersController } from './users.controller';
import { User } from './user.entity';

@Module({
  imports: [TypeOrmModule.forFeature([User])],
  providers: [UsersService],
  controllers: [UsersController],
})
export class UsersModule {}

Module này sử dụng phương thức forFeature() để xác định những repository nào được đăng ký trong scope hiện tại. Sau khi làm vậy, chúng ta có thể inject UsersRepository vào UsersService bằng decorator @InjectRepository():

users.service.ts
TSJS
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { User } from './user.entity';

@Injectable()
export class UsersService {
  constructor(
    @InjectRepository(User)
    private usersRepository: Repository<User>,
  ) {}

  findAll(): Promise<User[]> {
    return this.usersRepository.find();
  }

  findOne(id: number): Promise<User | null> {
    return this.usersRepository.findOneBy({ id });
  }

  async remove(id: number): Promise<void> {
    await this.usersRepository.delete(id);
  }
}
Đừng quên import UsersModule vào root AppModule.

Nếu bạn muốn sử dụng repository bên ngoài module đã import TypeOrmModule.forFeature, bạn cần re-export các provider được tạo ra từ đó. Bạn có thể làm điều này bằng cách export toàn bộ module như sau:

users.module.ts
TSJS
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './user.entity';

@Module({
  imports: [TypeOrmModule.forFeature([User])],
  exports: [TypeOrmModule]
})
export class UsersModule {}

Bây giờ nếu chúng ta import UsersModule vào UserHttpModule, chúng ta có thể dùng @InjectRepository(User) trong các provider của module sau.

users-http.module.ts
TSJS
import { Module } from '@nestjs/common';
import { UsersModule } from './users.module';
import { UsersService } from './users.service';
import { UsersController } from './users.controller';

@Module({
  imports: [UsersModule],
  providers: [UsersService],
  controllers: [UsersController]
})
export class UserHttpModule {}

Relations #

Relations (quan hệ) là các liên kết được thiết lập giữa hai hoặc nhiều bảng. Chúng được xây dựng dựa trên các trường chung của mỗi bảng, thường liên quan đến primary key và foreign key.

Có ba loại quan hệ:

KiểuMô tả
One-to-oneMỗi dòng trong bảng chính có duy nhất một dòng tương ứng trong bảng ngoại. Dùng decorator @OneToOne() để định nghĩa kiểu quan hệ này.
One-to-many / Many-to-oneMỗi dòng trong bảng chính có một hoặc nhiều dòng liên quan trong bảng ngoại. Dùng các decorator @OneToMany() và @ManyToOne().
Many-to-manyMỗi dòng trong bảng chính có nhiều dòng liên quan trong bảng ngoại, và ngược lại. Dùng decorator @ManyToMany().

Để định nghĩa quan hệ trong entity, dùng các decorator tương ứng. Ví dụ, để định nghĩa rằng mỗi User có thể có nhiều photo, dùng decorator @OneToMany().

user.entity.ts
TSJS
import { Entity, Column, PrimaryGeneratedColumn, OneToMany } from 'typeorm';
import { Photo } from '../photos/photo.entity';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  firstName: string;

  @Column()
  lastName: string;

  @Column({ default: true })
  isActive: boolean;

  @OneToMany(type => Photo, photo => photo.user)
  photos: Photo[];
}
Để tìm hiểu thêm về relations trong TypeORM, hãy truy cập tài liệu TypeORM.

Auto-load entities #

Việc thêm các entity thủ công vào mảng entities của tùy chọn data source có thể khá tẻ nhạt. Hơn nữa, việc tham chiếu entity từ root module sẽ phá vỡ ranh giới domain của ứng dụng và gây rò rỉ chi tiết triển khai sang các phần khác. Để giải quyết vấn đề này, một giải pháp thay thế được cung cấp: tự động nạp entity bằng cách đặt thuộc tính autoLoadEntities của object cấu hình (truyền vào phương thức forRoot()) thành true, như bên dưới:

app.module.ts
TSJS
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';

@Module({
  imports: [
    TypeOrmModule.forRoot({
      ...
      autoLoadEntities: true,
    }),
  ],
})
export class AppModule {}

Với tùy chọn này, mỗi entity được đăng ký thông qua phương thức forFeature() sẽ tự động được thêm vào mảng entities của object cấu hình.

Lưu ý rằng các entity không được đăng ký qua forFeature(), mà chỉ được tham chiếu từ entity khác (thông qua quan hệ), sẽ không được đưa vào bởi thiết lập autoLoadEntities.

Separating entity definition #

Bạn có thể định nghĩa một entity và các cột của nó trực tiếp trong model bằng cách dùng decorator. Tuy nhiên, một số người thích định nghĩa entity và cột của chúng trong các file riêng biệt bằng cách dùng "entity schemas". 

import { EntitySchema } from 'typeorm';
import { User } from './user.entity';

export const UserSchema = new EntitySchema<User>({
  name: 'User',
  target: User,
  columns: {
    id: {
      type: Number,
      primary: true,
      generated: true,
    },
    firstName: {
      type: String,
    },
    lastName: {
      type: String,
    },
    isActive: {
      type: Boolean,
      default: true,
    },
  },
  relations: {
    photos: {
      type: 'one-to-many',
      target: 'Photo', // tên của PhotoSchema
    },
  },
});
Nếu bạn cung cấp tùy chọn target, giá trị của tùy chọn name phải trùng với tên của class target. Nếu không cung cấp target, bạn có thể dùng bất kỳ tên nào.

Nest cho phép bạn dùng instance EntitySchema ở bất cứ đâu mà Entity được kỳ vọng, ví dụ:

import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { UserSchema } from './user.schema';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  imports: [TypeOrmModule.forFeature([UserSchema])],
  providers: [UsersService],
  controllers: [UsersController],
})
export class UsersModule {}

TypeORM Transactions #

Một database transaction (giao dịch) đại diện cho một đơn vị công việc được thực hiện trong một hệ thống quản lý cơ sở dữ liệu, được xử lý theo cách nhất quán và đáng tin cậy, độc lập với các giao dịch khác. Một giao dịch thường đại diện cho bất kỳ thay đổi nào trong cơ sở dữ liệu (tìm hiểu thêm).

Có nhiều chiến lược khác nhau để xử lý TypeORM transactions. Chúng tôi khuyến nghị sử dụng class QueryRunner vì nó cho phép kiểm soát toàn bộ giao dịch.

Đầu tiên, chúng ta cần inject object DataSource vào một class theo cách thông thường:

@Injectable()
export class UsersService {
  constructor(private dataSource: DataSource) {}
}
Class DataSource được import từ package typeorm.

Bây giờ chúng ta có thể dùng object này để tạo một giao dịch.

async createMany(users: User[]) {
  const queryRunner = this.dataSource.createQueryRunner();

  await queryRunner.connect();
  await queryRunner.startTransaction();
  try {
    await queryRunner.manager.save(users[0]);
    await queryRunner.manager.save(users[1]);

    await queryRunner.commitTransaction();
  } catch (err) {
    // vì có lỗi xảy ra, hãy rollback các thay đổi đã thực hiện
    await queryRunner.rollbackTransaction();
  } finally {
    // bạn cần release queryRunner đã được khởi tạo thủ công
    await queryRunner.release();
  }
}
Lưu ý rằng dataSource chỉ được dùng để tạo QueryRunner. Tuy nhiên, để test class này sẽ cần mock toàn bộ object DataSource (vốn có nhiều method). Do đó, chúng tôi khuyến nghị sử dụng một helper factory class (ví dụ: QueryRunnerFactory) và định nghĩa một interface với tập hợp method hạn chế cần thiết để duy trì giao dịch. Kỹ thuật này giúp việc mock các method đó trở nên đơn giản hơn.

Ngoài ra, bạn có thể dùng cách tiếp cận callback-style với phương thức transaction của object DataSource (đọc thêm).

async createMany(users: User[]) {
  await this.dataSource.transaction(async manager => {
    await manager.save(users[0]);
    await manager.save(users[1]);
  });
}

Subscribers #

Với TypeORM subscribers, bạn có thể lắng nghe các sự kiện (event) cụ thể của entity. 

import {
  DataSource,
  EntitySubscriberInterface,
  EventSubscriber,
  InsertEvent,
} from 'typeorm';
import { User } from './user.entity';

@EventSubscriber()
export class UserSubscriber implements EntitySubscriberInterface<User> {
  constructor(dataSource: DataSource) {
    dataSource.subscribers.push(this);
  }

  listenTo() {
    return User;
  }

  beforeInsert(event: InsertEvent<User>) {
    console.log(`BEFORE USER INSERTED: `, event.entity);
  }
}
Event subscriber không thể là request-scoped.

Bây giờ, thêm class UserSubscriber vào mảng providers:

import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './user.entity';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';
import { UserSubscriber } from './user.subscriber';

@Module({
  imports: [TypeOrmModule.forFeature([User])],
  providers: [UsersService, UserSubscriber],
  controllers: [UsersController],
})
export class UsersModule {}

Migrations #

Migrations cung cấp cách để cập nhật schema database theo từng bước nhỏ, giữ nó đồng bộ với data model của ứng dụng trong khi vẫn bảo toàn dữ liệu hiện có. Để tạo, chạy và hoàn tác migrations, TypeORM cung cấp một CLI chuyên dụng.

Các class migration tách biệt với source code của ứng dụng Nest. Vòng đời của chúng được quản lý bởi TypeORM CLI. Do đó, bạn không thể tận dụng dependency injection và các tính năng đặc thù của Nest với migrations. Để tìm hiểu thêm về migrations, hãy theo hướng dẫn trong tài liệu TypeORM.

Multiple databases #

Một số project yêu cầu nhiều kết nối database. Điều này cũng có thể thực hiện được với module này. Để làm việc với nhiều kết nối, đầu tiên hãy tạo các kết nối đó. Trong trường hợp này, việc đặt tên cho data source trở thành bắt buộc.

Giả sử bạn có một entity Album được lưu trong database riêng của nó.

const defaultOptions = {
  type: 'postgres',
  port: 5432,
  username: 'user',
  password: 'password',
  database: 'db',
  synchronize: true,
};

@Module({
  imports: [
    TypeOrmModule.forRoot({
      ...defaultOptions,
      host: 'user_db_host',
      entities: [User],
    }),
    TypeOrmModule.forRoot({
      ...defaultOptions,
      name: 'albumsConnection',
      host: 'album_db_host',
      entities: [Album],
    }),
  ],
})
export class AppModule {}
Nếu bạn không đặt name cho data source, tên mặc định sẽ là default. Lưu ý rằng bạn không nên có nhiều kết nối không có tên, hoặc có tên trùng nhau, vì chúng sẽ bị ghi đè lên nhau.
Nếu bạn dùng TypeOrmModule.forRootAsync, bạn cũng phải đặt tên data source bên ngoài useFactory. Ví dụ: 
TypeOrmModule.forRootAsync({
  name: 'albumsConnection',
  useFactory: ...,
  inject: ...,
}),
Xem issue này để biết thêm chi tiết.

Tại thời điểm này, bạn đã đăng ký các entity User và Album với data source riêng của chúng. Với thiết lập này, bạn phải cho phương thức TypeOrmModule.forFeature() và decorator @InjectRepository() biết data source nào nên được sử dụng. Nếu bạn không truyền tên data source nào, data source default sẽ được dùng.

@Module({
  imports: [
    TypeOrmModule.forFeature([User]),
    TypeOrmModule.forFeature([Album], 'albumsConnection'),
  ],
})
export class AppModule {}

Bạn cũng có thể inject DataSource hoặc EntityManager cho một data source cụ thể:

@Injectable()
export class AlbumsService {
  constructor(
    @InjectDataSource('albumsConnection')
    private dataSource: DataSource,
    @InjectEntityManager('albumsConnection')
    private entityManager: EntityManager,
  ) {}
}

Ngoài ra, bạn có thể inject bất kỳ DataSource nào vào providers:

@Module({
  providers: [
    {
      provide: AlbumsService,
      useFactory: (albumsConnection: DataSource) => {
        return new AlbumsService(albumsConnection);
      },
      inject: [getDataSourceToken('albumsConnection')],
    },
  ],
})
export class AlbumsModule {}

Testing #

Khi viết unit test cho ứng dụng, chúng ta thường muốn tránh việc kết nối thực sự đến database, để giữ cho các bộ test độc lập và quá trình thực thi nhanh nhất có thể. Nhưng các class của chúng ta có thể phụ thuộc vào các repository được lấy từ data source (connection) instance. Vậy chúng ta xử lý điều đó như thế nào? Giải pháp là tạo mock repository. Để đạt được điều này, chúng ta thiết lập custom providers. Mỗi repository đã đăng ký sẽ tự động được đại diện bởi token <EntityName>Repository, trong đó EntityName là tên của class entity.

Package @nestjs/typeorm cung cấp hàm getRepositoryToken() — hàm này trả về token đã được chuẩn bị dựa trên entity đã cho.

@Module({
  providers: [
    UsersService,
    {
      provide: getRepositoryToken(User),
      useValue: mockRepository,
    },
  ],
})
export class UsersModule {}

Bây giờ một mockRepository thay thế sẽ được sử dụng như là UsersRepository. Bất cứ khi nào một class nào yêu cầu UsersRepository bằng decorator @InjectRepository(), Nest sẽ sử dụng object mockRepository đã đăng ký.

Async configuration #

Bạn có thể muốn truyền các tùy chọn module repository theo cách bất đồng bộ thay vì tĩnh. Trong trường hợp này, hãy sử dụng phương thức forRootAsync(), cung cấp nhiều cách để xử lý cấu hình bất đồng bộ.

Một cách tiếp cận là sử dụng factory function:

TypeOrmModule.forRootAsync({
  useFactory: () => ({
    type: 'mysql',
    host: 'localhost',
    port: 3306,
    username: 'root',
    password: 'root',
    database: 'test',
    entities: [],
    synchronize: true,
  }),
});

Factory của chúng ta hoạt động giống bất kỳ asynchronous provider nào khác (ví dụ: nó có thể là async và có khả năng inject dependency thông qua inject).

TypeOrmModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: (configService: ConfigService) => ({
    type: 'mysql',
    host: configService.get('HOST'),
    port: +configService.get('PORT'),
    username: configService.get('USERNAME'),
    password: configService.get('PASSWORD'),
    database: configService.get('DATABASE'),
    entities: [],
    synchronize: true,
  }),
  inject: [ConfigService],
});

Ngoài ra, bạn có thể dùng cú pháp useClass:

TypeOrmModule.forRootAsync({
  useClass: TypeOrmConfigService,
});

Cấu trúc trên sẽ khởi tạo TypeOrmConfigService bên trong TypeOrmModule và sử dụng nó để cung cấp object tùy chọn bằng cách gọi createTypeOrmOptions(). Điều này có nghĩa là TypeOrmConfigService phải implement interface TypeOrmOptionsFactory, như được hiển thị dưới đây:

@Injectable()
export class TypeOrmConfigService implements TypeOrmOptionsFactory {
  createTypeOrmOptions(): TypeOrmModuleOptions {
    return {
      type: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      entities: [],
      synchronize: true,
    };
  }
}

Để ngăn việc tạo TypeOrmConfigService bên trong TypeOrmModule và dùng provider được import từ module khác, bạn có thể dùng cú pháp useExisting.

TypeOrmModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});

Cấu trúc này hoạt động tương tự như useClass với một điểm khác biệt quan trọng — TypeOrmModule sẽ tìm kiếm trong các module đã import để tái sử dụng ConfigService hiện có thay vì khởi tạo mới.

Hãy đảm bảo rằng thuộc tính name được định nghĩa cùng cấp với thuộc tính useFactory, useClass, hoặc useValue. Điều này cho phép Nest đăng ký đúng data source dưới injection token phù hợp.

Custom DataSource Factory #

Kết hợp với async configuration dùng useFactory, useClass, hoặc useExisting, bạn có thể tùy chọn chỉ định một hàm dataSourceFactory để tự cung cấp TypeORM data source của mình thay vì để TypeOrmModule tạo data source.

dataSourceFactory nhận vào DataSourceOptions của TypeORM được cấu hình trong async configuration thông qua useFactory, useClass, hoặc useExisting, và trả về một Promise resolve thành TypeORM DataSource. 

TypeOrmModule.forRootAsync({
  imports: [ConfigModule],
  inject: [ConfigService],
  // Dùng useFactory, useClass, hoặc useExisting
  // để cấu hình DataSourceOptions.
  useFactory: (configService: ConfigService) => ({
    type: 'mysql',
    host: configService.get('HOST'),
    port: +configService.get('PORT'),
    username: configService.get('USERNAME'),
    password: configService.get('PASSWORD'),
    database: configService.get('DATABASE'),
    entities: [],
    synchronize: true,
  }),
  // dataSource nhận vào DataSourceOptions đã được cấu hình
  // và trả về Promise<DataSource>.
  dataSourceFactory: async (options) => {
    const dataSource = await new DataSource(options).initialize();
    return dataSource;
  },
});
Class DataSource được import từ package typeorm.

Sequelize Integration #

Một lựa chọn thay thế cho TypeORM là sử dụng Sequelize ORM với package @nestjs/sequelize. Ngoài ra, chúng ta tận dụng package sequelize-typescript cung cấp tập hợp các decorator bổ sung để định nghĩa entity theo kiểu khai báo.

Để bắt đầu sử dụng, đầu tiên hãy cài đặt các dependency cần thiết. Trong chương này, chúng ta sẽ minh họa với MySQL, nhưng Sequelize hỗ trợ nhiều cơ sở dữ liệu quan hệ như PostgreSQL, MySQL, Microsoft SQL Server, SQLite và MariaDB. Quy trình trong chương này sẽ tương tự cho mọi database được Sequelize hỗ trợ. Bạn chỉ cần cài đặt thư viện client API liên quan đến database đã chọn.

npm install --save @nestjs/sequelize sequelize sequelize-typescript mysql2
npm install --save-dev @types/sequelize

Khi cài đặt hoàn tất, chúng ta có thể import SequelizeModule vào root AppModule.

app.module.ts
TSJS
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';

@Module({
  imports: [
    SequelizeModule.forRoot({
      dialect: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      models: [],
    }),
  ],
})
export class AppModule {}

Phương thức forRoot() hỗ trợ tất cả các thuộc tính cấu hình mà Sequelize constructor cung cấp (đọc thêm). Ngoài ra còn có một số thuộc tính cấu hình bổ sung được mô tả dưới đây.

Thuộc tínhMô tả
retryAttemptsSố lần thử kết nối lại đến database (mặc định: 10)
retryDelayThời gian chờ giữa các lần thử kết nối lại (ms) (mặc định: 3000)
autoLoadModelsNếu là true, các model sẽ được nạp tự động (mặc định: false)
keepConnectionAliveNếu là true, kết nối sẽ không bị đóng khi ứng dụng tắt (mặc định: false)
synchronizeNếu là true, các model được nạp tự động sẽ được đồng bộ hóa (mặc định: true)

Sau khi thiết lập xong, object Sequelize sẽ có thể được inject trong toàn bộ project (không cần import thêm module nào), ví dụ:

app.service.ts
TSJS
import { Injectable } from '@nestjs/common';
import { Sequelize } from 'sequelize-typescript';

@Injectable()
export class AppService {
  constructor(private sequelize: Sequelize) {}
}

Models #

Sequelize triển khai pattern Active Record. Với pattern này, bạn dùng các class model trực tiếp để tương tác với database. Để tiếp tục ví dụ, chúng ta cần ít nhất một model. Hãy định nghĩa model User.

user.model.ts
TSJS
import { Column, Model, Table } from 'sequelize-typescript';

@Table
export class User extends Model {
  @Column
  firstName: string;

  @Column
  lastName: string;

  @Column({ defaultValue: true })
  isActive: boolean;
}
Tìm hiểu thêm về các decorator có sẵn tại đây.

File model User nằm trong thư mục users. Thư mục này chứa tất cả các file liên quan đến UsersModule. Bạn có thể tự quyết định nơi lưu file model, nhưng chúng tôi khuyến nghị tạo chúng gần với domain của chúng, trong thư mục module tương ứng.

Để bắt đầu sử dụng model User, chúng ta cần cho Sequelize biết về nó bằng cách thêm vào mảng models trong tùy chọn của phương thức forRoot():

app.module.ts
TSJS
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';
import { User } from './users/user.model';

@Module({
  imports: [
    SequelizeModule.forRoot({
      dialect: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      models: [User],
    }),
  ],
})
export class AppModule {}

Tiếp theo, hãy xem UsersModule:

users.module.ts
TSJS
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';
import { User } from './user.model';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  imports: [SequelizeModule.forFeature([User])],
  providers: [UsersService],
  controllers: [UsersController],
})
export class UsersModule {}

Module này sử dụng phương thức forFeature() để xác định những model nào được đăng ký trong scope hiện tại. Với điều đó, chúng ta có thể inject UserModel vào UsersService bằng decorator @InjectModel():

users.service.ts
TSJS
import { Injectable } from '@nestjs/common';
import { InjectModel } from '@nestjs/sequelize';
import { User } from './user.model';

@Injectable()
export class UsersService {
  constructor(
    @InjectModel(User)
    private userModel: typeof User,
  ) {}

  async findAll(): Promise<User[]> {
    return this.userModel.findAll();
  }

  findOne(id: string): Promise<User> {
    return this.userModel.findOne({
      where: {
        id,
      },
    });
  }

  async remove(id: string): Promise<void> {
    const user = await this.findOne(id);
    await user.destroy();
  }
}
Đừng quên import UsersModule vào root AppModule.

Nếu bạn muốn sử dụng model bên ngoài module đã import SequelizeModule.forFeature, bạn cần re-export các provider được tạo ra từ đó. Bạn có thể làm điều này bằng cách export toàn bộ module:

users.module.ts
TSJS
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';
import { User } from './user.entity';

@Module({
  imports: [SequelizeModule.forFeature([User])],
  exports: [SequelizeModule]
})
export class UsersModule {}

Bây giờ nếu chúng ta import UsersModule vào UserHttpModule, chúng ta có thể dùng @InjectModel(User) trong các provider của module sau.

users-http.module.ts
TSJS
import { Module } from '@nestjs/common';
import { UsersModule } from './users.module';
import { UsersService } from './users.service';
import { UsersController } from './users.controller';

@Module({
  imports: [UsersModule],
  providers: [UsersService],
  controllers: [UsersController]
})
export class UserHttpModule {}

Relations #

Relations là các liên kết được thiết lập giữa hai hoặc nhiều bảng. Chúng được xây dựng dựa trên các trường chung của mỗi bảng, thường liên quan đến primary key và foreign key.

Có ba loại quan hệ:

KiểuMô tả
One-to-oneMỗi dòng trong bảng chính có duy nhất một dòng tương ứng trong bảng ngoại
One-to-many / Many-to-oneMỗi dòng trong bảng chính có một hoặc nhiều dòng liên quan trong bảng ngoại
Many-to-manyMỗi dòng trong bảng chính có nhiều dòng liên quan trong bảng ngoại, và ngược lại

Để định nghĩa quan hệ trong model, dùng các decorator tương ứng. Ví dụ, để định nghĩa rằng mỗi User có thể có nhiều photo, dùng decorator @HasMany().

user.model.ts
TSJS
import { Column, Model, Table, HasMany } from 'sequelize-typescript';
import { Photo } from '../photos/photo.model';

@Table
export class User extends Model {
  @Column
  firstName: string;

  @Column
  lastName: string;

  @Column({ defaultValue: true })
  isActive: boolean;

  @HasMany(() => Photo)
  photos: Photo[];
}
Để tìm hiểu thêm về associations trong Sequelize, đọc chương này.

Auto-load models #

Việc thêm model thủ công vào mảng models của tùy chọn kết nối có thể khá tẻ nhạt. Hơn nữa, việc tham chiếu model từ root module sẽ phá vỡ ranh giới domain của ứng dụng. Để giải quyết vấn đề này, hãy tự động nạp model bằng cách đặt cả hai thuộc tính autoLoadModels và synchronize của object cấu hình thành true:

app.module.ts
TSJS
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';

@Module({
  imports: [
    SequelizeModule.forRoot({
      ...
      autoLoadModels: true,
      synchronize: true,
    }),
  ],
})
export class AppModule {}

Với tùy chọn này, mỗi model được đăng ký qua phương thức forFeature() sẽ tự động được thêm vào mảng models của object cấu hình.

Lưu ý rằng các model không được đăng ký qua forFeature(), mà chỉ được tham chiếu từ model khác (thông qua association), sẽ không được đưa vào.

Sequelize Transactions #

Một database transaction đại diện cho một đơn vị công việc được thực hiện trong một hệ thống quản lý cơ sở dữ liệu, được xử lý theo cách nhất quán và đáng tin cậy, độc lập với các giao dịch khác. Một giao dịch thường đại diện cho bất kỳ thay đổi nào trong cơ sở dữ liệu (tìm hiểu thêm).

Có nhiều chiến lược khác nhau để xử lý Sequelize transactions. Dưới đây là ví dụ triển khai managed transaction (auto-callback).

Đầu tiên, chúng ta cần inject object Sequelize vào class theo cách thông thường:

@Injectable()
export class UsersService {
  constructor(private sequelize: Sequelize) {}
}
Class Sequelize được import từ package sequelize-typescript.

Bây giờ chúng ta có thể dùng object này để tạo một giao dịch.

async createMany() {
  try {
    await this.sequelize.transaction(async t => {
      const transactionHost = { transaction: t };

      await this.userModel.create(
          { firstName: 'Abraham', lastName: 'Lincoln' },
          transactionHost,
      );
      await this.userModel.create(
          { firstName: 'John', lastName: 'Boothe' },
          transactionHost,
      );
    });
  } catch (err) {
    // Giao dịch đã bị rollback
    // err là bất kỳ thứ gì đã reject promise chain trả về cho transaction callback
  }
}
Lưu ý rằng instance Sequelize chỉ được dùng để bắt đầu giao dịch. Tuy nhiên, để test class này sẽ cần mock toàn bộ object Sequelize. Do đó, chúng tôi khuyến nghị dùng một helper factory class (ví dụ: TransactionRunner) và định nghĩa interface với tập hợp method hạn chế cần thiết để duy trì giao dịch.

Migrations #

Migrations cung cấp cách để cập nhật schema database theo từng bước nhỏ, giữ nó đồng bộ với data model của ứng dụng trong khi vẫn bảo toàn dữ liệu hiện có. Để tạo, chạy và hoàn tác migrations, Sequelize cung cấp một CLI chuyên dụng.

Các class migration tách biệt với source code của ứng dụng Nest. Vòng đời của chúng được quản lý bởi Sequelize CLI. Do đó, bạn không thể tận dụng dependency injection và các tính năng đặc thù của Nest với migrations. Để tìm hiểu thêm về migrations, hãy theo hướng dẫn trong tài liệu Sequelize.

Multiple databases #

Một số project yêu cầu nhiều kết nối database. Điều này cũng có thể thực hiện với module này. Để làm việc với nhiều kết nối, đầu tiên hãy tạo các kết nối đó. Trong trường hợp này, việc đặt tên kết nối trở thành bắt buộc.

Giả sử bạn có entity Album được lưu trong database riêng của nó.

const defaultOptions = {
  dialect: 'postgres',
  port: 5432,
  username: 'user',
  password: 'password',
  database: 'db',
  synchronize: true,
};

@Module({
  imports: [
    SequelizeModule.forRoot({
      ...defaultOptions,
      host: 'user_db_host',
      models: [User],
    }),
    SequelizeModule.forRoot({
      ...defaultOptions,
      name: 'albumsConnection',
      host: 'album_db_host',
      models: [Album],
    }),
  ],
})
export class AppModule {}
Nếu bạn không đặt name cho một kết nối, tên mặc định sẽ là default. Lưu ý rằng bạn không nên có nhiều kết nối không có tên, hoặc có tên trùng nhau, vì chúng sẽ bị ghi đè lên nhau.

Tại thời điểm này, bạn đã đăng ký các model User và Album với kết nối riêng của chúng. Với thiết lập này, bạn phải cho phương thức SequelizeModule.forFeature() và decorator @InjectModel() biết kết nối nào nên được sử dụng. Nếu không truyền tên kết nối nào, kết nối default sẽ được dùng.

@Module({
  imports: [
    SequelizeModule.forFeature([User]),
    SequelizeModule.forFeature([Album], 'albumsConnection'),
  ],
})
export class AppModule {}

Bạn cũng có thể inject instance Sequelize cho một kết nối cụ thể:

@Injectable()
export class AlbumsService {
  constructor(
    @InjectConnection('albumsConnection')
    private sequelize: Sequelize,
  ) {}
}

Ngoài ra, bạn có thể inject bất kỳ instance Sequelize nào vào providers:

@Module({
  providers: [
    {
      provide: AlbumsService,
      useFactory: (albumsSequelize: Sequelize) => {
        return new AlbumsService(albumsSequelize);
      },
      inject: [getDataSourceToken('albumsConnection')],
    },
  ],
})
export class AlbumsModule {}

Testing #

Khi viết unit test cho ứng dụng, chúng ta thường muốn tránh việc kết nối thực sự đến database, giữ cho các bộ test độc lập và nhanh nhất có thể. Nhưng các class của chúng ta có thể phụ thuộc vào các model được lấy từ connection instance. Vậy chúng ta xử lý điều đó như thế nào? Giải pháp là tạo mock model. Để đạt được điều này, chúng ta thiết lập custom providers. Mỗi model đã đăng ký sẽ tự động được đại diện bởi token <ModelName>Model, trong đó ModelName là tên của class model.

Package @nestjs/sequelize cung cấp hàm getModelToken() — trả về token đã được chuẩn bị dựa trên model đã cho.

@Module({
  providers: [
    UsersService,
    {
      provide: getModelToken(User),
      useValue: mockModel,
    },
  ],
})
export class UsersModule {}

Bây giờ một mockModel thay thế sẽ được sử dụng như là UserModel. Bất cứ khi nào một class nào yêu cầu UserModel bằng decorator @InjectModel(), Nest sẽ sử dụng object mockModel đã đăng ký.

Async configuration #

Bạn có thể muốn truyền các tùy chọn SequelizeModule theo cách bất đồng bộ thay vì tĩnh. Trong trường hợp này, hãy dùng phương thức forRootAsync(), cung cấp nhiều cách để xử lý cấu hình bất đồng bộ.

Một cách tiếp cận là sử dụng factory function:

SequelizeModule.forRootAsync({
  useFactory: () => ({
    dialect: 'mysql',
    host: 'localhost',
    port: 3306,
    username: 'root',
    password: 'root',
    database: 'test',
    models: [],
  }),
});

Factory của chúng ta hoạt động giống bất kỳ asynchronous provider nào khác (ví dụ: nó có thể là async và có khả năng inject dependency thông qua inject).

SequelizeModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: (configService: ConfigService) => ({
    dialect: 'mysql',
    host: configService.get('HOST'),
    port: +configService.get('PORT'),
    username: configService.get('USERNAME'),
    password: configService.get('PASSWORD'),
    database: configService.get('DATABASE'),
    models: [],
  }),
  inject: [ConfigService],
});

Ngoài ra, bạn có thể dùng cú pháp useClass:

SequelizeModule.forRootAsync({
  useClass: SequelizeConfigService,
});

Cấu trúc trên sẽ khởi tạo SequelizeConfigService bên trong SequelizeModule và sử dụng nó để cung cấp object tùy chọn bằng cách gọi createSequelizeOptions(). Điều này có nghĩa là SequelizeConfigService phải implement interface SequelizeOptionsFactory:

@Injectable()
class SequelizeConfigService implements SequelizeOptionsFactory {
  createSequelizeOptions(): SequelizeModuleOptions {
    return {
      dialect: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      models: [],
    };
  }
}

Để ngăn việc tạo SequelizeConfigService bên trong SequelizeModule và dùng provider được import từ module khác, bạn có thể dùng cú pháp useExisting.

SequelizeModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});

Cấu trúc này hoạt động tương tự như useClass với một điểm khác biệt quan trọng — SequelizeModule sẽ tìm kiếm trong các module đã import để tái sử dụng ConfigService hiện có thay vì khởi tạo mới.
