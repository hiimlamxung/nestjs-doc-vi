Validation

Việc kiểm tra tính chính xác của bất kỳ dữ liệu nào được gửi vào ứng dụng web là một best practice. Để tự động validate các incoming request, Nest cung cấp một số pipe có sẵn ngay lập tức:

  • ValidationPipe
  • ParseIntPipe
  • ParseBoolPipe
  • ParseArrayPipe
  • ParseUUIDPipe

ValidationPipe sử dụng thư viện class-validator mạnh mẽ và các validation decorator mang tính khai báo (declarative) của nó. ValidationPipe cung cấp một cách tiếp cận thuận tiện để thực thi các quy tắc validation cho tất cả client payload đến, trong đó các quy tắc cụ thể được khai báo bằng các annotation đơn giản trong các class/DTO declaration cục bộ trong mỗi module.

Overview #

Trong chương Pipes, chúng ta đã tìm hiểu quy trình xây dựng các validation pipe đơn giản và bind chúng với controller, method hoặc ở cấp global để chứng minh cách hoạt động của quy trình. Hãy chắc chắn xem lại chương đó để hiểu rõ nhất các chủ đề trong chương này. Ở đây, chúng ta sẽ tập trung vào các use case thực tế khác nhau của ValidationPipe và cách sử dụng một số tính năng tùy chỉnh nâng cao của nó.

Using the built-in ValidationPipe #

Để bắt đầu sử dụng, trước tiên cài đặt các dependency cần thiết.

$ npm i --save class-validator class-transformer
💡 Hint

ValidationPipe được export từ package @nestjs/common.

Vì pipe này sử dụng thư viện class-validator và class-transformer, có nhiều option khả dụng. Bạn cấu hình các setting này thông qua configuration object được truyền vào pipe. Dưới đây là các option có sẵn:

export interface ValidationPipeOptions extends ValidatorOptions {
  transform?: boolean;
  disableErrorMessages?: boolean;
  exceptionFactory?: (errors: ValidationError[]) => any;
}

Tất cả các option class-validator khả dụng (kế thừa từ interface ValidatorOptions):

skipNullPropertiesbooleanNếu set thành true, validator sẽ bỏ qua validation của tất cả property mà null trong object đang được validate.
skipMissingPropertiesbooleanNếu set thành true, validator sẽ bỏ qua validation của tất cả property mà null hoặc undefined trong object đang được validate.
whitelistbooleanNếu set thành true, validator sẽ loại bỏ (strip) tất cả property của object được validate mà không sử dụng bất kỳ validation decorator nào.

Auto-validation #

Chúng ta sẽ bắt đầu bằng cách bind ValidationPipe ở cấp application, do đó đảm bảo tất cả endpoint được bảo vệ khỏi việc nhận dữ liệu không chính xác.

main.ts
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ValidationPipe());
  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();

Để kiểm tra pipe, hãy tạo một endpoint cơ bản.

@Post()
create(@Body() createUserDto: CreateUserDto) {
  return 'This action adds a new user';
}
💡 Hint

Vì TypeScript không lưu trữ metadata về generic hoặc interface, khi bạn sử dụng chúng trong DTO, ValidationPipe có thể không validate được incoming data đúng cách. Vì lý do này, hãy cân nhắc sử dụng concrete class trong DTO của bạn.

💡 Hint

Khi import DTO, bạn không thể sử dụng type-only import vì nó sẽ bị xóa tại runtime, nghĩa là hãy nhớ dùng import { CreateUserDto } thay vì import type { CreateUserDto }.

Bây giờ chúng ta có thể thêm một vài validation rule trong CreateUserDto. Chúng ta làm điều này bằng cách sử dụng các decorator được cung cấp bởi package class-validator, được mô tả chi tiết tại đây. Theo cách này, bất kỳ route nào sử dụng CreateUserDto sẽ tự động thực thi các validation rule này.

create-user.dto.ts
import { IsEmail, IsNotEmpty } from 'class-validator';

export class CreateUserDto {
  @IsEmail()
  email: string;

  @IsNotEmpty()
  password: string;
}

Với các rule này, nếu một request hit endpoint của chúng ta với property email không hợp lệ trong request body, ứng dụng sẽ tự động respond với mã 400 Bad Request, kèm theo response body như sau:

{
  "statusCode": 400,
  "error": "Bad Request",
  "message": ["email must be an email"]
}

Ngoài việc validate request body, ValidationPipe có thể được sử dụng với các request object property khác. Giả sử chúng ta muốn accept :id trong endpoint path. Để đảm bảo chỉ các số được chấp nhận cho request parameter này, chúng ta có thể sử dụng construct sau:

@Get(':id')
findOne(@Param() params: FindOneParams) {
  return 'This action returns a user';
}

FindOneParams, giống như một DTO, chỉ đơn giản là một class định nghĩa validation rule sử dụng class-validator. Nó sẽ trông như thế này:

import { IsNumberString } from 'class-validator';

export class FindOneParams {
  @IsNumberString()
  id: string;
}

Disable detailed errors #

Error message có thể hữu ích để giải thích request bị sai ở đâu. Tuy nhiên, một số môi trường production thích disable detailed error. Thực hiện điều này bằng cách truyền options object cho ValidationPipe:

app.useGlobalPipes(
  new ValidationPipe({
    disableErrorMessages: true,
  }),
);

Kết quả là, detailed error message sẽ không được hiển thị trong response body.

Stripping properties #

ValidationPipe của chúng ta cũng có thể filter ra các property mà không nên được nhận bởi method handler. Trong trường hợp này, chúng ta có thể whitelist các property được chấp nhận, và bất kỳ property nào không nằm trong whitelist sẽ tự động bị strip khỏi resulting object. Ví dụ: nếu handler của chúng ta expect property email và password, nhưng request cũng bao gồm property age, property này có thể tự động bị loại bỏ khỏi resulting DTO. Để enable behavior này, set whitelist thành true.

app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true,
  }),
);

Khi set thành true, điều này sẽ tự động loại bỏ các non-whitelisted property (những property không có bất kỳ decorator nào trong validation class).

Ngoài ra, bạn có thể dừng request khỏi việc xử lý khi có non-whitelisted property xuất hiện, và trả về error response cho user. Để enable điều này, set option forbidNonWhitelisted thành true, kết hợp với setting whitelist thành true.

Transform payload objects #

Payload đến qua network là các plain JavaScript object. ValidationPipe có thể tự động transform payload thành object được typed theo DTO class của chúng. Để enable auto-transformation, set transform thành true. Điều này có thể được thực hiện ở method level:

cats.controller.ts
@Post()
@UsePipes(new ValidationPipe({ transform: true }))
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}

Để enable behavior này globally, set option trên global pipe:

app.useGlobalPipes(
  new ValidationPipe({
    transform: true,
  }),
);

Với option auto-transformation được enable, ValidationPipe cũng sẽ thực hiện chuyển đổi các primitive type. Trong ví dụ sau, method findOne() nhận một argument đại diện cho id path parameter được extract:

@Get(':id')
findOne(@Param('id') id: number) {
  console.log(typeof id === 'number'); // true
  return 'This action returns a user';
}

Mặc định, mỗi path parameter và query parameter đến qua network dưới dạng string. Trong ví dụ trên, chúng ta chỉ định type id là number (trong method signature). Do đó, ValidationPipe sẽ cố gắng tự động chuyển đổi string identifier thành number.

Explicit conversion #

Trong section trên, chúng ta đã cho thấy ValidationPipe có thể implicitly transform query và path parameter dựa trên expected type. Tuy nhiên, tính năng này yêu cầu auto-transformation phải được enable.

Ngoài ra (với auto-transformation disabled), bạn có thể explicitly cast value sử dụng ParseIntPipe hoặc ParseBoolPipe (lưu ý rằng ParseStringPipe không cần thiết vì, như đã đề cập trước đó, mọi path parameter và query parameter đều đến qua network dưới dạng string theo mặc định).

@Get(':id')
findOne(
  @Param('id', ParseIntPipe) id: number,
  @Query('sort', ParseBoolPipe) sort: boolean,
) {
  console.log(typeof id === 'number'); // true
  console.log(typeof sort === 'boolean'); // true
  return 'This action returns a user';
}
💡 Hint

ParseIntPipe và ParseBoolPipe được export từ package @nestjs/common.

Mapped types #

Khi bạn xây dựng các tính năng như CRUD (Create/Read/Update/Delete), thường hữu ích khi xây dựng các variant trên một base entity type. Nest cung cấp một số utility function thực hiện type transformation để làm task này thuận tiện hơn.

⚠️ Warning

Nếu ứng dụng của bạn sử dụng package @nestjs/swagger, hãy xem chương này để biết thêm thông tin về Mapped Types. Tương tự, nếu bạn sử dụng package @nestjs/graphql hãy xem chương này. Cả hai package đều phụ thuộc nhiều vào type và do đó chúng yêu cầu import khác nhau để sử dụng. Do đó, nếu bạn sử dụng @nestjs/mapped-types (thay vì package thích hợp, hoặc @nestjs/swagger hoặc @nestjs/graphql) tùy thuộc vào type ứng dụng của bạn, bạn có thể gặp nhiều side-effect không được document.

Khi xây dựng input validation type (còn gọi là DTO), thường hữu ích khi xây dựng các biến thể create và update trên cùng một type. Ví dụ, biến thể create có thể yêu cầu tất cả field, trong khi biến thể update có thể làm tất cả field trở thành optional.

Nest cung cấp utility function PartialType() để làm task này dễ dàng hơn và giảm thiểu boilerplate.

Function PartialType() trả về một type (class) với tất cả property của input type được set thành optional. Ví dụ, giả sử chúng ta có create type như sau:

export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}

Mặc định, tất cả các field này đều là required. Để tạo type với cùng các field, nhưng mỗi field là optional, sử dụng PartialType() truyền class reference (CreateCatDto) làm argument:

export class UpdateCatDto extends PartialType(CreateCatDto) {}
💡 Hint

Function PartialType() được import từ package @nestjs/mapped-types.

Function PickType() xây dựng một type (class) mới bằng cách chọn một tập hợp property từ input type. Ví dụ, giả sử chúng ta bắt đầu với type:

export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}

Chúng ta có thể pick một tập property từ class này bằng utility function PickType():

export class UpdateCatAgeDto extends PickType(CreateCatDto, ['age'] as const) {}
💡 Hint

Function PickType() được import từ package @nestjs/mapped-types.

Function OmitType() xây dựng một type bằng cách chọn tất cả property từ input type rồi loại bỏ một tập key nhất định. Ví dụ, giả sử chúng ta bắt đầu với type:

export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}

Chúng ta có thể tạo ra derived type có mọi property ngoại trừ name như dưới đây. Trong construct này, argument thứ hai cho OmitType là mảng tên property.

export class UpdateCatDto extends OmitType(CreateCatDto, ['name'] as const) {}
💡 Hint

Function OmitType() được import từ package @nestjs/mapped-types.

Function IntersectionType() kết hợp hai type thành một type mới (class). Ví dụ, giả sử chúng ta bắt đầu với hai type:

export class CreateCatDto {
  name: string;
  breed: string;
}

export class AdditionalCatInfo {
  color: string;
}

Chúng ta có thể tạo new type kết hợp tất cả property trong cả hai type:

export class UpdateCatDto extends IntersectionType(
  CreateCatDto,
  AdditionalCatInfo,
) {}
💡 Hint

Function IntersectionType() được import từ package @nestjs/mapped-types.

Các type mapping utility function có thể composable. Ví dụ, code sau sẽ tạo ra type (class) có tất cả property của type CreateCatDto ngoại trừ name, và các property đó sẽ được set thành optional:

export class UpdateCatDto extends PartialType(
  OmitType(CreateCatDto, ['name'] as const),
) {}

Parsing and validating arrays #

TypeScript không lưu trữ metadata về generic hoặc interface, vì vậy khi bạn sử dụng chúng trong DTO, ValidationPipe có thể không validate incoming data đúng cách. Ví dụ, trong code sau, createUserDtos sẽ không được validate chính xác:

@Post()
createBulk(@Body() createUserDtos: CreateUserDto[]) {
  return 'This action adds new users';
}

Để validate array, tạo một class chuyên dụng chứa property wraps array, hoặc sử dụng ParseArrayPipe.

@Post()
createBulk(
  @Body(new ParseArrayPipe({ items: CreateUserDto }))
  createUserDtos: CreateUserDto[],
) {
  return 'This action adds new users';
}

Ngoài ra, ParseArrayPipe có thể hữu ích khi parsing query parameter. Hãy xem xét một method findByIds() trả về user dựa trên identifier được truyền dưới dạng query parameter.

@Get()
findByIds(
  @Query('ids', new ParseArrayPipe({ items: Number, separator: ',' }))
  ids: number[],
) {
  return 'This action returns users by ids';
}

Construct này validate incoming query parameter từ HTTP GET request như sau:

GET /?ids=1,2,3

WebSockets and Microservices #

Mặc dù chương này cho thấy ví dụ sử dụng HTTP style application (ví dụ: Express hoặc Fastify), ValidationPipe hoạt động giống nhau cho WebSocket và microservice, bất kể transport method nào được sử dụng.
