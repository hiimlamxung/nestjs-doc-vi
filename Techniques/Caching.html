Caching

Caching là một kỹ thuật mạnh mẽ và đơn giản giúp nâng cao hiệu năng ứng dụng của bạn. Bằng cách hoạt động như một lớp lưu trữ tạm thời, nó cho phép truy cập nhanh hơn đến dữ liệu được sử dụng thường xuyên, giảm nhu cầu phải lặp lại việc lấy hoặc tính toán cùng một thông tin. Điều này mang lại thời gian phản hồi nhanh hơn và cải thiện hiệu quả tổng thể.

Installation #

Để bắt đầu sử dụng caching trong Nest, bạn cần cài đặt package @nestjs/cache-manager cùng với package cache-manager.

$ npm install @nestjs/cache-manager cache-manager

Mặc định, mọi thứ được lưu trữ trong bộ nhớ in-memory. Vì cache-manager sử dụng Keyv bên dưới, bạn có thể dễ dàng chuyển sang giải pháp lưu trữ nâng cao hơn, chẳng hạn Redis, bằng cách cài đặt package adapter thích hợp. Chúng ta sẽ đề cập chi tiết hơn ở phần sau.

In-memory cache #

Để kích hoạt caching trong ứng dụng của bạn, import CacheModule và cấu hình nó bằng method register():

app.module.ts
TS
JS
import { Module } from '@nestjs/common';
import { CacheModule } from '@nestjs/cache-manager';
import { AppController } from './app.controller';

@Module({
  imports: [CacheModule.register()],
  controllers: [AppController],
})
export class AppModule {}
import { Module } from '@nestjs/common';
import { CacheModule } from '@nestjs/cache-manager';
import { AppController } from './app.controller';

@Module({
  imports: [CacheModule.register()],
  controllers: [AppController],
})
export class AppModule {}

Thiết lập này khởi tạo in-memory caching với cài đặt mặc định, cho phép bạn bắt đầu caching data ngay lập tức.

Interacting with the Cache store #

Để tương tác với cache manager instance, inject nó vào class của bạn bằng token CACHE_MANAGER, như sau:

constructor(@Inject(CACHE_MANAGER) private cacheManager: Cache) {}
💡 Hint

Class Cache và token CACHE_MANAGER đều được import từ package @nestjs/cache-manager.

Method get trên instance Cache (từ package cache-manager) được sử dụng để lấy item từ cache. Nếu item không tồn tại trong cache, null sẽ được trả về.

const value = await this.cacheManager.get('key');

Để thêm item vào cache, sử dụng method set:

await this.cacheManager.set('key', 'value');
📘 Note

In-memory cache storage chỉ có thể lưu trữ các giá trị có kiểu được hỗ trợ bởi structured clone algorithm.

Bạn có thể chỉ định TTL (expiration time tính bằng milliseconds) thủ công cho key cụ thể, như sau:

await this.cacheManager.set('key', 'value', 1000);

Trong đó 1000 là TTL tính bằng milliseconds - trong trường hợp này, cache item sẽ hết hạn sau một giây.

Để vô hiệu hóa expiration của cache, set configuration property ttl thành 0:

await this.cacheManager.set('key', 'value', 0);

Để xóa một item khỏi cache, sử dụng method del:

await this.cacheManager.del('key');

Để xóa toàn bộ cache, sử dụng method clear:

await this.cacheManager.clear();

Auto-caching responses #

⚠️ Warning

Trong các ứng dụng GraphQL, interceptor được thực thi riêng biệt cho từng field resolver. Do đó, CacheModule (sử dụng interceptor để cache response) sẽ không hoạt động đúng cách.

Để kích hoạt auto-caching response, chỉ cần bind CacheInterceptor vào nơi bạn muốn cache data.

@Controller()
@UseInterceptors(CacheInterceptor)
export class AppController {
  @Get()
  findAll(): string[] {
    return [];
  }
}
⚠️ Warning

Chỉ có GET endpoint mới được cache. Ngoài ra, các HTTP server route mà inject native response object (@Res()) không thể sử dụng Cache Interceptor. Xem response mapping để biết thêm chi tiết.

Để giảm lượng boilerplate cần thiết, bạn có thể bind CacheInterceptor cho tất cả endpoint globally:

app.module.ts
TS
JS
import { Module } from '@nestjs/common';
import { CacheModule, CacheInterceptor } from '@nestjs/cache-manager';
import { AppController } from './app.controller';
import { APP_INTERCEPTOR } from '@nestjs/core';

@Module({
  imports: [CacheModule.register()],
  controllers: [AppController],
  providers: [
    {
      provide: APP_INTERCEPTOR,
      useClass: CacheInterceptor,
    },
  ],
})
export class AppModule {}
import { Module } from '@nestjs/common';
import { CacheModule, CacheInterceptor } from '@nestjs/cache-manager';
import { AppController } from './app.controller';
import { APP_INTERCEPTOR } from '@nestjs/core';

@Module({
  imports: [CacheModule.register()],
  controllers: [AppController],
  providers: [
    {
      provide: APP_INTERCEPTOR,
      useClass: CacheInterceptor,
    },
  ],
})
export class AppModule {}

Time-to-live (TTL) #

Giá trị mặc định cho ttl là 0, nghĩa là cache sẽ không bao giờ hết hạn. Để chỉ định TTL tùy chỉnh, bạn có thể cung cấp option ttl trong method register(), như minh họa bên dưới:

CacheModule.register({
  ttl: 5000, // milliseconds
});

Use module globally #

Khi bạn muốn sử dụng CacheModule trong các module khác, bạn sẽ cần import nó (như tiêu chuẩn với bất kỳ Nest module nào). Ngoài ra, hãy khai báo nó như một global module bằng cách set property isGlobal trong options object thành true như bên dưới. Trong trường hợp đó, bạn sẽ không cần import CacheModule trong các module khác sau khi nó đã được load trong root module (v.d., AppModule).

CacheModule.register({
  isGlobal: true,
});

Global cache overrides #

Trong khi global cache được enable, cache entry được lưu dưới CacheKey được tự động tạo dựa trên route path. Bạn có thể override một số cache setting nhất định (@CacheKey() và @CacheTTL()) trên cơ sở per-method, cho phép tùy chỉnh caching strategy cho từng controller method riêng lẻ. Điều này có thể hữu ích nhất khi sử dụng different cache stores.

Bạn có thể áp dụng decorator @CacheTTL() trên cơ sở per-controller để set caching TTL cho toàn bộ controller. Trong trường hợp cả controller-level và method-level cache TTL setting đều được định nghĩa, cache TTL setting được chỉ định ở method level sẽ được ưu tiên hơn so với setting ở controller level.

@Controller()
@CacheTTL(50)
export class AppController {
  @CacheKey('custom_key')
  @CacheTTL(20)
  @Get()
  findAll(): string[] {
    return [];
  }
}
💡 Hint

Decorator @CacheKey() và @CacheTTL() được import từ package @nestjs/cache-manager.

Decorator @CacheKey() có thể được sử dụng có hoặc không có decorator @CacheTTL() tương ứng và ngược lại. Bạn có thể chọn chỉ override @CacheKey() hoặc chỉ @CacheTTL(). Setting nào không được override bằng decorator sẽ sử dụng giá trị mặc định đã đăng ký globally (xem Customize caching).

WebSockets and Microservices #

Bạn cũng có thể áp dụng CacheInterceptor cho WebSocket subscriber cũng như Microservice's pattern (bất kể transport method nào đang được sử dụng).

TS
JS
@CacheKey('events')
@UseInterceptors(CacheInterceptor)
@SubscribeMessage('events')
handleEvent(client: Client, data: string[]): Observable<string[]> {
  return [];
}
@CacheKey('events')
@UseInterceptors(CacheInterceptor)
@SubscribeMessage('events')
handleEvent(client: Client, data: string[]): Observable<string[]> {
  return [];
}

Tuy nhiên, decorator @CacheKey() bổ sung là bắt buộc để chỉ định key được sử dụng để lưu trữ và truy xuất cached data sau đó. Ngoài ra, xin lưu ý rằng bạn không nên cache mọi thứ. Các action thực hiện business operation thay vì chỉ đơn giản query data thì không bao giờ nên được cache.

Ngoài ra, bạn có thể chỉ định cache expiration time (TTL) bằng cách sử dụng decorator @CacheTTL(), decorator này sẽ override giá trị TTL mặc định global.

TS
JS
@CacheTTL(10)
@UseInterceptors(CacheInterceptor)
@SubscribeMessage('events')
handleEvent(client: Client, data: string[]): Observable<string[]> {
  return [];
}
@CacheTTL(10)
@UseInterceptors(CacheInterceptor)
@SubscribeMessage('events')
handleEvent(client: Client, data: string[]): Observable<string[]> {
  return [];
}
💡 Hint

Decorator @CacheTTL() có thể được sử dụng có hoặc không có decorator @CacheKey() tương ứng.

Adjust tracking #

Mặc định, Nest sử dụng request URL (trong HTTP app) hoặc cache key (trong websockets và microservices app, được set thông qua decorator @CacheKey()) để liên kết cache record với endpoint của bạn. Tuy nhiên, đôi khi bạn có thể muốn thiết lập tracking dựa trên các yếu tố khác nhau, ví dụ, sử dụng HTTP header (v.d. Authorization để xác định đúng profile endpoint).

Để thực hiện điều đó, tạo một subclass của CacheInterceptor và override method trackBy().

@Injectable()
class HttpCacheInterceptor extends CacheInterceptor {
  trackBy(context: ExecutionContext): string | undefined {
    return 'key';
  }
}

Using alternative Cache stores #

Chuyển sang cache store khác rất đơn giản. Đầu tiên, cài đặt package thích hợp. Ví dụ, để sử dụng Redis, cài đặt package @keyv/redis:

$ npm install @keyv/redis

Với package đã cài đặt, bạn có thể đăng ký CacheModule với multiple store như bên dưới:

import { Module } from '@nestjs/common';
import { CacheModule } from '@nestjs/cache-manager';
import { AppController } from './app.controller';
import KeyvRedis from '@keyv/redis';
import { Keyv } from 'keyv';
import { CacheableMemory } from 'cacheable';

@Module({
  imports: [
    CacheModule.registerAsync({
      useFactory: async () => {
        return {
          stores: [
            new Keyv({
              store: new CacheableMemory({ ttl: 60000, lruSize: 5000 }),
            }),
            new KeyvRedis('redis://localhost:6379'),
          ],
        };
      },
    }),
  ],
  controllers: [AppController],
})
export class AppModule {}

Trong ví dụ này, chúng ta đã đăng ký hai store: CacheableMemory và KeyvRedis. CacheableMemory store là một in-memory store đơn giản, trong khi KeyvRedis là Redis store. Mảng stores được sử dụng để chỉ định các store bạn muốn sử dụng. Store đầu tiên trong mảng là store mặc định, và phần còn lại là các fallback store.

Xem Keyv documentation để biết thêm thông tin về các store có sẵn.

Async configuration #

Bạn có thể muốn truyền module option một cách bất đồng bộ thay vì truyền tĩnh tại compile time. Trong trường hợp này, sử dụng method registerAsync(), cung cấp nhiều cách để xử lý async configuration.

Một cách tiếp cận là sử dụng factory function:

CacheModule.registerAsync({
  useFactory: () => ({
    ttl: 5,
  }),
});

Factory của chúng ta hoạt động giống như tất cả các asynchronous module factory khác (nó có thể là async và có thể inject dependency thông qua inject).

CacheModule.registerAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    ttl: configService.get('CACHE_TTL'),
  }),
  inject: [ConfigService],
});

Ngoài ra, bạn có thể sử dụng method useClass:

CacheModule.registerAsync({
  useClass: CacheConfigService,
});

Đoạn trên sẽ instantiate CacheConfigService bên trong CacheModule và sẽ sử dụng nó để lấy options object. CacheConfigService phải implement interface CacheOptionsFactory để cung cấp configuration option:

@Injectable()
class CacheConfigService implements CacheOptionsFactory {
  createCacheOptions(): CacheModuleOptions {
    return {
      ttl: 5,
    };
  }
}

Nếu bạn muốn sử dụng configuration provider hiện có được import từ module khác, sử dụng cú pháp useExisting:

CacheModule.registerAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});

Điều này hoạt động giống như useClass với một điểm khác biệt quan trọng - CacheModule sẽ tìm kiếm các module đã import để tái sử dụng bất kỳ ConfigService nào đã được tạo, thay vì instantiate một bản mới.

💡 Hint

CacheModule#register, CacheModule#registerAsync và CacheOptionsFactory có optional generic (type argument) để thu hẹp store-specific configuration option, giúp đảm bảo type safe.

Bạn cũng có thể truyền extraProviders vào method registerAsync(). Các provider này sẽ được merge với module provider.

CacheModule.registerAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    ttl: configService.get('CACHE_TTL'),
  }),
  inject: [ConfigService],
  extraProviders: [MyAdditionalProvider],
});
