Configuration

Ứng dụng thường chạy trong nhiều môi trường (environments) khác nhau. Tùy thuộc vào môi trường, các cài đặt cấu hình khác nhau nên được sử dụng. Ví dụ, thông thường môi trường local phụ thuộc vào các thông tin đăng nhập database cụ thể, chỉ hợp lệ với instance DB local. Môi trường production sẽ dùng một bộ thông tin đăng nhập DB riêng biệt. Vì các biến cấu hình thay đổi, best practice là lưu trữ các biến cấu hình trong môi trường.

Các biến môi trường được định nghĩa bên ngoài có thể nhìn thấy bên trong Node.js thông qua global process.env. Chúng ta có thể cố gắng giải quyết vấn đề đa môi trường bằng cách thiết lập riêng các biến môi trường trong từng môi trường. Tuy nhiên điều này nhanh chóng trở nên khó quản lý, đặc biệt trong môi trường development và testing khi các giá trị này cần được mock và/hoặc thay đổi dễ dàng.

Trong các ứng dụng Node.js, thông thường dùng các file .env chứa các cặp key-value, trong đó mỗi key đại diện cho một giá trị cụ thể, để biểu diễn từng môi trường. Chạy app trong các môi trường khác nhau chỉ đơn giản là thay đổi file .env đúng.

Một cách tiếp cận tốt khi dùng kỹ thuật này trong Nest là tạo một ConfigModule expose ra một ConfigService để load file .env phù hợp. Mặc dù bạn có thể tự viết module như vậy, nhưng để tiện lợi, Nest cung cấp sẵn package @nestjs/config. Chúng ta sẽ tìm hiểu package này trong chương này.

Installation #

Để bắt đầu sử dụng, trước tiên hãy cài đặt dependency cần thiết.

bash
$ npm i --save @nestjs/config
💡
HintPackage @nestjs/config sử dụng nội bộ dotenv.
💡
Note@nestjs/config yêu cầu TypeScript 4.1 trở lên.

Getting started #

Sau khi quá trình cài đặt hoàn tất, chúng ta có thể import ConfigModule. Thông thường, chúng ta sẽ import nó vào root AppModule và kiểm soát hành vi của nó bằng static method .forRoot(). Trong bước này, các cặp key/value biến môi trường được parse và resolved. Sau đó, chúng ta sẽ thấy một số tùy chọn để truy cập class ConfigService của ConfigModule trong các feature module khác.

app.module.ts
import { Module } from '@nestjs/common';
import { ConfigModule } from '@nestjs/config';

@Module({
  imports: [ConfigModule.forRoot()],
})
export class AppModule {}

Đoạn code trên sẽ load và parse file .env từ vị trí mặc định (thư mục gốc của project), hợp nhất các cặp key/value từ file .env với các biến môi trường được gán cho process.env, và lưu kết quả vào một cấu trúc private mà bạn có thể truy cập thông qua ConfigService. Method forRoot() đăng ký provider ConfigService, vốn cung cấp method get() để đọc các biến cấu hình đã được parse/merge. Vì @nestjs/config phụ thuộc vào dotenv, nó dùng các quy tắc của package đó để giải quyết xung đột trong tên biến môi trường. Khi một key tồn tại cả trong runtime environment dưới dạng biến môi trường (ví dụ, thông qua OS shell exports như export DATABASE_USER=test) và trong file .env, thì biến môi trường runtime sẽ được ưu tiên.

Một file .env mẫu trông như thế này:

.env
DATABASE_USER=test
DATABASE_PASSWORD=test

Nếu bạn cần một số biến môi trường phải có sẵn ngay cả trước khi ConfigModule được load và ứng dụng Nest được bootstrap (ví dụ, để truyền cấu hình microservice vào method NestFactory#createMicroservice), bạn có thể dùng option --env-file của Nest CLI. Option này cho phép bạn chỉ định đường dẫn đến file .env sẽ được load trước khi ứng dụng khởi động. Hỗ trợ flag --env-file được giới thiệu từ Node v20, xem tài liệu để biết thêm chi tiết.

bash
$ nest start --env-file .env

Custom env file path #

Theo mặc định, package tìm kiếm file .env trong thư mục gốc của ứng dụng. Để chỉ định đường dẫn khác cho file .env, hãy set thuộc tính envFilePath trong options object (tùy chọn) mà bạn truyền vào forRoot(), như sau:

ConfigModule.forRoot({
  envFilePath: '.development.env',
});

Bạn cũng có thể chỉ định nhiều đường dẫn cho file .env như thế này:

ConfigModule.forRoot({
  envFilePath: ['.env.development.local', '.env.development'],
});

Nếu một biến xuất hiện trong nhiều file, file đầu tiên sẽ được ưu tiên.

Disable env variables loading #

Nếu bạn không muốn load file .env, mà thay vào đó chỉ muốn truy cập các biến môi trường từ runtime environment (như với OS shell exports dạng export DATABASE_USER=test), hãy set thuộc tính ignoreEnvFile của options object thành true, như sau:

ConfigModule.forRoot({
  ignoreEnvFile: true,
});

Use module globally #

Khi bạn muốn dùng ConfigModule trong các module khác, bạn sẽ cần import nó (như với bất kỳ Nest module nào theo chuẩn). Ngoài ra, có thể khai báo nó là global module bằng cách set thuộc tính isGlobal của options object thành true, như bên dưới. Trong trường hợp đó, bạn sẽ không cần import ConfigModule trong các module khác một khi nó đã được load trong root module (ví dụ, AppModule).

ConfigModule.forRoot({
  isGlobal: true,
});

Custom configuration files #

Đối với các project phức tạp hơn, bạn có thể sử dụng các file cấu hình tùy chỉnh để trả về các configuration object lồng nhau. Điều này cho phép bạn nhóm các cài đặt cấu hình liên quan theo chức năng (ví dụ: các cài đặt liên quan đến database), và lưu trữ các cài đặt liên quan trong từng file riêng lẻ để giúp quản lý chúng độc lập.

Một file cấu hình tùy chỉnh export một factory function trả về một configuration object. Configuration object có thể là bất kỳ plain JavaScript object lồng nhau tùy ý nào. Object process.env sẽ chứa các cặp key/value biến môi trường đã được fully resolved (với file .env và các biến được định nghĩa bên ngoài đã được resolved và merge như mô tả ở trên). Vì bạn kiểm soát configuration object được trả về, bạn có thể thêm bất kỳ logic nào cần thiết để cast giá trị sang kiểu phù hợp, set giá trị mặc định, v.v. Ví dụ:

config/configuration.ts
export default () => ({
  port: parseInt(process.env.PORT, 10) || 3000,
  database: {
    host: process.env.DATABASE_HOST,
    port: parseInt(process.env.DATABASE_PORT, 10) || 5432
  }
});

Chúng ta load file này bằng thuộc tính load của options object truyền vào method ConfigModule.forRoot():

import configuration from './config/configuration';

@Module({
  imports: [
    ConfigModule.forRoot({
      load: [configuration],
    }),
  ],
})
export class AppModule {}
💡
NoticeGiá trị gán cho thuộc tính load là một array, cho phép bạn load nhiều file cấu hình (ví dụ: load: [databaseConfig, authConfig])

Với các file cấu hình tùy chỉnh, chúng ta cũng có thể quản lý các file tùy chỉnh như file YAML. Đây là ví dụ về cấu hình sử dụng định dạng YAML:

yaml
http:
  host: 'localhost'
  port: 8080

db:
  postgres:
    url: 'localhost'
    port: 5432
    database: 'yaml-db'

  sqlite:
    database: 'sqlite.db'

Để đọc và parse các file YAML, chúng ta có thể tận dụng package js-yaml.

bash
$ npm i js-yaml
$ npm i -D @types/js-yaml

Sau khi package được cài đặt, chúng ta dùng hàm yaml#load để load file YAML vừa tạo ở trên.

config/configuration.ts
import { readFileSync } from 'node:fs';
import { join } from 'node:path';
import * as yaml from 'js-yaml';

const YAML_CONFIG_FILENAME = 'config.yaml';

export default () => {
  return yaml.load(
    readFileSync(join(__dirname, YAML_CONFIG_FILENAME), 'utf8'),
  ) as Record<string, any>;
};
⚠️
NoteNest CLI không tự động chuyển các "assets" (các file không phải TS) sang thư mục dist trong quá trình build. Để đảm bảo các file YAML được sao chép, bạn phải chỉ định điều này trong object compilerOptions#assets trong file nest-cli.json. Ví dụ: nếu thư mục config nằm cùng cấp với thư mục src, hãy thêm compilerOptions#assets với giá trị "assets": [{"include": "../config/*.yaml", "outDir": "./dist/config"}]. Đọc thêm tại đây.

Một lưu ý nhanh — các file cấu hình không được tự động validate, ngay cả khi bạn đang dùng option validationSchema trong ConfigModule của NestJS. Nếu bạn cần validation hoặc muốn áp dụng bất kỳ phép biến đổi nào, bạn sẽ phải tự xử lý trong factory function nơi bạn có toàn quyền kiểm soát configuration object. Điều này cho phép bạn triển khai bất kỳ logic validation tùy chỉnh nào khi cần.

Ví dụ, nếu bạn muốn đảm bảo rằng port nằm trong một khoảng nhất định, bạn có thể thêm một bước validation vào factory function:

config/configuration.ts
export default () => {
  const config = yaml.load(
    readFileSync(join(__dirname, YAML_CONFIG_FILENAME), 'utf8'),
  ) as Record<string, any>;

  if (config.http.port < 1024 || config.http.port > 49151) {
    throw new Error('HTTP port must be between 1024 and 49151');
  }

  return config;
};

Bây giờ, nếu port nằm ngoài khoảng được chỉ định, ứng dụng sẽ throw một lỗi trong quá trình khởi động.

Using the ConfigService #

Để truy cập các giá trị cấu hình từ ConfigService, trước tiên chúng ta cần inject ConfigService. Như với bất kỳ provider nào, chúng ta cần import module chứa nó — ConfigModule — vào module sẽ sử dụng nó (trừ khi bạn đã set thuộc tính isGlobal thành true). Import nó vào một feature module như bên dưới.

feature.module.ts
@Module({
  imports: [ConfigModule],
  // ...
})

Sau đó chúng ta có thể inject nó bằng constructor injection thông thường:

constructor(private configService: ConfigService) {}
💡
HintConfigService được import từ package @nestjs/config.

Và sử dụng nó trong class của chúng ta:

// lấy một biến môi trường
const dbUser = this.configService.get<string>('DATABASE_USER');

// lấy một giá trị cấu hình tùy chỉnh
const dbHost = this.configService.get<string>('database.host');

Như minh họa ở trên, dùng method configService.get() để lấy một biến môi trường đơn giản bằng cách truyền tên biến. Bạn có thể TypeScript type hinting bằng cách truyền kiểu dữ liệu (ví dụ: get<string>(...)). Method get() cũng có thể duyệt qua một nested custom configuration object (được tạo qua Custom configuration file), như ví dụ thứ hai ở trên.

Bạn cũng có thể lấy toàn bộ nested custom configuration object bằng cách dùng một interface làm type hint:

interface DatabaseConfig {
  host: string;
  port: number;
}

const dbConfig = this.configService.get<DatabaseConfig>('database');

// giờ bạn có thể dùng `dbConfig.port` và `dbConfig.host`
const port = dbConfig.port;

Method get() cũng nhận một tham số thứ hai tùy chọn định nghĩa giá trị mặc định, sẽ được trả về khi key không tồn tại, như bên dưới:

// dùng "localhost" khi "database.host" không được định nghĩa
const dbHost = this.configService.get<string>('database.host', 'localhost');

ConfigService có hai optional generics (type arguments). Generic đầu tiên giúp ngăn chặn truy cập vào một config property không tồn tại. Dùng như bên dưới:

interface EnvironmentVariables {
  PORT: number;
  TIMEOUT: string;
}

// ở đâu đó trong code
constructor(private configService: ConfigService<EnvironmentVariables>) {
  const port = this.configService.get('PORT', { infer: true });

  // TypeScript Error: không hợp lệ vì URL không được định nghĩa trong EnvironmentVariables
  const url = this.configService.get('URL', { infer: true });
}

Khi thuộc tính infer được set thành true, method ConfigService#get sẽ tự động suy luận kiểu của property dựa trên interface, vì vậy ví dụ, typeof port === "number" (nếu bạn không dùng flag strictNullChecks của TypeScript) vì PORT có kiểu number trong interface EnvironmentVariables.

Ngoài ra, với tính năng infer, bạn có thể suy luận kiểu của property trong một nested custom configuration object, ngay cả khi dùng dot notation, như sau:

constructor(private configService: ConfigService<{ database: { host: string } }>) {
  const dbHost = this.configService.get('database.host', { infer: true })!;
  // typeof dbHost === "string"                                          |
  //                                                                     +--> non-null assertion operator
}

Generic thứ hai phụ thuộc vào generic đầu tiên, hoạt động như một type assertion để loại bỏ tất cả các kiểu undefined mà các method của ConfigService có thể trả về khi strictNullChecks được bật. Ví dụ:

// ...
constructor(private configService: ConfigService<{ PORT: number }, true>) {
  //                                                               ^^^^
  const port = this.configService.get('PORT', { infer: true });
  //    ^^^ Kiểu của port sẽ là 'number', bạn không cần TS type assertions nữa
}
💡
HintĐể đảm bảo method ConfigService#get chỉ lấy giá trị từ các custom configuration file và bỏ qua các biến process.env, hãy set option skipProcessEnv thành true trong options object của method forRoot() của ConfigModule.

Configuration namespaces #

ConfigModule cho phép bạn định nghĩa và load nhiều file cấu hình tùy chỉnh, như đã trình bày trong Custom configuration files ở trên. Bạn có thể quản lý các cấu trúc configuration object phức tạp với nested configuration objects như trong phần đó. Ngoài ra, bạn có thể trả về một "namespaced" configuration object với hàm registerAs() như sau:

config/database.config.ts
export default registerAs('database', () => ({
  host: process.env.DATABASE_HOST,
  port: process.env.DATABASE_PORT || 5432
}));

Như với các file cấu hình tùy chỉnh, bên trong factory function của registerAs(), object process.env sẽ chứa các cặp key/value biến môi trường đã được fully resolved (với file .env và các biến được định nghĩa bên ngoài đã được resolved và merge như mô tả ở trên).

💡
HintHàm registerAs được export từ package @nestjs/config.

Load một namespaced configuration với thuộc tính load trong options object của method forRoot(), theo cách tương tự như load một custom configuration file:

import databaseConfig from './config/database.config';

@Module({
  imports: [
    ConfigModule.forRoot({
      load: [databaseConfig],
    }),
  ],
})
export class AppModule {}

Bây giờ, để lấy giá trị host từ namespace database, hãy dùng dot notation. Dùng 'database' làm prefix cho tên property, tương ứng với tên của namespace (được truyền làm tham số đầu tiên cho hàm registerAs()):

const dbHost = this.configService.get<string>('database.host');

Một cách thay thế hợp lý là inject trực tiếp namespace database. Điều này cho phép chúng ta hưởng lợi từ strong typing:

constructor(
  @Inject(databaseConfig.KEY)
  private dbConfig: ConfigType<typeof databaseConfig>,
) {}
💡
HintConfigType được export từ package @nestjs/config.

Namespaced configurations in modules #

Để dùng một namespaced configuration như là configuration object cho một module khác trong ứng dụng của bạn, bạn có thể tận dụng method .asProvider() của configuration object. Method này chuyển đổi namespaced configuration của bạn thành một provider, sau đó có thể được truyền vào forRootAsync() (hoặc bất kỳ method tương đương nào) của module bạn muốn sử dụng.

Đây là một ví dụ:

import databaseConfig from './config/database.config';

@Module({
  imports: [
    TypeOrmModule.forRootAsync(databaseConfig.asProvider()),
  ],
})

Để hiểu cách method .asProvider() hoạt động, hãy xem giá trị trả về của nó:

// Giá trị trả về của method .asProvider()
{
  imports: [ConfigModule.forFeature(databaseConfig)],
  useFactory: (configuration: ConfigType<typeof databaseConfig>) => configuration,
  inject: [databaseConfig.KEY]
}

Cấu trúc này cho phép bạn tích hợp liền mạch các namespaced configuration vào các module của mình, đảm bảo ứng dụng vẫn được tổ chức gọn gàng và có tính module cao, mà không phải viết code boilerplate lặp đi lặp lại.

Cache environment variables #

Vì truy cập process.env có thể chậm, bạn có thể set thuộc tính cache của options object truyền vào ConfigModule.forRoot() thành true để tăng hiệu suất của method ConfigService#get khi làm việc với các biến được lưu trong process.env.

ConfigModule.forRoot({
  cache: true,
});

Partial registration #

Cho đến nay, chúng ta đã xử lý các file cấu hình trong root module (ví dụ: AppModule), với method forRoot(). Có thể bạn có một cấu trúc project phức tạp hơn, với các file cấu hình theo từng feature nằm trong nhiều thư mục khác nhau. Thay vì load tất cả các file này trong root module, package @nestjs/config cung cấp một tính năng gọi là partial registration, chỉ tham chiếu đến các file cấu hình liên quan đến từng feature module. Dùng static method forFeature() trong một feature module để thực hiện partial registration này, như sau:

import databaseConfig from './config/database.config';

@Module({
  imports: [ConfigModule.forFeature(databaseConfig)],
})
export class DatabaseModule {}
⚠️
WarningTrong một số trường hợp, bạn có thể cần truy cập các property được load thông qua partial registration bằng hook onModuleInit(), thay vì trong constructor. Điều này là do method forFeature() được chạy trong quá trình khởi tạo module, và thứ tự khởi tạo module là không xác định. Nếu bạn truy cập các giá trị được load theo cách này bởi một module khác trong constructor, module mà cấu hình phụ thuộc vào có thể chưa được khởi tạo. Method onModuleInit() chỉ chạy sau khi tất cả các module mà nó phụ thuộc vào đã được khởi tạo, vì vậy kỹ thuật này an toàn.

Schema validation #

Thực tế phổ biến là throw một exception trong quá trình khởi động ứng dụng nếu các biến môi trường bắt buộc chưa được cung cấp hoặc không đáp ứng một số quy tắc validation nhất định. Package @nestjs/config cho phép hai cách khác nhau để làm điều này:

  • Validator tích hợp sẵn của Joi. Với Joi, bạn định nghĩa một object schema và validate các JavaScript object với nó.
  • Một hàm validate() tùy chỉnh nhận các biến môi trường làm đầu vào.

Để dùng Joi, chúng ta phải cài đặt package Joi:

bash
$ npm install --save joi

Bây giờ chúng ta có thể định nghĩa một Joi validation schema và truyền nó qua thuộc tính validationSchema của options object trong method forRoot(), như bên dưới:

app.module.ts
import * as Joi from 'joi';

@Module({
  imports: [
    ConfigModule.forRoot({
      validationSchema: Joi.object({
        NODE_ENV: Joi.string()
          .valid('development', 'production', 'test', 'provision')
          .default('development'),
        PORT: Joi.number().port().default(3000),
      }),
    }),
  ],
})
export class AppModule {}

Theo mặc định, tất cả các schema key được coi là tùy chọn. Ở đây, chúng ta set giá trị mặc định cho NODE_ENV và PORT sẽ được sử dụng nếu chúng ta không cung cấp các biến này trong môi trường (file .env hoặc process environment). Ngoài ra, chúng ta có thể dùng method validation required() để yêu cầu một giá trị phải được định nghĩa trong môi trường. Trong trường hợp đó, bước validation sẽ throw một exception nếu chúng ta không cung cấp biến trong môi trường. Xem Joi validation methods để biết thêm về cách xây dựng validation schemas.

Theo mặc định, các biến môi trường không xác định (các biến môi trường có key không có trong schema) được cho phép và không trigger exception validation. Theo mặc định, tất cả lỗi validation đều được báo cáo. Bạn có thể thay đổi các hành vi này bằng cách truyền một options object qua key validationOptions trong options object của forRoot(). Options object này có thể chứa bất kỳ thuộc tính validation options tiêu chuẩn nào được cung cấp bởi Joi validation options. Ví dụ: để đảo ngược hai cài đặt trên, hãy truyền các options như sau:

app.module.ts
import * as Joi from 'joi';

@Module({
  imports: [
    ConfigModule.forRoot({
      validationSchema: Joi.object({
        NODE_ENV: Joi.string()
          .valid('development', 'production', 'test', 'provision')
          .default('development'),
        PORT: Joi.number().port().default(3000),
      }),
      validationOptions: {
        allowUnknown: false,
        abortEarly: true,
      },
    }),
  ],
})
export class AppModule {}

Package @nestjs/config sử dụng các cài đặt mặc định là:

  • allowUnknown: kiểm soát có cho phép các key không xác định trong biến môi trường hay không. Mặc định là true
  • abortEarly: nếu true, dừng validation ở lỗi đầu tiên; nếu false, trả về tất cả lỗi. Mặc định là false.

Lưu ý rằng một khi bạn quyết định truyền một validationOptions object, bất kỳ cài đặt nào bạn không truyền rõ ràng sẽ mặc định theo giá trị tiêu chuẩn của Joi (không phải giá trị mặc định của @nestjs/config). Ví dụ: nếu bạn để allowUnknowns không xác định trong validationOptions object tùy chỉnh, nó sẽ có giá trị mặc định của Joi là false. Do đó, tốt nhất là chỉ định cả hai cài đặt này trong object tùy chỉnh của bạn.

💡
HintĐể tắt validation của các predefined environment variables, hãy set thuộc tính validatePredefined thành false trong options object của method forRoot(). Predefined environment variables là các biến process (process.env) được set trước khi module được import. Ví dụ: nếu bạn khởi động ứng dụng với PORT=3000 node main.js, thì PORT là một predefined environment variable.

Custom validate function #

Ngoài ra, bạn có thể chỉ định một hàm validate đồng bộ (synchronous) nhận một object chứa các biến môi trường (từ env file và process) và trả về một object chứa các biến môi trường đã được validate để bạn có thể convert/mutate chúng nếu cần. Nếu hàm throw một lỗi, nó sẽ ngăn ứng dụng khởi động.

Trong ví dụ này, chúng ta sẽ sử dụng các package class-transformer và class-validator. Trước tiên, chúng ta phải định nghĩa:

  • một class với các validation constraints,
  • một hàm validate sử dụng các hàm plainToInstance và validateSync.
env.validation.ts
import { plainToInstance } from 'class-transformer';
import { IsEnum, IsNumber, Max, Min, validateSync } from 'class-validator';

enum Environment {
  Development = "development",
  Production  = "production",
  Test        = "test",
  Provision   = "provision",
}

class EnvironmentVariables {
  @IsEnum(Environment)
  NODE_ENV: Environment;

  @IsNumber()
  @Min(0)
  @Max(65535)
  PORT: number;
}

export function validate(config: Record<string, unknown>) {
  const validatedConfig = plainToInstance(
    EnvironmentVariables,
    config,
    { enableImplicitConversion: true },
  );
  const errors = validateSync(validatedConfig, { skipMissingProperties: false });

  if (errors.length > 0) {
    throw new Error(errors.toString());
  }
  return validatedConfig;
}

Với điều này, hãy dùng hàm validate như một configuration option của ConfigModule, như sau:

app.module.ts
import { validate } from './env.validation';

@Module({
  imports: [
    ConfigModule.forRoot({
      validate,
    }),
  ],
})
export class AppModule {}

Custom getter functions #

ConfigService định nghĩa một generic method get() để lấy một giá trị cấu hình theo key. Chúng ta cũng có thể thêm các hàm getter để cho phép một coding style tự nhiên hơn:

@Injectable()
export class ApiConfigService {
  constructor(private configService: ConfigService) {}

  get isAuthEnabled(): boolean {
    return this.configService.get('AUTH_ENABLED') === 'true';
  }
}

Bây giờ chúng ta có thể dùng getter function như sau:

app.service.ts
@Injectable()
export class AppService {
  constructor(apiConfigService: ApiConfigService) {
    if (apiConfigService.isAuthEnabled) {
      // Authentication được bật
    }
  }
}

Environment variables loaded hook #

Nếu cấu hình của một module phụ thuộc vào các biến môi trường, và các biến này được load từ file .env, bạn có thể dùng hook ConfigModule.envVariablesLoaded để đảm bảo rằng file đã được load trước khi tương tác với object process.env, xem ví dụ sau:

export async function getStorageModule() {
  await ConfigModule.envVariablesLoaded;
  return process.env.STORAGE === 'S3' ? S3StorageModule : DefaultStorageModule;
}

Cấu trúc này đảm bảo rằng sau khi Promise ConfigModule.envVariablesLoaded resolve, tất cả các biến cấu hình đều đã được load xong.

Conditional module configuration #

Đôi khi bạn muốn load có điều kiện một module và chỉ định điều kiện đó trong một biến môi trường. May mắn thay, @nestjs/config cung cấp một ConditionalModule cho phép bạn làm chính xác điều đó.

@Module({
  imports: [
    ConfigModule.forRoot(),
    ConditionalModule.registerWhen(FooModule, 'USE_FOO'),
  ],
})
export class AppModule {}

Module trên sẽ chỉ load FooModule nếu trong file .env không có giá trị false cho biến môi trường USE_FOO. Bạn cũng có thể tự truyền một điều kiện tùy chỉnh, một hàm nhận reference process.env và phải trả về một boolean để ConditionalModule xử lý:

@Module({
  imports: [
    ConfigModule.forRoot(),
    ConditionalModule.registerWhen(
      FooBarModule,
      (env: NodeJS.ProcessEnv) => !!env['foo'] && !!env['bar'],
    ),
  ],
})
export class AppModule {}

Điều quan trọng là phải đảm bảo rằng khi dùng ConditionalModule, bạn cũng phải có ConfigModule được load trong ứng dụng, để hook ConfigModule.envVariablesLoaded có thể được tham chiếu và sử dụng đúng cách. Nếu hook không được chuyển sang true trong vòng 5 giây, hoặc trong khoảng thời gian timeout tính bằng milliseconds do người dùng set trong tham số options thứ ba của method registerWhen, thì ConditionalModule sẽ throw một lỗi và Nest sẽ hủy việc khởi động ứng dụng.

Expandable variables #

Package @nestjs/config hỗ trợ mở rộng biến môi trường (environment variable expansion). Với kỹ thuật này, bạn có thể tạo các nested environment variables, nơi một biến được tham chiếu trong định nghĩa của biến khác. Ví dụ:

APP_URL=mywebsite.com
SUPPORT_EMAIL=support@${APP_URL}

Với cấu trúc này, biến SUPPORT_EMAIL resolve thành 'support@mywebsite.com'. Chú ý việc sử dụng cú pháp ${'{'}...{'}'} để kích hoạt việc resolve giá trị của biến APP_URL bên trong định nghĩa của SUPPORT_EMAIL.

💡
HintCho tính năng này, package @nestjs/config sử dụng nội bộ dotenv-expand.

Bật mở rộng biến môi trường bằng thuộc tính expandVariables trong options object truyền vào method forRoot() của ConfigModule, như bên dưới:

app.module.ts
@Module({
  imports: [
    ConfigModule.forRoot({
      // ...
      expandVariables: true,
    }),
  ],
})
export class AppModule {}

Using in the main.ts #

Mặc dù config của chúng ta được lưu trong một service, nó vẫn có thể được sử dụng trong file main.ts. Bằng cách này, bạn có thể dùng nó để lưu trữ các biến như application port hoặc CORS host.

Để truy cập nó, bạn phải dùng method app.get(), theo sau là service reference:

const configService = app.get(ConfigService);

Sau đó bạn có thể dùng nó như thường lệ, bằng cách gọi method get với configuration key:

const port = configService.get('PORT');
