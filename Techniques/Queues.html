Queues

Queues là một design pattern mạnh mẽ giúp bạn xử lý các thách thức phổ biến về application scaling và performance. Một số ví dụ về các vấn đề mà Queues có thể giúp bạn giải quyết:

  • Làm mượt các đỉnh xử lý (processing peak). Ví dụ, nếu người dùng có thể khởi tạo các task tốn tài nguyên vào thời điểm bất kỳ, bạn có thể add các task này vào queue thay vì thực hiện chúng đồng bộ. Sau đó, bạn có thể có các worker process lấy task từ queue theo cách có kiểm soát. Bạn có thể dễ dàng thêm Queue consumer mới để scale up việc xử lý backend khi ứng dụng mở rộng.
  • Chia nhỏ các monolithic task mà nếu không sẽ block Node.js event loop. Ví dụ, nếu một user request cần xử lý tốn CPU intensive như audio transcoding, bạn có thể delegate task này sang các process khác, giải phóng các user-facing process để duy trì tính responsive.
  • Cung cấp một kênh giao tiếp đáng tin cậy giữa các service khác nhau. Ví dụ, bạn có thể queue task (job) trong một process hoặc service, và consume chúng trong một process hoặc service khác. Bạn có thể được thông báo (bằng cách lắng nghe status event) khi completion, error hoặc các state change khác trong job life cycle từ bất kỳ process hoặc service nào. Khi Queue producer hoặc consumer gặp lỗi, state của chúng được bảo toàn và việc xử lý task có thể tự động restart khi các node được khởi động lại.

Nest cung cấp package @nestjs/bullmq cho BullMQ integration và @nestjs/bull cho Bull integration. Cả hai package đều là abstraction/wrapper trên top of các thư viện tương ứng, được phát triển bởi cùng một team. Bull hiện đang ở chế độ maintenance, trong khi BullMQ đang được phát triển tích cực, với một TypeScript implementation hiện đại và bộ tính năng khác biệt. Nếu Bull đáp ứng yêu cầu của bạn, nó vẫn là một lựa chọn đáng tin cậy và đã được thử nghiệm kỹ. Nest package giúp dễ dàng tích hợp cả BullMQ lẫn Bull Queues vào ứng dụng Nest của bạn một cách thân thiện.

Cả BullMQ và Bull đều sử dụng Redis để lưu trữ job data, vì vậy bạn cần cài đặt Redis trên hệ thống của mình. Vì chúng được hỗ trợ bởi Redis, kiến trúc Queue của bạn có thể được phân tán hoàn toàn và không phụ thuộc vào platform. Ví dụ, bạn có thể có một số Queue producer và consumer và listener chạy trong Nest trên một (hoặc nhiều) node, và các producer, consumer và listener khác chạy trên các Node.js platform khác trên các network node khác.

Chapter này bao gồm các package @nestjs/bullmq và @nestjs/bull. Chúng tôi cũng khuyến nghị đọc tài liệu BullMQ và Bull để biết thêm thông tin nền và chi tiết implementation cụ thể.

BullMQ installation

Để bắt đầu sử dụng BullMQ, chúng ta trước tiên cài đặt các dependency cần thiết.

bash
$ npm install --save @nestjs/bullmq bullmq

Sau khi quá trình cài đặt hoàn tất, chúng ta có thể import BullModule vào root AppModule.

app.module.ts
JS
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bullmq';

@Module({
  imports: [
    BullModule.forRoot({
      connection: {
        host: 'localhost',
        port: 6379,
      },
    }),
  ],
})
export class AppModule {}

Method forRoot() được sử dụng để đăng ký một bullmq package configuration object sẽ được sử dụng bởi tất cả queue đã đăng ký trong application (trừ khi được chỉ định khác). Để tham khảo, sau đây là một số properties trong configuration object:

  • connection: ConnectionOptions - Options để configure Redis connection. Xem Connections để biết thêm. Optional.
  • prefix: string - Prefix cho tất cả queue key. Optional.
  • defaultJobOptions: JobOpts - Options để control default settings cho new job. Xem JobOpts để biết thêm. Optional.
  • settings: AdvancedSettings - Advanced Queue configuration settings. Thông thường không nên thay đổi. Xem AdvancedSettings để biết thêm. Optional.
  • extraOptions - Extra options cho module init. Xem Manual Registration.

Tất cả options đều là optional, cung cấp quyền kiểm soát chi tiết hành vi của queue. Chúng được truyền trực tiếp cho BullMQ Queue constructor. Đọc thêm về các option này và các option khác tại đây.

Để đăng ký một queue, import BullModule.registerQueue() dynamic module, như sau:

BullModule.registerQueue({
  name: 'audio',
});
💡 Hint

Tạo nhiều queue bằng cách truyền nhiều comma-separated configuration object cho method registerQueue().

Method registerQueue() được sử dụng để instantiate và/hoặc đăng ký queue. Queue được chia sẻ giữa các module và process kết nối đến cùng Redis database cơ bản với cùng credential. Mỗi queue là duy nhất theo property name của nó. Tên queue được sử dụng vừa là injection token (để inject queue vào controller/provider), vừa là argument cho decorator để liên kết consumer class và listener với queue.

Bạn cũng có thể override một số pre-configured option cho một queue cụ thể, như sau:

BullModule.registerQueue({
  name: 'audio',
  connection: {
    port: 6380,
  },
});

BullMQ cũng hỗ trợ quan hệ parent - child giữa các job. Chức năng này cho phép tạo các flow trong đó job là node của tree có độ sâu tùy ý. Để đọc thêm về chúng, xem tại đây.

Để thêm một flow, bạn có thể làm như sau:

BullModule.registerFlowProducer({
  name: 'flowProducerName',
});

Vì job được lưu trong Redis, mỗi khi một specific named queue được instantiate (ví dụ: khi app được start/restart), nó sẽ cố gắng xử lý bất kỳ old job nào có thể tồn tại từ session trước chưa hoàn thành.

Mỗi queue có thể có một hoặc nhiều producer, consumer, và listener. Consumer lấy job từ queue theo một thứ tự cụ thể: FIFO (mặc định), LIFO, hoặc theo priority. Điều khiển thứ tự xử lý queue được thảo luận tại đây.

Named configurations #

Nếu queue của bạn kết nối đến nhiều Redis instance khác nhau, bạn có thể sử dụng kỹ thuật gọi là named configurations. Tính năng này cho phép bạn đăng ký nhiều configuration dưới các key được chỉ định, sau đó bạn có thể tham chiếu trong queue option.

Ví dụ, giả sử rằng bạn có một Redis instance bổ sung (ngoài instance mặc định) được sử dụng bởi một vài queue đã đăng ký trong application, bạn có thể đăng ký configuration của nó như sau:

BullModule.forRoot('alternative-config', {
  connection: {
    port: 6381,
  },
});

Trong ví dụ trên, 'alternative-config' chỉ là một configuration key (có thể là bất kỳ arbitrary string nào).

Với thiết lập này, bạn bây giờ có thể trỏ đến configuration này trong registerQueue() options object:

BullModule.registerQueue({
  configKey: 'alternative-config',
  name: 'video',
});

Producers #

Job producer add job vào queue. Producer thường là application service (Nest provider). Để add job vào queue, trước tiên inject queue vào service như sau:

import { Injectable } from '@nestjs/common';
import { Queue } from 'bullmq';
import { InjectQueue } from '@nestjs/bullmq';

@Injectable()
export class AudioService {
  constructor(@InjectQueue('audio') private audioQueue: Queue) {}
}
💡 Hint

Decorator @InjectQueue() xác định queue theo tên của nó, như đã cung cấp trong lệnh gọi method registerQueue() (ví dụ: 'audio').

Bây giờ, add một job bằng cách gọi method add() của queue, truyền một user-defined job object. Job được biểu diễn dưới dạng serializable JavaScript object (vì đó là cách chúng được lưu trong Redis database). Hình dạng của job bạn truyền là tùy ý; sử dụng nó để biểu diễn semantics của job object. Bạn cũng cần đặt tên cho nó. Điều này cho phép bạn tạo các consumer chuyên biệt chỉ xử lý job với tên đã cho.

const job = await this.audioQueue.add('transcode', {
  foo: 'bar',
});

Job options #

Job có thể có các option bổ sung đi kèm. Truyền một options object sau argument job trong method Queue.add(). Một số job options properties bao gồm:

  • priority: number - Optional priority value. Ranges from 1 (highest priority) to MAX_INT (lowest priority). Lưu ý rằng việc sử dụng priority có ảnh hưởng nhẹ đến performance, vì vậy hãy sử dụng chúng cẩn thận.
  • delay: number - Khoảng thời gian (millisecond) chờ trước khi job có thể được xử lý. Lưu ý rằng để delay chính xác, cả server và client nên có clock đồng bộ.
  • attempts: number - Tổng số lần thử để thực hiện job cho đến khi hoàn thành.
  • repeat: RepeatOpts - Lặp lại job theo cron specification. Xem RepeatOpts.
  • backoff: number | BackoffOpts - Backoff setting cho automatic retry khi job fail. Xem BackoffOpts.
  • lifo: boolean - Nếu true, add job vào right end của queue thay vì left (mặc định false).
  • jobId: number | string - Override job ID - mặc định, job ID là một unique integer, nhưng bạn có thể sử dụng setting này để override nó. Nếu sử dụng option này, bạn phải đảm bảo jobId duy nhất. Nếu bạn cố add job với id đã tồn tại, nó sẽ không được thêm.
  • removeOnComplete: boolean | number - Nếu true, xóa job khi nó hoàn thành thành công. Nếu là number, chỉ định số lượng job cần giữ. Default behavior là giữ job trong completed set.
  • removeOnFail: boolean | number - Nếu true, xóa job khi nó fail sau tất cả attempt. Nếu là number, chỉ định số lượng job cần giữ. Default behavior là giữ job trong failed set.
  • stackTraceLimit: number - Giới hạn số dòng stack trace sẽ được ghi lại trong stacktrace.

Dưới đây là một vài ví dụ về customizing job với job option.

Để delay start của job, sử dụng delay configuration property.

const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { delay: 3000 }, // 3 seconds delayed
);

Để add job vào right end của queue (xử lý job theo LIFO (Last In First Out)), set property lifo của configuration object thành true.

const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { lifo: true },
);

Để ưu tiên một job, sử dụng property priority.

const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { priority: 2 },
);

Để xem danh sách đầy đủ các option, hãy kiểm tra API documentation tại đây và tại đây.

Consumers #

Consumer là một class định nghĩa các method để xử lý job được add vào queue, hoặc lắng nghe event trên queue, hoặc cả hai. Khai báo một consumer class bằng decorator @Processor() như sau:

import { Processor } from '@nestjs/bullmq';

@Processor('audio')
export class AudioConsumer {}
💡 Hint

Consumer phải được đăng ký dưới dạng providers để package @nestjs/bullmq có thể pick chúng lên.

Trong đó decorator's string argument (ví dụ: 'audio') là tên của queue sẽ được liên kết với class method.

import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';

@Processor('audio')
export class AudioConsumer extends WorkerHost {
  async process(job: Job<any, any, string>): Promise<any> {
    let progress = 0;
    for (let i = 0; i < 100; i++) {
      await doSomething(job.data);
      progress += 1;
      await job.updateProgress(progress);
    }
    return {};
  }
}

Method process được gọi bất cứ khi nào worker ở trạng thái idle và có job cần xử lý trong queue. Handler method này nhận job object làm argument duy nhất. Giá trị được trả về bởi handler method được lưu trong job object và có thể được truy cập sau đó, ví dụ trong listener cho completed event.

Job object có nhiều method cho phép bạn tương tác với state của chúng. Ví dụ, code trên sử dụng method updateProgress() để cập nhật progress của job. Xem tại đây để biết complete Job object API reference.

Trong phiên bản cũ Bull, bạn có thể chỉ định rằng một job handler method sẽ chỉ xử lý only job của một type nhất định (job có name cụ thể) bằng cách truyền name đó cho @Process() decorator như bên dưới.

⚠️ Warning

Điều này không hoạt động với BullMQ, hãy đọc tiếp.

@Process('transcode')
async transcode(job: Job<unknown>) { ... }

Hành vi này không được hỗ trợ trong BullMQ do sự nhầm lẫn nó tạo ra. Thay vào đó, bạn cần switch case hoặc logic cho mỗi job name:

import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';

@Processor('audio')
export class AudioConsumer extends WorkerHost {
  async process(job: Job<any, any, string>): Promise<any> {
    switch (job.name) {
      case 'transcode': {
        let progress = 0;
        for (let i = 0; i < 100; i++) {
          await doSomething(job.data);
          progress += 1;
          await job.progress(progress);
        }
        return {};
      }
      case 'concatenate': {
        await doSomeLogic2();
        break;
      }
    }
  }
}

Điều này được trình bày trong phần named processor của tài liệu BullMQ.

Request-scoped consumers #

Khi một consumer được đánh dấu là request-scoped (tìm hiểu thêm về injection scope tại đây), một instance mới của class sẽ được tạo riêng cho mỗi job. Instance sẽ bị garbage-collected sau khi job hoàn thành.

@Processor({
  name: 'audio',
  scope: Scope.REQUEST,
})

Vì request-scoped consumer class được instantiate động và scoped đến một single job, bạn có thể inject JOB_REF thông qua constructor bằng cách sử dụng standard approach. 

constructor(@Inject(JOB_REF) jobRef: Job) {
  console.log(jobRef);
}
💡 Hint

Token JOB_REF được import từ package @nestjs/bullmq.

Event listeners #

BullMQ tạo ra một tập hợp các event hữu ích khi queue và/hoặc job state thay đổi. Các event này có thể được subscribe ở Worker level bằng decorator @OnWorkerEvent(event), hoặc ở Queue level với một dedicated listener class và decorator @OnQueueEvent(event).

Worker event phải được khai báo trong một consumer class (tức là, bên trong class decorated với @Processor() decorator). Để listen for một event, sử dụng decorator @OnWorkerEvent(event) với event bạn muốn xử lý. Ví dụ, để listen event khi job vào active state trong audio queue, sử dụng construct sau:

import { Processor, Process, OnWorkerEvent } from '@nestjs/bullmq';
import { Job } from 'bullmq';

@Processor('audio')
export class AudioConsumer {
  @OnWorkerEvent('active')
  onActive(job: Job) {
    console.log(
      `Processing job ${job.id} of type ${job.name} with data ${job.data}...`,
    );
  }

  // ...
}

Bạn có thể xem danh sách đầy đủ các event và argument của chúng dưới dạng property của WorkerListener tại đây.

QueueEvent listener phải sử dụng decorator @QueueEventsListener(queue) và extend class QueueEventsHost được cung cấp bởi @nestjs/bullmq. Để listen for event, sử dụng decorator @OnQueueEvent(event) với event bạn muốn xử lý. Ví dụ, để listen event khi job vào active state trong audio queue, sử dụng construct sau: 

import {
  QueueEventsHost,
  QueueEventsListener,
  OnQueueEvent,
} from '@nestjs/bullmq';

@QueueEventsListener('audio')
export class AudioEventsListener extends QueueEventsHost {
  @OnQueueEvent('active')
  onActive(job: { jobId: string; prev?: string }) {
    console.log(`Processing job ${job.jobId}...`);
  }

  // ...
}
💡 Hint

QueueEvent Listener phải được đăng ký dưới dạng providers để package @nestjs/bullmq có thể pick chúng lên.

Bạn có thể xem danh sách đầy đủ các event và argument của chúng dưới dạng property của QueueEventsListener tại đây.

Queue management #

Queue có một API cho phép bạn thực hiện các management function như pausing và resuming, lấy count of job ở các state khác nhau, và nhiều hơn nữa. Bạn có thể tìm full queue API tại đây. Invoke bất kỳ method nào trực tiếp trên Queue object, như hiển thị bên dưới với các ví dụ pause/resume.

Pause một queue với lệnh gọi method pause(). Một paused queue sẽ không xử lý new job cho đến khi được resumed, nhưng các job đang được xử lý sẽ tiếp tục cho đến khi chúng hoàn thành.

await audioQueue.pause();

Để resume một paused queue, sử dụng method resume(), như sau:

await audioQueue.resume();

Separate processes #

Job handler cũng có thể được chạy trong một separate (forked) process (source). Điều này có một số lợi thế:

  • Process được sandboxed nên nếu nó crash sẽ không ảnh hưởng đến worker.
  • Bạn có thể chạy blocking code mà không ảnh hưởng đến queue (job sẽ không bị stall).
  • Sử dụng tốt hơn multi-core CPU.
  • Ít connection đến redis hơn.
app.module.ts
JS
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bullmq';
import { join } from 'node:path';

@Module({
  imports: [
    BullModule.registerQueue({
      name: 'audio',
      processors: [join(__dirname, 'processor.js')],
    }),
  ],
})
export class AppModule {}
⚠️ Warning

Xin lưu ý rằng vì function của bạn đang được thực thi trong một forked process, Dependency Injection (và IoC container) sẽ không khả dụng. Điều đó có nghĩa là processor function của bạn sẽ cần chứa (hoặc tạo) tất cả instance của external dependency mà nó cần.

Async configuration #

Bạn có thể muốn truyền bullmq option một cách bất đồng bộ thay vì tĩnh. Trong trường hợp này, sử dụng method forRootAsync() cung cấp nhiều cách để xử lý async configuration. Tương tự, nếu bạn muốn truyền queue option một cách bất đồng bộ, sử dụng method registerQueueAsync().

Một cách tiếp cận là sử dụng factory function:

BullModule.forRootAsync({
  useFactory: () => ({
    connection: {
      host: 'localhost',
      port: 6379,
    },
  }),
});

Factory của chúng ta hoạt động giống như bất kỳ asynchronous provider nào khác (ví dụ: nó có thể là async và có khả năng inject dependency thông qua inject). 

BullModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    connection: {
      host: configService.get('QUEUE_HOST'),
      port: configService.get('QUEUE_PORT'),
    },
  }),
  inject: [ConfigService],
});

Ngoài ra, bạn có thể sử dụng cú pháp useClass:

BullModule.forRootAsync({
  useClass: BullConfigService,
});

Construction trên sẽ instantiate BullConfigService bên trong BullModule và sử dụng nó để cung cấp options object bằng cách gọi createSharedConfiguration(). Lưu ý rằng điều này có nghĩa là BullConfigService phải implement interface SharedBullConfigurationFactory, như hiển thị bên dưới: 

@Injectable()
class BullConfigService implements SharedBullConfigurationFactory {
  createSharedConfiguration(): BullModuleOptions {
    return {
      connection: {
        host: 'localhost',
        port: 6379,
      },
    };
  }
}

Để ngăn việc tạo BullConfigService bên trong BullModule và sử dụng một provider được import từ module khác, bạn có thể sử dụng cú pháp useExisting.

BullModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});

Construction này hoạt động giống useClass với một khác biệt quan trọng - BullModule sẽ tìm trong các imported module để tái sử dụng ConfigService hiện có thay vì instantiate một cái mới.

Tương tự, nếu bạn muốn truyền queue option bất đồng bộ, sử dụng method registerQueueAsync(), chỉ cần nhớ chỉ định attribute name bên ngoài factory function. 

BullModule.registerQueueAsync({
  name: 'audio',
  useFactory: () => ({
    redis: {
      host: 'localhost',
      port: 6379,
    },
  }),
});

Manual registration #

Mặc định, BullModule tự động đăng ký BullMQ component (queue, processor, và event listener service) trong lifecycle function onModuleInit. Tuy nhiên, trong một số trường hợp, hành vi này có thể không lý tưởng. Để ngăn automatic registration, enable manualRegistration trong BullModule như sau: 

BullModule.forRoot({
  extraOptions: {
    manualRegistration: true,
  },
});

Để đăng ký các component này theo cách thủ công, inject BullRegistrar và gọi function register, lý tưởng nhất trong OnModuleInit hoặc OnApplicationBootstrap. 

import { Injectable, OnModuleInit } from '@nestjs/common';
import { BullRegistrar } from '@nestjs/bullmq';

@Injectable()
export class AudioService implements OnModuleInit {
  constructor(private bullRegistrar: BullRegistrar) {}

  onModuleInit() {
    if (yourConditionHere) {
      this.bullRegistrar.register();
    }
  }
}

Trừ khi bạn gọi function BullRegistrar#register, không có BullMQ component nào sẽ hoạt động — nghĩa là sẽ không có job nào được xử lý.

Bull installation (Legacy)

Phần dưới đây mô tả cách sử dụng package @nestjs/bull (Bull - phiên bản legacy). Nếu bạn đang bắt đầu dự án mới, chúng tôi khuyến nghị sử dụng BullMQ (phần trên). Tuy nhiên, nếu dự án của bạn đang sử dụng Bull, các hướng dẫn bên dưới vẫn áp dụng.

Để bắt đầu sử dụng Bull, chúng ta cài đặt các dependency cần thiết:

bash
$ npm install --save @nestjs/bull bull
$ npm install --save-dev @types/bull

Sau khi cài đặt hoàn tất, import BullModule vào root AppModule:

app.module.ts
JS
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bull';

@Module({
  imports: [
    BullModule.forRoot({
      redis: {
        host: 'localhost',
        port: 6379,
      },
    }),
  ],
})
export class AppModule {}

Method forRoot() được sử dụng để đăng ký một bull package configuration object sẽ được sử dụng bởi tất cả queue đăng ký trong application (trừ khi chỉ định khác). Configuration object bao gồm các property sau:

  • limiter: RateLimiter - Options để control rate tại đó job của queue processor được xử lý. Xem RateLimiter để biết thêm. Optional.
  • redis: RedisOpts - Options để configure Redis connection. Xem RedisOpts để biết thêm. Optional.
  • prefix: string - Prefix cho tất cả queue key. Optional.
  • defaultJobOptions: JobOpts - Options để control default settings cho new job. Xem JobOpts để biết thêm. Optional.
  • settings: AdvancedSettings - Advanced Queue configuration settings. Thông thường không nên thay đổi. Xem AdvancedSettings để biết thêm. Optional.

Tất cả options đều optional, cung cấp quyền kiểm soát chi tiết hành vi queue. Chúng được truyền trực tiếp đến Bull Queue constructor. Đọc thêm tại đây.

Để đăng ký queue, import BullModule.registerQueue() dynamic module:

BullModule.registerQueue({
  name: 'audio',
});
💡 Hint

Tạo nhiều queue bằng cách truyền nhiều comma-separated configuration object cho method registerQueue().

Method registerQueue() được sử dụng để instantiate và/hoặc đăng ký queue. Queue được chia sẻ giữa các module và process kết nối đến cùng Redis database cơ bản với cùng credential. Mỗi queue là duy nhất theo property name.

Named configurations (Bull) #

Nếu queue của bạn kết nối đến nhiều Redis instance khác nhau, bạn có thể sử dụng kỹ thuật gọi là named configurations. Tính năng này cho phép đăng ký nhiều configuration dưới các key được chỉ định. 

BullModule.forRoot('alternative-config', {
  redis: {
    port: 6381,
  },
});

Sau đó bạn có thể trỏ đến configuration này trong registerQueue() options object:

BullModule.registerQueue({
  configKey: 'alternative-config',
  name: 'video',
});

Producers (Bull) #

Job producer add job vào queue. Để add job vào queue, inject queue vào service:

import { Injectable } from '@nestjs/common';
import { Queue } from 'bull';
import { InjectQueue } from '@nestjs/bull';

@Injectable()
export class AudioService {
  constructor(@InjectQueue('audio') private audioQueue: Queue) {}
}

Sau đó add job bằng method add():

const job = await this.audioQueue.add({
  foo: 'bar',
});

Named jobs #

Job có thể có tên duy nhất. Điều này cho phép bạn tạo các consumer chuyên biệt chỉ xử lý job với tên đã cho.

const job = await this.audioQueue.add('transcode', {
  foo: 'bar',
});
⚠️ Warning

Khi sử dụng named job, bạn phải tạo processor cho mỗi tên duy nhất được add vào queue, nếu không queue sẽ thông báo rằng bạn thiếu processor cho job đã cho. Xem phần Consumers để biết thêm.

Job options (Bull) #

Job có thể có các option bổ sung đi kèm. Truyền một options object sau argument job trong method Queue.add(). Một số job options properties bao gồm:

  • priority: number - Optional priority value. Ranges from 1 (highest priority) to MAX_INT (lowest priority). Lưu ý rằng việc sử dụng priority có ảnh hưởng nhẹ đến performance, vì vậy hãy sử dụng chúng cẩn thận.
  • delay: number - Khoảng thời gian (millisecond) chờ trước khi job có thể được xử lý. Lưu ý rằng để delay chính xác, cả server và client nên có clock đồng bộ.
  • attempts: number - Tổng số lần thử để thực hiện job cho đến khi hoàn thành.
  • repeat: RepeatOpts - Lặp lại job theo cron specification. Xem RepeatOpts.
  • backoff: number | BackoffOpts - Backoff setting cho automatic retry khi job fail. Xem BackoffOpts.
  • lifo: boolean - Nếu true, add job vào right end của queue thay vì left (mặc định false).
  • timeout: number - Số millisecond mà job này có thể chạy trước khi bị timeout và fail.
  • jobId: number | string - Override job ID - mặc định, job ID là một unique integer, nhưng bạn có thể sử dụng setting này để override nó. Nếu sử dụng option này, bạn phải đảm bảo jobId duy nhất. Nếu bạn cố add job với id đã tồn tại, nó sẽ không được thêm.
  • removeOnComplete: boolean | number - Nếu true, xóa job khi nó hoàn thành thành công. Nếu là number, chỉ định số lượng job cần giữ. Default behavior là giữ job trong completed set.
  • removeOnFail: boolean | number - Nếu true, xóa job khi nó fail sau tất cả attempt. Nếu là number, chỉ định số lượng job cần giữ. Default behavior là giữ job trong failed set.
  • stackTraceLimit: number - Giới hạn số dòng stack trace sẽ được ghi lại trong stacktrace.

Dưới đây là một vài ví dụ về customizing job với job option.

Để delay start của job, sử dụng delay configuration property.

const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { delay: 3000 }, // 3 seconds delayed
);

Để add job vào right end của queue (xử lý job theo LIFO (Last In First Out)), set property lifo của configuration object thành true.

const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { lifo: true },
);

Để ưu tiên một job, sử dụng property priority.

const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { priority: 2 },
);

Để xem danh sách đầy đủ các option, kiểm tra API documentation tại đây.

Consumers (Bull) #

Consumer là một class định nghĩa các method để xử lý job được add vào queue, hoặc lắng nghe event, hoặc cả hai. Khai báo consumer class bằng decorator @Processor():

import { Processor, Process } from '@nestjs/bull';

@Processor('audio')
export class AudioConsumer {
  @Process()
  async transcode(job: Job<unknown>) {
    let progress = 0;
    for (let i = 0; i < 100; i++) {
      await doSomething(job.data);
      progress += 1;
      await job.progress(progress);
    }
    return {};
  }
}
💡 Hint

Consumer phải được đăng ký dưới dạng providers để package @nestjs/bull có thể pick chúng lên.

Với named job, truyền tên cho @Process() decorator:

@Process('transcode')
async transcode(job: Job<unknown>) { ... }

Event listeners (Bull) #

Bull tạo một tập hợp các event hữu ích khi queue và/hoặc job state thay đổi. Nest cung cấp bộ decorator cho phép subscribe các core event set. Chúng được export từ @nestjs/bull package.

Event listener phải được khai báo trong consumer class (tức là, bên trong class được decorate với @Processor() decorator). Để listen for một event, sử dụng một trong các decorator dưới đây. Ví dụ, để listen event khi job vào active state trong audio queue, sử dụng construct sau: 

import { Processor, Process, OnQueueActive } from '@nestjs/bull';
import { Job } from 'bull';

@Processor('audio')
export class AudioConsumer {
  @OnQueueActive()
  onActive(job: Job) {
    console.log(
      `Processing job ${job.id} of type ${job.name} with data ${job.data}...`,
    );
  }

  // ...
}

Vì lắng nghe event là global-scoped (ví dụ: nếu một event xảy ra trên bất kỳ queue nào), hãy lưu ý rằng argument signature của handler method có thể khác so với local event handler tương ứng. Ví dụ:

Event Local event handler Global event handler
error @OnQueueError() @OnGlobalQueueError()
waiting @OnQueueWaiting() @OnGlobalQueueWaiting()
active @OnQueueActive() @OnGlobalQueueActive()
stalled @OnQueueStalled() @OnGlobalQueueStalled()
progress @OnQueueProgress() @OnGlobalQueueProgress()
completed @OnQueueCompleted() @OnGlobalQueueCompleted()
failed @OnQueueFailed() @OnGlobalQueueFailed()
paused @OnQueuePaused() @OnGlobalQueuePaused()
resumed @OnQueueResumed() @OnGlobalQueueResumed()
cleaned @OnQueueCleaned() @OnGlobalQueueCleaned()
drained @OnQueueDrained() @OnGlobalQueueDrained()
removed @OnQueueRemoved() @OnGlobalQueueRemoved()

Khi listening cho global event, method signature có thể hơi khác so với local counterpart. Cụ thể, bất kỳ method signature nào nhận job object trong local version sẽ nhận jobId (number) thay vì job (Job object) trong global version. Để lấy reference đến job thực tế trong trường hợp như vậy, sử dụng method Queue#getJob. Lệnh gọi này nên được await, và do đó handler nên được khai báo async. Ví dụ: 

@OnGlobalQueueCompleted()
async onGlobalCompleted(jobId: number, result: any) {
  const job = await this.immediateQueue.getJob(jobId);
  console.log('(Global) on completed: job ', job.id, ' -> result: ', result);
}
💡 Hint

Để truy cập Queue object (để thực hiện lệnh gọi getJob()), bạn phải inject nó. Ngoài ra, Queue phải được đăng ký trong module mà nó đang được inject vào.

Request-scoped consumers (Bull) #

Khi một consumer được đánh dấu là request-scoped (tìm hiểu thêm về injection scope tại đây), một instance mới của class sẽ được tạo riêng cho mỗi job. Instance sẽ bị garbage-collected sau khi job hoàn thành.

@Processor({
  name: 'audio',
  scope: Scope.REQUEST,
})

Vì request-scoped consumer class được instantiate động và scoped đến một single job, bạn có thể inject JOB_REF thông qua constructor bằng cách sử dụng standard approach. 

constructor(@Inject(JOB_REF) jobRef: Job) {
  console.log(jobRef);
}
💡 Hint

Token JOB_REF được import từ package @nestjs/bull.

Queue management (Bull) #

Queue có một API cho phép bạn thực hiện các management function như pausing và resuming, lấy count of job ở các state khác nhau, và nhiều hơn nữa. Bạn có thể tìm full queue API tại đây. Invoke bất kỳ method nào trực tiếp trên Queue object, như hiển thị bên dưới với các ví dụ pause/resume.

Pause một queue với lệnh gọi method pause(). Một paused queue sẽ không xử lý new job cho đến khi được resumed, nhưng các job đang được xử lý sẽ tiếp tục cho đến khi chúng hoàn thành.

await audioQueue.pause();

Để resume một paused queue, sử dụng method resume(), như sau:

await audioQueue.resume();

Separate processes (Bull) #

Job handler cũng có thể được chạy trong một separate (forked) process (source). Điều này có một số lợi thế:

  • Process được sandboxed nên nếu nó crash sẽ không ảnh hưởng đến worker.
  • Bạn có thể chạy blocking code mà không ảnh hưởng đến queue (job sẽ không bị stall).
  • Sử dụng tốt hơn multi-core CPU.
  • Ít connection đến Redis hơn.
app.module.ts
JS
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bull';
import { join } from 'path';

@Module({
  imports: [
    BullModule.registerQueue({
      name: 'audio',
      processors: [join(__dirname, 'processor.js')],
    }),
  ],
})
export class AppModule {}

Xin lưu ý rằng vì function của bạn đang được thực thi trong một forked process, Dependency Injection (và IoC container) sẽ không khả dụng. Điều đó có nghĩa là processor function của bạn sẽ cần chứa (hoặc tạo) tất cả instance của external dependency mà nó cần.

processor.ts
JS
import { Job, DoneCallback } from 'bull';

export default function (job: Job, cb: DoneCallback) {
  console.log(`[${process.pid}] ${JSON.stringify(job.data)}`);
  cb(null, 'It works');
}

Async configuration (Bull) #

Bạn có thể muốn truyền bull option một cách bất đồng bộ thay vì tĩnh. Trong trường hợp này, sử dụng method forRootAsync() cung cấp nhiều cách để xử lý async configuration. Tương tự, nếu bạn muốn truyền queue option một cách bất đồng bộ, sử dụng method registerQueueAsync().

Một cách tiếp cận là sử dụng factory function:

BullModule.forRootAsync({
  useFactory: () => ({
    redis: {
      host: 'localhost',
      port: 6379,
    },
  }),
});

Factory của chúng ta hoạt động giống như bất kỳ asynchronous provider nào khác (ví dụ: nó có thể là async và có khả năng inject dependency thông qua inject). 

BullModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    redis: {
      host: configService.get('QUEUE_HOST'),
      port: configService.get('QUEUE_PORT'),
    },
  }),
  inject: [ConfigService],
});

Ngoài ra, bạn có thể sử dụng cú pháp useClass:

BullModule.forRootAsync({
  useClass: BullConfigService,
});

Construction trên sẽ instantiate BullConfigService bên trong BullModule và sử dụng nó để cung cấp options object bằng cách gọi createSharedConfiguration(). Lưu ý rằng điều này có nghĩa là BullConfigService phải implement interface SharedBullConfigurationFactory, như hiển thị bên dưới: 

@Injectable()
class BullConfigService implements SharedBullConfigurationFactory {
  createSharedConfiguration(): BullModuleOptions {
    return {
      redis: {
        host: 'localhost',
        port: 6379,
      },
    };
  }
}

Để ngăn việc tạo BullConfigService bên trong BullModule và sử dụng một provider được import từ module khác, bạn có thể sử dụng cú pháp useExisting.

BullModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});

Construction này hoạt động giống useClass với một khác biệt quan trọng - BullModule sẽ tìm trong các imported module để tái sử dụng ConfigService hiện có thay vì instantiate một cái mới.
