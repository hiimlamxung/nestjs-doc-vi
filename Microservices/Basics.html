Overview

Ngoài kiến trúc ứng dụng truyền thống (đôi khi gọi là monolithic), Nest hỗ trợ nguyên bản phong cách phát triển kiến trúc microservice. Hầu hết các khái niệm được thảo luận ở những phần khác trong tài liệu này, chẳng hạn như dependency injection, decorators, exception filters, pipes, guards và interceptors, đều áp dụng như nhau cho microservices. Bất cứ khi nào có thể, Nest trừu tượng hóa chi tiết triển khai để các component giống nhau có thể chạy trên các nền tảng dựa trên HTTP, WebSockets và Microservices. Phần này đề cập đến các khía cạnh của Nest dành riêng cho microservices.

Trong Nest, một microservice về cơ bản là một ứng dụng sử dụng transport layer khác với HTTP.

Nest hỗ trợ nhiều triển khai transport layer tích hợp sẵn, được gọi là transporters, chịu trách nhiệm truyền tải messages giữa các microservice instances khác nhau. Hầu hết các transporters đều hỗ trợ nguyên bản cả hai kiểu message request-response và event-based. Nest trừu tượng hóa chi tiết triển khai của mỗi transporter đằng sau một interface chuẩn cho cả messaging request-response và event-based. Điều này giúp dễ dàng chuyển đổi từ transport layer này sang transport layer khác -- ví dụ để tận dụng các tính năng reliability hoặc performance cụ thể của một transport layer nhất định -- mà không ảnh hưởng đến code ứng dụng của bạn.

Installation #

Để bắt đầu xây dựng microservices, trước tiên hãy cài đặt package cần thiết:

$ npm i --save @nestjs/microservices

Getting started #

Để khởi tạo một microservice, sử dụng phương thức createMicroservice() của class NestFactory:

main
import { NestFactory } from '@nestjs/core';
import { Transport, MicroserviceOptions } from '@nestjs/microservices';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.createMicroservice<MicroserviceOptions>(
    AppModule,
    {
      transport: Transport.TCP,
    },
  );
  await app.listen();
}
bootstrap();

import { Transport } from '@nestjs/microservices';
import { AppModule } from './app.module';

  const app = await NestFactory.createMicroservice(AppModule, {
    transport: Transport.TCP,
  });
  await app.listen();
}
bootstrap();
Gợi ý

Microservices sử dụng TCP transport layer theo mặc định.

Tham số thứ hai của phương thức createMicroservice() là một object options. Object này có thể bao gồm hai thành viên:

transportChỉ định transporter (ví dụ: Transport.NATS)
optionsMột object options dành riêng cho transporter để xác định hành vi của transporter

Object options dành riêng cho transporter đã chọn. Transporter TCP cung cấp các thuộc tính được mô tả bên dưới. Đối với các transporters khác (ví dụ: Redis, MQTT, v.v.), hãy xem chương tương ứng để biết mô tả các options có sẵn.

hostConnection hostname
portConnection port
retryAttemptsSố lần retry message (mặc định: 0)
retryDelayDelay giữa các lần retry message (ms) (mặc định: 0)
serializerCustom serializer cho outgoing messages
deserializerCustom deserializer cho incoming messages
socketClassMột custom Socket kế thừa TcpSocket (mặc định: JsonSocket)
tlsOptionsCác options để cấu hình giao thức tls

Các thuộc tính trên dành riêng cho TCP transporter. Để biết thông tin về các options có sẵn cho các transporters khác, hãy tham khảo chương tương ứng.

Message and Event Patterns #

Microservices nhận diện cả messages và events theo patterns. Một pattern là một giá trị đơn giản, ví dụ, một literal object hoặc một string. Patterns được tự động serialize và gửi qua mạng cùng với phần data của message. Bằng cách này, message senders và consumers có thể phối hợp xác định requests nào được xử lý bởi handlers nào.

Request-response #

Kiểu message request-response hữu ích khi bạn cần trao đổi messages giữa các external services khác nhau. Mô hình này đảm bảo rằng service đã thực sự nhận được message (mà không yêu cầu bạn phải tự triển khai một giao thức acknowledgment). Tuy nhiên, cách tiếp cận request-response không phải lúc nào cũng phù hợp nhất. Ví dụ, các streaming transporters như Kafka hoặc NATS streaming, sử dụng log-based persistence, được tối ưu hóa để giải quyết một tập hợp các thách thức khác, phù hợp hơn với mô hình event messaging (xem event-based messaging để biết thêm chi tiết).

Để kích hoạt kiểu message request-response, Nest tạo hai logical channels: một để truyền data và một để chờ incoming responses. Đối với một số underlying transports, như NATS, hỗ trợ dual-channel này được cung cấp sẵn. Đối với những transports khác, Nest bù đắp bằng cách tạo thủ công các channels riêng biệt. Mặc dù điều này hiệu quả, nhưng có thể tạo ra một số overhead. Do đó, nếu bạn không cần kiểu message request-response, bạn có thể muốn cân nhắc sử dụng phương thức event-based.

Để tạo một message handler dựa trên mô hình request-response, sử dụng decorator @MessagePattern(), được import từ package @nestjs/microservices. Decorator này chỉ nên được sử dụng trong các controller classes, vì chúng đóng vai trò là entry points cho ứng dụng của bạn. Sử dụng nó trong providers sẽ không có tác dụng, vì chúng sẽ bị Nest runtime bỏ qua.

math.controller
import { Controller } from '@nestjs/common';
import { MessagePattern } from '@nestjs/microservices';

@Controller()
export class MathController {
  @MessagePattern({ cmd: 'sum' })
  accumulate(data: number[]): number {
    return (data || []).reduce((a, b) => a + b);
  }
}
import { Controller } from '@nestjs/common';
import { MessagePattern } from '@nestjs/microservices';

@Controller()
export class MathController {
  @MessagePattern({ cmd: 'sum' })
  accumulate(data) {
    return (data || []).reduce((a, b) => a + b);
  }
}

Trong đoạn code trên, message handler accumulate() lắng nghe các messages khớp với message pattern {{ '{' }} cmd: 'sum' {{ '}' }}. Message handler nhận một tham số duy nhất, data được truyền từ client. Trong trường hợp này, data là một mảng các số cần được tính tổng.

Asynchronous responses #

Message handlers có thể phản hồi đồng bộ hoặc bất đồng bộ, nghĩa là các phương thức async được hỗ trợ.

@MessagePattern({ cmd: 'sum' })
async accumulate(data: number[]): Promise<number> {
  return (data || []).reduce((a, b) => a + b);
}
@MessagePattern({ cmd: 'sum' })
async accumulate(data) {
  return (data || []).reduce((a, b) => a + b);
}

Một message handler cũng có thể trả về một Observable, trong trường hợp đó các giá trị kết quả sẽ được emit cho đến khi stream hoàn thành.

@MessagePattern({ cmd: 'sum' })
accumulate(data: number[]): Observable<number> {
  return from([1, 2, 3]);
}
@MessagePattern({ cmd: 'sum' })
accumulate(data: number[]): Observable<number> {
  return from([1, 2, 3]);
}

Trong ví dụ trên, message handler sẽ phản hồi ba lần, mỗi lần cho một item trong mảng.

Event-based #

Mặc dù phương thức request-response hoàn hảo cho việc trao đổi messages giữa các services, nhưng nó ít phù hợp hơn cho event-based messaging -- khi bạn chỉ muốn publish events mà không cần chờ response. Trong những trường hợp như vậy, overhead của việc duy trì hai channels cho request-response là không cần thiết.

Ví dụ, nếu bạn muốn thông báo cho một service khác rằng một điều kiện cụ thể đã xảy ra trong phần này của hệ thống, kiểu message event-based là lý tưởng.

Để tạo một event handler, sử dụng decorator @EventPattern(), được import từ package @nestjs/microservices.

@EventPattern('user_created')
async handleUserCreated(data: Record<string, unknown>) {
  // business logic
}
@EventPattern('user_created')
async handleUserCreated(data) {
  // business logic
}
Gợi ý

Bạn có thể đăng ký nhiều event handlers cho một event pattern duy nhất, và tất cả chúng sẽ được tự động kích hoạt song song.

Event handler handleUserCreated() lắng nghe event 'user_created'. Event handler nhận một tham số duy nhất, data được truyền từ client (trong trường hợp này, một event payload đã được gửi qua mạng).

Additional request details #

Trong các kịch bản nâng cao hơn, bạn có thể cần truy cập thêm chi tiết về incoming request. Ví dụ, khi sử dụng NATS với wildcard subscriptions, bạn có thể muốn lấy subject gốc mà producer đã gửi message tới. Tương tự, với Kafka, bạn có thể cần truy cập message headers. Để đạt được điều này, bạn có thể sử dụng các built-in decorators như bên dưới:

@MessagePattern('time.us.*')
getDate(@Payload() data: number[], @Ctx() context: NatsContext) {
  console.log(`Subject: ${context.getSubject()}`); // e.g. "time.us.east"
  return new Date().toLocaleTimeString(...);
}
@Bind(Payload(), Ctx())
@MessagePattern('time.us.*')
getDate(data, context) {
  console.log(`Subject: ${context.getSubject()}`); // e.g. "time.us.east"
  return new Date().toLocaleTimeString(...);
}
Gợi ý

@Payload(), @Ctx() và NatsContext được import từ @nestjs/microservices.

Gợi ý

Bạn cũng có thể truyền một property key vào decorator @Payload() để trích xuất một thuộc tính cụ thể từ incoming payload object, ví dụ, @Payload('id').

Client (producer class) #

Một ứng dụng Nest client có thể trao đổi messages hoặc publish events tới một Nest microservice bằng class ClientProxy. Class này cung cấp nhiều phương thức, như send() (cho request-response messaging) và emit() (cho event-driven messaging), cho phép giao tiếp với một remote microservice. Bạn có thể lấy một instance của class này theo những cách sau:

Một cách tiếp cận là import ClientsModule, cung cấp phương thức tĩnh register(). Phương thức này nhận một mảng các objects đại diện cho microservice transporters. Mỗi object phải bao gồm thuộc tính name, và tùy chọn thuộc tính transport (mặc định là Transport.TCP), cũng như thuộc tính options tùy chọn.

Thuộc tính name hoạt động như một injection token, bạn có thể sử dụng để inject một instance của ClientProxy ở bất cứ đâu cần thiết. Giá trị của thuộc tính name có thể là bất kỳ string hoặc JavaScript symbol tùy ý nào, như được mô tả trong tài liệu custom providers.

Thuộc tính options là một object bao gồm các thuộc tính giống như những gì chúng ta đã thấy trong phương thức createMicroservice() trước đó.

@Module({
  imports: [
    ClientsModule.register([
      { name: 'MATH_SERVICE', transport: Transport.TCP },
    ]),
  ],
})

Ngoài ra, bạn có thể sử dụng phương thức registerAsync() nếu bạn cần cung cấp cấu hình hoặc thực hiện bất kỳ quy trình bất đồng bộ nào trong quá trình thiết lập.

@Module({
  imports: [
    ClientsModule.registerAsync([
      {
        imports: [ConfigModule],
        name: 'MATH_SERVICE',
        useFactory: async (configService: ConfigService) => ({
          transport: Transport.TCP,
          options: {
            url: configService.get('URL'),
          },
        }),
        inject: [ConfigService],
      },
    ]),
  ],
})

Sau khi module đã được import, bạn có thể inject một instance của ClientProxy được cấu hình với các options đã chỉ định cho transporter 'MATH_SERVICE' bằng decorator @Inject().

constructor(
  @Inject('MATH_SERVICE') private client: ClientProxy,
) {}
Gợi ý

Các class ClientsModule và ClientProxy được import từ package @nestjs/microservices.

Đôi khi, bạn có thể cần lấy cấu hình transporter từ một service khác (chẳng hạn như ConfigService), thay vì hard-code nó trong ứng dụng client. Để đạt được điều này, bạn có thể đăng ký một custom provider sử dụng class ClientProxyFactory. Class này cung cấp phương thức tĩnh create() nhận một transporter options object và trả về một instance ClientProxy tùy chỉnh.

@Module({
  providers: [
    {
      provide: 'MATH_SERVICE',
      useFactory: (configService: ConfigService) => {
        const mathSvcOptions = configService.getMathSvcOptions();
        return ClientProxyFactory.create(mathSvcOptions);
      },
      inject: [ConfigService],
    }
  ]
  ...
})
Gợi ý

ClientProxyFactory được import từ package @nestjs/microservices.

Một tùy chọn khác là sử dụng property decorator @Client().

@Client({ transport: Transport.TCP })
client: ClientProxy;
Gợi ý

Decorator @Client() được import từ package @nestjs/microservices.

Sử dụng decorator @Client() không phải là kỹ thuật được ưu tiên, vì nó khó test hơn và khó chia sẻ một client instance.

ClientProxy là lazy. Nó không khởi tạo kết nối ngay lập tức. Thay vào đó, nó sẽ được thiết lập trước lần gọi microservice đầu tiên, và sau đó được tái sử dụng qua mỗi lần gọi tiếp theo. Tuy nhiên, nếu bạn muốn trì hoãn quá trình bootstrapping ứng dụng cho đến khi kết nối được thiết lập, bạn có thể khởi tạo kết nối thủ công bằng phương thức connect() của object ClientProxy bên trong lifecycle hook OnApplicationBootstrap.

async onApplicationBootstrap() {
  await this.client.connect();
}

Nếu kết nối không thể được tạo, phương thức connect() sẽ reject với error object tương ứng.

Sending messages #

ClientProxy cung cấp phương thức send(). Phương thức này được dùng để gọi microservice và trả về một Observable với response của nó. Do đó, chúng ta có thể subscribe vào các giá trị được emit một cách dễ dàng.

accumulate(): Observable<number> {
  const pattern = { cmd: 'sum' };
  const payload = [1, 2, 3];
  return this.client.send<number>(pattern, payload);
}
accumulate() {
  const pattern = { cmd: 'sum' };
  const payload = [1, 2, 3];
  return this.client.send(pattern, payload);
}

Phương thức send() nhận hai tham số, pattern và payload. pattern phải khớp với một pattern đã được định nghĩa trong decorator @MessagePattern(). payload là một message mà chúng ta muốn truyền tới remote microservice. Phương thức này trả về một cold Observable, nghĩa là bạn phải explicitly subscribe vào nó trước khi message được gửi.

Publishing events #

Để gửi một event, sử dụng phương thức emit() của object ClientProxy. Phương thức này publish một event tới message broker.

async publish() {
  this.client.emit<number>('user_created', new UserCreatedEvent());
}
async publish() {
  this.client.emit('user_created', new UserCreatedEvent());
}

Phương thức emit() nhận hai tham số: pattern và payload. pattern phải khớp với một pattern đã được định nghĩa trong decorator @EventPattern(), trong khi payload đại diện cho event data mà bạn muốn truyền tới remote microservice. Phương thức này trả về một hot Observable (trái ngược với cold Observable được trả về bởi send()), nghĩa là bất kể bạn có explicitly subscribe vào observable hay không, proxy sẽ ngay lập tức cố gắng deliver event.

Request-scoping #

Đối với những người đến từ các ngôn ngữ lập trình khác, có thể ngạc nhiên khi biết rằng trong Nest, hầu hết mọi thứ được chia sẻ giữa các incoming requests. Điều này bao gồm connection pool tới database, singleton services với global state, và nhiều hơn nữa. Hãy nhớ rằng Node.js không tuân theo mô hình request/response multi-threaded stateless, trong đó mỗi request được xử lý bởi một thread riêng biệt. Do đó, việc sử dụng singleton instances là an toàn cho các ứng dụng của chúng ta.

Tuy nhiên, có những trường hợp đặc biệt khi lifetime dựa trên request cho handler có thể là mong muốn. Điều này có thể bao gồm các kịch bản như per-request caching trong các ứng dụng GraphQL, request tracking, hoặc multi-tenancy. Bạn có thể tìm hiểu thêm về cách kiểm soát scopes trong tài liệu injection scopes.

Request-scoped handlers và providers có thể inject RequestContext bằng decorator @Inject() kết hợp với token CONTEXT:

import { Injectable, Scope, Inject } from '@nestjs/common';
import { CONTEXT, RequestContext } from '@nestjs/microservices';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(CONTEXT) private ctx: RequestContext) {}
}

Điều này cung cấp quyền truy cập vào object RequestContext, có hai thuộc tính:

export interface RequestContext<T = any> {
  pattern: string | Record<string, any>;
  data: T;
}

Thuộc tính data là message payload được gửi bởi message producer. Thuộc tính pattern là pattern được sử dụng để xác định handler phù hợp để xử lý incoming message.

Instance status updates #

Để nhận cập nhật theo thời gian thực về connection và trạng thái của underlying driver instance, bạn có thể subscribe vào status stream. Stream này cung cấp các status updates dành riêng cho driver đã chọn. Ví dụ, nếu bạn đang sử dụng TCP transporter (mặc định), status stream phát ra các event connected và disconnected.

this.client.status.subscribe((status: TcpStatus) => {
  console.log(status);
});
Gợi ý

Type TcpStatus được import từ package @nestjs/microservices.

Tương tự, bạn có thể subscribe vào status stream của server để nhận thông báo về trạng thái server.

const server = app.connectMicroservice<MicroserviceOptions>(...);
server.status.subscribe((status: TcpStatus) => {
  console.log(status);
});

Listening to internal events #

Trong một số trường hợp, bạn có thể muốn lắng nghe các internal events được phát ra bởi microservice. Ví dụ, bạn có thể lắng nghe event error để kích hoạt các thao tác bổ sung khi có lỗi xảy ra. Để làm điều này, sử dụng phương thức on(), như bên dưới:

this.client.on('error', (err) => {
  console.error(err);
});

Tương tự, bạn có thể lắng nghe các internal events của server:

server.on<TcpEvents>('error', (err) => {
  console.error(err);
});
Gợi ý

Type TcpEvents được import từ package @nestjs/microservices.

Underlying driver access #

Đối với các use cases nâng cao hơn, bạn có thể cần truy cập underlying driver instance. Điều này có thể hữu ích cho các kịch bản như đóng kết nối thủ công hoặc sử dụng các phương thức dành riêng cho driver. Tuy nhiên, hãy lưu ý rằng trong hầu hết trường hợp, bạn không cần truy cập driver trực tiếp.

Để làm vậy, bạn có thể sử dụng phương thức unwrap(), trả về underlying driver instance. Generic type parameter nên chỉ định type của driver instance mà bạn mong đợi.

const netServer = this.client.unwrap<Server>();

Ở đây, Server là một type được import từ module net.

Tương tự, bạn có thể truy cập underlying driver instance của server:

const netServer = server.unwrap<Server>();

Handling timeouts #

Trong các hệ thống phân tán, microservices đôi khi có thể bị down hoặc không khả dụng. Để tránh chờ đợi vô thời hạn, bạn có thể sử dụng timeouts. Timeout là một pattern rất hữu ích khi giao tiếp với các services khác. Để áp dụng timeouts cho các microservice calls của bạn, bạn có thể sử dụng RxJS timeout operator. Nếu microservice không phản hồi trong thời gian được chỉ định, một exception sẽ được throw, mà bạn có thể catch và xử lý phù hợp.

Để triển khai điều này, bạn cần sử dụng package rxjs. Chỉ cần sử dụng timeout operator trong pipe:

this.client
  .send<TResult, TInput>(pattern, data)
  .pipe(timeout(5000));
this.client
  .send(pattern, data)
  .pipe(timeout(5000));
Gợi ý

timeout operator được import từ package rxjs/operators.

Sau 5 giây, nếu microservice không phản hồi, nó sẽ throw một error.

TLS support #

Khi giao tiếp bên ngoài private network, điều quan trọng là phải encrypt traffic để đảm bảo an toàn. Trong NestJS, điều này có thể đạt được với TLS over TCP sử dụng module TLS tích hợp sẵn của Node. Nest cung cấp hỗ trợ TLS tích hợp trong TCP transport, cho phép chúng ta encrypt giao tiếp giữa các microservices hoặc clients.

Để kích hoạt TLS cho một TCP server, bạn cần cả private key và certificate ở định dạng PEM. Chúng được thêm vào server options bằng cách thiết lập tlsOptions và chỉ định các file key và cert, như bên dưới:

import * as fs from 'fs';
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';
import { MicroserviceOptions, Transport } from '@nestjs/microservices';

async function bootstrap() {
  const key = fs.readFileSync('<pathToKeyFile>', 'utf8').toString();
  const cert = fs.readFileSync('<pathToCertFile>', 'utf8').toString();

  const app = await NestFactory.createMicroservice<MicroserviceOptions>(
    AppModule,
    {
      transport: Transport.TCP,
      options: {
        tlsOptions: {
          key,
          cert,
        },
      },
    },
  );

  await app.listen();
}
bootstrap();

Để client giao tiếp an toàn qua TLS, chúng ta cũng định nghĩa object tlsOptions nhưng lần này với CA certificate. Đây là certificate của authority đã ký certificate của server. Điều này đảm bảo rằng client tin tưởng certificate của server và có thể thiết lập kết nối an toàn.

import { Module } from '@nestjs/common';
import { ClientsModule, Transport } from '@nestjs/microservices';

@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'MATH_SERVICE',
        transport: Transport.TCP,
        options: {
          tlsOptions: {
            ca: [fs.readFileSync('<pathToCaFile>', 'utf-8').toString()],
          },
        },
      },
    ]),
  ],
})
export class AppModule {}

Bạn cũng có thể truyền một mảng CAs nếu thiết lập của bạn sử dụng nhiều trusted authorities.

Khi mọi thứ đã được thiết lập, bạn có thể inject ClientProxy như bình thường bằng decorator @Inject() để sử dụng client trong services của bạn. Điều này đảm bảo giao tiếp được encrypt trên các NestJS microservices, với module TLS của Node xử lý các chi tiết encryption.

Để biết thêm thông tin, tham khảo tài liệu TLS của Node.

Dynamic configuration #

Khi một microservice cần được cấu hình sử dụng ConfigService (từ package @nestjs/config), nhưng injection context chỉ khả dụng sau khi microservice instance được tạo, AsyncMicroserviceOptions cung cấp một giải pháp. Cách tiếp cận này cho phép cấu hình động, đảm bảo tích hợp mượt mà với ConfigService.

import { ConfigService } from '@nestjs/config';
import { AsyncMicroserviceOptions, Transport } from '@nestjs/microservices';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.createMicroservice<AsyncMicroserviceOptions>(
    AppModule,
    {
      useFactory: (configService: ConfigService) => ({
        transport: Transport.TCP,
        options: {
          host: configService.get<string>('HOST'),
          port: configService.get<number>('PORT'),
        },
      }),
      inject: [ConfigService],
    },
  );

  await app.listen();
}
bootstrap();
