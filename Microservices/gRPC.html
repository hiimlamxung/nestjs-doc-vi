gRPC

gRPC là một framework RPC hiệu suất cao, mã nguồn mở, hiện đại, có thể chạy trong bất kỳ môi trường nào. Nó có thể kết nối hiệu quả các service trong và giữa các data center với hỗ trợ có thể cắm thêm (pluggable) cho load balancing, tracing, health checking và authentication.

Giống như nhiều hệ thống RPC khác, gRPC dựa trên khái niệm định nghĩa một service bằng các function (method) có thể được gọi từ xa. Với mỗi method, bạn định nghĩa các tham số và kiểu trả về. Các service, tham số và kiểu trả về được định nghĩa trong các file .proto sử dụng cơ chế protocol buffers mã nguồn mở, trung lập ngôn ngữ của Google.

Với gRPC transporter, Nest sử dụng các file .proto để liên kết động các client và server nhằm dễ dàng triển khai remote procedure call, tự động serialize và deserialize dữ liệu có cấu trúc.

Installation #

Để bắt đầu xây dựng microservice dựa trên gRPC, trước tiên hãy cài đặt các package cần thiết:

$ npm i --save @grpc/grpc-js @grpc/proto-loader

Overview #

Giống như các triển khai transport layer microservice khác của Nest, bạn chọn cơ chế gRPC transporter bằng thuộc tính transport của object options được truyền vào phương thức createMicroservice(). Trong ví dụ sau, chúng ta sẽ thiết lập một hero service. Thuộc tính options cung cấp metadata về service đó; các thuộc tính của nó được mô tả bên dưới.

main
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.GRPC,
  options: {
    package: 'hero',
    protoPath: join(__dirname, 'hero/hero.proto'),
  },
});
const app = await NestFactory.createMicroservice(AppModule, {
  transport: Transport.GRPC,
  options: {
    package: 'hero',
    protoPath: join(__dirname, 'hero/hero.proto'),
  },
});
Gợi ý

Hàm join() được import từ package path; enum Transport được import từ package @nestjs/microservices.

Trong file nest-cli.json, chúng ta thêm thuộc tính assets cho phép phân phối các file không phải TypeScript, và watchAssets để bật theo dõi tất cả các asset không phải TypeScript. Trong trường hợp này, chúng ta muốn các file .proto được tự động sao chép vào thư mục dist.

nest-cli.json

    "assets": ["**/*.proto"],
    "watchAssets": true
  }
}

Options #

Object options của gRPC transporter hiển thị các thuộc tính được mô tả bên dưới.

packageTên Protobuf package (khớp với thiết lập package trong file .proto). Bắt buộc
protoPathĐường dẫn tuyệt đối (hoặc tương đối với thư mục root) tới file .proto. Bắt buộc
urlURL kết nối. Chuỗi có định dạng ip address/dns name:port (ví dụ: '0.0.0.0:50051' cho Docker server) xác định địa chỉ/cổng mà transporter thiết lập kết nối. Tùy chọn. Mặc định là 'localhost:5000'
protoLoaderTên NPM package cho tiện ích tải file .proto. Tùy chọn. Mặc định là '@grpc/proto-loader'
loaderCác tùy chọn của @grpc/proto-loader. Cung cấp quyền kiểm soát chi tiết hành vi của file .proto. Tùy chọn. Xem tại đây để biết thêm chi tiết
credentialsServer credentials. Tùy chọn. Đọc thêm tại đây

Sample gRPC service #

Hãy định nghĩa service gRPC mẫu có tên HeroesService. Trong object options ở trên, thuộc tính protoPath thiết lập đường dẫn tới file định nghĩa .proto hero.proto. File hero.proto được cấu trúc sử dụng protocol buffers. Đây là nội dung của nó:

hero/hero.proto
// hero/hero.proto
syntax = "proto3";

package hero;

service HeroesService {
  rpc FindOne (HeroById) returns (Hero) {}
}

message HeroById {
  int32 id = 1;
}

message Hero {
  int32 id = 1;
  string name = 2;
}

HeroesService của chúng ta hiển thị một phương thức FindOne(). Phương thức này nhận đối số đầu vào kiểu HeroById và trả về message Hero (protocol buffers sử dụng phần tử message để định nghĩa cả kiểu tham số và kiểu trả về).

Tiếp theo, chúng ta cần triển khai service. Để định nghĩa một handler thực hiện định nghĩa này, chúng ta sử dụng decorator @GrpcMethod() trong controller, như được hiển thị bên dưới. Decorator này cung cấp metadata cần thiết để khai báo một method là gRPC service method.

Gợi ý

Decorator @MessagePattern() (đọc thêm) được giới thiệu trong các chương microservice trước không được sử dụng với microservice dựa trên gRPC. Decorator @GrpcMethod() thay thế hiệu quả cho microservice dựa trên gRPC.

heroes.controller
@Controller()
export class HeroesController {
  @GrpcMethod('HeroesService', 'FindOne')
  findOne(data: HeroById, metadata: Metadata, call: ServerUnaryCall<any, any>): Hero {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}
@Controller()
export class HeroesController {
  @GrpcMethod('HeroesService', 'FindOne')
  findOne(data, metadata, call) {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}
Gợi ý

Decorator @GrpcMethod() được import từ package @nestjs/microservices, trong khi Metadata và ServerUnaryCall từ package grpc.

Decorator được hiển thị ở trên nhận hai đối số. Đối số đầu tiên là tên service (ví dụ: 'HeroesService'), tương ứng với định nghĩa service HeroesService trong hero.proto. Đối số thứ hai (chuỗi 'FindOne') tương ứng với phương thức rpc FindOne() được định nghĩa trong HeroesService trong file hero.proto.

Phương thức handler findOne() nhận ba đối số: data được truyền từ caller, metadata lưu trữ gRPC request metadata và call để lấy các thuộc tính object GrpcCall như sendMetadata để gửi metadata tới client.

Cả hai đối số của decorator @GrpcMethod() đều là tùy chọn. Nếu gọi không có đối số thứ hai (ví dụ: 'FindOne'), Nest sẽ tự động liên kết phương thức rpc trong file .proto với handler dựa trên việc chuyển đổi tên handler sang upper camel case (ví dụ: handler findOne được liên kết với định nghĩa rpc call FindOne). Điều này được hiển thị bên dưới.

Ví dụ, trong đoạn code sau, class HeroesService liên kết các phương thức handler của nó với định nghĩa service HeroesService trong file hero.proto dựa trên khớp tên 'HeroesService'.

Client #

Ứng dụng Nest có thể hoạt động như gRPC client, sử dụng các service được định nghĩa trong file .proto. Bạn truy cập các remote service thông qua object ClientGrpc. Bạn có thể lấy object ClientGrpc theo nhiều cách.

Kỹ thuật được ưu tiên là import ClientsModule. Sử dụng phương thức register() để bind một package service được định nghĩa trong file .proto với một injection token, và để cấu hình service. Thuộc tính name là injection token. Đối với gRPC service, sử dụng transport: Transport.GRPC. Thuộc tính options là object có cùng các thuộc tính đã mô tả ở trên.

imports: [
  ClientsModule.register([
    {
      name: 'HERO_PACKAGE',
      transport: Transport.GRPC,
      options: {
        package: 'hero',
        protoPath: join(__dirname, 'hero/hero.proto'),
      },
    },
  ]),
];
Gợi ý

Phương thức register() nhận một mảng các object. Đăng ký nhiều package bằng cách cung cấp một danh sách các object đăng ký phân tách bằng dấu phẩy.

Sau khi đăng ký, chúng ta có thể inject object ClientGrpc đã cấu hình với @Inject(). Sau đó chúng ta sử dụng phương thức getService() của object ClientGrpc để lấy instance service, như được hiển thị bên dưới.

@Injectable()
export class AppService implements OnModuleInit {
  private heroesService: HeroesService;

  constructor(@Inject('HERO_PACKAGE') private client: ClientGrpc) {}

  onModuleInit() {
    this.heroesService = this.client.getService<HeroesService>('HeroesService');
  }

  getHero(): Observable<string> {
    return this.heroesService.findOne({ id: 1 });
  }
}
Lưu ý

gRPC Client sẽ không gửi các field chứa dấu gạch dưới _ trong tên của chúng trừ khi tùy chọn keepCase được thiết lập thành true trong cấu hình proto loader (options.loader.keepcase trong cấu hình microservice transporter).

Lưu ý rằng có một sự khác biệt nhỏ so với kỹ thuật được sử dụng trong các phương thức transport microservice khác. Thay vì class ClientProxy, chúng ta sử dụng class ClientGrpc, cung cấp phương thức getService(). Phương thức generic getService() nhận tên service làm đối số và trả về instance của nó (nếu có sẵn).

Ngoài ra, bạn có thể sử dụng decorator @Client() để khởi tạo object ClientGrpc, như sau:

@Injectable()
export class AppService implements OnModuleInit {
  @Client({
    transport: Transport.GRPC,
    options: {
      package: 'hero',
      protoPath: join(__dirname, 'hero/hero.proto'),
    },
  })
  client: ClientGrpc;

  private heroesService: HeroesService;

  onModuleInit() {
    this.heroesService = this.client.getService<HeroesService>('HeroesService');
  }

  getHero(): Observable<string> {
    return this.heroesService.findOne({ id: 1 });
  }
}

Cuối cùng, đối với các tình huống phức tạp hơn, chúng ta có thể inject một client được cấu hình động sử dụng class ClientProxyFactory như đã mô tả tại đây.

Trong cả hai trường hợp, chúng ta đều có tham chiếu tới object proxy HeroesService, hiển thị cùng tập hợp các method được định nghĩa bên trong file .proto. Bây giờ, khi chúng ta truy cập object proxy này (tức là heroesService), hệ thống gRPC tự động serialize request, chuyển tiếp chúng tới hệ thống từ xa, trả về response và deserialize response. Vì gRPC bảo vệ chúng ta khỏi các chi tiết truyền thông mạng, heroesService trông và hoạt động giống như một local provider.

Lưu ý, tất cả các service method đều ở dạng lower camel case (để tuân theo quy ước tự nhiên của ngôn ngữ). Vì vậy, ví dụ, trong khi định nghĩa HeroesService trong file .proto chứa hàm FindOne(), instance heroesService sẽ cung cấp phương thức findOne().

interface HeroesService {
  findOne(data: { id: number }): Observable<any>;
}

Một message handler cũng có thể trả về Observable, trong trường hợp đó các giá trị kết quả sẽ được phát ra cho đến khi stream hoàn thành.

heroes.controller
@Get()
call(): Observable<any> {
  return this.heroesService.findOne({ id: 1 });
}
@Get()
call() {
  return this.heroesService.findOne({ id: 1 });
}

Để gửi gRPC metadata (cùng với request), bạn có thể truyền đối số thứ hai, như sau:

call(): Observable<any> {
  const metadata = new Metadata();
  metadata.add('Set-Cookie', 'yummy_cookie=choco');

  return this.heroesService.findOne({ id: 1 }, metadata);
}
Gợi ý

Class Metadata được import từ package grpc.

Xin lưu ý rằng điều này yêu cầu cập nhật interface HeroesService mà chúng ta đã định nghĩa ở các bước trước.

Example #

Một ví dụ hoạt động có sẵn tại đây.

gRPC Reflection #

gRPC Server Reflection Specification là một tiêu chuẩn cho phép gRPC client yêu cầu chi tiết về API mà server hiển thị, tương tự như việc hiển thị tài liệu OpenAPI cho REST API. Điều này có thể giúp việc làm việc với các công cụ debug dành cho developer như grpc-ui hoặc postman trở nên dễ dàng hơn đáng kể.

Để thêm hỗ trợ gRPC reflection vào server, trước tiên hãy cài đặt package triển khai cần thiết:

$ npm i --save @grpc/reflection

Sau đó nó có thể được hook vào gRPC server sử dụng hook onLoadPackageDefinition trong các tùy chọn gRPC server, như sau:

main
import { ReflectionService } from '@grpc/reflection';

const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  options: {
    onLoadPackageDefinition: (pkg, server) => {
      new ReflectionService(pkg).addToServer(server);
    },
  },
});

Bây giờ server của bạn sẽ phản hồi các message yêu cầu chi tiết API sử dụng reflection specification.

gRPC Streaming #

gRPC tự bản thân hỗ trợ các kết nối sống lâu dài, thường được gọi là streams. Stream rất hữu ích cho các trường hợp như Chat, Observations hoặc truyền dữ liệu theo Chunk. Tìm thêm chi tiết trong tài liệu chính thức tại đây.

Nest hỗ trợ GRPC stream handler theo hai cách có thể:

  • RxJS Subject + Observable handler: có thể hữu ích để viết response ngay bên trong phương thức Controller hoặc được truyền xuống consumer Subject/Observable
  • Pure GRPC call stream handler: có thể hữu ích để được truyền tới một executor sẽ xử lý phần còn lại của dispatch cho Node standard Duplex stream handler.

Streaming sample #

Hãy định nghĩa một gRPC service mẫu mới có tên HelloService. File hello.proto được cấu trúc sử dụng protocol buffers. Đây là nội dung của nó:

hello/hello.proto
// hello/hello.proto
syntax = "proto3";

package hello;

service HelloService {
  rpc BidiHello(stream HelloRequest) returns (stream HelloResponse);
  rpc LotsOfGreetings(stream HelloRequest) returns (HelloResponse);
}

message HelloRequest {
  string greeting = 1;
}

message HelloResponse {
  string reply = 1;
}
Gợi ý

Phương thức LotsOfGreetings có thể được triển khai đơn giản với decorator @GrpcMethod (như trong các ví dụ ở trên) vì stream trả về có thể phát ra nhiều giá trị.

Dựa trên file .proto này, hãy định nghĩa interface HelloService:

interface HelloService {
  bidiHello(upstream: Observable<HelloRequest>): Observable<HelloResponse>;
  lotsOfGreetings(
    upstream: Observable<HelloRequest>,
  ): Observable<HelloResponse>;
}

interface HelloRequest {
  greeting: string;
}

interface HelloResponse {
  reply: string;
}
Gợi ý

Proto interface có thể được tạo tự động bởi package ts-proto, tìm hiểu thêm tại đây.

Subject strategy #

Decorator @GrpcStreamMethod() cung cấp tham số function dưới dạng RxJS Observable. Do đó, chúng ta có thể nhận và xử lý nhiều message.

@GrpcStreamMethod()
bidiHello(messages: Observable<any>, metadata: Metadata, call: ServerDuplexStream<any, any>): Observable<any> {
  const subject = new Subject();

  const onNext = message => {
    console.log(message);
    subject.next({
      reply: 'Hello, world!'
    });
  };
  const onComplete = () => subject.complete();
  messages.subscribe({
    next: onNext,
    complete: onComplete,
  });

  return subject.asObservable();
}
Cảnh báo

Để hỗ trợ tương tác full-duplex với decorator @GrpcStreamMethod(), phương thức controller phải trả về RxJS Observable.

Gợi ý

Class/interface Metadata và ServerUnaryCall được import từ package grpc.

Theo định nghĩa service (trong file .proto), phương thức BidiHello nên stream request tới service. Để gửi nhiều message bất đồng bộ tới stream từ client, chúng ta sử dụng class ReplaySubject của RxJS.

const helloService = this.client.getService<HelloService>('HelloService');
const helloRequest$ = new ReplaySubject<HelloRequest>();

helloRequest$.next({ greeting: 'Hello (1)!' });
helloRequest$.next({ greeting: 'Hello (2)!' });
helloRequest$.complete();

return helloService.bidiHello(helloRequest$);

Trong ví dụ trên, chúng ta đã viết hai message vào stream (các lệnh gọi next()) và thông báo cho service rằng chúng ta đã hoàn thành gửi dữ liệu (lệnh gọi complete()).

Call stream handler #

Khi giá trị trả về của method được định nghĩa là stream, decorator @GrpcStreamCall() cung cấp tham số function dưới dạng grpc.ServerDuplexStream, hỗ trợ các phương thức tiêu chuẩn như .on('data', callback), .write(message) hoặc .cancel(). Tài liệu đầy đủ về các phương thức có sẵn có thể tìm thấy tại đây.

Ngoài ra, khi giá trị trả về của method không phải là stream, decorator @GrpcStreamCall() cung cấp hai tham số function, lần lượt là grpc.ServerReadableStream (đọc thêm tại đây) và callback.

Hãy bắt đầu với triển khai BidiHello hỗ trợ tương tác full-duplex.

@GrpcStreamCall()
bidiHello(requestStream: any) {
  requestStream.on('data', message => {
    console.log(message);
    requestStream.write({
      reply: 'Hello, world!'
    });
  });
}
Gợi ý

Decorator này không yêu cầu bất kỳ tham số trả về cụ thể nào. Stream được mong đợi sẽ được xử lý tương tự như bất kỳ kiểu stream tiêu chuẩn nào khác.

Trong ví dụ trên, chúng ta đã sử dụng phương thức write() để viết object vào response stream. Callback được truyền vào phương thức .on() làm tham số thứ hai sẽ được gọi mỗi khi service nhận được chunk dữ liệu mới.

Hãy triển khai phương thức LotsOfGreetings.

@GrpcStreamCall()
lotsOfGreetings(requestStream: any, callback: (err: unknown, value: HelloResponse) => void) {
  requestStream.on('data', message => {
    console.log(message);
  });
  requestStream.on('end', () => callback(null, { reply: 'Hello, world!' }));
}

Ở đây chúng ta đã sử dụng hàm callback để gửi response sau khi xử lý requestStream đã hoàn thành.

Health checks #

Khi chạy ứng dụng gRPC trong một orchestrator như Kubernetes, bạn có thể cần biết liệu nó có đang chạy và ở trạng thái healthy hay không. gRPC Health Check specification là một tiêu chuẩn cho phép gRPC client hiển thị trạng thái health để cho phép orchestrator hành động phù hợp.

Để thêm hỗ trợ gRPC health check, trước tiên hãy cài đặt package grpc-node:

$ npm i --save grpc-health-check

Sau đó nó có thể được hook vào gRPC service sử dụng hook onLoadPackageDefinition trong các tùy chọn gRPC server, như sau. Lưu ý rằng protoPath cần có cả health check và hero package.

main
import { HealthImplementation, protoPath as healthCheckProtoPath } from 'grpc-health-check';

const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  options: {
    protoPath: [
      healthCheckProtoPath,
      join(__dirname, 'hero/hero.proto'),
    ],
    onLoadPackageDefinition: (pkg, server) => {
      const healthImpl = new HealthImplementation({
        '': 'UNKNOWN',
      });

      healthImpl.addToServer(server);
      healthImpl.setStatus('', 'SERVING');
    },
  },
});
Gợi ý

gRPC health probe là một CLI hữu ích để kiểm tra gRPC health check trong môi trường containerized.

gRPC Metadata #

Metadata là thông tin về một lời gọi RPC cụ thể dưới dạng danh sách các cặp key-value, trong đó key là string và value thường là string nhưng có thể là dữ liệu nhị phân. Metadata là opaque (không trong suốt) đối với bản thân gRPC - nó cho phép client cung cấp thông tin liên quan đến lời gọi cho server và ngược lại. Metadata có thể bao gồm authentication token, request identifier và tag cho mục đích monitoring, và thông tin dữ liệu như số lượng record trong một tập dữ liệu.

Để đọc metadata trong handler @GrpcMethod(), sử dụng đối số thứ hai (metadata), có kiểu Metadata (được import từ package grpc).

Để gửi lại metadata từ handler, sử dụng phương thức ServerUnaryCall#sendMetadata() (đối số handler thứ ba).

heroes.controller
@Controller()
export class HeroesService {
  @GrpcMethod()
  findOne(data: HeroById, metadata: Metadata, call: ServerUnaryCall<any, any>): Hero {
    const serverMetadata = new Metadata();
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];

    serverMetadata.add('Set-Cookie', 'yummy_cookie=choco');
    call.sendMetadata(serverMetadata);

    return items.find(({ id }) => id === data.id);
  }
}

Tương tự, để đọc metadata trong handler được gắn decorator @GrpcStreamMethod() (subject strategy), sử dụng đối số thứ hai (metadata), có kiểu Metadata (được import từ package grpc).

Để gửi lại metadata từ handler, sử dụng phương thức ServerDuplexStream#sendMetadata() (đối số handler thứ ba).

Để đọc metadata từ bên trong call stream handler (handler được gắn decorator @GrpcStreamCall()), lắng nghe event metadata trên tham chiếu requestStream, như sau:

requestStream.on('metadata', (metadata: Metadata) => {
  const meta = metadata.get('X-Meta');
});
