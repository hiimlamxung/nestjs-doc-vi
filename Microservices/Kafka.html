Kafka

Kafka là một nền tảng streaming phân tán mã nguồn mở có ba khả năng chính:

  • Publish và subscribe các stream of records, tương tự như message queue hoặc hệ thống enterprise messaging.
  • Lưu trữ các stream of records một cách bền vững và chịu lỗi.
  • Xử lý các stream of records khi chúng xảy ra.

Dự án Kafka nhằm cung cấp một nền tảng thống nhất, throughput cao, độ trễ thấp để xử lý các data feed thời gian thực. Nó tích hợp rất tốt với Apache Storm và Spark để phân tích dữ liệu streaming thời gian thực.

Installation #

Để bắt đầu xây dựng các microservice dựa trên Kafka, trước tiên hãy cài đặt package cần thiết:

$ npm i --save kafkajs

Overview #

Giống như các implementation transport layer microservice khác của Nest, bạn chọn cơ chế Kafka transporter bằng cách sử dụng thuộc tính transport của object options được truyền vào phương thức createMicroservice(), cùng với thuộc tính options tùy chọn, như được hiển thị dưới đây:

main
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      brokers: ['localhost:9092'],
    }
  }
});
const app = await NestFactory.createMicroservice(AppModule, {

Client #

Có một sự khác biệt nhỏ trong Kafka so với các microservice transporter khác. Thay vì class ClientProxy, chúng ta sử dụng class ClientKafkaProxy.

Giống như các microservice transporter khác, bạn có một số tùy chọn để tạo instance ClientKafkaProxy.

Một phương pháp để tạo instance là sử dụng ClientsModule. Để tạo client instance với ClientsModule, hãy import nó và sử dụng phương thức register() để truyền vào một object options có cùng các thuộc tính như được hiển thị ở trên trong phương thức createMicroservice(), cùng với thuộc tính name được sử dụng làm injection token. Đọc thêm về ClientsModule tại đây.

@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'HERO_SERVICE',
        transport: Transport.KAFKA,
        options: {
          client: {
            clientId: 'hero',
            brokers: ['localhost:9092'],
          },
          consumer: {
            groupId: 'hero-consumer'
          }
        }
      },
    ]),
  ]
  ...
})

Các tùy chọn khác để tạo client (hoặc ClientProxyFactory hoặc @Client()) cũng có thể được sử dụng. Bạn có thể đọc về chúng tại đây.

Sử dụng decorator @Client() như sau:

@Client({
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero',
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer'
    }
  }
})
client: ClientKafkaProxy;

Message pattern #

Message pattern của Kafka microservice sử dụng hai topic cho các kênh request và reply. Phương thức ClientKafkaProxy.send() gửi message với một return address bằng cách liên kết một correlation id, reply topic, và reply partition với request message. Điều này yêu cầu instance ClientKafkaProxy phải subscribe vào reply topic và được gán ít nhất một partition trước khi gửi message.

Do đó, bạn cần có ít nhất một reply topic partition cho mỗi ứng dụng Nest đang chạy. Ví dụ, nếu bạn đang chạy 4 ứng dụng Nest nhưng reply topic chỉ có 3 partition, thì 1 trong 4 ứng dụng Nest sẽ gặp lỗi khi cố gắng gửi message.

Khi các instance ClientKafkaProxy mới được khởi chạy, chúng sẽ tham gia consumer group và subscribe vào các topic tương ứng. Quá trình này kích hoạt việc rebalance các topic partition được gán cho các consumer trong consumer group.

Thông thường, các topic partition được gán bằng round robin partitioner, gán các topic partition cho một tập hợp consumer được sắp xếp theo tên consumer được đặt ngẫu nhiên khi khởi chạy ứng dụng. Tuy nhiên, khi một consumer mới tham gia consumer group, consumer mới có thể được đặt ở bất kỳ vị trí nào trong tập hợp consumer. Điều này tạo ra một điều kiện mà các consumer đã tồn tại có thể bị gán các partition khác nhau khi consumer hiện tại nằm sau consumer mới. Kết quả là, các consumer bị gán partition khác nhau sẽ mất các response message của các request đã gửi trước khi rebalance.

Để ngăn các consumer ClientKafkaProxy mất response message, một custom partitioner tích hợp sẵn dành riêng cho Nest được sử dụng. Custom partitioner này gán partition cho một tập hợp consumer được sắp xếp theo high-resolution timestamp (process.hrtime()) được đặt khi khởi chạy ứng dụng.

Message response subscription #

Lưu ý

Phần này chỉ liên quan nếu bạn sử dụng kiểu message request-response (với decorator @MessagePattern và phương thức ClientKafkaProxy.send). Việc subscribe vào response topic không cần thiết cho giao tiếp event-based (decorator @EventPattern và phương thức ClientKafkaProxy.emit).

Class ClientKafkaProxy cung cấp phương thức subscribeToResponseOf(). Phương thức subscribeToResponseOf() nhận tên topic của request làm đối số và thêm tên reply topic dẫn xuất vào một tập hợp các reply topic. Phương thức này là bắt buộc khi triển khai message pattern.

heroes.controller
onModuleInit() {
  this.client.subscribeToResponseOf('hero.kill.dragon');
}

Nếu instance ClientKafkaProxy được tạo bất đồng bộ, phương thức subscribeToResponseOf() phải được gọi trước khi gọi phương thức connect().

heroes.controller
async onModuleInit() {
  this.client.subscribeToResponseOf('hero.kill.dragon');
  await this.client.connect();
}

Incoming #

Nest nhận các Kafka message đến dưới dạng object có các thuộc tính key, value, và headers với giá trị kiểu Buffer. Sau đó Nest phân tích các giá trị này bằng cách chuyển đổi buffer thành string. Nếu string có dạng "object like", Nest sẽ cố gắng phân tích string đó dưới dạng JSON. Giá trị value sau đó được truyền cho handler tương ứng.

Outgoing #

Nest gửi các Kafka message đi sau quá trình serialization khi publish event hoặc gửi message. Điều này xảy ra trên các đối số được truyền cho các phương thức emit() và send() của ClientKafkaProxy hoặc trên các giá trị được trả về từ phương thức @MessagePattern. Quá trình serialization này "stringify" các object không phải string hoặc buffer bằng cách sử dụng JSON.stringify() hoặc phương thức prototype toString().

heroes.controller
@Controller()
export class HeroesController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const dragonId = message.dragonId;
    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];
    return items;
  }
}
Gợi ý

@Payload() được import từ package @nestjs/microservices.

Các message đi cũng có thể được key bằng cách truyền một object có các thuộc tính key và value. Việc key message rất quan trọng để đáp ứng yêu cầu co-partitioning.

heroes.controller
@Controller()
export class HeroesController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const realm = 'Nest';
    const heroId = message.heroId;
    const dragonId = message.dragonId;

    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];

    return {
      headers: {
        realm
      },
      key: heroId,
      value: items
    }
  }
}

Ngoài ra, các message được truyền theo format này cũng có thể chứa custom header được đặt trong thuộc tính hash headers. Giá trị thuộc tính hash header phải có kiểu string hoặc kiểu Buffer.

heroes.controller
@Controller()
export class HeroesController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const realm = 'Nest';
    const heroId = message.heroId;
    const dragonId = message.dragonId;

    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];

    return {
      headers: {
        kafka_nestRealm: realm
      },
      key: heroId,
      value: items
    }
  }
}

Event-based #

Mặc dù phương pháp request-response là lý tưởng cho việc trao đổi message giữa các service, nó ít phù hợp hơn khi kiểu message của bạn là event-based (điều này lại là lý tưởng cho Kafka) - khi bạn chỉ muốn publish event mà không cần chờ response. Trong trường hợp đó, bạn không muốn overhead cần thiết của request-response để duy trì hai topic.

Hãy xem hai phần này để tìm hiểu thêm: Overview: Event-based và Overview: Publishing events.

Context #

Trong các tình huống phức tạp hơn, bạn có thể cần truy cập thêm thông tin về request đến. Khi sử dụng Kafka transporter, bạn có thể truy cập object KafkaContext.

@MessagePattern('hero.kill.dragon')
killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  console.log(`Topic: ${context.getTopic()}`);
}
@Bind(Payload(), Ctx())
@MessagePattern('hero.kill.dragon')
killDragon(message, context) {
  console.log(`Topic: ${context.getTopic()}`);
}
Gợi ý

@Payload(), @Ctx() và KafkaContext được import từ package @nestjs/microservices.

Để truy cập object Kafka IncomingMessage gốc, sử dụng phương thức getMessage() của object KafkaContext, như sau:

@MessagePattern('hero.kill.dragon')
killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  const originalMessage = context.getMessage();
  const partition = context.getPartition();
  const { headers, timestamp } = originalMessage;
}
@Bind(Payload(), Ctx())
@MessagePattern('hero.kill.dragon')
killDragon(message, context) {
  const originalMessage = context.getMessage();
  const partition = context.getPartition();
  const { headers, timestamp } = originalMessage;
}

Trong đó IncomingMessage thực hiện interface sau:

interface IncomingMessage {
  topic: string;
  partition: number;
  timestamp: string;
  size: number;
  attributes: number;
  offset: string;
  key: any;
  value: any;
  headers: Record<string, any>;
}

Nếu handler của bạn có thời gian xử lý chậm cho mỗi message nhận được, bạn nên cân nhắc sử dụng callback heartbeat. Để lấy function heartbeat, sử dụng phương thức getHeartbeat() của KafkaContext, như sau:

@MessagePattern('hero.kill.dragon')
async killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  const heartbeat = context.getHeartbeat();

  // Do some slow processing
  await doWorkPart1();

  // Send heartbeat to not exceed the sessionTimeout
  await heartbeat();

  // Do some slow processing again
  await doWorkPart2();
}

Naming conventions #

Các thành phần Kafka microservice thêm mô tả vai trò tương ứng vào các tùy chọn client.clientId và consumer.groupId để ngăn xung đột giữa các thành phần client và server của Nest microservice. Theo mặc định, các thành phần ClientKafkaProxy thêm hậu tố -client và các thành phần ServerKafka thêm hậu tố -server vào cả hai tùy chọn này. Lưu ý cách các giá trị được cung cấp bên dưới được chuyển đổi theo cách đó (như hiển thị trong comments).

main
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero', // hero-server
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer' // hero-consumer-server
    },
  }
});

Và cho client:

heroes.controller
@Client({
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero', // hero-client
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer' // hero-consumer-client
    }
  }
})
client: ClientKafkaProxy;
Gợi ý

Quy ước đặt tên Kafka client và consumer có thể được tùy chỉnh bằng cách extend ClientKafkaProxy và KafkaServer trong custom provider của bạn và override constructor.

Vì message pattern của Kafka microservice sử dụng hai topic cho các kênh request và reply, nên một reply pattern nên được dẫn xuất từ request topic. Theo mặc định, tên của reply topic là tên request topic với .reply được thêm vào cuối.

heroes.controller
onModuleInit() {
  this.client.subscribeToResponseOf('hero.get'); // hero.get.reply
}
Gợi ý

Quy ước đặt tên Kafka reply topic có thể được tùy chỉnh bằng cách extend ClientKafkaProxy trong custom provider của bạn và override phương thức getResponsePatternName.

Retriable exceptions #

Tương tự như các transporter khác, tất cả unhandled exception đều được tự động bọc trong RpcException và chuyển đổi sang format "thân thiện với người dùng". Tuy nhiên, có những edge-case mà bạn có thể muốn bỏ qua cơ chế này và để exception được consume bởi driver kafkajs. Việc throw exception khi xử lý message sẽ chỉ thị cho kafkajs retry nó (redeliver nó), có nghĩa là ngay cả khi message (hoặc event) handler đã được kích hoạt, offset sẽ không được commit vào Kafka.

Cảnh báo

Đối với event handler (giao tiếp event-based), tất cả unhandled exception được coi là retriable exception theo mặc định.

Để thực hiện điều này, bạn có thể sử dụng class chuyên dụng KafkaRetriableException, như sau:

throw new KafkaRetriableException('...');
Gợi ý

Class KafkaRetriableException được export từ package @nestjs/microservices.

Custom exception handling #

Cùng với các cơ chế xử lý lỗi mặc định, bạn có thể tạo một custom Exception Filter cho các Kafka event để quản lý retry logic. Ví dụ, ví dụ dưới đây minh họa cách bỏ qua một event có vấn đề sau một số lần retry có thể cấu hình:

import { Catch, ArgumentsHost, Logger } from '@nestjs/common';
import { BaseExceptionFilter } from '@nestjs/core';
import { KafkaContext } from '../ctx-host';

@Catch()
export class KafkaMaxRetryExceptionFilter extends BaseExceptionFilter {
  private readonly logger = new Logger(KafkaMaxRetryExceptionFilter.name);

  constructor(
    private readonly maxRetries: number,
    // Optional custom function executed when max retries are exceeded
    private readonly skipHandler?: (message: any) => Promise<void>,
  ) {
    super();
  }

  async catch(exception: unknown, host: ArgumentsHost) {
    const kafkaContext = host.switchToRpc().getContext<KafkaContext>();
    const message = kafkaContext.getMessage();
    const currentRetryCount = this.getRetryCountFromContext(kafkaContext);

    if (currentRetryCount >= this.maxRetries) {
      this.logger.warn(
        `Max retries (${this.maxRetries}) exceeded for message: ${JSON.stringify(message)}`
      );

      if (this.skipHandler) {
        try {
          await this.skipHandler(message);
        } catch (err) {
          this.logger.error('Error in skipHandler:', err);
        }
      }

      try {
        await this.commitOffset(kafkaContext);
      } catch (commitError) {
        this.logger.error('Failed to commit offset:', commitError);
      }
      return; // Stop propagating the exception
    }

    // If retry count is below the maximum, proceed with the default Exception Filter logic
    super.catch(exception, host);
  }

  private getRetryCountFromContext(context: KafkaContext): number {
    const headers = context.getMessage().headers || {};
    const retryHeader = headers['retryCount'] || headers['retry-count'];
    return retryHeader ? Number(retryHeader) : 0;
  }

  private async commitOffset(context: KafkaContext): Promise<void> {
    const consumer = context.getConsumer && context.getConsumer();
    if (!consumer) {
      throw new Error('Consumer instance is not available from KafkaContext.');
    }

    const topic = context.getTopic && context.getTopic();
    const partition = context.getPartition && context.getPartition();
    const message = context.getMessage();
    const offset = message.offset;

    if (!topic || partition === undefined || offset === undefined) {
      throw new Error(
        'Incomplete Kafka message context for committing offset.'
      );
    }

    await consumer.commitOffsets([
      {
        topic,
        partition,
        // When committing an offset, commit the next number (i.e., current offset + 1)
        offset: (Number(offset) + 1).toString(),
      },
    ]);
  }
}

Filter này cung cấp một cách để retry xử lý Kafka event lên đến một số lần có thể cấu hình. Khi đạt đến số retry tối đa, nó sẽ kích hoạt custom skipHandler (nếu được cung cấp) và commit offset, hiệu quả là bỏ qua event có vấn đề. Điều này cho phép các event tiếp theo được xử lý mà không bị gián đoạn.

Bạn có thể tích hợp filter này bằng cách thêm nó vào event handler:

@UseFilters(new KafkaMaxRetryExceptionFilter(5))
export class MyEventHandler {
  @EventPattern('your-topic')
  async handleEvent(@Payload() data: any, @Ctx() context: KafkaContext) {
    // Your event processing logic...
  }
}

Commit offsets #

Việc commit offset là rất quan trọng khi làm việc với Kafka. Theo mặc định, message sẽ được tự động commit sau một khoảng thời gian nhất định. Để biết thêm thông tin, hãy truy cập KafkaJS docs. KafkaContext cung cấp một cách để truy cập active consumer để commit offset thủ công. Consumer chính là KafkaJS consumer và hoạt động như native KafkaJS implementation.

@EventPattern('user.created')
async handleUserCreated(@Payload() data: IncomingMessage, @Ctx() context: KafkaContext) {
  // business logic

  const { offset } = context.getMessage();
  const partition = context.getPartition();
  const topic = context.getTopic();
  const consumer = context.getConsumer();
  await consumer.commitOffsets([{ topic, partition, offset }])
}
@Bind(Payload(), Ctx())
@EventPattern('user.created')
async handleUserCreated(data, context) {
  // business logic

  const { offset } = context.getMessage();
  const partition = context.getPartition();
  const topic = context.getTopic();
  const consumer = context.getConsumer();
  await consumer.commitOffsets([{ topic, partition, offset }])
}

Để tắt auto-committing message, hãy đặt autoCommit: false trong cấu hình run, như sau:

main
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      brokers: ['localhost:9092'],
    },
    run: {
      autoCommit: false
    }
  }
});
const app = await NestFactory.createMicroservice(AppModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      brokers: ['localhost:9092'],
    },
    run: {
      autoCommit: false
    }
  }
});

Instance status updates #

Để nhận các cập nhật thời gian thực về kết nối và trạng thái của driver instance bên dưới, bạn có thể subscribe vào stream status. Stream này cung cấp các cập nhật trạng thái dành riêng cho driver được chọn. Đối với Kafka driver, stream status phát ra các event connected, disconnected, rebalancing, crashed, và stopped.

this.client.status.subscribe((status: KafkaStatus) => {
  console.log(status);
});
Gợi ý

Kiểu KafkaStatus được import từ package @nestjs/microservices.

Tương tự, bạn có thể subscribe vào stream status của server để nhận thông báo về trạng thái server.

const server = app.connectMicroservice<MicroserviceOptions>(...);
server.status.subscribe((status: KafkaStatus) => {
  console.log(status);
});

Underlying producer and consumer #

Đối với các use case nâng cao hơn, bạn có thể cần truy cập các instance producer và consumer bên dưới. Điều này có thể hữu ích cho các tình huống như đóng kết nối thủ công hoặc sử dụng các phương thức dành riêng cho driver. Tuy nhiên, hãy lưu ý rằng trong hầu hết các trường hợp, bạn không nên cần truy cập driver trực tiếp.

Để làm điều này, bạn có thể sử dụng các getter producer và consumer được expose bởi instance ClientKafkaProxy.

const producer = this.client.producer;
const consumer = this.client.consumer;
