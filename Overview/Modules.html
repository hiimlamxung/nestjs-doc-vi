Modules

Module là một class được annotate bằng decorator @Module(). Decorator này cung cấp metadata mà Nest sử dụng để tổ chức và quản lý cấu trúc ứng dụng một cách hiệu quả.

Application graph — Nest dùng để resolve mối quan hệ và dependencies giữa các module

Mọi ứng dụng Nest đều có ít nhất một module — root module, dùng làm điểm khởi đầu để Nest xây dựng application graph. Đây là cấu trúc nội bộ Nest dùng để resolve các mối quan hệ và dependency giữa module và provider. Trong khi ứng dụng nhỏ có thể chỉ cần root module, nhìn chung thì không phải như vậy. Module được khuyến khích dùng như một cách hiệu quả để tổ chức component. Với hầu hết ứng dụng, bạn sẽ có nhiều module, mỗi module đóng gói một tập các tính năng liên quan chặt chẽ với nhau.

Decorator @Module() nhận một object với các property mô tả module:

Property Mô tả
providers Các provider sẽ được Nest injector khởi tạo và có thể được chia sẻ ít nhất trong module này.
controllers Tập các controller được định nghĩa trong module này, cần được khởi tạo.
imports Danh sách các module được import — các module này export các provider mà module hiện tại cần dùng.
exports Tập con của providers được cung cấp bởi module này và cần khả dụng trong các module khác khi import module này. Có thể dùng chính provider hoặc chỉ token (provide value) của nó.

Module đóng gói provider theo mặc định — nghĩa là bạn chỉ có thể inject các provider thuộc module hiện tại hoặc được export rõ ràng từ các module được import. Các provider được export từ một module về bản chất là public interface hay API của module đó.

Feature modules #

Trong ví dụ của chúng ta, CatsController và CatsService có liên quan chặt chẽ và phục vụ cùng một domain của ứng dụng. Việc nhóm chúng vào một feature module là hợp lý. Feature module tổ chức code liên quan đến một tính năng cụ thể, giúp duy trì ranh giới rõ ràng và tổ chức tốt hơn. Điều này đặc biệt quan trọng khi ứng dụng hoặc team phát triển lớn dần, và phù hợp với các nguyên tắc SOLID.

Tiếp theo, chúng ta sẽ tạo CatsModule để minh hoạ cách nhóm controller và service.

cats/cats.module.ts
import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers:   [CatsService],
})
export class CatsModule {}
💡 Hint

Để tạo module bằng CLI, chạy lệnh $ nest g module cats.

Ở trên, chúng ta đã định nghĩa CatsModule trong file cats.module.ts và chuyển tất cả những gì liên quan đến module này vào thư mục cats. Việc cuối cùng cần làm là import module này vào root module (AppModule, định nghĩa trong app.module.ts).

app.module.ts
import { Module } from '@nestjs/common';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule {}

Shared modules #

Trong Nest, module là singleton theo mặc định — do đó bạn có thể dễ dàng chia sẻ cùng một instance của bất kỳ provider nào giữa nhiều module.

CatsModule (exports: CatsService)
↙   ↓   ↘ chia sẻ cùng 1 instance
ModuleA
ModuleB
ModuleC

Mọi module đều tự động là một shared module. Một khi được tạo, nó có thể được tái sử dụng bởi bất kỳ module nào. Giả sử chúng ta muốn chia sẻ một instance của CatsService giữa nhiều module khác. Để làm điều đó, trước tiên cần export provider CatsService bằng cách thêm nó vào mảng exports của module:

cats.module.ts
import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers:   [CatsService],
  exports:     [CatsService],
})
export class CatsModule {}

Bây giờ bất kỳ module nào import CatsModule đều có quyền truy cập CatsService và sẽ chia sẻ cùng một instance với tất cả các module khác cũng import nó.

Nếu đăng ký trực tiếp CatsService vào từng module riêng lẻ, điều đó sẽ hoạt động nhưng mỗi module sẽ nhận một instance CatsService riêng biệt. Điều này dẫn đến tốn bộ nhớ hơn vì nhiều instance của cùng một service được tạo ra, và có thể gây ra các hành vi không mong đợi như state không nhất quán nếu service duy trì trạng thái nội bộ.

Bằng cách đóng gói CatsService vào CatsModule và export nó, chúng ta đảm bảo rằng cùng một instance của CatsService được tái sử dụng trên tất cả các module import CatsModule. Điều này không chỉ giảm tiêu thụ bộ nhớ mà còn dẫn đến hành vi dễ dự đoán hơn — đây là một trong những lợi ích chính của tính module hoá và dependency injection trong NestJS.

Module re-exporting #

Như đã thấy ở trên, module có thể export các provider nội bộ của mình. Ngoài ra, chúng cũng có thể re-export các module mà chúng import. Trong ví dụ dưới, CommonModule vừa được import vào và export từ CoreModule, làm cho nó khả dụng cho các module khác import CoreModule.

@Module({
  imports: [CommonModule],
  exports: [CommonModule],
})
export class CoreModule {}

Dependency injection #

Bản thân module class cũng có thể inject provider (ví dụ: phục vụ mục đích cấu hình):

cats.module.ts
TS
JS
import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers:   [CatsService],
})
export class CatsModule {
  constructor(private catsService: CatsService) {}
}
import { Module, Dependencies } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers:   [CatsService],
})
@Dependencies(CatsService)
export class CatsModule {
  constructor(catsService) {
    this.catsService = catsService;
  }
}

Tuy nhiên, bản thân module class không thể được inject như một provider do vấn đề circular dependency.

Global modules #

Nếu bạn phải import cùng một tập module ở khắp nơi, điều đó có thể trở nên lặp đi lặp lại. Không giống như Angular — nơi providers được đăng ký ở global scope và khả dụng ở mọi nơi — Nest đóng gói provider bên trong module scope. Bạn không thể dùng provider của một module ở nơi khác nếu không import module đó trước.

Khi muốn cung cấp một tập provider nên khả dụng ở mọi nơi (ví dụ: helper, database connection, v.v.), hãy làm module đó thành global bằng decorator @Global().

import { Module, Global } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Global()
@Module({
  controllers: [CatsController],
  providers:   [CatsService],
  exports:     [CatsService],
})
export class CatsModule {}

Decorator @Global() làm cho module có global scope. Global module chỉ nên được đăng ký một lần duy nhất, thường là bởi root hoặc core module. Trong ví dụ trên, provider CatsService sẽ có mặt ở khắp nơi — các module muốn inject service này sẽ không cần import CatsModule vào mảng imports của chúng.

💡 Hint

Làm mọi thứ thành global không được khuyến khích như một thực hành thiết kế. Mặc dù global module có thể giúp giảm boilerplate, nhìn chung tốt hơn nên dùng mảng imports để cung cấp API của module cho các module khác một cách có kiểm soát và rõ ràng. Cách tiếp cận này mang lại cấu trúc và khả năng bảo trì tốt hơn, đảm bảo chỉ những phần cần thiết của module được chia sẻ và tránh coupling không cần thiết giữa các phần không liên quan.

Dynamic modules #

Dynamic module trong Nest cho phép bạn tạo ra các module có thể được cấu hình tại runtime. Điều này đặc biệt hữu ích khi cần cung cấp các module linh hoạt, có thể tuỳ biến — nơi provider có thể được tạo dựa trên một số tuỳ chọn hoặc cấu hình nhất định. Đây là tổng quan ngắn gọn về cách dynamic module hoạt động.

TS
JS
import { Module, DynamicModule } from '@nestjs/common';
import { createDatabaseProviders } from './database.providers';
import { Connection } from './connection.provider';

@Module({
  providers: [Connection],
  exports:   [Connection],
})
export class DatabaseModule {
  static forRoot(entities = [], options?): DynamicModule {
    const providers = createDatabaseProviders(options, entities);
    return {
      module:    DatabaseModule,
      providers: providers,
      exports:   providers,
    };
  }
}
import { Module } from '@nestjs/common';
import { createDatabaseProviders } from './database.providers';
import { Connection } from './connection.provider';

@Module({
  providers: [Connection],
  exports:   [Connection],
})
export class DatabaseModule {
  static forRoot(entities = [], options) {
    const providers = createDatabaseProviders(options, entities);
    return {
      module:    DatabaseModule,
      providers: providers,
      exports:   providers,
    };
  }
}
💡 Hint

Phương thức forRoot() có thể return dynamic module synchronously hoặc asynchronously (ví dụ: qua Promise).

Module này định nghĩa provider Connection theo mặc định (trong metadata của decorator @Module()), nhưng ngoài ra — tuỳ thuộc vào các object entities và options truyền vào forRoot() — nó còn expose một tập các provider bổ sung, ví dụ như repository. Lưu ý rằng các property trả về từ dynamic module mở rộng (thay vì ghi đè) metadata module cơ bản định nghĩa trong decorator @Module(). Đó là lý do cả provider Connection khai báo tĩnh lẫn các repository provider được tạo động đều được export từ module.

Nếu muốn đăng ký dynamic module trong global scope, set property global thành true:

{
  global:    true,
  module:    DatabaseModule,
  providers: providers,
  exports:   providers,
}
⚠️ Warning

Như đã đề cập ở trên, làm mọi thứ thành global không phải là quyết định thiết kế tốt.

DatabaseModule có thể được import và cấu hình theo cách sau:

import { Module } from '@nestjs/common';
import { DatabaseModule } from './database/database.module';
import { User } from './users/entities/user.entity';

@Module({
  imports: [DatabaseModule.forRoot([User])],
})
export class AppModule {}

Nếu muốn re-export dynamic module, bạn có thể bỏ qua lời gọi forRoot() trong mảng exports:

import { Module } from '@nestjs/common';
import { DatabaseModule } from './database/database.module';
import { User } from './users/entities/user.entity';

@Module({
  imports: [DatabaseModule.forRoot([User])],
  exports: [DatabaseModule],
})
export class AppModule {}

Chương Dynamic modules đề cập chủ đề này chi tiết hơn và bao gồm ví dụ hoạt động thực tế.

💡 Hint

Tìm hiểu cách build dynamic module có thể tuỳ biến cao với ConfigurableModuleBuilder trong chương này.
