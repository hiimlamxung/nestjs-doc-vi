Pipes

Pipe là một class được annotate bằng decorator @Injectable(), implement interface PipeTransform.

Pipe có hai trường hợp sử dụng điển hình:

  • transformation: chuyển đổi input data sang dạng mong muốn (ví dụ: từ string sang integer)
  • validation: kiểm tra input data, nếu hợp lệ thì truyền tiếp không đổi; ngược lại throw exception

Trong cả hai trường hợp, pipe hoạt động trên các arguments được xử lý bởi controller route handler. Nest đặt pipe ngay trước khi method được gọi — pipe nhận các argument dành cho method và thực hiện thao tác trên chúng. Mọi thao tác transformation hoặc validation diễn ra tại thời điểm đó, sau đó route handler mới được gọi với các argument đã (có thể) được biến đổi.

Nest đi kèm một số built-in pipe mà bạn có thể sử dụng ngay. Bạn cũng có thể tự build custom pipe. Trong chương này, chúng ta sẽ giới thiệu các built-in pipe và cách bind chúng vào route handler. Sau đó, chúng ta sẽ xem qua một số custom pipe để thấy cách build pipe từ đầu.

Hint

Pipe chạy bên trong exceptions zone. Nghĩa là khi Pipe throw exception, nó được xử lý bởi exceptions layer (global exceptions filter và bất kỳ exceptions filter nào được áp dụng cho context hiện tại). Với điều trên, rõ ràng là khi exception được throw trong Pipe, không có controller method nào được thực thi sau đó. Đây là kỹ thuật best-practice để validate dữ liệu đến từ nguồn bên ngoài tại system boundary.

Built-in pipes #

Nest đi kèm một số pipe có sẵn:

  • ValidationPipe
  • ParseIntPipe
  • ParseFloatPipe
  • ParseBoolPipe
  • ParseArrayPipe
  • ParseUUIDPipe
  • ParseEnumPipe
  • DefaultValuePipe
  • ParseFilePipe
  • ParseDatePipe

Chúng được export từ package @nestjs/common.

Hãy xem nhanh cách dùng ParseIntPipe. Đây là ví dụ về trường hợp transformation, nơi pipe đảm bảo rằng parameter của method handler được convert sang JavaScript integer (hoặc throw exception nếu convert thất bại). Ở phần sau của chương này, chúng ta sẽ trình bày một custom implementation đơn giản cho ParseIntPipe. Các kỹ thuật ví dụ bên dưới cũng áp dụng cho các built-in transformation pipe khác (ParseBoolPipe, ParseFloatPipe, ParseEnumPipe, ParseArrayPipe, ParseDatePipe, và ParseUUIDPipe — mà chúng ta sẽ gọi chung là các Parse* pipe trong chương này).

Binding pipes #

Để dùng pipe, chúng ta cần bind một instance của pipe class vào context phù hợp. Trong ví dụ ParseIntPipe, chúng ta muốn liên kết pipe với một route handler method cụ thể và đảm bảo nó chạy trước khi method được gọi. Chúng ta thực hiện bằng cấu trúc sau, được gọi là binding pipe ở cấp method parameter:

@Get(':id')
async findOne(@Param('id', ParseIntPipe) id: number) {
  return this.catsService.findOne(id);
}

Điều này đảm bảo một trong hai điều kiện sau là đúng: hoặc parameter nhận được trong method findOne() là number (như mong đợi trong lời gọi this.catsService.findOne()), hoặc exception được throw trước khi route handler được gọi.

Ví dụ, giả sử route được gọi như:

GET localhost:3000/abc

Nest sẽ throw exception như sau:

{
  "statusCode": 400,
  "message": "Validation failed (numeric string is expected)",
  "error": "Bad Request"
}

Exception sẽ ngăn body của method findOne() thực thi.

Trong ví dụ trên, chúng ta truyền class (ParseIntPipe), không phải instance, để framework đảm nhận việc khởi tạo và cho phép dependency injection. Giống như với pipe và guard, chúng ta có thể truyền in-place instance thay vào đó. Truyền in-place instance hữu ích nếu muốn tuỳ biến hành vi của built-in pipe bằng cách truyền options:

@Get(':id')
async findOne(
  @Param('id', new ParseIntPipe({ errorHttpStatusCode: HttpStatus.NOT_ACCEPTABLE }))
  id: number,
) {
  return this.catsService.findOne(id);
}

Binding các transformation pipe khác (tất cả Parse* pipe) hoạt động tương tự. Các pipe này đều hoạt động trong ngữ cảnh validate route parameter, query string parameter và request body value.

Ví dụ với query string parameter:

@Get()
async findOne(@Query('id', ParseIntPipe) id: number) {
  return this.catsService.findOne(id);
}

Đây là ví dụ dùng ParseUUIDPipe để parse string parameter và validate xem đó có phải UUID không.

TS
JS
@Get(':uuid')
async findOne(@Param('uuid', new ParseUUIDPipe()) uuid: string) {
  return this.catsService.findOne(uuid);
}
@Get(':uuid')
@Bind(Param('uuid', new ParseUUIDPipe()))
async findOne(uuid) {
  return this.catsService.findOne(uuid);
}
Hint

Khi dùng ParseUUIDPipe(), bạn đang parse UUID version 3, 4 hoặc 5. Nếu chỉ yêu cầu version cụ thể của UUID, có thể truyền version vào pipe options.

Ở trên chúng ta đã thấy các ví dụ binding các Parse* pipe built-in. Binding validation pipe hơi khác một chút; chúng ta sẽ thảo luận trong phần tiếp theo.

Hint

Ngoài ra, xem Validation techniques để biết các ví dụ mở rộng về validation pipe.

Custom pipes #

Như đã đề cập, bạn có thể tự build custom pipe. Trong khi Nest cung cấp ParseIntPipe và ValidationPipe built-in mạnh mẽ, hãy build các phiên bản custom đơn giản của mỗi loại từ đầu để thấy cách custom pipe được xây dựng.

Chúng ta bắt đầu với một ValidationPipe đơn giản. Ban đầu, nó chỉ nhận input value và trả về ngay giá trị đó — hoạt động như một identity function.

validation.pipe
TS
JS
import { PipeTransform, Injectable, ArgumentMetadata } from '@nestjs/common';

@Injectable()
export class ValidationPipe implements PipeTransform {
  transform(value: any, metadata: ArgumentMetadata) {
    return value;
  }
}
import { Injectable } from '@nestjs/common';

@Injectable()
export class ValidationPipe {
  transform(value, metadata) {
    return value;
  }
}
Hint

PipeTransform<T, R> là generic interface phải được implement bởi mọi pipe. Generic interface dùng T để biểu thị type của input value, và R để biểu thị kiểu trả về của method transform().

Mọi pipe phải implement method transform() để hoàn thành contract của interface PipeTransform. Method này có hai parameter:

  • value
  • metadata

Parameter value là method argument hiện tại đang được xử lý (trước khi được route handling method nhận), và metadata là metadata của method argument đang được xử lý. Object metadata có các property sau:

export interface ArgumentMetadata {
  type: 'body' | 'query' | 'param' | 'custom';
  metatype?: Type<unknown>;
  data?: string;
}
PropertyMô tả
type Cho biết argument là body @Body(), query @Query(), param @Param(), hay custom parameter.
metatype Cung cấp metatype của argument, ví dụ String. Lưu ý: giá trị là undefined nếu bạn bỏ khai báo type trong chữ ký method của route handler, hoặc dùng vanilla JavaScript.
data String được truyền vào decorator, ví dụ @Body('string'). Là undefined nếu bạn để trống dấu ngoặc của decorator.
Warning

TypeScript interface biến mất sau khi transpile. Do đó, nếu type của method parameter được khai báo là interface thay vì class, giá trị metatype sẽ là Object.

Schema based validation #

Hãy làm validation pipe hữu ích hơn. Xem xét method create() của CatsController, nơi chúng ta có lẽ muốn đảm bảo post body object hợp lệ trước khi chạy service method.

TS
JS
@Post()
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
async create(@Body() createCatDto) {
  this.catsService.create(createCatDto);
}

Hãy tập trung vào body parameter createCatDto. Type của nó là CreateCatDto:

create-cat.dto
export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}

Chúng ta muốn đảm bảo mọi request đến method create chứa body hợp lệ. Vì vậy chúng ta phải validate ba member của object createCatDto. Chúng ta có thể làm điều này bên trong route handler method, nhưng làm vậy không lý tưởng vì sẽ vi phạm single responsibility principle (SRP).

Một cách tiếp cận khác là tạo validator class và ủy thác công việc cho nó. Điều này có nhược điểm là phải nhớ gọi validator này ở đầu mỗi method.

Vậy tạo validation middleware thì sao? Cách này có thể hoạt động, nhưng không may là không thể tạo generic middleware có thể sử dụng xuyên suốt mọi context trong toàn bộ ứng dụng. Bởi vì middleware không biết về execution context, bao gồm handler sẽ được gọi và bất kỳ parameter nào của nó.

Tất nhiên, đây chính xác là use case mà pipe được thiết kế để xử lý. Vì vậy hãy tiếp tục tinh chỉnh validation pipe của chúng ta.

Object schema validation #

Có một số cách tiếp cận để thực hiện object validation theo cách clean, DRY. Một cách phổ biến là sử dụng validation dựa trên schema. Hãy thử cách tiếp cận đó.

Thư viện Zod cho phép tạo schema theo cách đơn giản, với API dễ đọc. Hãy build validation pipe sử dụng Zod-based schema.

Bắt đầu bằng cách cài đặt package cần thiết:

$ npm install --save zod

Trong đoạn code bên dưới, chúng ta tạo một class đơn giản nhận schema làm argument của constructor. Sau đó áp dụng method schema.parse() để validate incoming argument theo schema đã cung cấp.

Như đã lưu ý trước đó, validation pipe hoặc trả về giá trị không đổi hoặc throw exception.

Trong phần tiếp theo, bạn sẽ thấy cách cung cấp schema phù hợp cho một controller method cụ thể bằng decorator @UsePipes(). Làm vậy giúp validation pipe của chúng ta có thể tái sử dụng xuyên suốt các context, đúng như mục tiêu ban đầu.

TS
JS
import { PipeTransform, ArgumentMetadata, BadRequestException } from '@nestjs/common';
import { ZodSchema } from 'zod';

export class ZodValidationPipe implements PipeTransform {
  constructor(private schema: ZodSchema) {}

  transform(value: unknown, metadata: ArgumentMetadata) {
    try {
      const parsedValue = this.schema.parse(value);
      return parsedValue;
    } catch (error) {
      throw new BadRequestException('Validation failed');
    }
  }
}
import { BadRequestException } from '@nestjs/common';

export class ZodValidationPipe {
  constructor(private schema) {}

  transform(value, metadata) {
    try {
      const parsedValue = this.schema.parse(value);
      return parsedValue;
    } catch (error) {
      throw new BadRequestException('Validation failed');
    }
  }
}

Binding validation pipes #

Trước đó, chúng ta đã thấy cách bind transformation pipe (như ParseIntPipe và các Parse* pipe còn lại).

Binding validation pipe cũng rất đơn giản.

Trong trường hợp này, chúng ta muốn bind pipe ở cấp method call. Trong ví dụ hiện tại, chúng ta cần thực hiện các bước sau để dùng ZodValidationPipe:

  1. Tạo instance của ZodValidationPipe
  2. Truyền context-specific Zod schema vào class constructor của pipe
  3. Bind pipe vào method

Ví dụ Zod schema:

import { z } from 'zod';

export const createCatSchema = z
  .object({
    name:  z.string(),
    age:   z.number(),
    breed: z.string(),
  })
  .required();

export type CreateCatDto = z.infer<typeof createCatSchema>;

Chúng ta thực hiện bằng decorator @UsePipes() như bên dưới:

cats.controller
TS
JS
@Post()
@UsePipes(new ZodValidationPipe(createCatSchema))
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
@Bind(Body())
@UsePipes(new ZodValidationPipe(createCatSchema))
async create(createCatDto) {
  this.catsService.create(createCatDto);
}
Hint

Decorator @UsePipes() được import từ package @nestjs/common.

Warning

Thư viện zod yêu cầu bật cấu hình strictNullChecks trong file tsconfig.json.

Class validator #

Warning

Các kỹ thuật trong phần này yêu cầu TypeScript và không khả dụng nếu ứng dụng của bạn được viết bằng vanilla JavaScript.

Hãy xem một implementation thay thế cho kỹ thuật validation của chúng ta.

Nest hoạt động tốt với thư viện class-validator. Thư viện mạnh mẽ này cho phép bạn dùng decorator-based validation. Decorator-based validation đặc biệt mạnh mẽ, nhất là khi kết hợp với khả năng Pipe của Nest vì chúng ta có quyền truy cập vào metatype của property đang xử lý. Trước khi bắt đầu, cần cài đặt các package cần thiết:

$ npm i --save class-validator class-transformer

Sau khi cài đặt, chúng ta có thể thêm một số decorator vào class CreateCatDto. Ở đây chúng ta thấy một ưu điểm đáng kể của kỹ thuật này: class CreateCatDto vẫn là nguồn sự thật duy nhất cho Post body object (thay vì phải tạo một validation class riêng).

create-cat.dto
import { IsString, IsInt } from 'class-validator';

export class CreateCatDto {
  @IsString()
  name: string;

  @IsInt()
  age: number;

  @IsString()
  breed: string;
}
Hint

Đọc thêm về các class-validator decorator tại đây.

Bây giờ chúng ta có thể tạo class ValidationPipe sử dụng các annotation này.

validation.pipe
import { PipeTransform, Injectable, ArgumentMetadata, BadRequestException } from '@nestjs/common';
import { validate } from 'class-validator';
import { plainToInstance } from 'class-transformer';

@Injectable()
export class ValidationPipe implements PipeTransform<any> {
  async transform(value: any, { metatype }: ArgumentMetadata) {
    if (!metatype || !this.toValidate(metatype)) {
      return value;
    }
    const object = plainToInstance(metatype, value);
    const errors = await validate(object);
    if (errors.length > 0) {
      throw new BadRequestException('Validation failed');
    }
    return value;
  }

  private toValidate(metatype: Function): boolean {
    const types: Function[] = [String, Boolean, Number, Array, Object];
    return !types.includes(metatype);
  }
}
Hint

Nhắc nhở, bạn không cần tự build generic validation pipe vì ValidationPipe đã được Nest cung cấp sẵn. Built-in ValidationPipe cung cấp nhiều option hơn ví dụ chúng ta tự build trong chương này, được giữ ở mức cơ bản để minh họa cơ chế hoạt động của custom pipe. Bạn có thể tìm chi tiết đầy đủ cùng nhiều ví dụ tại đây.

Notice

Chúng ta đã dùng thư viện class-transformer ở trên, được tạo bởi cùng tác giả với class-validator, và do đó chúng phối hợp rất tốt với nhau.

Hãy đi qua đoạn code này. Đầu tiên, lưu ý rằng method transform() được đánh dấu async. Điều này có thể vì Nest hỗ trợ cả pipe đồng bộ lẫn bất đồng bộ. Chúng ta đặt method này là async vì một số class-validator validation có thể là async (sử dụng Promise).

Tiếp theo, lưu ý rằng chúng ta dùng destructuring để trích xuất trường metatype (chỉ trích xuất member này từ ArgumentMetadata) vào parameter metatype. Đây chỉ là cách viết tắt cho việc lấy toàn bộ ArgumentMetadata rồi có thêm câu lệnh gán biến metatype.

Tiếp theo, lưu ý helper function toValidate(). Nó chịu trách nhiệm bỏ qua bước validation khi argument hiện tại đang xử lý là native JavaScript type (những type này không thể gắn validation decorator, nên không có lý do chạy chúng qua bước validation).

Tiếp theo, chúng ta dùng function plainToInstance() của class-transformer để transform plain JavaScript argument object thành typed object để có thể áp dụng validation. Lý do phải làm vậy là incoming post body object, khi được deserialize từ network request, không có bất kỳ type information nào (đây là cách nền tảng underlying như Express hoạt động). Class-validator cần sử dụng các validation decorator mà chúng ta đã định nghĩa cho DTO trước đó, nên cần thực hiện transformation này để xử lý incoming body như một object được decorate đúng cách, chứ không phải một plain vanilla object.

Cuối cùng, như đã lưu ý trước đó, vì đây là validation pipe nên nó hoặc trả về giá trị không đổi, hoặc throw exception.

Bước cuối cùng là bind ValidationPipe. Pipe có thể là parameter-scoped, method-scoped, controller-scoped, hoặc global-scoped. Trước đó, với Zod-based validation pipe, chúng ta đã thấy ví dụ binding pipe ở cấp method. Trong ví dụ bên dưới, chúng ta sẽ bind pipe instance vào decorator @Body() của route handler để pipe được gọi để validate post body.

cats.controller
@Post()
async create(
  @Body(new ValidationPipe()) createCatDto: CreateCatDto,
) {
  this.catsService.create(createCatDto);
}

Parameter-scoped pipe hữu ích khi logic validation chỉ liên quan đến một parameter cụ thể.

Global scoped pipes #

Vì ValidationPipe được tạo để generic nhất có thể, chúng ta có thể phát huy tối đa lợi ích của nó bằng cách thiết lập như global-scoped pipe để nó được áp dụng cho mọi route handler trong toàn bộ ứng dụng.

main
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ValidationPipe());
  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
Notice

Trong trường hợp hybrid app, method useGlobalPipes() không thiết lập pipe cho gateway và microservice. Với ứng dụng microservice "tiêu chuẩn" (non-hybrid), useGlobalPipes() sẽ mount pipe globally.

Global pipe được sử dụng trong toàn bộ ứng dụng, cho mọi controller và mọi route handler.

Lưu ý rằng về mặt dependency injection, global pipe được đăng ký từ bên ngoài bất kỳ module nào (với useGlobalPipes() như ví dụ trên) không thể inject dependency vì binding được thực hiện ngoài context của bất kỳ module nào. Để giải quyết vấn đề này, bạn có thể thiết lập global pipe trực tiếp từ bất kỳ module nào bằng cấu trúc sau:

app.module
import { Module } from '@nestjs/common';
import { APP_PIPE } from '@nestjs/core';

@Module({
  providers: [
    {
      provide:  APP_PIPE,
      useClass: ValidationPipe,
    },
  ],
})
export class AppModule {}
Hint

Khi sử dụng cách tiếp cận này để thực hiện dependency injection cho pipe, lưu ý rằng bất kể module nào sử dụng cấu trúc này, pipe thực tế là global. Nên thực hiện ở đâu? Chọn module nơi pipe (ValidationPipe trong ví dụ trên) được định nghĩa. Ngoài ra, useClass không phải là cách duy nhất để đăng ký custom provider. Tìm hiểu thêm tại đây.

Transformation use case #

Validation không phải là use case duy nhất cho custom pipe. Ở đầu chương này, chúng ta đã đề cập rằng pipe cũng có thể transform input data sang format mong muốn. Điều này có thể vì value trả về từ hàm transform hoàn toàn ghi đè giá trị trước đó của argument.

Khi nào điều này hữu ích? Xem xét rằng đôi khi dữ liệu từ client cần trải qua một số thay đổi — ví dụ chuyển string thành integer — trước khi có thể được xử lý đúng cách bởi route handler method. Hơn nữa, một số trường dữ liệu bắt buộc có thể bị thiếu, và chúng ta muốn áp dụng giá trị mặc định. Transformation pipe có thể thực hiện các chức năng này bằng cách đặt một hàm xử lý giữa client request và request handler.

Đây là ParseIntPipe đơn giản chịu trách nhiệm parse string thành giá trị integer. (Như đã lưu ý ở trên, Nest có built-in ParseIntPipe phức tạp hơn; chúng ta bao gồm ví dụ này như một custom transformation pipe đơn giản).

parse-int.pipe
TS
JS
import { PipeTransform, Injectable, ArgumentMetadata, BadRequestException } from '@nestjs/common';

@Injectable()
export class ParseIntPipe implements PipeTransform<string, number> {
  transform(value: string, metadata: ArgumentMetadata): number {
    const val = parseInt(value, 10);
    if (isNaN(val)) {
      throw new BadRequestException('Validation failed');
    }
    return val;
  }
}

Sau đó chúng ta có thể bind pipe này vào param đã chọn như bên dưới:

TS
JS
@Get(':id')
async findOne(@Param('id', new ParseIntPipe()) id) {
  return this.catsService.findOne(id);
}
@Get(':id')
@Bind(Param('id', new ParseIntPipe()))
async findOne(id) {
  return this.catsService.findOne(id);
}

Một trường hợp transformation hữu ích khác là chọn một existing user entity từ database bằng id được cung cấp trong request:

TS
JS
@Get(':id')
findOne(@Param('id', UserByIdPipe) userEntity: UserEntity) {
  return userEntity;
}
@Get(':id')
@Bind(Param('id', UserByIdPipe))
findOne(userEntity) {
  return userEntity;
}

Chúng ta để việc implement pipe này cho người đọc, nhưng lưu ý rằng giống như tất cả transformation pipe khác, nó nhận input value (một id) và trả về output value (một UserEntity object). Điều này có thể làm code của bạn trở nên declarative và DRY hơn bằng cách trừu tượng hoá boilerplate code ra khỏi handler và đưa vào một pipe chung.

Providing defaults #

Các Parse* pipe kỳ vọng giá trị của parameter phải được định nghĩa. Chúng throw exception khi nhận giá trị null hoặc undefined. Để cho phép endpoint xử lý các giá trị querystring parameter bị thiếu, chúng ta phải cung cấp giá trị mặc định trước khi các Parse* pipe hoạt động trên các giá trị này. DefaultValuePipe phục vụ mục đích đó. Chỉ cần khởi tạo DefaultValuePipe trong decorator @Query() trước Parse* pipe liên quan, như bên dưới:

@Get()
async findAll(
  @Query('activeOnly', new DefaultValuePipe(false), ParseBoolPipe) activeOnly: boolean,
  @Query('page', new DefaultValuePipe(0), ParseIntPipe) page: number,
) {
  return this.catsService.findAll({ activeOnly, page });
}
