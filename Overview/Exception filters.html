Exception filters

Nest đi kèm với một exceptions layer tích hợp sẵn, chịu trách nhiệm xử lý tất cả exception chưa được xử lý trong toàn bộ ứng dụng. Khi một exception không được xử lý bởi code ứng dụng, nó sẽ bị bắt bởi layer này, sau đó tự động gửi về một response thân thiện phù hợp cho người dùng.

Exception layer tự động bắt và xử lý tất cả exception chưa được xử lý

Theo mặc định, tác vụ này được thực hiện bởi global exception filter tích hợp sẵn, xử lý các exception thuộc type HttpException (và các subclass của nó). Khi một exception không được nhận dạng (không phải HttpException và không kế thừa từ HttpException), exception filter tích hợp sẽ tạo ra JSON response mặc định sau:

Default error response
{
  "statusCode": 500,
  "message": "Internal server error"
}
💡 Hint

Global exception filter có hỗ trợ một phần thư viện http-errors. Về cơ bản, bất kỳ exception nào được throw mà chứa các property statusCode và message sẽ được điền đúng thông tin và gửi về như response (thay vì InternalServerErrorException mặc định cho các exception không nhận dạng được).

Throwing standard exceptions #

Nest cung cấp class HttpException tích hợp sẵn, được export từ package @nestjs/common. Đối với các ứng dụng HTTP REST/GraphQL API điển hình, best practice là gửi các HTTP response object chuẩn khi có điều kiện lỗi nhất định xảy ra.

Ví dụ: trong CatsController có method findAll() (một GET route handler). Giả sử route handler này throw một exception vì lý do nào đó. Để minh hoạ, chúng ta hard-code như sau:

cats.controller.ts
TS
JS
@Get()
async findAll() {
  throw new HttpException('Forbidden', HttpStatus.FORBIDDEN);
}
@Get()
async findAll() {
  throw new HttpException('Forbidden', HttpStatus.FORBIDDEN);
}
💡 Hint

Chúng ta dùng HttpStatus ở đây — một helper enum được import từ package @nestjs/common.

Khi client gọi endpoint này, response trông như sau:

{
  "statusCode": 403,
  "message": "Forbidden"
}

Constructor của HttpException nhận hai argument bắt buộc để xác định response:

  • Argument response — định nghĩa JSON response body. Có thể là string hoặc object.
  • Argument status — định nghĩa HTTP status code.

Theo mặc định, JSON response body chứa hai property: statusCode (mặc định theo HTTP status code trong argument status) và message (mô tả ngắn về HTTP error dựa trên status).

Để chỉ override phần message, truyền một string vào argument response. Để override toàn bộ JSON response body, truyền một object. Nest sẽ serialize object và trả về như JSON response body.

Còn có argument constructor thứ ba (optional) — options — có thể dùng để cung cấp error cause. Object cause không được serialize vào response object, nhưng hữu ích cho mục đích logging.

Đây là ví dụ override toàn bộ response body và cung cấp error cause:

cats.controller.ts
TS
JS
@Get()
async findAll() {
  try {
    await this.service.findAll()
  } catch (error) {
    throw new HttpException({
      status: HttpStatus.FORBIDDEN,
      error: 'This is a custom message',
    }, HttpStatus.FORBIDDEN, {
      cause: error
    });
  }
}
@Get()

  try {
    await this.service.findAll()
  } catch (error) {
    throw new HttpException({
      status: HttpStatus.FORBIDDEN,
      error: 'This is a custom message',
    }, HttpStatus.FORBIDDEN, {
      cause: error
    });
  }
}

Với cách trên, response sẽ trông như sau:

{
  "status": 403,
  "error": "This is a custom message"
}

Exceptions logging #

Theo mặc định, exception filter không log các exception tích hợp như HttpException (và các exception kế thừa từ nó). Khi các exception này được throw, chúng sẽ không xuất hiện trong console vì chúng được coi là một phần của luồng ứng dụng bình thường. Hành vi tương tự áp dụng cho các exception tích hợp khác như WsException và RpcException.

Tất cả các exception này đều kế thừa từ base class IntrinsicException, được export từ package @nestjs/common. Class này giúp phân biệt các exception là một phần của vận hành ứng dụng bình thường với các exception không phải vậy.

Nếu muốn log các exception này, bạn có thể tạo một custom exception filter — sẽ được giải thích trong phần tiếp theo.

Custom exceptions #

Trong nhiều trường hợp, bạn sẽ không cần viết custom exception mà có thể dùng Nest HTTP exception tích hợp sẵn. Nếu cần tạo exception tuỳ biến, practice tốt là tạo exceptions hierarchy riêng, nơi các custom exception kế thừa từ base class HttpException. Với cách tiếp cận này, Nest sẽ nhận dạng exception và tự động xử lý error response. Hãy implement một custom exception như sau:

forbidden.exception.ts
TS
JS
export class ForbiddenException extends HttpException {
  constructor() {
    super('Forbidden', HttpStatus.FORBIDDEN);
  }
}
export class ForbiddenException extends HttpException {
  constructor() {
    super('Forbidden', HttpStatus.FORBIDDEN);
  }
}

Vì ForbiddenException extend base HttpException, nó sẽ hoạt động liền mạch với exception handler tích hợp sẵn, và do đó chúng ta có thể dùng nó bên trong method findAll().

cats.controller.ts
TS
JS
@Get()
async findAll() {
  throw new ForbiddenException();
}
@Get()

  throw new ForbiddenException();
}

Tất cả built-in exception đều có thể cung cấp cả error cause và error description thông qua parameter options:

  "error": "Some error description",

}

Exception filters #

Trong khi exception filter base (tích hợp sẵn) có thể tự động xử lý nhiều trường hợp, bạn có thể muốn kiểm soát hoàn toàn exceptions layer. Ví dụ: thêm logging hoặc dùng JSON schema khác dựa trên một số yếu tố động. Exception filter được thiết kế chính xác cho mục đích này — chúng cho phép kiểm soát luồng xử lý và nội dung response gửi về client.

Hãy tạo một exception filter chịu trách nhiệm bắt exception là instance của class HttpException và implement logic response tuỳ biến cho chúng. Để làm điều này, cần truy cập vào các đối tượng platform Request và Response.

http-exception.filter.ts
TS
JS
import { ExceptionFilter, Catch, ArgumentsHost, HttpException } from '@nestjs/common';
import { Request, Response } from 'express';

@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const ctx      = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request  = ctx.getRequest<Request>();
    const status   = exception.getStatus();

    response
      .status(status)
      .json({
        statusCode: status,
        timestamp: new Date().toISOString(),
        path: request.url,
      });
  }
}
import { Catch, HttpException } from '@nestjs/common';

@Catch(HttpException)
export class HttpExceptionFilter {
  catch(exception, host) {
    const ctx      = host.switchToHttp();
    const response = ctx.getResponse();
    const request  = ctx.getRequest();
    const status   = exception.getStatus();

    response
      .status(status)
      .json({
        statusCode: status,
        timestamp: new Date().toISOString(),
        path: request.url,
      });
  }
}
💡 Hint

Tất cả exception filter nên implement generic interface ExceptionFilter<T>. Điều này yêu cầu cung cấp method catch(exception: T, host: ArgumentsHost) với chữ ký đã chỉ định. T biểu thị type của exception.

⚠️ Warning

Nếu dùng @nestjs/platform-fastify, có thể dùng response.send() thay vì response.json(). Đừng quên import đúng type từ fastify.

Decorator @Catch(HttpException) gắn metadata cần thiết vào exception filter, báo cho Nest biết rằng filter cụ thể này tìm kiếm exception thuộc type HttpException và không gì khác. Decorator @Catch() có thể nhận một parameter hoặc danh sách phân cách bởi dấu phẩy — cho phép thiết lập filter cho nhiều type exception cùng lúc.

Arguments host #

Hãy xem xét các parameter của method catch(). Parameter exception là đối tượng exception đang được xử lý. Parameter host là đối tượng ArgumentsHost — một utility object mạnh mẽ mà chúng ta sẽ tìm hiểu thêm trong chương execution context.

Trong code mẫu này, chúng ta dùng nó để lấy tham chiếu đến các đối tượng Request và Response đang được truyền đến request handler gốc (trong controller nơi exception phát sinh). Chúng ta dùng một số helper method trên ArgumentsHost để lấy các đối tượng Request và Response mong muốn.

Lý do cho mức độ trừu tượng này là ArgumentsHost hoạt động trong mọi context (ví dụ: HTTP server context mà chúng ta đang làm việc, nhưng cũng cả Microservices và WebSockets). Trong chương execution context, chúng ta sẽ thấy cách truy cập các argument cơ bản phù hợp cho bất kỳ execution context nào — điều này cho phép viết các generic exception filter hoạt động trên tất cả context.

Binding filters #

Hãy gắn HttpExceptionFilter mới vào method create() của CatsController.

cats.controller.ts
TS
JS
@Post()
@UseFilters(new HttpExceptionFilter())
async create(@Body() createCatDto: CreateCatDto) {
  throw new ForbiddenException();
}
@Post()
@UseFilters(new HttpExceptionFilter())
@Bind(Body())
async create(createCatDto) {
  throw new ForbiddenException();
}
💡 Hint

Decorator @UseFilters() được import từ package @nestjs/common.

Ở đây chúng ta dùng decorator @UseFilters(). Tương tự @Catch(), nó có thể nhận một filter instance hoặc danh sách phân cách bởi dấu phẩy. Ngoài ra, có thể truyền class (thay vì instance), để framework chịu trách nhiệm khởi tạo và cho phép dependency injection.

Global filter được đăng ký từ bên ngoài bất kỳ module nào (với useGlobalFilters()) không thể inject dependency vì điều này được thực hiện ngoài context của bất kỳ module nào. Để giải quyết vấn đề này, bạn có thể đăng ký global-scoped filter trực tiếp từ bất kỳ module nào bằng cách sau:

Nên thực hiện điều này ở module nơi filter được định nghĩa. Ngoài ra, useClass không phải là cách duy nhất để đăng ký custom provider. Tìm hiểu thêm tại đây.

Catch everything #

Để bắt mọi exception chưa được xử lý (bất kể type exception), hãy để trống danh sách parameter của decorator @Catch(), ví dụ: @Catch().

Trong ví dụ dưới, code độc lập với platform vì nó dùng HTTP adapter để deliver response và không trực tiếp dùng bất kỳ đối tượng platform-specific nào (Request và Response):

import {
  ExceptionFilter, Catch, ArgumentsHost,
  HttpException, HttpStatus,
} from '@nestjs/common';
import { HttpAdapterHost } from '@nestjs/core';

@Catch()
export class CatchEverythingFilter implements ExceptionFilter {
  constructor(private readonly httpAdapterHost: HttpAdapterHost) {}

  catch(exception: unknown, host: ArgumentsHost): void {
    // httpAdapter có thể chưa có trong constructor, resolve ở đây
    const { httpAdapter } = this.httpAdapterHost;
    const ctx = host.switchToHttp();

    const httpStatus =
      exception instanceof HttpException
        ? exception.getStatus()
        : HttpStatus.INTERNAL_SERVER_ERROR;

    const responseBody = {
      statusCode: httpStatus,
      timestamp: new Date().toISOString(),
      path: httpAdapter.getRequestUrl(ctx.getRequest()),
    };

    httpAdapter.reply(ctx.getResponse(), responseBody, httpStatus);
  }
}
⚠️ Warning

Khi kết hợp exception filter bắt tất cả với một filter được gắn vào type cụ thể, filter "Catch anything" nên được khai báo trước để cho phép filter cụ thể xử lý đúng type được gắn.

Inheritance #

Thông thường, bạn sẽ tạo các exception filter tuỳ biến hoàn toàn để đáp ứng yêu cầu ứng dụng. Tuy nhiên, có những trường hợp bạn chỉ muốn extend global exception filter mặc định tích hợp sẵn và override hành vi dựa trên một số yếu tố nhất định.

Để uỷ thác xử lý exception cho base filter, cần extend BaseExceptionFilter và gọi method catch() được kế thừa:

all-exceptions.filter.ts
TS
JS
import { Catch, ArgumentsHost } from '@nestjs/common';
import { BaseExceptionFilter } from '@nestjs/core';

@Catch()
export class AllExceptionsFilter extends BaseExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    super.catch(exception, host);
  }
}
import { Catch } from '@nestjs/common';
import { BaseExceptionFilter } from '@nestjs/core';

@Catch()
export class AllExceptionsFilter extends BaseExceptionFilter {
  catch(exception, host) {
    super.catch(exception, host);
  }
}
⚠️ Warning

Method-scoped và Controller-scoped filter extend BaseExceptionFilter không nên được khởi tạo với new. Thay vào đó, hãy để framework tự động khởi tạo chúng.

Global filter có thể extend base filter. Điều này có thể thực hiện theo một trong hai cách. Cách thứ nhất là inject tham chiếu HttpAdapter khi khởi tạo custom global filter:

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  const { httpAdapter } = app.get(HttpAdapterHost);
  app.useGlobalFilters(new AllExceptionsFilter(httpAdapter));

  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();

Cách thứ hai là dùng token APP_FILTER như đã trình bày ở trên.
