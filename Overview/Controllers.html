Routing mechanism điều hướng request đến đúng controller

Controller có trách nhiệm xử lý các request đến và trả về response cho client.

Mục đích của một controller là nhận các request cụ thể cho ứng dụng. Cơ chế routing quyết định controller nào sẽ xử lý mỗi request. Thông thường, mỗi controller có nhiều route, và mỗi route có thể thực hiện các hành động khác nhau.

Để tạo một controller cơ bản, chúng ta dùng class và decorator. Decorator liên kết class với metadata cần thiết, cho phép Nest tạo routing map kết nối request với controller tương ứng.

💡 Hint

Để nhanh chóng tạo CRUD controller với validation tích hợp sẵn, dùng CRUD generator của CLI: nest g resource [name].

Routing #

Trong ví dụ dưới, chúng ta dùng decorator @Controller() — thứ bắt buộc để định nghĩa một controller cơ bản. Chúng ta chỉ định path prefix tuỳ chọn là cats. Dùng path prefix trong @Controller() giúp nhóm các route liên quan lại và giảm code lặp. Ví dụ, nếu muốn nhóm các route quản lý entity cat dưới path /cats, ta đặt prefix cats trong @Controller() — và không cần lặp lại đoạn path đó cho từng route trong file.

cats.controller.ts
TS
JS
import { Controller, Get } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Get()
  findAll(): string {
    return 'This action returns all cats';
  }
}
import { Controller, Get } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Get()
  findAll() {
    return 'This action returns all cats';
  }
}
💡 Hint

Để tạo controller bằng CLI, chạy lệnh $ nest g controller [name].

Decorator HTTP request method @Get() đặt trước method findAll() báo Nest tạo một handler cho endpoint HTTP cụ thể, xác định bởi HTTP method (GET) và route path. Route path của handler được xác định bằng cách kết hợp prefix (tuỳ chọn) của controller với path chỉ định trong decorator của method. Vì chúng ta đặt prefix cats và không thêm path trong decorator method, Nest map GET /cats đến handler này.

Ví dụ: nếu prefix controller là cats và decorator method là @Get('breed'), route kết quả sẽ là GET /cats/breed.

Trong ví dụ trên, khi có GET request đến endpoint này, Nest routes đến method findAll() do người dùng định nghĩa. Tên method hoàn toàn tuỳ ý — Nest không gắn ý nghĩa cụ thể nào cho tên method.

Method này trả về status code 200 và response liên quan — ở đây là một string. Để giải thích tại sao, cần giới thiệu hai tuỳ chọn Nest dùng để thao tác response:

Phương thứcMô tả
Standard (khuyến nghị) Với phương thức tích hợp này, khi request handler trả về JavaScript object hoặc array, nó tự động được serialize sang JSON. Khi trả về kiểu JavaScript primitive (string, number, boolean), Nest gửi giá trị đó mà không serialize. Điều này giúp xử lý response đơn giản: chỉ cần return giá trị, Nest lo phần còn lại.

Hơn nữa, status code mặc định luôn là 200, ngoại trừ POST request dùng 201. Có thể thay đổi bằng decorator @HttpCode(...) ở cấp handler.
Library-specific Có thể dùng response object đặc thù của thư viện (như Express), inject bằng decorator @Res() trong chữ ký method handler (ví dụ: findAll(@Res() response)). Với cách này, bạn dùng được các phương thức xử lý response native của object đó. Với Express: response.status(200).send().
⚠️ Warning

Nest phát hiện khi handler dùng @Res() hoặc @Next(), tức bạn đã chọn library-specific option. Nếu dùng cả hai cùng lúc, Standard approach sẽ tự động bị vô hiệu hoá cho route đó. Để dùng cả hai (ví dụ: inject response object chỉ để set cookies/headers nhưng vẫn để framework xử lý phần còn lại), phải set passthrough: true trong @Res({ passthrough: true }).

Request object #

Handler thường cần truy cập thông tin request từ client. Nest cung cấp quyền truy cập vào request object từ platform bên dưới (mặc định là Express). Inject nó bằng decorator @Req() trong chữ ký handler.

cats.controller.ts
TS
JS
import { Controller, Get, Req } from '@nestjs/common';
import type { Request } from 'express';

@Controller('cats')
export class CatsController {
  @Get()
  findAll(@Req() request: Request): string {
    return 'This action returns all cats';
  }
}
import { Controller, Bind, Get, Req } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Get()
  @Bind(Req())
  findAll(request) {
    return 'This action returns all cats';
  }
}
💡 Hint

Để tận dụng typings của express (như ví dụ request: Request), hãy cài đặt package @types/express.

Request object đại diện cho HTTP request và chứa các property cho query string, parameters, HTTP headers và body (xem thêm tại đây). Thông thường không cần truy cập thủ công — dùng các decorator chuyên dụng như @Body() hoặc @Query() thay thế. Dưới đây là danh sách decorator được cung cấp và platform-specific object tương ứng:

DecoratorPlatform Object tương ứng
@Request(), @Req()req
@Response(), @Res()*res
@Next()next
@Session()req.session
@Param(key?: string)req.params / req.params[key]
@Body(key?: string)req.body / req.body[key]
@Query(key?: string)req.query / req.query[key]
@Headers(name?: string)req.headers / req.headers[name]
@Ip()req.ip
@HostParam()req.hosts

* Để tương thích với typings giữa các HTTP platform (Express và Fastify), Nest cung cấp @Res() và @Response() — @Res() chỉ là alias. Cả hai expose native platform response object trực tiếp. Khi inject @Res() hoặc @Response() trong method handler, Nest vào Library-specific mode và bạn chịu trách nhiệm quản lý response bằng cách gọi res.json(...) hoặc res.send(...), nếu không HTTP server sẽ bị treo.

💡 Hint

Tìm hiểu cách tạo custom decorator, xem chương này.

Resources #

Trước đó, chúng ta định nghĩa endpoint để fetch resource cats (GET route). Thông thường cũng cần endpoint tạo bản ghi mới. Hãy tạo POST handler:

cats.controller.ts
TS
JS
import { Controller, Get, Post } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Post()
  create(): string {
    return 'This action adds a new cat';
  }

  @Get()
  findAll(): string {
    return 'This action returns all cats';
  }
}
import { Controller, Get, Post } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Post()
  create() {
    return 'This action adds a new cat';
  }

  @Get()
  findAll() {
    return 'This action returns all cats';
  }
}

Nest cung cấp decorator cho tất cả HTTP method tiêu chuẩn: @Get(), @Post(), @Put(), @Delete(), @Patch(), @Options(), và @Head(). Ngoài ra, @All() định nghĩa endpoint xử lý tất cả chúng.

Route wildcards #

NestJS cũng hỗ trợ pattern-based route. Ví dụ, dấu hoa thị (*) dùng như wildcard để match bất kỳ tổ hợp ký tự nào ở cuối path. Method findAll() dưới đây sẽ được thực thi với bất kỳ route nào bắt đầu bằng abcd/:

@Get('abcd/*')
findAll() {
  return 'This route uses a wildcard';
}

Path 'abcd/*' sẽ match abcd/, abcd/123, abcd/abc, v.v. Dấu gạch ngang (-) và dấu chấm (.) được hiểu theo nghĩa đen trong string-based path.

Cách này hoạt động trên cả Express và Fastify. Tuy nhiên, với Express v5, hệ thống routing đã nghiêm ngặt hơn. Trong Express thuần, bạn phải dùng named wildcard — ví dụ: abcd/*splat (có thể đặt tên bất kỳ). Dù vậy, Nest cung cấp compatibility layer nên vẫn dùng được dấu * thuần.

Đối với dấu hoa thị ở giữa route, Express yêu cầu named wildcard (ví dụ: ab{*splat}cd), trong khi Fastify không hỗ trợ chúng.

Status code #

Status code mặc định của response luôn là 200, ngoại trừ POST request mặc định là 201. Dùng decorator @HttpCode(...) ở cấp handler để thay đổi:

@Post()
@HttpCode(204)
create() {
  return 'This action adds a new cat';
}
💡 Hint

Import HttpCode từ package @nestjs/common.

Nếu status code không cố định mà phụ thuộc vào nhiều yếu tố, dùng library-specific response object (inject bằng @Res()), hoặc throw exception.

Response headers #

Để chỉ định custom response header, dùng decorator @Header() hoặc response object đặc thù của thư viện (gọi res.header() trực tiếp).

@Post()
@Header('Cache-Control', 'no-store')
create() {
  return 'This action adds a new cat';
}
💡 Hint

Import Header từ package @nestjs/common.

Redirection #

Để redirect response đến URL cụ thể, dùng decorator @Redirect() hoặc res.redirect(). @Redirect() nhận hai tham số tuỳ chọn: url và statusCode. Mặc định statusCode là 302 (Found).

@Get()
@Redirect('https://nestjs.com', 301)
💡 Hint

Đôi khi cần xác định HTTP status code hoặc redirect URL một cách động. Thực hiện điều này bằng cách return một object theo interface HttpRedirectResponse (từ @nestjs/common).

Giá trị được return sẽ override tham số truyền vào @Redirect(). Ví dụ:

@Get('docs')
@Redirect('https://docs.nestjs.com', 302)
getDocs(@Query('version') version) {
  if (version && version === '5') {
    return { url: 'https://docs.nestjs.com/v5/' };
  }
}

Route parameters #

Route với static path không hoạt động khi cần nhận dữ liệu động (ví dụ: GET /cats/1 để lấy cat có id 1). Để định nghĩa route với parameter, thêm token route parameter vào route path. Các token này có thể truy cập bằng decorator @Param() trong chữ ký method.

💡 Hint

Route với parameter nên khai báo sau bất kỳ static path nào, tránh parameterized path chặn traffic dành cho static path.

TS
JS
@Get(':id')
findOne(@Param() params: any): string {
  console.log(params.id);
  return `This action returns a #${params.id} cat`;
}
@Get(':id')
@Bind(Param())
findOne(params) {
  console.log(params.id);
  return `This action returns a #${params.id} cat`;
}

Decorator @Param() decorate một tham số method (ở trên là params), giúp route parameter truy cập được như property của tham số đó. Bạn có thể truy cập id qua params.id. Hoặc truyền token cụ thể vào decorator và tham chiếu trực tiếp:

💡 Hint

Import Param từ package @nestjs/common.

TS
JS
@Get(':id')
findOne(@Param('id') id: string): string {
  return `This action returns a #${id} cat`;
}
@Get(':id')
@Bind(Param('id'))
findOne(id) {
  return `This action returns a #${id} cat`;
}

Sub-domain routing #

Decorator @Controller có thể nhận tuỳ chọn host để yêu cầu HTTP host của incoming request phải khớp với một giá trị cụ thể.

@Controller({ host: 'admin.example.com' })
export class AdminController {
  @Get()
  index(): string {
    return 'Admin page';
  }
}
⚠️ Warning

Vì Fastify không hỗ trợ nested router, nếu dùng sub-domain routing, hãy dùng Express adapter mặc định.

Tương tự route path, tuỳ chọn host có thể dùng token để bắt giá trị động trong host name. Host parameter có thể truy cập bằng decorator @HostParam():

@Controller({ host: ':account.example.com' })
export class AccountController {
  @Get()
  getInfo(@HostParam('account') account: string) {
    return account;
  }
}

State sharing #

Với developer đến từ ngôn ngữ khác, có thể ngạc nhiên khi biết rằng trong Nest, hầu hết mọi thứ được chia sẻ giữa các incoming request — bao gồm database connection pool, singleton service với global state, v.v. Node.js không dùng mô hình Multi-Threaded Stateless, nơi mỗi request được xử lý bởi thread riêng. Do đó, dùng singleton instance trong Nest hoàn toàn an toàn.

Dù vậy, có edge case mà request-based lifetime cho controller có thể cần thiết: per-request caching trong GraphQL, request tracking, hay multi-tenancy. Xem thêm về kiểm soát injection scope tại đây.

Asynchronicity #

Nest hỗ trợ đầy đủ hàm async. Mọi hàm async phải return một Promise, cho phép return giá trị deferred mà Nest tự động resolve. Ví dụ:

cats.controller.ts
TS
JS
@Get()
async findAll(): Promise<any[]> {
  return [];
}
@Get()
async findAll() {
  return [];
}

Code này hoàn toàn hợp lệ. Nest còn cho phép route handler return RxJS observable stream. Nest xử lý subscription nội bộ và resolve giá trị emit cuối cùng khi stream hoàn tất.

cats.controller.ts
TS
JS
@Get()
findAll(): Observable<any[]> {
  return of([]);
}
@Get()
findAll() {
  return of([]);
}

Cả hai cách đều hợp lệ — chọn cái phù hợp nhất với nhu cầu của bạn.

Request payloads #

Ví dụ POST route handler trước đây không nhận tham số nào. Hãy sửa bằng cách thêm decorator @Body().

Khi dùng TypeScript, cần định nghĩa schema DTO (Data Transfer Object) — object chỉ định cách dữ liệu gửi qua mạng. Nên dùng class thay vì TypeScript interface. Tại sao? Class là chuẩn JavaScript ES6, tồn tại trong compiled JavaScript. Interface TypeScript bị loại bỏ trong quá trình transpilation — Nest không thể tham chiếu chúng tại runtime. Điều này quan trọng vì các tính năng như Pipes cần truy cập metatype của biến tại runtime, chỉ khả thi với class.

Tạo class CreateCatDto:

create-cat.dto.ts
export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}

Chỉ có ba property cơ bản. Sau đó dùng DTO này bên trong CatsController:

cats.controller.ts
TS
JS
@Post()
async create(@Body() createCatDto: CreateCatDto) {
  return 'This action adds a new cat';
}
@Post()
@Bind(Body())
async create(createCatDto) {
  return 'This action adds a new cat';
}
💡 Hint

ValidationPipe có thể lọc các property không nên được method handler nhận. Chúng ta có thể whitelist các property chấp nhận được — bất kỳ property nào không trong whitelist sẽ bị loại tự động. Trong CreateCatDto, whitelist là name, age, và breed. Tìm hiểu thêm tại đây.

Query parameters #

Dùng decorator @Query() để trích xuất query parameter từ incoming request. Ví dụ filter danh sách cats theo age và breed:

cats.controller.ts
@Get()
async findAll(@Query('age') age: number, @Query('breed') breed: string) {
  return `This action returns all cats filtered by age: ${age} and breed: ${breed}`;
}

Ví dụ request GET /cats?age=2&breed=Persian sẽ cho age = 2 và breed = "Persian".

Nếu cần xử lý query parameter phức tạp hơn (nested object hoặc array):

?filter[where][name]=John&filter[where][age]=30
?item[]=1&item[]=2

Cần cấu hình HTTP adapter để dùng query parser phù hợp. Với Express, dùng extended parser:

const app = await NestFactory.create<NestExpressApplication>(AppModule);
app.set('query parser', 'extended');

Với Fastify, dùng tuỳ chọn querystringParser:

const app = await NestFactory.create<NestFastifyApplication>(
  AppModule,
  new FastifyAdapter({
    querystringParser: (str) => qs.parse(str),
  }),
);
💡 Hint

qs là querystring parser hỗ trợ nesting và array. Cài đặt bằng npm install qs.

Full resource sample #

Ví dụ dưới minh hoạ việc dùng nhiều decorator để tạo một controller cơ bản với các method truy cập và thao tác dữ liệu nội bộ.

cats.controller.ts
TS
JS
import { Controller, Get, Query, Post, Body, Put, Param, Delete } from '@nestjs/common';
import { CreateCatDto, UpdateCatDto, ListAllEntities } from './dto';

@Controller('cats')
export class CatsController {
  @Post()
  create(@Body() createCatDto: CreateCatDto) {
    return 'This action adds a new cat';
  }

  @Get()
  findAll(@Query() query: ListAllEntities) {
    return `This action returns all cats (limit: ${query.limit} items)`;
  }

  @Get(':id')
  findOne(@Param('id') id: string) {
    return `This action returns a #${id} cat`;
  }

  @Put(':id')
  update(@Param('id') id: string, @Body() updateCatDto: UpdateCatDto) {
    return `This action updates a #${id} cat`;
  }

  @Delete(':id')
  remove(@Param('id') id: string) {
    return `This action removes a #${id} cat`;
  }
}
import { Controller, Get, Query, Post, Body, Put, Param, Delete, Bind } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Post()
  @Bind(Body())
  create(createCatDto) {
    return 'This action adds a new cat';
  }

  @Get()
  @Bind(Query())
  findAll(query) {
    console.log(query);
    return `This action returns all cats (limit: ${query.limit} items)`;
  }

  @Get(':id')
  @Bind(Param('id'))
  findOne(id) {
    return `This action returns a #${id} cat`;
  }

  @Put(':id')
  @Bind(Param('id'), Body())
  update(id, updateCatDto) {
    return `This action updates a #${id} cat`;
  }

  @Delete(':id')
  @Bind(Param('id'))
  remove(id) {
    return `This action removes a #${id} cat`;
  }
}
💡 Hint

Nest CLI cung cấp generator (schematic) tự động tạo toàn bộ boilerplate code, cải thiện developer experience. Xem thêm tại đây.

Getting up and running #

Dù CatsController đã được định nghĩa đầy đủ, Nest vẫn chưa biết về nó và sẽ không tự tạo instance của class đó.

Controller phải luôn là một phần của module — đó là lý do chúng ta bao gồm mảng controllers trong @Module(). Vì chưa có module nào khác ngoài root AppModule, ta dùng nó để đăng ký CatsController:

app.module.ts
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';

@Module({
  controllers: [CatsController],
})
export class AppModule {}

Chúng ta gắn metadata vào module class bằng @Module(), và Nest có thể dễ dàng xác định controller nào cần được mount.

Library-specific approach #

Cho đến nay chúng ta đã đề cập cách Nest tiêu chuẩn để thao tác response. Một cách khác là dùng response object đặc thù của thư viện, inject bằng decorator @Res(). Hãy viết lại CatsController:

TS
JS
import { Controller, Get, Post, Res, HttpStatus } from '@nestjs/common';
import { Response } from 'express';

@Controller('cats')
export class CatsController {
  @Post()
  create(@Res() res: Response) {
    res.status(HttpStatus.CREATED).send();
  }

  @Get()
  findAll(@Res() res: Response) {
    res.status(HttpStatus.OK).json([]);
  }
}
import { Controller, Get, Post, Bind, Res, Body, HttpStatus } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Post()
  @Bind(Res(), Body())
  create(res, createCatDto) {
    res.status(HttpStatus.CREATED).send();
  }

  @Get()
  @Bind(Res())
  findAll(res) {
    res.status(HttpStatus.OK).json([]);
  }
}

Mặc dù cách này hoạt động và linh hoạt hơn (kiểm soát đầy đủ response object: thao tác header, truy cập tính năng đặc thù của thư viện), nó nên được dùng cẩn thận. Nhược điểm chính: code trở nên phụ thuộc vào platform (các thư viện khác nhau có API response khác nhau), và việc testing trở nên khó hơn vì cần mock response object.

Hơn nữa, bạn mất tính tương thích với các tính năng Nest dựa vào xử lý response tiêu chuẩn như Interceptor và các decorator @HttpCode() / @Header(). Để giải quyết, bật tuỳ chọn passthrough:

TS
JS
@Get()
findAll(@Res({ passthrough: true }) res: Response) {
  res.status(HttpStatus.OK);
  return [];
}
@Get()
@Bind(Res({ passthrough: true }))
findAll(res) {
  res.status(HttpStatus.OK);
  return [];
}

Với cách này, bạn có thể tương tác với native response object (set cookie hoặc header theo điều kiện cụ thể), đồng thời vẫn để framework xử lý phần còn lại.
