Guards

Guard là một class được annotate bằng decorator @Injectable(), implement interface CanActivate.

Middleware
auth, logger…
→

canActivate()
→
Interceptor
+ Pipe
→
Route Handler
@Get() @Post()…
Guard thực thi SAU tất cả middleware nhưng TRƯỚC bất kỳ interceptor hay pipe nào

Guard có một trách nhiệm duy nhất: xác định xem một request đã cho có được route handler xử lý hay không, dựa trên các điều kiện nhất định (như permission, role, ACL…) có mặt tại runtime. Điều này thường được gọi là authorization.

Authorization (và người anh em authentication, thường phối hợp cùng) thường được xử lý bởi middleware trong các ứng dụng Express truyền thống. Middleware là lựa chọn tốt cho authentication vì những thứ như token validation và attach property vào request không gắn chặt với route context cụ thể.

Nhưng middleware, về bản chất, là "ngu ngốc". Nó không biết handler nào sẽ được thực thi sau khi gọi next(). Ngược lại, Guard có quyền truy cập vào instance ExecutionContext và do đó biết chính xác điều gì sẽ được thực thi tiếp theo. Guard được thiết kế giống như exception filter, pipe và interceptor — cho phép bạn xen vào logic xử lý tại đúng điểm trong chu kỳ request/response một cách declarative.

💡 Hint

Guard thực thi sau tất cả middleware, nhưng trước bất kỳ interceptor hay pipe nào.

Authorization guard #

Như đã đề cập, authorization là use case tuyệt vời cho Guard vì các route cụ thể chỉ khả dụng khi caller (thường là user đã xác thực) có đủ permission. AuthGuard chúng ta sẽ build giả sử rằng user đã được xác thực (và do đó, token được attach vào request header). Nó sẽ trích xuất và validate token, rồi dùng thông tin đó để xác định request có thể tiếp tục hay không.

auth.guard.ts
TS
JS
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Observable } from 'rxjs';

@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(
    context: ExecutionContext,
  ): boolean | Promise<boolean> | Observable<boolean> {
    const request = context.switchToHttp().getRequest();
    return validateRequest(request);
  }
}
import { Injectable } from '@nestjs/common';

@Injectable()
export class AuthGuard {
  async canActivate(context) {
    const request = context.switchToHttp().getRequest();
    return validateRequest(request);
  }
}
💡 Hint

Nếu bạn đang tìm ví dụ thực tế về cách implement cơ chế authentication trong ứng dụng, hãy xem chương này. Tương tự, để có ví dụ authorization phức tạp hơn, hãy xem trang này.

Logic bên trong hàm validateRequest() có thể đơn giản hoặc phức tạp tuỳ ý. Trọng tâm của ví dụ này là minh hoạ cách guard phù hợp vào chu kỳ request/response.

Mỗi guard phải implement hàm canActivate(). Hàm này trả về boolean cho biết request hiện tại được cho phép hay không. Nó có thể trả về response đồng bộ hoặc bất đồng bộ (qua Promise hoặc Observable). Nest dùng giá trị trả về để điều khiển hành động tiếp theo:

✅ Trả về true

Request sẽ được xử lý bình thường — route handler được gọi.

❌ Trả về false

Nest sẽ từ chối request — framework tự động throw ForbiddenException.

Execution context #

Hàm canActivate() nhận một argument duy nhất là instance ExecutionContext. ExecutionContext kế thừa từ ArgumentsHost. Chúng ta đã gặp ArgumentsHost trong chương exception filters. Trong ví dụ trên, chúng ta chỉ dùng các helper method đã được định nghĩa trên ArgumentsHost để lấy reference đến Request object. Bạn có thể tham khảo lại phần Arguments host của chương exception filters để biết thêm.

Bằng cách extend ArgumentsHost, ExecutionContext còn thêm một số helper method mới cung cấp thêm chi tiết về quá trình thực thi hiện tại. Những chi tiết này hữu ích để build các guard generic có thể hoạt động trên nhiều loại controller, method và execution context. Tìm hiểu thêm về ExecutionContext tại đây.

Role-based authentication #

Hãy build một guard chức năng hơn — chỉ cho phép truy cập với user có role cụ thể. Bắt đầu với một guard template cơ bản, hiện tại nó cho phép tất cả request đi qua:

roles.guard.ts
TS
JS
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Observable } from 'rxjs';

@Injectable()
export class RolesGuard implements CanActivate {
  canActivate(
    context: ExecutionContext,
  ): boolean | Promise<boolean> | Observable<boolean> {
    return true;
  }
}
import { Injectable } from '@nestjs/common';

@Injectable()
export class RolesGuard {
  canActivate(context) {
    return true;
  }
}

Binding guards #

Giống như pipe và exception filter, guard có thể có scope ở các cấp độ khác nhau:

Method-scoped
Controller-scoped
Global-scoped

Dưới đây, chúng ta thiết lập controller-scoped guard bằng decorator @UseGuards(). Decorator này có thể nhận một argument duy nhất hoặc danh sách phân cách bởi dấu phẩy. Điều này cho phép bạn dễ dàng áp dụng tập hợp guard phù hợp chỉ với một khai báo.

@Controller('cats')
@UseGuards(RolesGuard)
export class CatsController {}
💡 Hint

Decorator @UseGuards() được import từ package @nestjs/common.

Ở trên, chúng ta truyền class RolesGuard (không phải instance), để framework đảm nhận việc khởi tạo và cho phép dependency injection. Cũng có thể truyền in-place instance:

@Controller('cats')
@UseGuards(new RolesGuard())
export class CatsController {}

Cấu trúc trên gắn guard vào mọi handler được khai báo trong controller này. Để áp dụng guard chỉ cho một method, dùng decorator @UseGuards() ở cấp method.

Để thiết lập global guard, dùng method useGlobalGuards() của Nest application instance:

const app = await NestFactory.create(AppModule);
app.useGlobalGuards(new RolesGuard());
⚠️ Notice

Với hybrid app, method useGlobalGuards() mặc định không thiết lập guard cho gateway và microservice (xem Hybrid application để biết cách thay đổi hành vi này). Với microservice app "chuẩn" (non-hybrid), useGlobalGuards() mount guard globally.

Global guard được sử dụng trên toàn bộ ứng dụng, cho mọi controller và mọi route handler. Về mặt dependency injection, global guard được đăng ký từ bên ngoài bất kỳ module nào (với useGlobalGuards() như ví dụ trên) không thể inject dependency vì điều này được thực hiện ngoài context của bất kỳ module nào. Để giải quyết vấn đề này, bạn có thể thiết lập guard trực tiếp từ bất kỳ module nào bằng cấu trúc sau:

app.module.ts
import { Module } from '@nestjs/common';
import { APP_GUARD } from '@nestjs/core';

@Module({
  providers: [
    {
      provide:  APP_GUARD,
      useClass: RolesGuard,
    },
  ],
})
export class AppModule {}
💡 Hint

Khi dùng cách này, guard thực chất là global bất kể module nào sử dụng cấu trúc này. Nên thực hiện ở module nơi guard (RolesGuard) được định nghĩa. Ngoài ra, useClass không phải cách duy nhất để đăng ký custom provider. Tìm hiểu thêm tại đây.

Setting roles per handler #

RolesGuard đang hoạt động, nhưng chưa đủ thông minh. Chúng ta chưa tận dụng tính năng quan trọng nhất của guard — execution context. Nó chưa biết về role, hoặc role nào được phép với mỗi handler. CatsController có thể có các permission scheme khác nhau cho từng route.

Đây là lúc custom metadata phát huy tác dụng. Nest cung cấp khả năng attach custom metadata vào route handler thông qua decorator được tạo bằng Reflector.createDecorator hoặc decorator tích hợp @SetMetadata().

Hãy tạo decorator @Roles() bằng method Reflector.createDecorator — Reflector được cung cấp sẵn bởi framework và export từ package @nestjs/core:

roles.decorator.ts
import { Reflector } from '@nestjs/core';

export const Roles = Reflector.createDecorator<string[]>();

Decorator Roles ở đây là một function nhận một argument duy nhất có type string[]. Để dùng decorator này, chỉ cần annotate handler với nó:

cats.controller.ts
TS
JS
@Post()
@Roles(['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
@Roles(['admin'])
@Bind(Body())
async create(createCatDto) {
  this.catsService.create(createCatDto);
}

Ở đây chúng ta đã attach metadata của decorator Roles vào method create(), chỉ ra rằng chỉ user có role admin mới được phép truy cập route này.

Ngoài ra, thay vì dùng Reflector.createDecorator, chúng ta có thể dùng decorator tích hợp @SetMetadata(). Tìm hiểu thêm tại đây.

Putting it all together #

Bây giờ hãy quay lại và kết hợp mọi thứ với RolesGuard. Hiện tại nó trả về true trong mọi trường hợp. Chúng ta muốn làm giá trị trả về có điều kiện dựa trên việc so sánh role được gán cho user hiện tại với role thực tế được yêu cầu bởi route đang được xử lý. Để truy cập role(s) của route (custom metadata), dùng helper class Reflector:

roles.guard.ts
TS
JS
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { Roles } from './roles.decorator';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const roles = this.reflector.get(Roles, context.getHandler());
    if (!roles) {
      return true;
    }
    const request = context.switchToHttp().getRequest();
    const user = request.user;
    return matchRoles(roles, user.roles);
  }
}
import { Injectable, Dependencies } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { Roles } from './roles.decorator';

@Injectable()
@Dependencies(Reflector)
export class RolesGuard {
  constructor(reflector) {
    this.reflector = reflector;
  }

  canActivate(context) {
    const roles = this.reflector.get(Roles, context.getHandler());
    if (!roles) {
      return true;
    }
    const request = context.switchToHttp().getRequest();
    const user = request.user;
    return matchRoles(roles, user.roles);
  }
}
💡 Hint

Trong Node.js, thông thường người ta attach authenticated user vào request object. Do đó trong code mẫu trên, chúng ta giả sử request.user chứa user instance và allowed role. Trong app của bạn, bạn sẽ tạo association đó trong custom authentication guard (hoặc middleware). Xem chương này để biết thêm.

⚠️ Warning

Logic bên trong hàm matchRoles() có thể đơn giản hoặc phức tạp tuỳ ý. Trọng tâm của ví dụ này là minh hoạ cách guard phù hợp vào chu kỳ request/response.

Tham khảo phần Reflection and metadata của chương Execution context để biết thêm chi tiết về cách dùng Reflector theo context.

Khi user không có đủ privilege request một endpoint, Nest tự động trả về response sau:

{
  "statusCode": 403,
  "message": "Forbidden resource",
  "error": "Forbidden"
}

Lưu ý rằng đằng sau hậu trường, khi guard trả về false, framework throw ForbiddenException. Nếu muốn trả về error response khác, bạn nên throw exception cụ thể của mình:

throw new UnauthorizedException();

Mọi exception được throw bởi guard sẽ được xử lý bởi exceptions layer (global exceptions filter và bất kỳ exceptions filter nào được áp dụng cho context hiện tại).

💡 Hint

Nếu bạn đang tìm ví dụ thực tế về cách implement authorization, hãy xem chương này.
