Providers

Provider là một khái niệm cốt lõi trong Nest. Nhiều class cơ bản của Nest — như service, repository, factory, helper — đều có thể được coi là provider. Ý tưởng then chốt là một provider có thể được inject như một dependency, cho phép các object tạo ra các mối quan hệ đa dạng với nhau. Việc "kết nối" các object này phần lớn được Nest runtime system đảm nhiệm.

Ở chương trước, chúng ta đã tạo một CatsController đơn giản. Controller nên xử lý HTTP request và ủy thác các tác vụ phức tạp hơn cho provider. Provider là các plain JavaScript class được khai báo là providers trong một NestJS module. Tham khảo chương "Modules" để biết thêm chi tiết.

💡 Hint

Vì Nest cho phép bạn thiết kế và tổ chức dependencies theo cách hướng đối tượng, chúng tôi khuyến khích mạnh mẽ việc tuân theo các nguyên tắc SOLID.

Services #

Hãy bắt đầu bằng cách tạo một CatsService đơn giản. Service này sẽ xử lý việc lưu trữ và truy xuất dữ liệu, và sẽ được CatsController sử dụng. Do vai trò quản lý logic của ứng dụng, nó là ứng cử viên lý tưởng để định nghĩa như một provider.

cats.service.ts
TS
JS
import { Injectable } from '@nestjs/common';
import { Cat } from './interfaces/cat.interface';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  create(cat: Cat) {
    this.cats.push(cat);
  }

  findAll(): Cat[] {
    return this.cats;
  }
}
import { Injectable } from '@nestjs/common';

@Injectable()
export class CatsService {
  constructor() {
    this.cats = [];
  }

  create(cat) {
    this.cats.push(cat);
  }

  findAll() {
    return this.cats;
  }
}
💡 Hint

Để tạo service bằng CLI, chỉ cần chạy lệnh $ nest g service cats.

CatsService là một class cơ bản với một property và hai method. Điểm quan trọng ở đây là decorator @Injectable(). Decorator này gắn metadata vào class, báo hiệu rằng CatsService là class có thể được quản lý bởi IoC container của Nest.

Ngoài ra, ví dụ này sử dụng interface Cat, trông có thể như sau:

interfaces/cat.interface.ts
TS
JS
export interface Cat {
  name: string;
  age: number;
  breed: string;
}
// JavaScript không có interface — dùng JSDoc hoặc class thay thế

Bây giờ chúng ta đã có service class để lấy dữ liệu cats, hãy sử dụng nó bên trong CatsController:

cats.controller.ts
TS
JS
import { Controller, Get, Post, Body } from '@nestjs/common';
import { CreateCatDto } from './dto/create-cat.dto';
import { CatsService } from './cats.service';
import { Cat } from './interfaces/cat.interface';

@Controller('cats')
export class CatsController {
  constructor(private catsService: CatsService) {}

  @Post()
  async create(@Body() createCatDto: CreateCatDto) {
    this.catsService.create(createCatDto);
  }

  @Get()
  async findAll(): Promise<Cat[]> {
    return this.catsService.findAll();
  }
}
import { Controller, Get, Post, Body, Bind, Dependencies } from '@nestjs/common';
import { CatsService } from './cats.service';

@Controller('cats')
@Dependencies(CatsService)
export class CatsController {
  constructor(catsService) {
    this.catsService = catsService;
  }

  @Post()
  @Bind(Body())
  async create(createCatDto) {
    this.catsService.create(createCatDto);
  }

  @Get()
  async findAll() {
    return this.catsService.findAll();
  }
}

CatsService được inject thông qua constructor của class. Lưu ý việc sử dụng từ khoá private. Shorthand này cho phép chúng ta vừa khai báo vừa khởi tạo member catsService ngay trên cùng một dòng, đơn giản hoá toàn bộ quá trình.

Dependency injection #

Nest được xây dựng xung quanh design pattern mạnh mẽ được gọi là Dependency Injection. Chúng tôi khuyến khích đọc một bài viết hay về khái niệm này trong tài liệu chính thức của Angular.

Trong Nest, nhờ khả năng của TypeScript, việc quản lý dependency rất đơn giản vì chúng được resolve dựa trên type. Trong ví dụ dưới, Nest sẽ resolve catsService bằng cách tạo và trả về một instance của CatsService (hoặc trong trường hợp singleton, trả về instance đã tồn tại nếu đã được yêu cầu ở nơi khác). Dependency này sau đó được inject vào constructor của controller (hoặc gán vào property đã chỉ định):

constructor(private catsService: CatsService) {}

Scopes #

Provider thông thường có một lifetime ("scope") đồng bộ với vòng đời ứng dụng. Khi ứng dụng được bootstrap, mọi dependency đều phải được resolve — nghĩa là mọi provider đều được khởi tạo. Tương tự, khi ứng dụng tắt, tất cả provider sẽ bị huỷ. Tuy nhiên, cũng có thể làm cho provider có scope theo request, nghĩa là lifetime của nó được gắn với một request cụ thể thay vì vòng đời ứng dụng. Bạn có thể tìm hiểu thêm về các kỹ thuật này trong chương Injection Scopes.

Custom providers #

Nest đi kèm với một IoC (inversion of control) container tích hợp sẵn để quản lý mối quan hệ giữa các provider. Tính năng này là nền tảng của dependency injection, nhưng thực ra nó mạnh mẽ hơn nhiều so với những gì chúng ta đã đề cập. Có nhiều cách để định nghĩa provider: có thể dùng plain value, class, hoặc cả factory bất đồng bộ lẫn đồng bộ. Xem thêm ví dụ định nghĩa provider trong chương Dependency Injection.

Optional providers #

Đôi khi bạn có thể có các dependency không nhất thiết phải luôn được resolve. Ví dụ: class của bạn có thể phụ thuộc vào một configuration object, nhưng nếu không có, nên dùng giá trị mặc định. Trong trường hợp đó, dependency được coi là optional và việc thiếu configuration provider không nên gây ra lỗi.

Để đánh dấu provider là optional, dùng decorator @Optional() trong chữ ký constructor.

import { Injectable, Optional, Inject } from '@nestjs/common';

@Injectable()
export class HttpService<T> {
  constructor(@Optional() @Inject('HTTP_OPTIONS') private httpClient: T) {}
}

Trong ví dụ trên, chúng ta đang dùng custom provider, đó là lý do bao gồm custom token HTTP_OPTIONS. Các ví dụ trước minh hoạ constructor-based injection, nơi dependency được chỉ định thông qua một class trong constructor. Xem thêm chi tiết về custom provider và cách token của chúng hoạt động trong chương Custom Providers.

Property-based injection #

Kỹ thuật chúng ta đã dùng từ trước đến nay được gọi là constructor-based injection — provider được inject thông qua constructor method. Trong một số trường hợp cụ thể, property-based injection có thể hữu ích. Ví dụ: nếu top-level class phụ thuộc vào một hoặc nhiều provider, việc truyền tất cả chúng lên qua super() trong subclass có thể trở nên cồng kềnh. Để tránh điều này, có thể dùng decorator @Inject() trực tiếp ở cấp property.

import { Injectable, Inject } from '@nestjs/common';

@Injectable()
export class HttpService<T> {
  @Inject('HTTP_OPTIONS')
  private readonly httpClient: T;
}
⚠️ Warning

Nếu class của bạn không extend class nào khác, thông thường nên ưu tiên constructor-based injection. Constructor chỉ rõ ràng những dependency nào được yêu cầu, giúp code dễ đọc và dễ hiểu hơn so với class property được annotate bằng @Inject.

Provider registration #

Bây giờ chúng ta đã định nghĩa provider (CatsService) và một consumer (CatsController), chúng ta cần đăng ký service với Nest để nó có thể thực hiện injection. Điều này được thực hiện bằng cách chỉnh sửa file module (app.module.ts) và thêm service vào mảng providers trong decorator @Module().

app.module.ts
TS
JS
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
  controllers: [CatsController],
  providers:   [CatsService],
})
export class AppModule {}
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
  controllers: [CatsController],
  providers:   [CatsService],
})
export class AppModule {}

Nest giờ đây có thể resolve các dependency của class CatsController.

Manual instantiation #

Cho đến nay, chúng ta đã thấy Nest tự động xử lý hầu hết các chi tiết của việc resolve dependency. Tuy nhiên, trong một số trường hợp, bạn có thể cần bước ra ngoài hệ thống Dependency Injection tích hợp sẵn và thủ công lấy hoặc khởi tạo provider. Hai kỹ thuật như vậy được đề cập ngắn gọn dưới đây:

  • Để lấy các instance đang tồn tại hoặc khởi tạo provider động, dùng Module reference.
  • Để lấy provider bên trong hàm bootstrap() (ví dụ: cho standalone application hoặc dùng configuration service trong quá trình bootstrap), xem Standalone applications.
