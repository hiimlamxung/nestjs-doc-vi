Middleware

Middleware là một function được gọi trước route handler. Các middleware function có quyền truy cập vào đối tượng request và response, cùng với middleware function next() trong chu kỳ request-response của ứng dụng. Middleware function tiếp theo thường được ký hiệu bằng biến tên next.

Nest middleware mặc định tương đương với express middleware. Mô tả từ tài liệu chính thức của express nêu các khả năng của middleware:

Middleware function có thể thực hiện các tác vụ sau:

  • Thực thi bất kỳ đoạn code nào.
  • Thay đổi đối tượng request và response.
  • Kết thúc chu kỳ request-response.
  • Gọi middleware function tiếp theo trong stack.
  • Nếu middleware hiện tại không kết thúc chu kỳ request-response, nó bắt buộc phải gọi next() để chuyển điều khiển sang middleware tiếp theo. Nếu không, request sẽ bị treo.

Bạn có thể implement custom Nest middleware trong một function, hoặc trong một class với decorator @Injectable(). Class nên implement interface NestMiddleware, trong khi function không có yêu cầu đặc biệt nào. Hãy bắt đầu bằng cách implement một middleware đơn giản sử dụng phương pháp class.

⚠️ Warning

Express và fastify xử lý middleware theo cách khác nhau và cung cấp các chữ ký method khác nhau. Đọc thêm tại đây.

logger.middleware.ts
TS
JS
import { Injectable, NestMiddleware } from '@nestjs/common';
import { Request, Response, NextFunction } from 'express';

@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    console.log('Request...');
    next();
  }
}
import { Injectable } from '@nestjs/common';

@Injectable()
export class LoggerMiddleware {
  use(req, res, next) {
    console.log('Request...');
    next();
  }
}

Dependency injection #

Nest middleware hỗ trợ đầy đủ Dependency Injection. Cũng giống như với provider và controller, chúng có thể inject dependency khả dụng trong cùng module. Như thường lệ, điều này được thực hiện thông qua constructor.

Applying middleware #

Không có chỗ cho middleware trong decorator @Module(). Thay vào đó, chúng ta thiết lập chúng bằng method configure() của module class. Module chứa middleware phải implement interface NestModule. Hãy thiết lập LoggerMiddleware ở cấp AppModule.

app.module.ts
TS
JS
import { Module, NestModule, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes('cats');
  }
}
import { Module } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule {
  configure(consumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes('cats');
  }
}

Trong ví dụ trên, chúng ta đã thiết lập LoggerMiddleware cho các route handler /cats đã định nghĩa trước đó trong CatsController. Chúng ta cũng có thể giới hạn middleware cho một request method cụ thể bằng cách truyền một object chứa path và method vào method forRoutes() khi cấu hình middleware. Trong ví dụ dưới, lưu ý rằng chúng ta import enum RequestMethod để tham chiếu đến loại request method mong muốn.

app.module.ts
TS
JS
import { Module, NestModule, RequestMethod, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes({ path: 'cats', method: RequestMethod.GET });
  }
}
import { Module, RequestMethod } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule {
  configure(consumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes({ path: 'cats', method: RequestMethod.GET });
  }
}
💡 Hint

Method configure() có thể được làm bất đồng bộ bằng async/await (ví dụ: bạn có thể await hoàn thành một thao tác bất đồng bộ bên trong body của configure()).

⚠️ Warning

Khi dùng express adapter, NestJS app sẽ đăng ký json và urlencoded từ package body-parser theo mặc định. Điều này có nghĩa là nếu muốn tuỳ biến middleware đó qua MiddlewareConsumer, bạn cần tắt global middleware bằng cách đặt flag bodyParser thành false khi tạo application với NestFactory.create().

Route wildcards #

Route dựa trên pattern cũng được hỗ trợ trong NestJS middleware. Ví dụ: named wildcard (*splat) có thể được dùng để khớp bất kỳ tổ hợp ký tự nào trong một route. Trong ví dụ dưới, middleware sẽ được thực thi cho bất kỳ route nào bắt đầu bằng abcd/, bất kể số lượng ký tự theo sau.

forRoutes({
  path:   'abcd/*splat',
  method: RequestMethod.ALL,
});
💡 Hint

splat chỉ là tên của wildcard parameter và không có ý nghĩa đặc biệt. Bạn có thể đặt tên bất kỳ, ví dụ: *wildcard.

Route path 'abcd/*' sẽ khớp với abcd/1, abcd/123, abcd/abc, v.v. Dấu gạch ngang (-) và dấu chấm (.) được hiểu theo nghĩa đen trong string-based path. Tuy nhiên, abcd/ không có ký tự bổ sung sẽ không khớp với route. Để làm điều này, hãy bọc wildcard trong dấu ngoặc nhọn để làm nó optional:

forRoutes({
  path:   'abcd/{*splat}',
  method: RequestMethod.ALL,
});

Middleware consumer #

MiddlewareConsumer là một helper class. Nó cung cấp một số built-in method để quản lý middleware. Tất cả đều có thể được chain theo fluent style. Method forRoutes() có thể nhận một string, nhiều string, một object RouteInfo, một controller class và thậm chí nhiều controller class. Trong hầu hết trường hợp bạn sẽ chỉ truyền danh sách các controller phân cách bởi dấu phẩy. Dưới đây là ví dụ với một controller duy nhất:

app.module.ts
TS
JS
import { Module, NestModule, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';
import { CatsController } from './cats/cats.controller';

@Module({
  imports: [CatsModule],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes(CatsController);
  }
}
import { Module } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';
import { CatsController } from './cats/cats.controller';

@Module({
  imports: [CatsModule],
})
export class AppModule {
  configure(consumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes(CatsController);
  }
}
💡 Hint

Method apply() có thể nhận một middleware duy nhất, hoặc nhiều argument để chỉ định nhiều middleware.

Excluding routes #

Đôi khi chúng ta muốn loại trừ một số route khỏi việc áp dụng middleware. Điều này có thể dễ dàng thực hiện bằng method exclude(). Method exclude() nhận một string, nhiều string, hoặc một object RouteInfo để xác định các route cần loại trừ.

Đây là ví dụ về cách sử dụng:

consumer
  .apply(LoggerMiddleware)
  .exclude(
    { path: 'cats', method: RequestMethod.GET },
    { path: 'cats', method: RequestMethod.POST },
    'cats/{*splat}',
  )
  .forRoutes(CatsController);
💡 Hint

Method exclude() hỗ trợ wildcard parameter sử dụng package path-to-regexp.

Với ví dụ trên, LoggerMiddleware sẽ được gắn vào tất cả route định nghĩa trong CatsController ngoại trừ ba route được truyền vào exclude().

Cách tiếp cận này cung cấp tính linh hoạt trong việc áp dụng hoặc loại trừ middleware dựa trên các route hoặc pattern route cụ thể.

Functional middleware #

Class LoggerMiddleware chúng ta đang dùng khá đơn giản. Nó không có member, không có method bổ sung, và không có dependency. Tại sao không định nghĩa nó trong một function đơn giản thay vì class? Trên thực tế, hoàn toàn có thể. Loại middleware này được gọi là functional middleware. Hãy chuyển logger middleware từ class-based sang functional middleware để minh hoạ sự khác biệt:

logger.middleware.ts
TS
JS
import { Request, Response, NextFunction } from 'express';

export function logger(req: Request, res: Response, next: NextFunction) {
  console.log(`Request...`);
  next();
};
export function logger(req, res, next) {
  console.log(`Request...`);
  next();
};

Và sử dụng nó trong AppModule:

app.module.ts
TS
JS
consumer
  .apply(logger)
  .forRoutes(CatsController);
consumer
  .apply(logger)
  .forRoutes(CatsController);
💡 Hint

Hãy xem xét dùng functional middleware đơn giản hơn bất cứ khi nào middleware của bạn không cần bất kỳ dependency nào.

Multiple middleware #

Như đã đề cập ở trên, để gắn nhiều middleware thực thi tuần tự, chỉ cần cung cấp danh sách phân cách bằng dấu phẩy bên trong method apply():

consumer.apply(cors(), helmet(), logger).forRoutes(CatsController);

Global middleware #

Nếu muốn gắn middleware vào tất cả route đã đăng ký cùng một lúc, có thể dùng method use() được cung cấp bởi instance INestApplication:

main.ts
TS
JS
const app = await NestFactory.create(AppModule);
app.use(logger);
await app.listen(process.env.PORT ?? 3000);
const app = await NestFactory.create(AppModule);
app.use(logger);
await app.listen(process.env.PORT ?? 3000);
💡 Hint

Không thể truy cập DI container trong global middleware. Bạn có thể dùng functional middleware khi sử dụng app.use(). Hoặc, có thể dùng class middleware và consume nó bằng .forRoutes('*') trong AppModule (hoặc bất kỳ module nào khác).
