Custom decorators

Nest được xây dựng xung quanh một tính năng ngôn ngữ gọi là decorators. Decorator là khái niệm được biết đến rộng rãi trong nhiều ngôn ngữ lập trình phổ biến, nhưng trong thế giới JavaScript, chúng vẫn còn tương đối mới. Để hiểu rõ hơn cách decorator hoạt động, hãy đọc bài viết này.

Một ES2016 decorator là một expression trả về function và có thể nhận target, name và property descriptor làm argument. Bạn áp dụng nó bằng cách đặt ký tự @ ở phía trước decorator và đặt nó ở ngay trên những gì bạn muốn decorate. Decorator có thể được định nghĩa cho class, method hoặc property.

Param decorators #

Nest cung cấp một tập param decorator hữu ích để dùng cùng với HTTP route handler. Dưới đây là danh sách các decorator được cung cấp và các Express (hoặc Fastify) object tương ứng:

DecoratorExpress / Fastify object
@Request(), @Req()req
@Response(), @Res()res
@Next()next
@Session()req.session
@Param(param?: string)req.params / req.params[param]
@Body(param?: string)req.body / req.body[param]
@Query(param?: string)req.query / req.query[param]
@Headers(param?: string)req.headers / req.headers[param]
@Ip()req.ip
@HostParam()req.hosts

Ngoài ra, bạn có thể tạo custom decorator của riêng mình. Tại sao lại hữu ích?

Trong thế giới Node.js, thông thường người ta attach property vào request object, rồi trích xuất thủ công trong mỗi route handler với code như sau:

const user = req.user;

Để code dễ đọc và minh bạch hơn, bạn có thể tạo decorator @User() và tái sử dụng nó trên tất cả controller:

user.decorator.ts
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const User = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    return request.user;
  },
);

Sau đó, dùng nó ở bất cứ đâu phù hợp:

TS
JS
@Get()
async findOne(@User() user: UserEntity) {
  console.log(user);
}
@Get()
@Bind(User())
async findOne(user) {
  console.log(user);
}

Passing data #

Khi hành vi của decorator phụ thuộc vào một số điều kiện, bạn có thể dùng tham số data để truyền argument cho factory function của decorator. Một use case là custom decorator trích xuất property từ request object theo key.

Giả sử authentication layer validate request và attach user entity vào request object. User entity của authenticated request có thể trông như sau:

{
  "id": 101,
  "firstName": "Alan",
  "lastName": "Turing",
  "email": "alan@email.com",
  "roles": ["admin"]
}

Hãy định nghĩa decorator nhận property name làm key và trả về giá trị liên quan nếu tồn tại (hoặc undefined nếu không tồn tại, hoặc nếu user object chưa được tạo):

user.decorator.ts
TS
JS
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const User = createParamDecorator(
  (data: string, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    const user = request.user;

    return data ? user?.[data] : user;
  },
);
import { createParamDecorator } from '@nestjs/common';

export const User = createParamDecorator((data, ctx) => {
  const request = ctx.switchToHttp().getRequest();
  const user = request.user;

  return data ? user && user[data] : user;
});

Đây là cách truy cập property cụ thể qua decorator @User() trong controller:

TS
JS
@Get()
async findOne(@User('firstName') firstName: string) {
  console.log(`Hello ${firstName}`);
}
@Get()
@Bind(User('firstName'))
async findOne(firstName) {
  console.log(`Hello ${firstName}`);
}

Bạn có thể dùng cùng một decorator với các key khác nhau để truy cập các property khác nhau. Nếu user object sâu hoặc phức tạp, điều này giúp implementation của request handler dễ đọc hơn.

💡 Hint

Với TypeScript, createParamDecorator<T>() là generic — bạn có thể enforce type safety tường minh, ví dụ createParamDecorator<string>((data, ctx) => ...). Hoặc chỉ định kiểu parameter trong factory function: createParamDecorator((data: string, ctx) => ...). Nếu bỏ qua cả hai, type của data sẽ là any.

Working with pipes #

Nest xử lý custom param decorator giống hệt như các decorator tích hợp (@Body(), @Param() và @Query()). Điều này có nghĩa là pipe cũng được thực thi cho custom annotated parameter (trong ví dụ của chúng ta, argument user). Hơn nữa, bạn có thể áp dụng pipe trực tiếp vào custom decorator:

TS
JS
@Get()
async findOne(
  @User(new ValidationPipe({ validateCustomDecorators: true }))
  user: UserEntity,
) {
  console.log(user);
}
@Get()
@Bind(User(new ValidationPipe({ validateCustomDecorators: true })))
async findOne(user) {
  console.log(user);
}
💡 Hint

Lưu ý rằng option validateCustomDecorators phải được đặt thành true. ValidationPipe mặc định không validate argument được annotate bằng custom decorator.

Decorator composition #

Nest cung cấp helper method applyDecorators để kết hợp nhiều decorator thành một. Ví dụ, giả sử bạn muốn kết hợp tất cả decorator liên quan đến authentication thành một decorator duy nhất:

auth.decorator.ts
TS
JS
import { applyDecorators } from '@nestjs/common';

export function Auth(...roles: Role[]) {
  return applyDecorators(
    SetMetadata('roles', roles),
    UseGuards(AuthGuard, RolesGuard),
    ApiBearerAuth(),
    ApiUnauthorizedResponse({ description: 'Unauthorized' }),
  );
}
import { applyDecorators } from '@nestjs/common';

export function Auth(...roles) {
  return applyDecorators(
    SetMetadata('roles', roles),
    UseGuards(AuthGuard, RolesGuard),
    ApiBearerAuth(),
    ApiUnauthorizedResponse({ description: 'Unauthorized' }),
  );
}

Sau đó dùng custom @Auth() decorator như sau:

@Get('users')
@Auth('admin')
findAllUsers() {}

Điều này có tác dụng áp dụng cả bốn decorator chỉ với một lần khai báo.

⚠️ Warning

Decorator @ApiHideProperty() từ package @nestjs/swagger không composable và sẽ không hoạt động đúng với hàm applyDecorators.
