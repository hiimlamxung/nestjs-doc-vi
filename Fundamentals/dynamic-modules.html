Dynamic modules

Chương Modules bao gồm các khái niệm cơ bản về Nest module, và có giới thiệu sơ lược về dynamic modules. Chương này mở rộng chủ đề dynamic modules. Sau khi hoàn thành, bạn sẽ hiểu rõ chúng là gì và cách thức cũng như khi nào nên sử dụng chúng.

Introduction #

Hầu hết các ví dụ code trong phần Overview của tài liệu sử dụng module thông thường, hay còn gọi là static module. Module định nghĩa các nhóm component như provider và controller kết hợp thành phần module của ứng dụng tổng thể. Chúng cung cấp execution context, hay scope, cho các component này. Ví dụ, các provider được định nghĩa trong module sẽ hiển thị với các thành viên khác trong module mà không cần export. Khi provider cần hiển thị bên ngoài module, trước tiên nó được export từ host module, sau đó import vào consuming module.

Hãy đi qua ví dụ quen thuộc.

Đầu tiên, chúng ta định nghĩa UsersModule để cung cấp và export UsersService. UsersModule là host module của UsersService.

users.module.ts
import { Module } from '@nestjs/common';
import { UsersService } from './users.service';

@Module({
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}

Tiếp theo, chúng ta định nghĩa AuthModule, import UsersModule, giúp exported provider của UsersModule khả dụng bên trong AuthModule:

auth.module.ts
import { Module } from '@nestjs/common';
import { AuthService } from './auth.service';
import { UsersModule } from '../users/users.module';

@Module({
  imports: [UsersModule],
  providers: [AuthService],
  exports: [AuthService],
})
export class AuthModule {}

Cấu trúc này cho phép chúng ta inject UsersService vào, ví dụ, AuthService được host trong AuthModule:

auth.service.ts
import { Injectable } from '@nestjs/common';
import { UsersService } from '../users/users.service';

@Injectable()
export class AuthService {
  constructor(private usersService: UsersService) {}
  /*
    implementation that makes use of this.usersService
  */
}

Chúng ta gọi đây là static module binding. Tất cả thông tin Nest cần để kết nối các module đã được khai báo sẵn trong host module và consuming module. Hãy xem điều gì xảy ra trong quá trình này. Nest làm cho UsersService khả dụng bên trong AuthModule bằng cách:

  1. Khởi tạo UsersModule, bao gồm import transitive các module khác mà UsersModule consume, và resolve transitive bất kỳ dependency nào (xem Custom providers).
  2. Khởi tạo AuthModule, và làm cho exported provider của UsersModule khả dụng cho các component trong AuthModule (giống như chúng đã được khai báo trong AuthModule).
  3. Inject instance của UsersService vào AuthService.

Dynamic module use case #

Với static module binding, consuming module không có cơ hội ảnh hưởng đến cách provider từ host module được cấu hình. Tại sao điều này quan trọng? Xem xét trường hợp chúng ta có module đa năng cần hoạt động khác nhau trong các use case khác nhau. Tương tự khái niệm "plugin" trong nhiều hệ thống, nơi tiện ích chung cần cấu hình trước khi consumer sử dụng.

Ví dụ tốt với Nest là configuration module. Nhiều ứng dụng thấy hữu ích khi externalize chi tiết cấu hình qua configuration module. Điều này giúp thay đổi application setting linh hoạt cho các môi trường khác nhau: database development, database staging/testing, database production, v.v. Bằng cách ủy thác quản lý configuration parameter cho configuration module, application source code độc lập với configuration parameter.

Thách thức là bản thân configuration module, vì nó là generic, cần được tùy chỉnh bởi consuming module. Đây là lúc dynamic module phát huy tác dụng. Sử dụng tính năng dynamic module, chúng ta có thể làm configuration module trở nên dynamic để consuming module sử dụng API truyền cấu hình vào khi import.

Nói cách khác, dynamic module cung cấp API để import module vào module khác, và tùy chỉnh property cũng như behavior của module đó khi import, trái ngược với static binding đã thấy trước đó.

Config module example #

Chúng ta sẽ sử dụng phiên bản cơ bản của ConfigModule trong section này. Đây là bản đơn giản hóa — nếu cần version đầy đủ, hãy tham khảo @nestjs/config package (xem chương Configuration).

config.service.ts
import { Injectable } from '@nestjs/common';
import * as dotenv from 'dotenv';
import * as fs from 'fs';
import { EnvConfig } from './interfaces';

@Injectable()
export class ConfigService {
  private readonly envConfig: EnvConfig;

  constructor() {
    const options = { folder: './config' };
    const filePath = `${options.folder}/${process.env.NODE_ENV || 'development'}.env`;
    this.envConfig = dotenv.parse(fs.readFileSync(filePath));
  }

  get(key: string): string {
    return this.envConfig[key];
  }
}
config.module.ts
import { Module } from '@nestjs/common';
import { ConfigService } from './config.service';

@Module({
  providers: [ConfigService],
  exports: [ConfigService],
})
export class ConfigModule {}

Đây là static module đơn giản. Đường dẫn đến file .env đang bị hardcode. Mục tiêu là làm cho nó dynamic, để đường dẫn có thể tùy chỉnh được.

Configurable module builder #

Để làm ConfigModule trở nên dynamic, chúng ta cần thêm static method trả về object DynamicModule. Static method đó theo convention thường được đặt tên là register hoặc forRoot.

config.module.ts
import { DynamicModule, Module } from '@nestjs/common';
import { ConfigService } from './config.service';

@Module({})
export class ConfigModule {
  static register(): DynamicModule {
    return {
      module: ConfigModule,
      providers: [ConfigService],
      exports: [ConfigService],
    };
  }
}
💡 Hint

Dynamic module phải trả về object có chính xác cùng interface với @Module() decorator, cộng thêm property module bổ sung — chính là class module đó.

Method register() nhận argument tùy chỉnh. Trong ví dụ này, nó nhận object options. Chúng ta sẽ tạo CONFIG_OPTIONS provider với useValue:

config.module.ts
@Module({})
export class ConfigModule {
  static register(options): DynamicModule {
    return {
      module: ConfigModule,
      providers: [
        {
          provide: 'CONFIG_OPTIONS',
          useValue: options,
        },
        ConfigService,
      ],
      exports: [ConfigService],
    };
  }
}

Bây giờ chúng ta inject 'CONFIG_OPTIONS' provider vào ConfigService:

config.service.ts
@Injectable()
export class ConfigService {
  private readonly envConfig: EnvConfig;

  constructor(@Inject('CONFIG_OPTIONS') private options) {
    const filePath = `${options.folder}/${process.env.NODE_ENV || 'development'}.env`;
    this.envConfig = dotenv.parse(fs.readFileSync(filePath));
  }

  get(key: string): string {
    return this.envConfig[key];
  }
}

Cuối cùng, sử dụng dynamic module trong AppModule:

app.module.ts
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';
import { ConfigModule } from './config/config.module';

@Module({
  imports: [ConfigModule.register({ folder: './config' })],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}

Community guidelines #

Bạn có thể đã thấy phương thức như forRoot, register, forFeature ở một số package @nestjs/ và tự hỏi sự khác biệt giữa chúng. Không có convention cứng nào, nhưng @nestjs/ package tuân theo các guideline sau:

  • Khi bạn expect sử dụng register, nghĩa là bạn đang cấu hình dynamic module với cấu hình cụ thể chỉ cho calling module sử dụng.
  • Khi bạn expect sử dụng forRoot, bạn đang cấu hình dynamic module một lần và tái sử dụng cấu hình đó ở nhiều nơi (mặc dù có thể không ý thức được).
  • Khi bạn expect sử dụng forFeature, bạn đang sử dụng cấu hình từ forRoot của dynamic module, nhưng cần modify một số cấu hình cụ thể cho nhu cầu của calling module.

Tất cả chúng thường có counterpart async tương ứng: registerAsync, forRootAsync, forFeatureAsync, có cùng ý nghĩa nhưng sử dụng Nest Dependency Injection cho cấu hình.

ConfigurableModuleBuilder #

Vì việc tạo thủ công dynamic module có tính cấu hình cao với async provider khá phức tạp, Nest cung cấp class ConfigurableModuleBuilder giúp xử lý boilerplate và xây dựng module interface chỉ trong vài dòng code.

Ví dụ, sử dụng ConfigurableModuleBuilder để tạo ConfigModule:

config.module-definition.ts
import { ConfigurableModuleBuilder } from '@nestjs/common';

export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>().build();

Cấu trúc này tự động tạo class ConfigurableModuleClass cung cấp cả method register (phương thức đồng bộ) và registerAsync (phương thức bất đồng bộ):

config.module.ts
import { Module } from '@nestjs/common';
import { ConfigService } from './config.service';
import { ConfigurableModuleClass } from './config.module-definition';

@Module({
  providers: [ConfigService],
  exports: [ConfigService],
})
export class ConfigModule extends ConfigurableModuleClass {}

Extending ConfigurableModuleClass nghĩa là ConfigModule giờ không chỉ cung cấp method register mà còn registerAsync cho consumer cấu hình module bất đồng bộ:

@Module({
  imports: [
    ConfigModule.register({ folder: './config' }),
    // hoặc alternatively:
    // ConfigModule.registerAsync({
    //   useFactory: () => {
    //     return { folder: './config' }
    //   },
    //   inject: [...any dependencies that factory may need]
    // }),
  ],
})
export class AppModule {}

Custom method key #

ConfigurableModuleClass theo mặc định cung cấp method register và counterpart registerAsync tương ứng. Để sử dụng tên method khác, hãy dùng method ConfigurableModuleBuilder#setClassMethodName, như sau:

config.module-definition.ts
export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>()
    .setClassMethodName('forRoot')
    .build();

Cấu trúc này sẽ hướng dẫn ConfigurableModuleBuilder tạo ra class cung cấp forRoot và forRootAsync thay thế. Ví dụ sử dụng:

app.module.ts
@Module({
  imports: [
    ConfigModule.forRoot({ folder: './config' }),
  ],
})
export class AppModule {}
💡 Hint

Nếu bạn muốn sử dụng forRoot thay vì register, hãy xem phần Community guidelines ở trên.

Custom options factory class #

Vì method registerAsync (hoặc forRootAsync hoặc tên tùy chỉnh khác tùy cấu hình) cho phép consumer truyền provider definition tham chiếu đến class factory, theo mặc định class factory đó phải cung cấp method create() trả về options object. Tuy nhiên, nếu convention tên khác, bạn có thể thay đổi behavior này bằng method ConfigurableModuleBuilder#setFactoryMethodName, và hướng dẫn ConfigurableModuleBuilder mong đợi method khác, ví dụ createConfigOptions:

config.module-definition.ts
export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>()
    .setFactoryMethodName('createConfigOptions')
    .build();

Bây giờ, class ConfigModuleOptionsFactory phải cung cấp method createConfigOptions (thay vì create):

app.module.ts
@Module({
  imports: [
    ConfigModule.registerAsync({
      useClass: ConfigModuleOptionsFactory,
    }),
  ],
})
export class AppModule {}

Extra options #

Có những trường hợp bạn muốn truyền extra option chỉ dùng bởi module (ví dụ: flag isGlobal xác định module có phải global hay không) nhưng không nên được include trong MODULE_OPTIONS_TOKEN provider (vì chúng không liên quan đến service/provider bên trong module, ví dụ ConfigService không cần biết module có global hay không). Trong trường hợp đó, sử dụng method ConfigurableModuleBuilder#setExtras:

config.module-definition.ts
export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>()
    .setExtras(
      {
        isGlobal: true,
      },
      (definition, extras) => ({
        ...definition,
        global: extras.isGlobal,
      }),
    )
    .build();

Trong ví dụ trên, argument đầu tiên của setExtras là object chứa giá trị mặc định cho các property "extra". Argument thứ hai là function nhận auto-generated module definition (với provider, exports, v.v.) và object extras đại diện cho extra properties (do consumer chỉ định hoặc lấy giá trị mặc định). Giá trị trả về của function này là module definition đã được modify. Trong ví dụ cụ thể này, chúng ta lấy property extras.isGlobal và gán vào property global của module definition (xác định module có phải là global hay không, đọc thêm tại đây).

Bây giờ khi sử dụng module này, flag isGlobal có thể được truyền cùng:

app.module.ts
@Module({
  imports: [
    ConfigModule.register({
      isGlobal: true,
      folder: './config',
    }),
  ],
})
export class AppModule {}

Extending auto-generated methods #

Các static method được tạo tự động (register, registerAsync, v.v.) có thể được extend nếu cần, như sau:

config.module.ts
import { DynamicModule, Module } from '@nestjs/common';
import { ConfigService } from './config.service';
import {
  ConfigurableModuleClass,
  MODULE_OPTIONS_TOKEN,
  OPTIONS_TYPE,
  ASYNC_OPTIONS_TYPE,
} from './config.module-definition';

@Module({
  providers: [ConfigService],
  exports: [ConfigService],
})
export class ConfigModule extends ConfigurableModuleClass {
  static register(options: typeof OPTIONS_TYPE): DynamicModule {
    return {
      ...super.register(options),
      // custom logic here
    };
  }
}

Lưu ý việc sử dụng type OPTIONS_TYPE và ASYNC_OPTIONS_TYPE — chúng phải được export từ module definition file:

config.module-definition.ts
export const {
  ConfigurableModuleClass,
  MODULE_OPTIONS_TOKEN,
  OPTIONS_TYPE,
  ASYNC_OPTIONS_TYPE,
} = new ConfigurableModuleBuilder<ConfigModuleOptions>()
  .build();
