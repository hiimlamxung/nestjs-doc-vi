Execution context

Nest cung cấp một số lớp tiện ích giúp dễ dàng viết các ứng dụng hoạt động trên nhiều application context khác nhau (ví dụ: các application context dựa trên Nest HTTP server, microservices và WebSockets). Các tiện ích này cung cấp thông tin về execution context hiện tại, có thể được sử dụng để xây dựng các guards, filters và interceptors generic có thể hoạt động trên một tập hợp rộng các controllers, methods và execution contexts.

Chúng ta sẽ tìm hiểu hai lớp như vậy trong chương này: ArgumentsHost và ExecutionContext.

ArgumentsHost class #

Lớp ArgumentsHost cung cấp các phương thức để truy xuất các arguments được truyền vào một handler. Nó cho phép chọn application context phù hợp (ví dụ: HTTP, RPC (microservice), hoặc WebSockets) để truy xuất các arguments từ đó. Framework cung cấp một instance của ArgumentsHost, thường được tham chiếu dưới dạng tham số host, ở những nơi bạn có thể muốn truy cập nó. Ví dụ, phương thức catch() của một exception filter được gọi với instance ArgumentsHost.

ArgumentsHost đơn giản đóng vai trò như một lớp trừu tượng hóa (abstraction) trên các arguments của handler. Ví dụ, đối với các ứng dụng HTTP server (khi sử dụng @nestjs/platform-express), đối tượng host đóng gói mảng [request, response, next] của Express, trong đó request là đối tượng request, response là đối tượng response, và next là hàm điều khiển chu kỳ request-response của ứng dụng. Mặt khác, đối với các ứng dụng GraphQL, đối tượng host chứa mảng [root, args, context, info].

Current application context #

Khi xây dựng các guards, filters và interceptors generic nhằm chạy trên nhiều application context, chúng ta cần một cách để xác định loại ứng dụng mà method của chúng ta đang chạy trong đó. Hãy thực hiện điều này với phương thức getType() của ArgumentsHost:

if (host.getType() === 'http') {
  // do something that is only important in the context of regular HTTP requests (REST)
} else if (host.getType() === 'rpc') {
  // do something that is only important in the context of Microservice requests
} else if (host.getType<GqlContextType>() === 'graphql') {
  // do something that is only important in the context of GraphQL requests
}
💡 Hint

GqlContextType được import từ package @nestjs/graphql.

Với application type có sẵn, bạn có thể viết các component tổng quát hơn, như được trình bày bên dưới.

Host handler arguments #

Để truy xuất mảng arguments được truyền vào handler, một cách tiếp cận là sử dụng phương thức getArgs() của đối tượng host.

const [req, res, next] = host.getArgs();

Bạn có thể lấy một argument cụ thể theo index bằng phương thức getArgByIndex():

const request = host.getArgByIndex(0);
const response = host.getArgByIndex(1);

Trong các ví dụ này, chúng ta đã truy xuất các đối tượng request và response theo index, điều này thường không được khuyến khích vì nó gắn ứng dụng với một application context cụ thể. Thay vào đó, bạn có thể làm cho code của mình mạnh mẽ hơn và tái sử dụng được bằng cách sử dụng một trong các phương thức tiện ích của đối tượng host để chuyển sang application context phù hợp cho ứng dụng của bạn. Các phương thức chuyển đổi context được trình bày bên dưới.

/**
 * Switch context to RPC.
 */
switchToRpc(): RpcArgumentsHost;
/**
 * Switch context to HTTP.
 */
switchToHttp(): HttpArgumentsHost;
/**
 * Switch context to WebSockets.
 */
switchToWs(): WsArgumentsHost;

Hãy viết lại ví dụ trước bằng phương thức switchToHttp(). Phương thức tiện ích host.switchToHttp() trả về một đối tượng HttpArgumentsHost phù hợp với HTTP application context. Đối tượng HttpArgumentsHost có hai phương thức hữu ích mà chúng ta có thể sử dụng để trích xuất các đối tượng mong muốn. Trong trường hợp này, chúng ta cũng sử dụng type assertion của Express để trả về các đối tượng Express request và response nguyên bản (native):

const ctx = host.switchToHttp();
const request = ctx.getRequest<Request>();
const response = ctx.getResponse<Response>();

Tương tự, WsArgumentsHost và RpcArgumentsHost có các phương thức để trả về các đối tượng phù hợp trong mỗi context microservices và WebSockets. Dưới đây là type declarations cho WsArgumentsHost:

export interface WsArgumentsHost {
  getData<T>(): T;
  getClient<T>(): T;
}

Dưới đây là type declarations cho RpcArgumentsHost:

export interface RpcArgumentsHost {
  getData<T>(): T;
  getContext<T>(): T;
}

ExecutionContext class #

ExecutionContext kế thừa (extends) ArgumentsHost, cung cấp thêm các chi tiết về quá trình thực thi hiện tại. Giống như ArgumentsHost, Nest cung cấp một instance của ExecutionContext ở những nơi bạn có thể cần, chẳng hạn như trong phương thức canActivate() của một guard và phương thức intercept() của một interceptor. Nó cung cấp các phương thức sau:

export interface ExecutionContext extends ArgumentsHost {
  /**
   * Returns the type of the controller class which the current handler belongs to.
   */
  getClass<T = any>(): Type<T>;
  /**
   * Returns a reference to the handler (method) that will be invoked next in the
   * request pipeline.
   */
  getHandler(): Function;
}

Phương thức getHandler() trả về một tham chiếu đến handler sắp được gọi. Phương thức getClass() trả về type của lớp Controller mà handler cụ thể này thuộc về. Ví dụ, trong một HTTP context, nếu request hiện tại đang được xử lý là một POST request, gắn với phương thức create() trên CatsController, thì getHandler() trả về tham chiếu đến phương thức create() và getClass() trả về type CatsController (không phải instance):

const methodKey = ctx.getHandler().name; // "create"
const className = ctx.getClass().name; // "CatsController"

Khả năng truy cập tham chiếu đến cả class hiện tại và handler method mang lại sự linh hoạt rất lớn. Quan trọng nhất, nó cho chúng ta cơ hội truy cập metadata được thiết lập thông qua decorator Reflector#createDecorator hoặc decorator built-in @SetMetadata() từ bên trong guards hoặc interceptors. Chúng ta sẽ tìm hiểu use case này bên dưới.

Reflection and metadata #

Nest cung cấp khả năng đính kèm custom metadata vào route handlers thông qua decorator Reflector#createDecorator, và decorator built-in @SetMetadata(). Trong phần này, hãy so sánh hai cách tiếp cận và xem cách truy cập metadata từ bên trong guard hoặc interceptor.

Để tạo strongly-typed decorators bằng Reflector#createDecorator, chúng ta cần chỉ định type argument. Ví dụ, hãy tạo một decorator Roles nhận một mảng strings làm argument.

roles.decorator.ts
TS
JS
import { Reflector } from '@nestjs/core';

export const Roles = Reflector.createDecorator<string[]>();
import { Reflector } from '@nestjs/core';

export const Roles = Reflector.createDecorator();

Decorator Roles ở đây là một function nhận một argument duy nhất kiểu string[].

Bây giờ, để sử dụng decorator này, chúng ta chỉ cần gán nó vào handler:

cats.controller.ts
TS
JS
@Post()
@Roles(['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
@Roles(['admin'])
async create(@Body() createCatDto) {
  this.catsService.create(createCatDto);
}

Ở đây chúng ta đã gán metadata decorator Roles vào phương thức create(), chỉ ra rằng chỉ những user có role admin mới được phép truy cập route này.

Để truy cập role(s) của route (custom metadata), chúng ta sẽ sử dụng lại lớp helper Reflector. Reflector có thể được inject vào class theo cách thông thường:

roles.guard.ts
TS
JS
@Injectable()
export class RolesGuard {
  constructor(private reflector: Reflector) {}
}
@Injectable()
export class RolesGuard {
  constructor(private reflector) {}
}
💡 Hint

Lớp Reflector được import từ package @nestjs/core.

Bây giờ, để đọc metadata của handler, sử dụng phương thức get():

const roles = this.reflector.get(Roles, context.getHandler());

Phương thức Reflector#get cho phép chúng ta dễ dàng truy cập metadata bằng cách truyền vào hai arguments: một decorator reference và một context (target của decorator) để truy xuất metadata. Trong ví dụ này, decorator reference được chỉ định là Roles (tham khảo lại file roles.decorator.ts ở trên). Context được cung cấp bởi lời gọi context.getHandler(), dẫn đến việc trích xuất metadata cho route handler đang được xử lý hiện tại. Nhớ rằng, getHandler() cho chúng ta một tham chiếu đến hàm route handler.

Ngoài ra, chúng ta có thể tổ chức controller của mình bằng cách áp dụng metadata ở cấp controller, áp dụng cho tất cả routes trong controller class:

cats.controller.ts
TS
JS
@Roles(['admin'])
@Controller('cats')
export class CatsController {}
@Roles(['admin'])
@Controller('cats')
export class CatsController {}

Trong trường hợp này, để trích xuất metadata ở cấp controller, chúng ta truyền context.getClass() làm argument thứ hai (để cung cấp controller class làm context cho việc trích xuất metadata) thay vì context.getHandler():

roles.guard.ts
const roles = this.reflector.get(Roles, context.getClass());

Với khả năng cung cấp metadata ở nhiều cấp, bạn có thể cần trích xuất và gộp metadata từ nhiều context. Lớp Reflector cung cấp hai phương thức tiện ích dùng để hỗ trợ điều này. Các phương thức này trích xuất metadata cùng lúc từ cả hai target (controller và method), và kết hợp chúng theo các cách khác nhau.

Xem xét tình huống sau, nơi bạn đã cung cấp metadata Roles ở cả hai cấp.

cats.controller.ts
TS
JS
@Roles(['user'])
@Controller('cats')
export class CatsController {
  @Post()
  @Roles(['admin'])
  async create(@Body() createCatDto: CreateCatDto) {
    this.catsService.create(createCatDto);
  }
}
@Roles(['user'])
@Controller('cats')
export class CatsController {
  @Post()
  @Roles(['admin'])
  async create(@Body() createCatDto) {
    this.catsService.create(createCatDto);
  }
}

Nếu ý định của bạn là chỉ định 'user' làm role mặc định, và ghi đè một cách có chọn lọc cho một số method nhất định, bạn có thể sẽ sử dụng phương thức getAllAndOverride().

const roles = this.reflector.getAllAndOverride(Roles, [context.getHandler(), context.getClass()]);

Một guard với code này, chạy trong context của phương thức create(), với metadata ở trên, sẽ cho kết quả roles chứa ['admin'].

Để lấy metadata cho cả hai và gộp lại (phương thức này gộp cả arrays và objects), sử dụng phương thức getAllAndMerge():

const roles = this.reflector.getAllAndMerge(Roles, [context.getHandler(), context.getClass()]);

Điều này sẽ cho kết quả roles chứa ['admin', 'user'].

Đối với cả hai phương thức gộp này, bạn truyền decorator reference làm argument đầu tiên, và một mảng các context target của metadata (tức là các lời gọi đến phương thức getHandler() và/hoặc getClass()) làm argument thứ hai.

Low-level approach #

Như đã đề cập, thay vì sử dụng Reflector#createDecorator, bạn cũng có thể sử dụng decorator built-in @SetMetadata() để gán metadata vào handler.

cats.controller.ts
TS
JS
@Post()
@SetMetadata('roles', ['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
@SetMetadata('roles', ['admin'])
async create(@Body() createCatDto) {
  this.catsService.create(createCatDto);
}
💡 Hint

Decorator @SetMetadata() được import từ package @nestjs/common.

Với cách xây dựng trên, chúng ta đã gán metadata roles (roles là key của metadata, trong khi ['admin'] là giá trị tương ứng) vào phương thức create(). Mặc dù cách này hoạt động, nhưng sử dụng trực tiếp @SetMetadata() trong routes không phải là best practice. Thay vào đó, bạn có thể tạo decorators riêng của mình, như trình bày bên dưới:

roles.decorator.ts
TS
JS
import { SetMetadata } from '@nestjs/common';

export const Roles = (...roles: string[]) => SetMetadata('roles', roles);
import { SetMetadata } from '@nestjs/common';

export const Roles = (...roles) => SetMetadata('roles', roles);

Cách tiếp cận này gọn hơn và dễ đọc hơn nhiều, đồng thời strongly-typed. Bây giờ chúng ta đã có custom decorator @Roles(), có thể sử dụng nó để trang trí phương thức create():

cats.controller.ts
TS
JS
@Post()
@Roles('admin')
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
@Roles('admin')
async create(@Body() createCatDto) {
  this.catsService.create(createCatDto);
}
