Lifecycle Events

Một ứng dụng Nest, cũng như mọi phần tử trong ứng dụng, đều có một lifecycle được quản lý bởi Nest. Nest cung cấp các lifecycle hooks cho phép bạn quan sát các sự kiện lifecycle quan trọng, và khả năng thực thi code (chạy registered code trên các modules, providers hoặc controllers) khi các sự kiện đó xảy ra.

Lifecycle sequence #

Sơ đồ sau mô tả trình tự các sự kiện lifecycle chính của ứng dụng, từ thời điểm ứng dụng được bootstrapped cho đến khi tiến trình Node thoát. Toàn bộ lifecycle có thể được chia thành ba giai đoạn: initializing, running và terminating. Sử dụng lifecycle này, bạn có thể lên kế hoạch khởi tạo phù hợp cho modules và services, quản lý các kết nối đang hoạt động, và shutdown ứng dụng một cách graceful khi nhận được tín hiệu termination.

Lifecycle events #

Các lifecycle events xảy ra trong quá trình bootstrapping và shutdown ứng dụng. Nest gọi các lifecycle hook methods đã đăng ký trên modules, providers và controllers tại mỗi sự kiện lifecycle sau đây (các shutdown hooks cần được bật trước, như mô tả bên dưới). Như minh họa trong sơ đồ trên, Nest cũng gọi các phương thức nền tảng tương ứng để bắt đầu lắng nghe kết nối, và ngừng lắng nghe kết nối.

Trong bảng dưới đây, onModuleInit và onApplicationBootstrap chỉ được kích hoạt khi bạn gọi rõ ràng app.init() hoặc app.listen().

Trong bảng dưới đây, onModuleDestroy, beforeApplicationShutdown và onApplicationShutdown chỉ được kích hoạt khi bạn gọi rõ ràng app.close() hoặc nếu tiến trình nhận được một system signal đặc biệt (như SIGTERM) và bạn đã gọi đúng enableShutdownHooks khi application bootstrap (xem phần Application shutdown bên dưới).

Lifecycle hook methodLifecycle event kích hoạt hook method call
onModuleInit()Được gọi một lần sau khi các dependencies của host module đã được resolve.
onApplicationBootstrap()Được gọi một lần sau khi tất cả modules đã được khởi tạo, nhưng trước khi lắng nghe kết nối.
onModuleDestroy()*Được gọi sau khi nhận được tín hiệu termination (ví dụ: SIGTERM).
beforeApplicationShutdown()*Được gọi sau khi tất cả các handler onModuleDestroy() hoàn thành (Promises resolved hoặc rejected); sau khi hoàn tất (Promises resolved hoặc rejected), tất cả các kết nối hiện tại sẽ được đóng (gọi app.close()).
onApplicationShutdown()*Được gọi sau khi các kết nối đã đóng (app.close() resolves).

* Đối với các events này, nếu bạn không gọi app.close() một cách rõ ràng, bạn phải opt-in để chúng hoạt động với các system signals như SIGTERM. Xem phần Application shutdown bên dưới.

⚠️ Warning

Các lifecycle hooks được liệt kê ở trên không được kích hoạt cho các class có scope là request-scoped. Các class request-scoped không gắn với application lifecycle và thời gian tồn tại của chúng là không thể đoán trước. Chúng được tạo riêng cho mỗi request và tự động được garbage-collected sau khi response được gửi đi.

💡 Hint

Thứ tự thực thi của onModuleInit() và onApplicationBootstrap() phụ thuộc trực tiếp vào thứ tự import modules, chờ đợi hook trước đó hoàn thành.

Usage #

Mỗi lifecycle hook được đại diện bởi một interface. Các interfaces về mặt kỹ thuật là tùy chọn vì chúng không tồn tại sau khi TypeScript biên dịch. Tuy nhiên, best practice là sử dụng chúng để có được lợi ích từ strong typing và editor tooling. Để đăng ký một lifecycle hook, hãy implement interface tương ứng. Ví dụ, để đăng ký một method được gọi trong quá trình module initialization trên một class cụ thể (ví dụ: Controller, Provider hoặc Module), hãy implement interface OnModuleInit bằng cách cung cấp phương thức onModuleInit(), như trình bày bên dưới:

users.service.ts
TS
JS
import { Injectable, OnModuleInit } from '@nestjs/common';

@Injectable()
export class UsersService implements OnModuleInit {
  onModuleInit() {
    console.log(`The module has been initialized.`);
  }
}
import { Injectable } from '@nestjs/common';

@Injectable()
export class UsersService {
  onModuleInit() {
    console.log(`The module has been initialized.`);
  }
}

Asynchronous initialization #

Cả hai hooks OnModuleInit và OnApplicationBootstrap đều cho phép bạn trì hoãn quá trình initialization của ứng dụng (trả về một Promise hoặc đánh dấu method là async và await việc hoàn thành một phương thức bất đồng bộ trong thân method).

TS
JS
async onModuleInit(): Promise<void> {
  await this.fetch();
}
async onModuleInit() {
  await this.fetch();
}

Application shutdown #

Các hooks onModuleDestroy(), beforeApplicationShutdown() và onApplicationShutdown() được gọi trong giai đoạn terminating (để phản hồi lời gọi rõ ràng đến app.close() hoặc khi nhận được các system signals như SIGTERM nếu đã opt-in). Tính năng này thường được sử dụng với Kubernetes để quản lý lifecycle của containers, bởi Heroku cho dynos hoặc các services tương tự.

Các shutdown hook listeners tiêu thụ tài nguyên hệ thống, nên chúng bị tắt theo mặc định. Để sử dụng shutdown hooks, bạn phải bật listeners bằng cách gọi enableShutdownHooks():

main.ts
TS
JS
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // Starts listening for shutdown hooks
  app.enableShutdownHooks();

  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // Starts listening for shutdown hooks
  app.enableShutdownHooks();

  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
⚠️ Warning

Do các hạn chế cố hữu của platform, NestJS có hỗ trợ hạn chế cho application shutdown hooks trên Windows. Bạn có thể kỳ vọng SIGINT hoạt động, cũng như SIGBREAK và ở một mức độ nào đó là SIGHUP - đọc thêm. Tuy nhiên SIGTERM sẽ không bao giờ hoạt động trên Windows vì việc kill một tiến trình trong task manager là vô điều kiện, "tức là, không có cách nào để bất kỳ ứng dụng nào phát hiện hoặc ngăn chặn điều đó". Đây là một số tài liệu liên quan từ libuv để tìm hiểu thêm về cách SIGINT, SIGBREAK và các tín hiệu khác được xử lý trên Windows. Ngoài ra, xem tài liệu Node.js về Process Signal Events

ℹ️ Info

enableShutdownHooks tiêu thụ bộ nhớ bằng cách khởi động listeners. Trong trường hợp bạn đang chạy nhiều ứng dụng Nest trong một tiến trình Node duy nhất (ví dụ: khi chạy parallel tests với Jest), Node có thể cảnh báo về quá nhiều listener processes. Vì lý do này, enableShutdownHooks không được bật theo mặc định. Hãy lưu ý điều kiện này khi bạn đang chạy nhiều instances trong một tiến trình Node duy nhất.

Khi ứng dụng nhận được tín hiệu termination, nó sẽ gọi bất kỳ phương thức onModuleDestroy(), beforeApplicationShutdown(), rồi onApplicationShutdown() đã đăng ký (theo trình tự mô tả ở trên) với signal tương ứng làm tham số đầu tiên. Nếu một function đã đăng ký chờ đợi một lời gọi bất đồng bộ (trả về một promise), Nest sẽ không tiếp tục trình tự cho đến khi promise được resolved hoặc rejected.

TS
JS
@Injectable()
class UsersService implements OnApplicationShutdown {
  onApplicationShutdown(signal: string) {
    console.log(signal); // e.g. "SIGINT"
  }
}
@Injectable()
class UsersService {
  onApplicationShutdown(signal) {
    console.log(signal); // e.g. "SIGINT"
  }
}
ℹ️ Info

Việc gọi app.close() không kết thúc tiến trình Node mà chỉ kích hoạt các hooks onModuleDestroy() và onApplicationShutdown(), nên nếu có một số intervals, long-running background tasks, v.v. thì tiến trình sẽ không tự động được kết thúc.
