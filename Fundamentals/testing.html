Testing

Automated testing được coi là một phần thiết yếu của bất kỳ nỗ lực phát triển phần mềm nghiêm túc nào. Automation giúp dễ dàng lặp lại các test riêng lẻ hoặc test suites một cách nhanh chóng và dễ dàng trong quá trình phát triển. Điều này giúp đảm bảo rằng các bản release đáp ứng các mục tiêu chất lượng và hiệu năng. Automation giúp tăng coverage và cung cấp feedback loop nhanh hơn cho developers. Automation vừa tăng năng suất của từng developer vừa đảm bảo rằng các tests được chạy tại các thời điểm quan trọng trong development lifecycle, chẳng hạn như source code control check-in, feature integration, và version release.

Các tests này thường bao gồm nhiều loại, bao gồm unit tests, end-to-end (e2e) tests, integration tests, v.v. Mặc dù lợi ích là không thể phủ nhận, nhưng việc thiết lập chúng có thể rất tẻ nhạt. Nest cố gắng thúc đẩy các best practices phát triển, bao gồm testing hiệu quả, vì vậy nó bao gồm các tính năng sau để giúp developers và teams xây dựng và tự động hóa tests. Nest:

  • tự động scaffold các default unit tests cho components và e2e tests cho applications
  • cung cấp default tooling (chẳng hạn như test runner xây dựng một isolated module/application loader)
  • cung cấp tích hợp với Jest và Supertest out-of-the-box, trong khi vẫn giữ tính agnostic với testing tools
  • cung cấp Nest dependency injection system trong testing environment để dễ dàng mocking components

Như đã đề cập, bạn có thể sử dụng bất kỳ testing framework nào bạn thích, vì Nest không ép buộc bất kỳ tooling cụ thể nào. Chỉ cần thay thế các thành phần cần thiết (chẳng hạn như test runner), và bạn vẫn tận hưởng được lợi ích của các testing facilities sẵn có của Nest.

Installation #

Để bắt đầu, trước tiên cài đặt package cần thiết:

npm i --save-dev @nestjs/testing

Unit testing #

Trong ví dụ sau, chúng ta test hai classes: CatsController và CatsService. Như đã đề cập, Jest được cung cấp làm testing framework mặc định. Nó đóng vai trò test-runner và cũng cung cấp assert functions và test-double utilities hỗ trợ mocking, spying, v.v. Trong bài test cơ bản sau, chúng ta khởi tạo thủ công các classes này, và đảm bảo rằng controller và service thực hiện đúng API contract của chúng.

cats.controller.spec.ts
TS
JS
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(() => {
    catsService = new CatsService();
    catsController = new CatsController(catsService);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});

Testing framework mặc định (Jest) cung cấp assert functions và test utilities sẵn có. Trong trường hợp cơ bản như trên, chúng ta không cần các Nest testing tools. Khi phần mềm sử dụng dependency injection nhiều hơn, chúng ta sẽ cần dùng @nestjs/testing.

Testing utilities #

Package @nestjs/testing cung cấp một tập hợp các utilities cho phép xây dựng một quy trình testing mạnh mẽ hơn. Hãy viết lại ví dụ trước sử dụng class Test có sẵn:

cats.controller.spec.ts
TS
JS
import { Test } from '@nestjs/testing';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      controllers: [CatsController],
      providers: [CatsService],
    }).compile();

    catsService = moduleRef.get<CatsService>(CatsService);
    catsController = moduleRef.get<CatsController>(CatsController);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});
import { Test } from '@nestjs/testing';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController;
  let catsService;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      controllers: [CatsController],
      providers: [CatsService],
    }).compile();

    catsService = moduleRef.get(CatsService);
    catsController = moduleRef.get(CatsController);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});

Class Test hữu ích trong việc cung cấp một application execution context mô phỏng toàn bộ Nest runtime, nhưng cho phép bạn dễ dàng quản lý class instances, bao gồm mocking và overriding. Class Test có phương thức createTestingModule() nhận module metadata object làm đối số (cùng object mà bạn truyền vào @Module() decorator). Phương thức này trả về một instance của TestingModule mà lần lượt cung cấp một số methods. Đối với unit tests, method quan trọng là compile(). Method này bootstraps một module cùng với các dependencies của nó (tương tự cách ứng dụng được bootstrapped trong file main.ts thông thường sử dụng NestFactory.create()), và trả về module đã sẵn sàng cho testing.

💡 Hint

Phương thức compile() là asynchronous và do đó phải được awaited. Khi module đã được compiled, bạn có thể truy xuất bất kỳ static instance nào nó khai báo (controllers và providers) sử dụng phương thức get().

TestingModule kế thừa từ module reference class, và do đó có khả năng resolve dynamically các scoped providers (transient hoặc request-scoped). Làm điều này với phương thức resolve() (phương thức get() chỉ có thể truy xuất static instances).

const moduleRef = await Test.createTestingModule({
  controllers: [CatsController],
  providers: [CatsService],
}).compile();

catsService = await moduleRef.resolve(CatsService);
⚠️ Warning

Phương thức resolve() trả về một unique instance của provider, từ DI container sub-tree của chính nó. Mỗi sub-tree có một unique context identifier. Do đó, nếu bạn gọi method này nhiều lần và so sánh instance references, bạn sẽ thấy chúng không bằng nhau.

💡 Hint

Tìm hiểu thêm về các tính năng module reference tại đây.

Thay vì sử dụng production version của bất kỳ provider nào, bạn có thể override nó bằng một custom provider cho mục đích testing. Ví dụ, bạn có thể mock một database service thay vì kết nối đến live database. Chúng ta sẽ đề cập đến overrides trong phần tiếp theo, nhưng chúng cũng có sẵn cho unit tests.

Class TestingModule cung cấp các methods sau:

MethodMô tả
createNestApplication()Tạo và trả về một Nest application (INestApplication instance) dựa trên module đã cho. Lưu ý rằng ứng dụng phải được khởi tạo thủ công bằng phương thức init().
createNestMicroservice()Tạo và trả về một Nest microservice (INestMicroservice instance) dựa trên module đã cho.
get()Truy xuất một static instance của controller hoặc provider (bao gồm guards, filters, v.v.) có sẵn trong application context. Kế thừa từ module reference class.
resolve()Truy xuất một dynamically created scoped instance (request hoặc transient) của controller hoặc provider (bao gồm guards, filters, v.v.) có sẵn trong application context. Kế thừa từ module reference class.
select()Điều hướng qua dependency graph của module; có thể được sử dụng để truy xuất một specific instance từ selected module (sử dụng cùng với strict mode (strict: true) trong phương thức get()).
