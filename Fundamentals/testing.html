Testing

Automated testing được coi là một phần thiết yếu của bất kỳ nỗ lực phát triển phần mềm nghiêm túc nào. Automation giúp dễ dàng lặp lại các test riêng lẻ hoặc test suites một cách nhanh chóng và dễ dàng trong quá trình phát triển. Điều này giúp đảm bảo rằng các bản release đáp ứng các mục tiêu chất lượng và hiệu năng. Automation giúp tăng coverage và cung cấp feedback loop nhanh hơn cho developers. Automation vừa tăng năng suất của từng developer vừa đảm bảo rằng các tests được chạy tại các thời điểm quan trọng trong development lifecycle, chẳng hạn như source code control check-in, feature integration, và version release.

Các tests này thường bao gồm nhiều loại, bao gồm unit tests, end-to-end (e2e) tests, integration tests, v.v. Mặc dù lợi ích là không thể phủ nhận, nhưng việc thiết lập chúng có thể rất tẻ nhạt. Nest cố gắng thúc đẩy các best practices phát triển, bao gồm testing hiệu quả, vì vậy nó bao gồm các tính năng sau để giúp developers và teams xây dựng và tự động hóa tests. Nest:

  • tự động scaffold các default unit tests cho components và e2e tests cho applications
  • cung cấp default tooling (chẳng hạn như test runner xây dựng một isolated module/application loader)
  • cung cấp tích hợp với Jest và Supertest out-of-the-box, trong khi vẫn giữ tính agnostic với testing tools
  • cung cấp Nest dependency injection system trong testing environment để dễ dàng mocking components

Như đã đề cập, bạn có thể sử dụng bất kỳ testing framework nào bạn thích, vì Nest không ép buộc bất kỳ tooling cụ thể nào. Chỉ cần thay thế các thành phần cần thiết (chẳng hạn như test runner), và bạn vẫn tận hưởng được lợi ích của các testing facilities sẵn có của Nest.

Installation #

Để bắt đầu, trước tiên cài đặt package cần thiết:

npm i --save-dev @nestjs/testing

Unit testing #

Trong ví dụ sau, chúng ta test hai classes: CatsController và CatsService. Như đã đề cập, Jest được cung cấp làm testing framework mặc định. Nó đóng vai trò test-runner và cũng cung cấp assert functions và test-double utilities hỗ trợ mocking, spying, v.v. Trong bài test cơ bản sau, chúng ta khởi tạo thủ công các classes này, và đảm bảo rằng controller và service thực hiện đúng API contract của chúng.

Testing utilities #

Package @nestjs/testing cung cấp một tập hợp các utilities cho phép xây dựng một quy trình testing mạnh mẽ hơn. Hãy viết lại ví dụ trước sử dụng class Test có sẵn:

cats.controller.spec.ts
TS
JS
import { Test } from '@nestjs/testing';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      controllers: [CatsController],
      providers: [CatsService],
    }).compile();

    catsService = moduleRef.get<CatsService>(CatsService);
    catsController = moduleRef.get<CatsController>(CatsController);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});

import { CatsService } from './cats.service';

  let catsController;
  let catsService;

      providers: [CatsService],
    }).compile();

    catsService = moduleRef.get(CatsService);
    catsController = moduleRef.get(CatsController);
  });

    });
  });
});

Class Test hữu ích trong việc cung cấp một application execution context mô phỏng toàn bộ Nest runtime, nhưng cho phép bạn dễ dàng quản lý class instances, bao gồm mocking và overriding. Class Test có phương thức createTestingModule() nhận module metadata object làm đối số (cùng object mà bạn truyền vào @Module() decorator). Phương thức này trả về một instance của TestingModule mà lần lượt cung cấp một số methods. Đối với unit tests, method quan trọng là compile(). Method này bootstraps một module cùng với các dependencies của nó (tương tự cách ứng dụng được bootstrapped trong file main.ts thông thường sử dụng NestFactory.create()), và trả về module đã sẵn sàng cho testing.

💡 Hint

Phương thức compile() là asynchronous và do đó phải được awaited. Khi module đã được compiled, bạn có thể truy xuất bất kỳ static instance nào nó khai báo (controllers và providers) sử dụng phương thức get().

TestingModule kế thừa từ module reference class, và do đó có khả năng resolve dynamically các scoped providers (transient hoặc request-scoped). Làm điều này với phương thức resolve() (phương thức get() chỉ có thể truy xuất static instances).

const moduleRef = await Test.createTestingModule({
  controllers: [CatsController],
  providers: [CatsService],
}).compile();

catsService = await moduleRef.resolve(CatsService);
⚠️ Warning

Phương thức resolve() trả về một unique instance của provider, từ DI container sub-tree của chính nó. Mỗi sub-tree có một unique context identifier. Do đó, nếu bạn gọi method này nhiều lần và so sánh instance references, bạn sẽ thấy chúng không bằng nhau.

💡 Hint

Tìm hiểu thêm về các tính năng module reference tại đây.

Thay vì sử dụng production version của bất kỳ provider nào, bạn có thể override nó bằng một custom provider cho mục đích testing. Ví dụ, bạn có thể mock một database service thay vì kết nối đến live database. Chúng ta sẽ đề cập đến overrides trong phần tiếp theo, nhưng chúng cũng có sẵn cho unit tests.

Auto mocking #

Nest cũng cho phép bạn định nghĩa một mock factory để áp dụng cho tất cả các dependencies bị thiếu. Điều này hữu ích cho các trường hợp bạn có một số lượng lớn dependencies trong một class và việc mock tất cả chúng sẽ mất nhiều thời gian và thiết lập. Để sử dụng tính năng này, createTestingModule() cần được kết nối với phương thức useMocker(), truyền vào một factory cho các dependency mocks của bạn. Factory này có thể nhận một token tùy chọn, đó là một instance token, bất kỳ token nào hợp lệ cho một Nest provider, và trả về một mock implementation. Dưới đây là ví dụ tạo mock chung sử dụng jest-mock-extended và một mock cụ thể cho CatsService sử dụng jest.fn().

// ...
import { createMock } from 'jest-mock-extended';

const moduleRef = await Test.createTestingModule({
  controllers: [CatsController],
})
  .useMocker((token) => {
    if (token === CatsService) {
      return {
        findAll: jest.fn().mockResolvedValue(catArray),
      };
    }
    if (typeof token === 'function') {
      const mockMetadata = jest.requireActual('jest-mock');
      return new (mockMetadata.ModuleMocker(global).generateFromMetadata(
        mockMetadata.ModuleMocker(global).getMetadata(token),
      ))();
    }
  })
  .compile();

Bạn cũng có thể truy xuất các custom mocks này từ testing container như thông thường với các custom providers, ví dụ moduleRef.get(CatsService).

💡 Hint

Một generic mock factory tổng quát, như createMock từ @golevelup/ts-jest, cũng có thể được truyền trực tiếp.

💡 Hint

REQUEST và INQUIRER providers không thể được tự động mock vì chúng đã được chỉ định sẵn là context-scoped providers. Tuy nhiên, chúng có thể được overwritten bằng cách sử dụng custom provider syntax, ví dụ .overrideProvider(REQUEST).useValue({}).

End-to-end testing #

Không giống unit tests tập trung vào các modules và classes riêng lẻ, end-to-end (e2e) testing bao phủ sự tương tác của các classes và modules ở mức tổng thể hơn -- gần với loại tương tác mà end-users sẽ có với production system. Khi một ứng dụng phát triển, rất khó để kiểm tra thủ công end-to-end behavior của mỗi API endpoint. Automated end-to-end tests giúp chúng ta đảm bảo rằng overall behavior của hệ thống là chính xác và đáp ứng project requirements. Để thực hiện e2e tests, chúng ta sử dụng cấu hình tương tự như chúng ta vừa đề cập trong unit testing. Ngoài ra, Nest giúp dễ dàng sử dụng thư viện Supertest để mô phỏng HTTP requests.

cats.e2e-spec.ts
TS
JS
import * as request from 'supertest';
import { Test } from '@nestjs/testing';
import { CatsModule } from '../../src/cats/cats.module';
import { CatsService } from '../../src/cats/cats.service';
import { INestApplication } from '@nestjs/common';

describe('Cats', () => {
  let app: INestApplication;
  let catsService = { findAll: () => ['test'] };

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({
      imports: [CatsModule],
    })
      .overrideProvider(CatsService)
      .useValue(catsService)
      .compile();

    app = moduleRef.createNestApplication();
    await app.init();
  });

  it(`/GET cats`, () => {
    return request(app.getHttpServer())
      .get('/cats')
      .expect(200)
      .expect({
        data: catsService.findAll(),
      });
  });

  afterAll(async () => {
    await app.close();
  });
});

Trong ví dụ này, chúng ta xây dựng trên một số khái niệm đã mô tả trước đó. Ngoài phương thức compile() mà chúng ta đã sử dụng trước đó, giờ đây chúng ta sử dụng phương thức createNestApplication() để khởi tạo một Nest runtime environment đầy đủ.

Một lưu ý cần xem xét là khi ứng dụng của bạn được compiled sử dụng phương thức compile(), HttpAdapterHost#httpAdapter sẽ là undefined tại thời điểm đó. Điều này là vì chưa có HTTP adapter hoặc server nào được tạo trong giai đoạn compilation này. Nếu test của bạn yêu cầu httpAdapter, bạn nên sử dụng phương thức createNestApplication() để tạo application instance, hoặc refactor project của bạn để tránh dependency này khi khởi tạo dependencies graph.

Được rồi, hãy phân tích ví dụ:

Chúng ta lưu reference đến running app trong biến app để có thể sử dụng nó mô phỏng HTTP requests.

Chúng ta mô phỏng HTTP tests sử dụng hàm request() từ Supertest. Chúng ta muốn các HTTP requests này route đến running Nest app, vì vậy chúng ta truyền hàm request() một reference đến HTTP listener nằm bên dưới Nest (mà lần lượt, có thể được cung cấp bởi Express platform). Do đó construction request(app.getHttpServer()). Lời gọi request() đưa cho chúng ta một wrapped HTTP Server, giờ đã kết nối với Nest app, cung cấp các methods để mô phỏng actual HTTP request. Ví dụ, sử dụng request(...).get('/cats') sẽ khởi tạo một request giống hệt HTTP request thực tế như get '/cats' đến qua network.

Trong ví dụ này, chúng ta cũng cung cấp một alternate (test-double) implementation của CatsService mà đơn giản trả về một giá trị hard-coded mà chúng ta có thể test. Sử dụng overrideProvider() để cung cấp alternate implementation như vậy. Tương tự, Nest cung cấp các methods để override modules, guards, interceptors, filters và pipes tương ứng với các phương thức overrideModule(), overrideGuard(), overrideInterceptor(), overrideFilter(), và overridePipe().

Mỗi phương thức override (ngoại trừ overrideModule()) trả về một object với 3 methods khác nhau phản chiếu những methods đã mô tả cho custom providers:

  • useClass: bạn cung cấp một class sẽ được instantiated để provide instance thay thế object (provider, guard, v.v.).
  • useValue: bạn cung cấp một instance sẽ override object.
  • useFactory: bạn cung cấp một function trả về một instance sẽ override object.

Mặt khác, overrideModule() trả về một object với phương thức useModule(), mà bạn có thể sử dụng để cung cấp một module sẽ override module gốc, như sau:

const moduleRef = await Test.createTestingModule({
  imports: [AppModule],
})
  .overrideModule(CatsModule)
  .useModule(AlternateCatsModule)
  .compile();

Mỗi loại phương thức override, nối tiếp nhau, trả về instance của TestingModule, và do đó có thể kết nối chuỗi với các method khác theo fluent style. Bạn nên sử dụng compile() ở cuối chuỗi như vậy để Nest khởi tạo và trả về module.

Ngoài ra, đôi khi bạn có thể muốn cung cấp một custom logger (ví dụ, khi tests chạy trên CI server). Sử dụng phương thức setLogger() và truyền vào một object thực hiện LoggerService interface để hướng dẫn TestModuleBuilder cách log trong quá trình tests (mặc định, chỉ "error" logs được log ra console).

Module đã compiled có một số methods hữu ích, như mô tả trong bảng sau:

MethodMô tả
createNestApplication()Tạo và trả về một Nest application (INestApplication instance) dựa trên module đã cho. Lưu ý rằng ứng dụng phải được khởi tạo thủ công bằng phương thức init().
createNestMicroservice()Tạo và trả về một Nest microservice (INestMicroservice instance) dựa trên module đã cho.

Overriding globally registered enhancers #

Nếu bạn có một globally registered guard (hoặc pipe, interceptor, hoặc filter), bạn cần thực hiện thêm một vài bước để override enhancer đó. Để ôn lại, registration ban đầu trông như thế này:

providers: [
  {
    provide: APP_GUARD,
    useClass: JwtAuthGuard,
  },
],

Đây là việc đăng ký guard dưới dạng "multi"-provider thông qua token APP_*. Để có thể thay thế JwtAuthGuard ở đây, registration cần sử dụng một existing provider trong slot này:

providers: [
  {
    provide: APP_GUARD,
    useExisting: JwtAuthGuard,
  },
  JwtAuthGuard,
],
💡 Hint

Thay đổi useClass thành useExisting để tham chiếu một registered provider thay vì để Nest khởi tạo nó đằng sau token.

Bây giờ JwtAuthGuard là một regular provider có thể nhìn thấy bởi Nest khi tạo TestingModule:

const moduleRef = await Test.createTestingModule({
  imports: [AppModule],
})
  .overrideProvider(JwtAuthGuard)
  .useClass(MockAuthGuard)
  .compile();

Bây giờ tất cả các tests sẽ sử dụng MockAuthGuard trên mọi request.

Testing request-scoped instances #

Request-scoped providers được tạo uniquely cho mỗi request đến. Instance được garbage-collected sau khi request đã hoàn tất quá trình xử lý. Điều này đặt ra một vấn đề, bởi vì chúng ta không thể truy cập một dependency injection sub-tree được tạo cụ thể cho một tested request.

Chúng ta biết (dựa trên các phần trên) rằng phương thức resolve() có thể được sử dụng để truy xuất một dynamically instantiated class. Ngoài ra, như đã mô tả tại đây, chúng ta biết rằng có thể truyền một unique context identifier để kiểm soát lifecycle của một DI container sub-tree. Làm cách nào sử dụng điều này trong testing context?

Chiến lược là tạo một context identifier trước và buộc Nest sử dụng ID cụ thể này để tạo sub-tree cho tất cả incoming requests. Bằng cách này, chúng ta sẽ có thể truy xuất instances được tạo cho một tested request.

Để thực hiện điều này, sử dụng jest.spyOn() trên ContextIdFactory:

TS
JS
const contextId = ContextIdFactory.create();
jest
  .spyOn(ContextIdFactory, 'getByRequest')
  .mockImplementation(() => contextId);
const contextId = ContextIdFactory.create();
jest
