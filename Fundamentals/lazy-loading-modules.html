Lazy-loading modules

Mặc định, module được eagerly loaded, nghĩa là ngay khi ứng dụng load, tất cả module cũng được load theo, bất kể chúng có cần thiết ngay lập tức hay không. Mặc dù điều này ổn cho hầu hết ứng dụng, nó có thể trở thành bottleneck cho app/worker chạy trong serverless environment, nơi startup latency ("cold start") là yếu tố quan trọng.

Lazy loading có thể giúp giảm thời gian bootstrap bằng cách chỉ load các module cần thiết cho từng lần gọi serverless function cụ thể. Ngoài ra, bạn cũng có thể load các module khác một cách bất đồng bộ sau khi serverless function đã "warm" để tăng tốc thời gian bootstrap cho các lần gọi tiếp theo (deferred modules registration).

💡 Hint

Nếu bạn quen thuộc với framework Angular, có thể bạn đã thấy thuật ngữ "lazy-loading modules" trước đó. Hãy lưu ý rằng kỹ thuật này khác biệt về chức năng trong Nest và hãy nghĩ về nó như một tính năng hoàn toàn khác chỉ có chung quy ước đặt tên.

⚠️ Warning

Lưu ý rằng các lifecycle hook method không được gọi trong lazy loaded module và service.

Getting started #

Để load module theo yêu cầu (on-demand), Nest cung cấp class LazyModuleLoader có thể được inject vào class theo cách thông thường:

cats.service
TS
JS
@Injectable()
export class CatsService {
  constructor(private lazyModuleLoader: LazyModuleLoader) {}
}
@Injectable()
@Dependencies(LazyModuleLoader)
export class CatsService {
  constructor(lazyModuleLoader) {
    this.lazyModuleLoader = lazyModuleLoader;
  }
}

Ngoài ra, bạn có thể lấy tham chiếu đến LazyModuleLoader từ bên trong file bootstrap ứng dụng (main.ts), như sau:

// "app" ở đây đại diện cho Nest application instance
const lazyModuleLoader = app.get(LazyModuleLoader);

Bây giờ, bạn có thể load bất kỳ module nào bằng cách sử dụng construction sau:

const { LazyModule } = await import('./lazy.module');
const moduleRef = await this.lazyModuleLoader.load(() => LazyModule);
💡 Hint

"Lazy loaded" module được cache từ lần gọi đầu tiên của method LazyModuleLoader#load. Điều đó nghĩa là mỗi lần cố gắng load LazyModule tiếp theo sẽ rất nhanh và trả về cached instance, thay vì load lại module.

Load "LazyModule" attempt: 1
time: 2.379ms
Load "LazyModule" attempt: 2
time: 0.294ms
Load "LazyModule" attempt: 3
time: 0.303ms

Ngoài ra, "lazy loaded" module chia sẻ cùng module graph với các module được eagerly loaded khi bootstrap ứng dụng cũng như bất kỳ lazy module nào khác được đăng ký sau đó trong app.

Trong đó lazy.module.ts là file TypeScript export một Nest module thông thường (không cần thay đổi thêm gì).

Method LazyModuleLoader#load trả về module reference (của LazyModule) cho phép bạn duyệt danh sách nội bộ các provider và lấy tham chiếu đến bất kỳ provider nào bằng injection token làm lookup key.

Ví dụ, giả sử chúng ta có LazyModule với định nghĩa sau:

@Module({
  providers: [LazyService],
  exports: [LazyService],
})
export class LazyModule {}
💡 Hint

Lazy loaded module không thể được đăng ký như global module vì điều đó đơn giản là không có ý nghĩa (vì chúng được đăng ký lazily, theo yêu cầu khi tất cả module được đăng ký tĩnh đã được khởi tạo). Tương tự, các global enhancer đã đăng ký (guard/interceptor/v.v.) cũng sẽ không hoạt động đúng cách.

Để lấy tham chiếu đến provider LazyService, chúng ta sử dụng instance moduleRef, như sau:

const { LazyModule } = await import('./lazy.module');
const moduleRef = await this.lazyModuleLoader.load(() => LazyModule);

const { LazyService } = await import('./lazy.service');
const lazyService = moduleRef.get(LazyService);

Lazy loading controllers, gateways, and resolvers #

Vì controller (hoặc resolver trong ứng dụng GraphQL) trong Nest đại diện cho các tập route/path/topic (hoặc query/mutation), bạn không thể lazy load chúng bằng class LazyModuleLoader.

⚠️ Warning

Controller, gateway, và resolver được đăng ký bên trong lazy loaded module sẽ không hoạt động như mong đợi.

Tương tự, bạn không thể đăng ký middleware function (bằng cách implement interface NestModule) on-the-fly.

Cuối cùng, package @nestjs/graphql với code first approach được bật sẽ tự động generate GraphQL schema on-the-fly dựa trên metadata. Điều đó nghĩa là nó yêu cầu tất cả class phải được load trước. Nếu không, sẽ không thể tạo ra schema phù hợp và hợp lệ.

Common use-cases #

Thông thường, bạn sẽ thấy lazy loaded module trong các tình huống khi worker/cron job/lambda & serverless function/webhook phải trigger các service khác nhau (logic khác nhau) dựa trên input argument (route path/date/query parameter, v.v.). Mặt khác, lazy loading module có thể không có nhiều ý nghĩa cho ứng dụng monolithic, nơi startup time tương đối không quan trọng.
