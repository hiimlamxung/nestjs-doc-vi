Injection scopes

Đối với những người đến từ các ngôn ngữ lập trình khác, có thể bất ngờ khi biết rằng trong Nest, hầu hết mọi thứ đều được chia sẻ giữa các incoming request. Chúng ta có connection pool đến database, singleton service với global state, v.v. Hãy nhớ rằng Node.js không tuân theo mô hình Request/Response Multi-Threaded Stateless Model nơi mỗi request được xử lý bởi một thread riêng biệt. Do đó, việc sử dụng singleton instance hoàn toàn an toàn cho ứng dụng của chúng ta.

Tuy nhiên, có những trường hợp ngoại lệ khi request-based lifetime có thể là behavior mong muốn, ví dụ: per-request caching trong ứng dụng GraphQL, request tracking, và multi-tenancy. Injection scope cung cấp cơ chế để đạt được provider lifetime behavior mong muốn.

Provider scope #

Một provider có thể có bất kỳ scope nào sau đây:

  • DEFAULT: Một instance duy nhất của provider được chia sẻ trong toàn bộ ứng dụng. Vòng đời instance gắn trực tiếp với vòng đời ứng dụng. Khi ứng dụng đã bootstrap xong, tất cả singleton provider đã được khởi tạo. Singleton scope được sử dụng mặc định.
  • REQUEST: Một instance mới của provider được tạo riêng cho mỗi incoming request. Instance này sẽ được garbage-collected sau khi request xử lý xong.
  • TRANSIENT: Transient provider không được chia sẻ giữa các consumer. Mỗi consumer inject một transient provider sẽ nhận được một instance mới, riêng biệt.
💡 Hint

Sử dụng singleton scope được khuyến nghị cho hầu hết các use case. Chia sẻ provider giữa các consumer và giữa các request nghĩa là instance có thể được cache và quá trình khởi tạo chỉ xảy ra một lần, trong lúc khởi động ứng dụng.

Usage #

Chỉ định injection scope bằng cách truyền property scope vào object options của @Injectable() decorator:

TS
JS
import { Injectable, Scope } from '@nestjs/common';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {}
import { Injectable, Scope } from '@nestjs/common';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {}

Tương tự, cho custom provider, set property scope trong long-hand form cho provider registration:

{
  provide: 'CACHE_MANAGER',
  useClass: CacheManager,
  scope: Scope.TRANSIENT,
}
💡 Hint

Import Scope enum từ @nestjs/common.

Singleton scope được sử dụng mặc định và không cần khai báo. Nếu bạn muốn khai báo rõ ràng provider là singleton scoped, sử dụng giá trị Scope.DEFAULT cho property scope.

📘 Notice

Websocket Gateway không nên sử dụng request-scoped provider vì chúng phải hoạt động như singleton. Mỗi gateway đóng gói một real socket và không thể được khởi tạo nhiều lần. Hạn chế này cũng áp dụng cho một số provider khác, như Passport strategies hoặc Cron controllers.

Controller scope #

Controller cũng có thể có scope, áp dụng cho tất cả request method handler được khai báo trong controller đó. Giống như provider scope, scope của controller khai báo vòng đời của nó. Với request-scoped controller, một instance mới được tạo cho mỗi inbound request, và được garbage-collected khi request hoàn thành xử lý.

Chỉ định controller scope với property scope của object ControllerOptions:

@Controller({
  path: 'cats',
  scope: Scope.REQUEST,
})
export class CatsController {}

Scope hierarchy #

Scope REQUEST lan truyền lên trên (bubble up) chuỗi injection. Một controller phụ thuộc vào request-scoped provider sẽ tự động trở thành request-scoped.

Hãy tưởng tượng dependency graph sau: CatsController <- CatsService <- CatsRepository. Nếu CatsService là request-scoped (và các thành phần khác là default singleton), CatsController sẽ trở thành request-scoped vì nó phụ thuộc vào injected service. CatsRepository, không phụ thuộc vào request-scoped provider, sẽ vẫn giữ singleton scoped.

Transient-scoped dependency không tuân theo pattern đó. Nếu singleton-scoped DogsService inject một transient LoggerService provider, nó sẽ nhận được instance mới của LoggerService. Tuy nhiên, DogsService vẫn giữ singleton scope, nên inject DogsService ở bất kỳ đâu sẽ không resolve ra instance mới của DogsService. Nếu đây là behavior mong muốn, DogsService phải được đánh dấu rõ ràng là TRANSIENT.

Request provider #

Trong ứng dụng HTTP server-based (ví dụ: sử dụng @nestjs/platform-express hoặc @nestjs/platform-fastify), bạn có thể muốn truy cập tham chiếu đến original request object khi sử dụng request-scoped provider. Bạn có thể làm điều này bằng cách inject REQUEST object.

cats.service.ts
TS
JS
import { Injectable, Scope, Inject } from '@nestjs/common';
import { REQUEST } from '@nestjs/core';
import { Request } from 'express';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(REQUEST) private request: Request) {}
}
import { Injectable, Scope, Inject } from '@nestjs/common';
import { REQUEST } from '@nestjs/core';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(REQUEST) private request) {}
}

REQUEST provider vốn là request-scoped, nghĩa là bạn không cần chỉ định scope REQUEST một cách rõ ràng khi sử dụng nó. Ngoài ra, ngay cả khi bạn cố tình làm vậy, nó sẽ bị bỏ qua. Bất kỳ provider nào phụ thuộc vào request-scoped provider sẽ tự động áp dụng request scope, và behavior này không thể thay đổi.

Do sự khác biệt về platform/protocol nền tảng, bạn truy cập request object hơi khác cho ứng dụng Microservice hoặc GraphQL. Trong ứng dụng GraphQL, bạn inject CONTEXT thay vì REQUEST:

cats.service.ts
TS
JS
import { Injectable, Scope, Inject } from '@nestjs/common';
import { CONTEXT } from '@nestjs/graphql';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(CONTEXT) private context) {}
}
import { Injectable, Scope, Inject } from '@nestjs/common';
import { CONTEXT } from '@nestjs/graphql';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(CONTEXT) private context) {}
}

Sau đó bạn cấu hình giá trị context (trong GraphQLModule) để chứa request làm property của nó.

Inquirer provider #

Nếu bạn muốn lấy ra class nơi mà provider đã được inject, ví dụ trong logging hoặc metrics provider, bạn có thể inject token INQUIRER.

hello.service.ts
TS
JS
import { Injectable, Scope, Inject } from '@nestjs/common';
import { INQUIRER } from '@nestjs/core';

@Injectable({ scope: Scope.TRANSIENT })
export class HelloService {
  constructor(@Inject(INQUIRER) private parentClass: object) {}

  sayHello(message: string) {
    console.log(`${this.parentClass?.constructor?.name}: ${message}`);
  }
}
import { Injectable, Scope, Inject } from '@nestjs/common';
import { INQUIRER } from '@nestjs/core';

@Injectable({ scope: Scope.TRANSIENT })
export class HelloService {
  constructor(@Inject(INQUIRER) private parentClass) {}

  sayHello(message) {
    console.log(`${this.parentClass?.constructor?.name}: ${message}`);
  }
}

Sau đó nó có thể được sử dụng như sau:

app.service.ts
TS
JS
import { Injectable } from '@nestjs/common';
import { HelloService } from './hello.service';

@Injectable()
export class AppService {
  constructor(private helloService: HelloService) {}

  getRoot() {
    this.helloService.sayHello('My message');
  }
}
import { Injectable } from '@nestjs/common';
import { HelloService } from './hello.service';

@Injectable()
export class AppService {
  constructor(private helloService) {}

  getRoot() {
    this.helloService.sayHello('My message');
  }
}

Trong ví dụ trên, khi AppService#getRoot được gọi, "AppService: My message" sẽ được log ra console.

Performance #

Sử dụng request-scoped provider sẽ ảnh hưởng đến hiệu năng ứng dụng. Mặc dù Nest cố gắng cache càng nhiều metadata càng tốt, nó vẫn phải tạo instance mới của class bạn trên mỗi request. Do đó, nó sẽ làm chậm thời gian response trung bình và kết quả benchmarking tổng thể. Trừ khi provider bắt buộc phải là request-scoped, bạn được khuyến nghị mạnh mẽ sử dụng default singleton scope.

💡 Hint

Mặc dù nghe có vẻ đáng lo ngại, một ứng dụng được thiết kế đúng cách với request-scoped provider không nên chậm hơn khoảng ~5% về latency.

Durable providers #

Request scope, như đã lưu ý ở section trên, có thể dẫn đến tăng latency vì có ít nhất một request-scoped provider (inject vào controller instance, hoặc sâu hơn — inject vào một trong các provider của nó) khiến controller cũng trở thành request-scoped. Điều đó nghĩa là nó phải được khởi tạo (create) cho mỗi request riêng lẻ (và garbage collected sau đó). Điều này cũng nghĩa là, ví dụ, với 30.000 parallel request, sẽ có 30.000 ephemeral instance của controller (và các request-scoped provider của nó).

Khi có một common provider mà phần lớn provider phụ thuộc vào (ví dụ như database connection, hoặc logger service), nó tự động chuyển tất cả các provider đó thành request-scoped. Điều này có thể gây ra vấn đề trong ứng dụng multi-tenant, nơi header hoặc cookie chứa "tenant ID" trung tâm và cần được dùng để trích xuất database-specific connection/schema cho mỗi request đó.

Nếu bạn không nhất thiết muốn mỗi request tạo DI sub-tree mới — bạn chỉ muốn tạo per-tenant thay vì per-request — thì không có lý do gì để đánh dấu tất cả các provider đó là REQUEST scoped. Đây là lúc durable provider phát huy tác dụng.

Trước khi đánh dấu provider là durable, trước tiên chúng ta cần đăng ký một strategy hướng dẫn Nest cách "aggregate" — nhóm request lại với nhau, liên kết chúng với các DI sub-tree tương ứng.

aggregate-by-tenant.strategy.ts
TS
JS
import {
  HostComponentInfo,
  ContextId,
  ContextIdFactory,
  ContextIdStrategy,
} from '@nestjs/core';
import { Request } from 'express';

const tenants = new Map<string, ContextId>();

export class AggregateByTenantContextIdStrategy
  implements ContextIdStrategy
{
  attach(contextId: ContextId, request: Request) {
    const tenantId = request.headers['x-tenant-id'] as string;
    if (!tenantId) {
      return () => contextId;
    }

    let tenantSubTreeId: ContextId;
    if (tenants.has(tenantId)) {
      tenantSubTreeId = tenants.get(tenantId);
    } else {
      tenantSubTreeId = ContextIdFactory.create();
      tenants.set(tenantId, tenantSubTreeId);
    }

    return (info: HostComponentInfo) =>
      info.isTreeDurable ? tenantSubTreeId : contextId;
  }
}
import {
  ContextIdFactory,
} from '@nestjs/core';

const tenants = new Map();

Sau đó, đăng ký strategy đó trong file main.ts:

main.ts
TS
JS
import { NestFactory, ContextIdFactory } from '@nestjs/core';
import { AppModule } from './app.module';
import { AggregateByTenantContextIdStrategy } from './aggregate-by-tenant.strategy';

ContextIdFactory.apply(new AggregateByTenantContextIdStrategy());

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  await app.listen(3000);
}
bootstrap();

💡 Hint

ContextIdFactory class được import từ package @nestjs/core.

Miễn là quá trình đăng ký xảy ra trước khi bất kỳ request nào đến, mọi thứ sẽ hoạt động như dự kiến.

Tiếp theo, để đánh dấu một provider là durable, chỉ cần set flag durable thành true và thay đổi scope sang Scope.REQUEST:

cats.service.ts
TS
JS
@Injectable({ scope: Scope.REQUEST, durable: true })
export class CatsService {}
@Injectable({ scope: Scope.REQUEST, durable: true })
export class CatsService {}

Tương tự, cho custom provider, set flag durable trong long-hand form cho provider registration:

{
  provide: 'CACHE_MANAGER',
  useClass: CacheManager,
  scope: Scope.REQUEST,
  durable: true,
}
💡 Hint

Tương tự request scope, durability cũng lan truyền lên trên (bubble up) chuỗi injection. Nghĩa là nếu A phụ thuộc vào B được đánh dấu là durable, A ngầm trở thành durable (trừ khi durable được đặt rõ ràng thành false cho provider A).

⚠️ Warning

Lưu ý strategy này không lý tưởng cho ứng dụng hoạt động với số lượng tenant rất lớn.
