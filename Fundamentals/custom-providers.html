Custom providers

Trong các chương trước, chúng ta đã đề cập đến nhiều khía cạnh khác nhau của Dependency Injection (DI) và cách nó được sử dụng trong Nest. Một ví dụ về điều này là constructor based dependency injection được sử dụng để inject các instance (thường là service provider) vào các class. Bạn sẽ không ngạc nhiên khi biết rằng Dependency Injection được xây dựng sâu vào lõi của Nest theo một cách cơ bản. Cho đến nay, chúng ta mới chỉ khám phá một pattern chính. Khi ứng dụng của bạn trở nên phức tạp hơn, bạn có thể cần tận dụng đầy đủ các tính năng của hệ thống DI, vì vậy hãy cùng khám phá chúng chi tiết hơn.

DI fundamentals #

Dependency injection là một kỹ thuật inversion of control (IoC), trong đó bạn ủy thác việc khởi tạo các dependency cho IoC container (trong trường hợp của chúng ta là NestJS runtime system), thay vì tự thực hiện nó trong code một cách imperative. Hãy xem điều gì xảy ra trong ví dụ này từ chương Providers.

Đầu tiên, chúng ta định nghĩa một provider. Decorator @Injectable() đánh dấu class CatsService là một provider.

cats.service.ts
TS
JS
import { Injectable } from '@nestjs/common';
import { Cat } from './interfaces/cat.interface';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  findAll(): Cat[] {
    return this.cats;
  }
}
import { Injectable } from '@nestjs/common';

@Injectable()
export class CatsService {
  constructor() {
    this.cats = [];
  }

  findAll() {
    return this.cats;
  }
}

Sau đó, chúng ta yêu cầu Nest inject provider vào class controller:

cats.controller.ts
TS
JS
import { Controller, Get } from '@nestjs/common';
import { CatsService } from './cats.service';
import { Cat } from './interfaces/cat.interface';

@Controller('cats')
export class CatsController {
  constructor(private catsService: CatsService) {}

  @Get()
  async findAll(): Promise<Cat[]> {
    return this.catsService.findAll();
  }
}
import { Controller, Get, Bind, Dependencies } from '@nestjs/common';
import { CatsService } from './cats.service';

@Controller('cats')
@Dependencies(CatsService)
export class CatsController {
  constructor(catsService) {
    this.catsService = catsService;
  }

  @Get()
  findAll() {
    return this.catsService.findAll();
  }
}

Cuối cùng, chúng ta đăng ký provider với Nest IoC container:

app.module.ts
TS
JS
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class AppModule {}
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class AppModule {}

Chính xác thì điều gì đang xảy ra trong quy trình này? Có ba bước chính:

  • Trong cats.service.ts, decorator @Injectable() khai báo class CatsService là một class có thể được quản lý bởi Nest IoC container.
  • Trong cats.controller.ts, CatsController khai báo dependency vào token CatsService bằng constructor injection: 
    constructor(private catsService: CatsService) {}
  • Trong app.module.ts, chúng ta liên kết token CatsService với class CatsService từ file cats.service.ts. Chúng ta sẽ thấy chính xác cách liên kết này (còn gọi là registration) xảy ra bên dưới.

Khi Nest IoC container khởi tạo CatsController, nó đầu tiên sẽ tìm kiếm bất kỳ dependency nào*. Khi nó tìm thấy dependency CatsService, nó thực hiện tra cứu trên token CatsService, trả về class CatsService, theo bước đăng ký (#3 ở trên). Giả sử scope SINGLETON (behavior mặc định), Nest sau đó sẽ tạo một instance của CatsService, cache nó và trả về, hoặc nếu đã có một instance được cache, trả về instance hiện có.

*Giải thích này được đơn giản hóa một chút để minh họa ý tưởng. Một lĩnh vực quan trọng mà chúng ta đã lược qua là quá trình phân tích code để tìm dependency rất sâu - nó xảy ra trong quá trình khởi tạo ứng dụng. Một đặc điểm chính là phân tích dependency (hay "tạo dependency graph") là transitive. Trong ví dụ trên, nếu bản thân CatsService có dependency, những dependency đó cũng sẽ được resolved. Dependency graph đảm bảo rằng các dependency được resolved đúng thứ tự — về cơ bản là "bottom up". Cơ chế này giúp developer không phải quản lý các dependency graph phức tạp.

Standard providers #

Hãy xem kỹ hơn decorator @Module(). Trong app.module, chúng ta khai báo:

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})

Property providers nhận một mảng các provider. Cho đến nay, chúng ta đã cung cấp provider thông qua danh sách tên class. Trên thực tế, cú pháp providers: [CatsService] là cách viết tắt của cú pháp đầy đủ hơn:

providers: [
  {
    provide: CatsService,
    useClass: CatsService,
  },
];

Bây giờ chúng ta thấy rõ construct này, chúng ta có thể hiểu quá trình đăng ký. Ở đây, chúng ta rõ ràng đang liên kết token CatsService với class CatsService. Ký hiệu viết tắt chỉ là sự tiện lợi để đơn giản hóa use-case phổ biến nhất, nơi token được sử dụng để yêu cầu một instance của class cùng tên.

Custom providers #

Điều gì xảy ra khi yêu cầu của bạn vượt quá những gì Standard providers cung cấp? Dưới đây là một vài ví dụ:

  • Bạn muốn tạo một custom instance thay vì để Nest khởi tạo (hoặc trả về instance đã cache) của class
  • Bạn muốn tái sử dụng một class hiện có trong dependency thứ hai
  • Bạn muốn override một class bằng mock version để testing

Nest cho phép bạn định nghĩa Custom providers để xử lý các trường hợp này. Nó cung cấp một vài cách để định nghĩa custom provider. Hãy cùng xem.

💡 Hint

Nếu bạn gặp vấn đề với dependency resolution, bạn có thể set biến NEST_DEBUG và nhận thêm dependency resolution log trong quá trình startup.

Value providers: useValue #

Cú pháp useValue hữu ích để inject một constant value, đưa thư viện bên ngoài vào Nest container, hoặc thay thế implementation thực tế bằng mock object. Giả sử bạn muốn buộc Nest sử dụng mock CatsService cho mục đích testing.

import { CatsService } from './cats.service';

const mockCatsService = {
  /* mock implementation
  ...
  */
};

@Module({
  imports: [CatsModule],
  providers: [
    {
      provide: CatsService,
      useValue: mockCatsService,
    },
  ],
})
export class AppModule {}

Trong ví dụ này, token CatsService sẽ resolve thành mock object mockCatsService. useValue yêu cầu một value — trong trường hợp này là literal object có cùng interface với class CatsService mà nó thay thế. Nhờ structural typing của TypeScript, bạn có thể sử dụng bất kỳ object nào có compatible interface, bao gồm literal object hoặc class instance được khởi tạo bằng new.

Non-class-based provider tokens #

Cho đến nay, chúng ta đã sử dụng class name làm provider token (giá trị của property provide trong provider object được liệt kê trong mảng providers). Đây là pattern tiêu chuẩn với constructor based injection, nơi token là class name. (Tham khảo DI Fundamentals để ôn lại về token). Đôi khi, chúng ta có thể muốn linh hoạt sử dụng string hoặc symbol làm DI token. Ví dụ:

import { connection } from './connection';

@Module({
  providers: [
    {
      provide: 'CONNECTION',
      useValue: connection,
    },
  ],
})
export class AppModule {}

Trong ví dụ này, chúng ta liên kết string-valued token ('CONNECTION') với object connection đã import từ file bên ngoài.

💡 Hint

Ngoài việc sử dụng string làm token value, bạn cũng có thể sử dụng JavaScript symbol hoặc TypeScript enum.

Trước đó chúng ta đã thấy cách inject provider sử dụng constructor based injection pattern tiêu chuẩn. Pattern này yêu cầu dependency được khai báo với class name. Custom provider 'CONNECTION' sử dụng string-valued token. Hãy xem cách inject provider như vậy. Để làm điều đó, chúng ta sử dụng decorator @Inject(). Decorator này nhận một argument duy nhất — token.

TS
JS
@Injectable()
export class CatsRepository {
  constructor(@Inject('CONNECTION') connection: Connection) {}
}
@Injectable()
@Inject('CONNECTION')
export class CatsRepository {
  constructor(connection) {}
}
💡 Hint

Decorator @Inject() được import từ package @nestjs/common.

Mặc dù chúng ta sử dụng trực tiếp string 'CONNECTION' trong ví dụ trên để minh họa, nhưng theo best practice, bạn nên định nghĩa token trong file riêng biệt, chẳng hạn constants.ts. Hãy coi chúng giống như symbol hoặc enum được định nghĩa trong file riêng của chúng và import khi cần.

Class providers: useClass #

Cú pháp useClass cho phép bạn xác định động class mà token sẽ resolve thành. Ví dụ, giả sử chúng ta có class ConfigService abstract (hoặc default). Tùy thuộc vào môi trường hiện tại, chúng ta muốn Nest cung cấp implementation khác nhau của configuration service.

const configServiceProvider = {
  provide: ConfigService,
  useClass:
    process.env.NODE_ENV === 'development'
      ? DevelopmentConfigService
      : ProductionConfigService,
};

@Module({
  providers: [configServiceProvider],
})
export class AppModule {}

Lưu ý rằng chúng ta đã sử dụng class name ConfigService làm token. Với bất kỳ class nào phụ thuộc vào ConfigService, Nest sẽ inject instance của class được cung cấp (DevelopmentConfigService hoặc ProductionConfigService) thay thế bất kỳ default implementation nào có thể đã được khai báo ở nơi khác (ví dụ: ConfigService được khai báo với decorator @Injectable()).

Factory providers: useFactory #

Cú pháp useFactory cho phép tạo provider dynamically. Provider thực tế sẽ được cung cấp bởi value trả về từ factory function. Factory function có thể đơn giản hoặc phức tạp tùy ý. Factory đơn giản có thể không phụ thuộc vào bất kỳ provider nào khác. Factory phức tạp hơn có thể tự inject các provider khác mà nó cần để tính toán kết quả. Đối với trường hợp sau, factory provider syntax có một cặp cơ chế liên quan:

  • Factory function có thể chấp nhận (optional) argument.
  • Property inject (optional) chấp nhận một mảng provider mà Nest sẽ resolve và truyền như argument vào factory function trong quá trình khởi tạo. Ngoài ra, các provider này có thể được đánh dấu là optional. Hai danh sách này có mối tương quan: Nest sẽ truyền instance từ danh sách inject như argument vào factory function theo cùng thứ tự.

Ví dụ dưới đây minh họa điều này.

const connectionProvider = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider, optionalProvider?: string) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider, { token: 'SomeOptionalProvider', optional: true }],
};

@Module({
  providers: [
    connectionProvider,
    OptionsProvider,
    // { provide: 'SomeOptionalProvider', useValue: 'anything' },
  ],
})
export class AppModule {}

Alias providers: useExisting #

Cú pháp useExisting cho phép bạn tạo alias cho các provider hiện có. Điều này tạo ra hai cách để truy cập cùng một provider. Trong ví dụ bên dưới, string-based token 'AliasedLoggerService' là alias cho class-based token LoggerService. Giả sử chúng ta có hai dependency khác nhau, một dùng 'AliasedLoggerService' và một dùng LoggerService. Nếu cả hai dependency đều được đăng ký với scope SINGLETON, chúng sẽ resolve vào cùng một instance.

@Injectable()
class LoggerService {
  /* implementation details */
}

const loggerAliasProvider = {
  provide: 'AliasedLoggerService',
  useExisting: LoggerService,
};

@Module({
  providers: [LoggerService, loggerAliasProvider],
})
export class AppModule {}

Non-service based providers #

Mặc dù provider thường cung cấp service, chúng không giới hạn ở mục đích sử dụng đó. Provider có thể cung cấp bất kỳ value nào. Ví dụ, provider có thể cung cấp mảng configuration object dựa trên môi trường hiện tại, như bên dưới:

const configFactory = {
  provide: 'CONFIG',
  useFactory: () => {
    return process.env.NODE_ENV === 'development' ? devConfig : prodConfig;
  },
};

Export custom provider #

Giống như bất kỳ provider nào, custom provider được giới hạn trong scope của module khai báo nó (declaring module). Để các module khác có thể nhìn thấy, nó phải được export. Để export custom provider, chúng ta có thể sử dụng token của nó hoặc toàn bộ provider object.

Ví dụ sau cho thấy cách export sử dụng token:

const connectionFactory = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider],
};

@Module({
  providers: [connectionFactory],
  exports: ['CONNECTION'],
})
export class AppModule {}

Ngoài ra, bạn có thể export với toàn bộ provider object:

const connectionFactory = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider],
};

@Module({
  providers: [connectionFactory],
  exports: [connectionFactory],
})
export class AppModule {}
