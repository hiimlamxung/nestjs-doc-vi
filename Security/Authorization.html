Tiếng Việt Tài liệu NestJS được dịch sang tiếng Việt

Authorization

Authorization đề cập đến quá trình xác định những gì một user có thể làm. Ví dụ, một user quản trị được phép tạo, chỉnh sửa và xóa bài viết. Một user không phải quản trị chỉ được phép đọc các bài viết.

Authorization là trực giao và độc lập với authentication. Tuy nhiên, authorization yêu cầu một cơ chế authentication.

Có nhiều cách tiếp cận và chiến lược khác nhau để xử lý authorization. Cách tiếp cận được chọn cho bất kỳ dự án nào phụ thuộc vào yêu cầu cụ thể của ứng dụng đó. Chương này trình bày một số cách tiếp cận authorization có thể được điều chỉnh cho nhiều yêu cầu khác nhau.

Triển khai RBAC cơ bản #

Role-based access control (RBAC) là một cơ chế kiểm soát truy cập trung lập về chính sách, được định nghĩa xoay quanh các role và quyền hạn. Trong phần này, chúng ta sẽ minh họa cách triển khai một cơ chế RBAC rất cơ bản sử dụng guards của Nest.

Đầu tiên, hãy tạo một Role enum đại diện cho các role trong hệ thống:

role.enum 
export enum Role {
  User = 'user',
  Admin = 'admin',
}
Gợi ý

Trong các hệ thống phức tạp hơn, bạn có thể lưu trữ các role trong cơ sở dữ liệu, hoặc lấy chúng từ nhà cung cấp authentication bên ngoài.

Với điều này, chúng ta có thể tạo một decorator @Roles(). Decorator này cho phép chỉ định những role nào được yêu cầu để truy cập các tài nguyên cụ thể.

roles.decorator 
import { SetMetadata } from '@nestjs/common';
import { Role } from '../enums/role.enum';

export const ROLES_KEY = 'roles';
export const Roles = (...roles: Role[]) => SetMetadata(ROLES_KEY, roles);
import { SetMetadata } from '@nestjs/common';

export const ROLES_KEY = 'roles';
export const Roles = (...roles) => SetMetadata(ROLES_KEY, roles);

Bây giờ chúng ta đã có decorator @Roles() tùy chỉnh, chúng ta có thể sử dụng nó để decorate bất kỳ route handler nào.

cats.controller 
@Post()
@Roles(Role.Admin)
create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
@Roles(Role.Admin)
@Bind(Body())
create(createCatDto) {
  this.catsService.create(createCatDto);
}

Cuối cùng, chúng ta tạo một class RolesGuard sẽ so sánh các role được gán cho user hiện tại với các role thực sự được yêu cầu bởi route đang được xử lý.

roles.guard 
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Reflector } from '@nestjs/core';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.getAllAndOverride<Role[]>(ROLES_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!requiredRoles) {
      return true;
    }
    const { user } = context.switchToHttp().getRequest();
    return requiredRoles.some((role) => user.roles?.includes(role));
  }
}
import { Injectable, Dependencies } from '@nestjs/common';
import { Reflector } from '@nestjs/core';

@Injectable()
@Dependencies(Reflector)
export class RolesGuard {
  constructor(reflector) {
    this.reflector = reflector;
  }

  canActivate(context) {
    const requiredRoles = this.reflector.getAllAndOverride(ROLES_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!requiredRoles) {
      return true;
    }
    const { user } = context.switchToHttp().getRequest();
    return requiredRoles.some((role) => user.roles.includes(role));
  }
}
Gợi ý

Tham khảo phần Reflection và metadata trong chương Execution context để biết thêm chi tiết về việc sử dụng Reflector theo cách nhận biết ngữ cảnh.

Lưu ý

Ví dụ này được đặt tên là "basic" vì chúng ta chỉ kiểm tra sự hiện diện của các role ở cấp route handler. Trong các ứng dụng thực tế, bạn có thể có các endpoint/handler liên quan đến nhiều thao tác, trong đó mỗi thao tác yêu cầu một tập hợp quyền cụ thể. Trong trường hợp này, bạn sẽ phải cung cấp cơ chế kiểm tra role ở đâu đó trong business-logic của bạn, khiến việc bảo trì trở nên khó khăn hơn vì sẽ không có nơi tập trung nào liên kết các quyền với các hành động cụ thể.

Trong ví dụ này, chúng ta giả định rằng request.user chứa instance của user và các role được phép (dưới thuộc tính roles). Trong ứng dụng của bạn, bạn có thể sẽ thực hiện liên kết đó trong authentication guard tùy chỉnh của bạn - xem chương authentication để biết thêm chi tiết.

Để đảm bảo ví dụ này hoạt động, class User của bạn phải có dạng như sau:

user 
class User {
  // ...other properties
  roles: Role[];
}

Cuối cùng, hãy đảm bảo đăng ký RolesGuard, ví dụ, ở cấp controller, hoặc toàn cục:

app.module 
providers: [
  {
    provide: APP_GUARD,
    useClass: RolesGuard,
  },
],

Khi một user với quyền không đủ yêu cầu một endpoint, Nest tự động trả về response sau:

response 
{
  "statusCode": 403,
  "message": "Forbidden resource",
  "error": "Forbidden"
}
Gợi ý

Nếu bạn muốn trả về một error response khác, bạn nên throw exception cụ thể của riêng bạn thay vì trả về giá trị boolean.

Claims-based authorization #

Khi một identity được tạo, nó có thể được gán một hoặc nhiều claim được cấp bởi một bên đáng tin cậy. Một claim là một cặp name-value đại diện cho những gì subject có thể làm, không phải subject là gì.

Để triển khai Claims-based authorization trong Nest, bạn có thể làm theo các bước tương tự mà chúng ta đã trình bày ở trên trong phần RBAC với một điểm khác biệt quan trọng: thay vì kiểm tra các role cụ thể, bạn nên so sánh permissions. Mỗi user sẽ có một tập hợp các permission được gán. Tương tự, mỗi resource/endpoint sẽ xác định những permission nào được yêu cầu (ví dụ, thông qua một decorator @RequirePermissions() chuyên dụng) để truy cập chúng.

cats.controller 
@Post()
@RequirePermissions(Permission.CREATE_CAT)
create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
@Post()
@RequirePermissions(Permission.CREATE_CAT)
@Bind(Body())
create(createCatDto) {
  this.catsService.create(createCatDto);
}
Gợi ý

Trong ví dụ trên, Permission (tương tự Role mà chúng ta đã trình bày trong phần RBAC) là một TypeScript enum chứa tất cả các permission có sẵn trong hệ thống của bạn.

Tích hợp CASL #

CASL là một thư viện authorization đẳng cấu (isomorphic) giới hạn những tài nguyên mà một client nhất định được phép truy cập. Nó được thiết kế để có thể áp dụng dần dần và có thể dễ dàng mở rộng từ authorization dựa trên claim đơn giản đến authorization dựa trên subject và attribute đầy đủ tính năng.

Để bắt đầu, trước tiên hãy cài đặt package @casl/ability:

terminal 
$ npm i @casl/ability
Gợi ý

Trong ví dụ này, chúng ta chọn CASL, nhưng bạn có thể sử dụng bất kỳ thư viện nào khác như accesscontrol hoặc acl, tùy thuộc vào sở thích và nhu cầu dự án của bạn.

Sau khi cài đặt xong, để minh họa cơ chế của CASL, chúng ta sẽ định nghĩa hai entity class: User và Article.

user.entity 
class User {
  id: number;
  isAdmin: boolean;
}
article.entity 
class Article {
  id: number;
  isPublished: boolean;
  authorId: number;
}

Bây giờ hãy xem xét và tinh chỉnh các yêu cầu cho ví dụ này:

  • Admin có thể quản lý (create/read/update/delete) tất cả entity
  • User có quyền truy cập chỉ đọc đối với mọi thứ
  • User có thể cập nhật bài viết của họ (article.authorId === userId)
  • Bài viết đã được xuất bản không thể bị xóa (article.isPublished === true)

Với điều này, chúng ta có thể bắt đầu bằng việc tạo một Action enum đại diện cho tất cả các hành động có thể mà user có thể thực hiện với các entity:

action.enum 
export enum Action {
  Manage = 'manage',
  Create = 'create',
  Read = 'read',
  Update = 'update',
  Delete = 'delete',
}
Lưu ý

manage là một từ khóa đặc biệt trong CASL đại diện cho "bất kỳ hành động nào".

Để đóng gói thư viện CASL, hãy tạo CaslModule và CaslAbilityFactory ngay bây giờ.

terminal 
$ nest g module casl
$ nest g class casl/casl-ability.factory

Với điều này, chúng ta có thể định nghĩa method createForUser() trên CaslAbilityFactory. Method này sẽ tạo object Ability cho một user nhất định:

casl-ability.factory 
type Subjects = InferSubjects<typeof Article | typeof User> | 'all';

export type AppAbility = MongoAbility<[Action, Subjects]>;

@Injectable()
export class CaslAbilityFactory {
  createForUser(user: User) {
    const { can, cannot, build } = new AbilityBuilder(createMongoAbility);

    if (user.isAdmin) {
      can(Action.Manage, 'all'); // read-write access to everything
    } else {
      can(Action.Read, 'all'); // read-only access to everything
    }

    can(Action.Update, Article, { authorId: user.id });
    cannot(Action.Delete, Article, { isPublished: true });

    return build({
      detectSubjectType: (item) =>
        item.constructor as ExtractSubjectType<Subjects>,
    });
  }
}
Lưu ý

all là một từ khóa đặc biệt trong CASL đại diện cho "bất kỳ subject nào".

Gợi ý

Các class MongoAbility, AbilityBuilder, AbilityClass, và ExtractSubjectType được export từ package @casl/ability.

Gợi ý

Tùy chọn detectSubjectType cho phép CASL hiểu cách lấy subject type từ một object. Để biết thêm thông tin, đọc tài liệu CASL để biết chi tiết.

Trong ví dụ trên, chúng ta đã tạo instance MongoAbility sử dụng class AbilityBuilder. Như bạn có thể đoán, can và cannot chấp nhận cùng các đối số nhưng có ý nghĩa khác nhau, can cho phép thực hiện một hành động trên subject được chỉ định và cannot cấm. Cả hai đều có thể chấp nhận tối đa 4 đối số. Để tìm hiểu thêm về các hàm này, hãy truy cập tài liệu chính thức của CASL.

Cuối cùng, hãy đảm bảo thêm CaslAbilityFactory vào mảng providers và exports trong định nghĩa module CaslModule:

casl.module 
import { Module } from '@nestjs/common';
import { CaslAbilityFactory } from './casl-ability.factory';

@Module({
  providers: [CaslAbilityFactory],
  exports: [CaslAbilityFactory],
})
export class CaslModule {}

Với điều này, chúng ta có thể inject CaslAbilityFactory vào bất kỳ class nào sử dụng constructor injection tiêu chuẩn miễn là CaslModule được import trong host context:

example 
constructor(private caslAbilityFactory: CaslAbilityFactory) {}

Sau đó sử dụng nó trong class như sau.

example 
const ability = this.caslAbilityFactory.createForUser(user);
if (ability.can(Action.Read, 'all')) {
  // "user" has read access to everything
}
Gợi ý

Tìm hiểu thêm về class MongoAbility trong tài liệu chính thức của CASL.

Ví dụ, giả sử chúng ta có một user không phải admin. Trong trường hợp này, user có thể đọc bài viết, nhưng việc tạo mới hoặc xóa các bài viết hiện có nên bị cấm.

example 
const user = new User();
user.isAdmin = false;

const ability = this.caslAbilityFactory.createForUser(user);
ability.can(Action.Read, Article); // true
ability.can(Action.Delete, Article); // false
ability.can(Action.Create, Article); // false
Gợi ý

Mặc dù cả class MongoAbility và AbilityBuilder đều cung cấp các method can và cannot, chúng có mục đích khác nhau và chấp nhận các đối số hơi khác nhau.

Ngoài ra, như chúng ta đã chỉ định trong các yêu cầu, user có thể cập nhật bài viết của mình:

example 
const user = new User();
user.id = 1;

const article = new Article();
article.authorId = user.id;

const ability = this.caslAbilityFactory.createForUser(user);
ability.can(Action.Update, article); // true

article.authorId = 2;
ability.can(Action.Update, article); // false

Như bạn có thể thấy, instance MongoAbility cho phép chúng ta kiểm tra quyền theo cách khá dễ đọc. Tương tự, AbilityBuilder cho phép chúng ta định nghĩa quyền (và chỉ định các điều kiện khác nhau) theo cách tương tự. Để tìm thêm ví dụ, hãy truy cập tài liệu chính thức.

Nâng cao: Triển khai PoliciesGuard #

Trong phần này, chúng ta sẽ minh họa cách xây dựng một guard phức tạp hơn, kiểm tra xem user có đáp ứng các authorization policy cụ thể có thể được cấu hình ở cấp method hay không.

Đầu tiên, hãy làm rõ các yêu cầu. Mục tiêu là cung cấp một cơ chế cho phép chỉ định kiểm tra policy cho mỗi route handler. Chúng ta sẽ hỗ trợ cả object và function.

Hãy bắt đầu bằng việc định nghĩa các interface cho policy handler:

policy-handler.interface 
import { AppAbility } from '../casl/casl-ability.factory';

interface IPolicyHandler {
  handle(ability: AppAbility): boolean;
}

type PolicyHandlerCallback = (ability: AppAbility) => boolean;

export type PolicyHandler = IPolicyHandler | PolicyHandlerCallback;

Với điều này, chúng ta có thể tạo decorator @CheckPolicies().

check-policies.decorator 
export const CHECK_POLICIES_KEY = 'check_policy';
export const CheckPolicies = (...handlers: PolicyHandler[]) =>
  SetMetadata(CHECK_POLICIES_KEY, handlers);

Bây giờ hãy tạo PoliciesGuard:

policies.guard 
@Injectable()
export class PoliciesGuard implements CanActivate {
  constructor(
    private reflector: Reflector,
    private caslAbilityFactory: CaslAbilityFactory,
  ) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const policyHandlers =
      this.reflector.get<PolicyHandler[]>(
        CHECK_POLICIES_KEY,
        context.getHandler(),
      ) || [];

    const { user } = context.switchToHttp().getRequest();
    const ability = this.caslAbilityFactory.createForUser(user);

    return policyHandlers.every((handler) =>
      this.execPolicyHandler(handler, ability),
    );
  }

  private execPolicyHandler(handler: PolicyHandler, ability: AppAbility) {
    if (typeof handler === 'function') {
      return handler(ability);
    }
    return handler.handle(ability);
  }
}
Gợi ý

Trong ví dụ này, chúng ta giả định rằng request.user chứa instance của user. Trong ứng dụng của bạn, bạn có thể sẽ thực hiện liên kết đó trong authentication guard tùy chỉnh của bạn - xem chương authentication để biết thêm chi tiết.

Cuối cùng, để kiểm tra guard này, hãy gắn nó vào bất kỳ route handler nào:

articles.controller 
@Get()
@UseGuards(PoliciesGuard)
@CheckPolicies((ability: AppAbility) => ability.can(Action.Read, Article))
findAll() {
  return this.articlesService.findAll();
}

Ngoài ra, chúng ta có thể định nghĩa một class implement interface IPolicyHandler:

read-article-policy.handler 
export class ReadArticlePolicyHandler implements IPolicyHandler {
  handle(ability: AppAbility) {
    return ability.can(Action.Read, Article);
  }
}

Và sử dụng nó như sau:

articles.controller 
@Get()
@UseGuards(PoliciesGuard)
@CheckPolicies(new ReadArticlePolicyHandler())
findAll() {
  return this.articlesService.findAll();
}
Lưu ý

Vì chúng ta phải khởi tạo policy handler tại chỗ bằng từ khóa new, class ReadArticlePolicyHandler không thể sử dụng Dependency Injection. Điều này có thể được giải quyết với method ModuleRef#get (đọc thêm tại đây). Về cơ bản, thay vì đăng ký function và instance thông qua decorator @CheckPolicies(), bạn phải cho phép truyền một Type<IPolicyHandler>. Sau đó, bên trong guard của bạn, bạn có thể lấy instance sử dụng type reference: moduleRef.get(YOUR_HANDLER_TYPE) hoặc thậm chí khởi tạo động bằng method ModuleRef#create.
