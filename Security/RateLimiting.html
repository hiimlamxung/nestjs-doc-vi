Rate Limiting

Một kỹ thuật phổ biến để bảo vệ ứng dụng khỏi các cuộc tấn công brute-force là rate-limiting. Để bắt đầu, bạn cần cài đặt package @nestjs/throttler.

$ npm i --save @nestjs/throttler

Sau khi cài đặt xong, ThrottlerModule có thể được cấu hình như bất kỳ package Nest nào khác với các phương thức forRoot hoặc forRootAsync.

app.module
@Module({
  imports: [
     ThrottlerModule.forRoot({
      throttlers: [
        {
          ttl: 60000,
          limit: 10,
        },
      ],
    }),
  ],
})
export class AppModule {}

Đoạn code trên sẽ thiết lập các tùy chọn toàn cục cho ttl, thời gian sống tính bằng milliseconds, và limit, số lượng request tối đa trong khoảng thời gian ttl, cho các route của ứng dụng được bảo vệ bởi guard.

Sau khi module đã được import, bạn có thể chọn cách bind ThrottlerGuard. Bất kỳ kiểu binding nào như đã đề cập trong phần guards đều được. Ví dụ, nếu bạn muốn bind guard một cách toàn cục, bạn có thể thực hiện bằng cách thêm provider này vào bất kỳ module nào:

{
  provide: APP_GUARD,
  useClass: ThrottlerGuard
}

Multiple Throttler Definitions #

Có thể sẽ có lúc bạn muốn thiết lập nhiều định nghĩa throttling, ví dụ không quá 3 lần gọi trong một giây, 20 lần gọi trong 10 giây, và 100 lần gọi trong một phút.

app.module
@Module({
  imports: [
    ThrottlerModule.forRoot([
      {
        name: 'short',
        ttl: 1000,
        limit: 3,
      },
      {
        name: 'medium',
        ttl: 10000,
        limit: 20
      },
      {
        name: 'long',
        ttl: 60000,
        limit: 100
      }
    ]),
  ],
})
export class AppModule {}

Customization #

Có thể sẽ có lúc bạn muốn bind guard vào một controller hoặc toàn cục, nhưng muốn tắt rate limiting cho một hoặc nhiều endpoint của bạn. Để làm điều đó, bạn có thể sử dụng decorator @SkipThrottle(), để bỏ qua throttler cho toàn bộ class hoặc một route đơn lẻ. Decorator @SkipThrottle() cũng có thể nhận vào một object với các key là string và value là boolean cho trường hợp bạn muốn loại trừ hầu hết controller, nhưng không phải mọi route, và cấu hình theo từng throttler set nếu bạn có nhiều hơn một. Nếu bạn không truyền object, mặc định sẽ sử dụng { default: true }.

@SkipThrottle()
@Controller('users')
export class UsersController {}

Decorator @SkipThrottle() này có thể được sử dụng để bỏ qua một route hoặc một class hoặc để phủ định việc bỏ qua một route trong một class đã được bỏ qua.

@SkipThrottle()
@Controller('users')
export class UsersController {
  // Rate limiting is applied to this route.
  @SkipThrottle({ default: false })
  dontSkip() {
    return 'List users work with Rate limiting.';
  }
  // This route will skip rate limiting.
  doSkip() {
    return 'List users work without Rate limiting.';
  }
}

Ngoài ra còn có decorator @Throttle() có thể được sử dụng để ghi đè limit và ttl đã thiết lập trong module toàn cục, để cung cấp các tùy chọn bảo mật chặt chẽ hơn hoặc lỏng hơn.

// Override default configuration for Rate limiting and duration.
@Throttle({ default: { limit: 3, ttl: 60000 } })
@Get()
findAll() {
  return "List users works with custom rate limiting.";
}

Proxies #

Nếu ứng dụng của bạn đang chạy phía sau một proxy server, điều cần thiết là phải cấu hình HTTP adapter để tin tưởng proxy. Bạn có thể tham khảo các tùy chọn HTTP adapter cụ thể cho Express và Fastify để kích hoạt thiết lập trust proxy.

Đây là một ví dụ:

main
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';
import { NestExpressApplication } from '@nestjs/platform-express';

async function bootstrap() {
  const app = await NestFactory.create<NestExpressApplication>(AppModule);
  app.set('trust proxy', 'loopback'); // Trust requests from the loopback address
  await app.listen(3000);
}

bootstrap();
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.set('trust proxy', 'loopback'); // Trust requests from the loopback address
  await app.listen(3000);
}

bootstrap();

Việc bật trust proxy cho phép bạn lấy địa chỉ IP gốc từ header X-Forwarded-For. Bạn cũng có thể tùy chỉnh hành vi của ứng dụng bằng cách ghi đè phương thức getTracker():

throttler-behind-proxy.guard
import { ThrottlerGuard } from '@nestjs/throttler';
import { Injectable } from '@nestjs/common';

@Injectable()
export class ThrottlerBehindProxyGuard extends ThrottlerGuard {
  protected async getTracker(req: Record<string, any>): Promise<string> {
    return req.ips.length ? req.ips[0] : req.ip; // individualize IP extraction to meet your own needs
  }
}
Gợi ý

Bạn có thể tìm API của object req Request cho express tại đây và cho fastify tại đây.

Websockets #

Module này có thể hoạt động với websockets, nhưng cần một số mở rộng class. Bạn có thể mở rộng ThrottlerGuard và ghi đè phương thức handleRequest như sau:

@Injectable()
export class WsThrottlerGuard extends ThrottlerGuard {
  async handleRequest(requestProps: ThrottlerRequest): Promise<boolean> {
    const {
      context,
      limit,
      ttl,
      throttler,
      blockDuration,
      getTracker,
      generateKey,
    } = requestProps;

    const client = context.switchToWs().getClient();
    const tracker = client._socket.remoteAddress;
    const key = generateKey(context, tracker, throttler.name);
    const { totalHits, timeToExpire, isBlocked, timeToBlockExpire } =
      await this.storageService.increment(
        key,
        ttl,
        limit,
        blockDuration,
        throttler.name,
      );

    const getThrottlerSuffix = (name: string) =>
      name === 'default' ? '' : `-${name}`;

    // Throw an error when the user reached their limit.
    if (isBlocked) {
      await this.throwThrottlingException(context, {
        limit,
        ttl,
        key,
        tracker,
        totalHits,
        timeToExpire,
        isBlocked,
        timeToBlockExpire,
      });
    }

    return true;
  }
}
Gợi ý

Nếu bạn đang sử dụng ws, cần thay thế _socket bằng conn

Có một vài điều cần lưu ý khi làm việc với WebSockets:

  • Guard không thể được đăng ký với APP_GUARD hoặc app.useGlobalGuards()
  • Khi đạt đến giới hạn, Nest sẽ phát ra một event exception, vì vậy hãy đảm bảo có một listener sẵn sàng cho điều này
Gợi ý

Nếu bạn đang sử dụng package @nestjs/platform-ws, bạn có thể sử dụng client._socket.remoteAddress thay thế.

GraphQL #

ThrottlerGuard cũng có thể được sử dụng để làm việc với các request GraphQL. Một lần nữa, guard có thể được mở rộng, nhưng lần này phương thức getRequestResponse sẽ được ghi đè:

@Injectable()
export class GqlThrottlerGuard extends ThrottlerGuard {
  getRequestResponse(context: ExecutionContext) {
    const gqlCtx = GqlExecutionContext.create(context);
    const ctx = gqlCtx.getContext();
    return { req: ctx.req, res: ctx.res };
  }
}

Configuration #

Các tùy chọn sau đây hợp lệ cho object được truyền vào mảng options của ThrottlerModule:

nametên để theo dõi nội bộ throttler set nào đang được sử dụng. Mặc định là default nếu không được truyền
ttlsố milliseconds mà mỗi request sẽ tồn tại trong storage
limitsố lượng request tối đa trong khoảng thời gian TTL
blockDurationsố milliseconds mà request sẽ bị chặn trong khoảng thời gian đó
ignoreUserAgentsmột mảng các biểu thức chính quy của user-agents cần bỏ qua khi throttling request
skipIfmột function nhận vào ExecutionContext và trả về boolean để bỏ qua logic throttler. Giống như @SkipThrottler(), nhưng dựa trên request

Nếu bạn cần thiết lập storage, hoặc muốn sử dụng một số tùy chọn trên theo nghĩa toàn cục hơn, áp dụng cho mỗi throttler set, bạn có thể truyền các tùy chọn trên qua key throttlers và sử dụng bảng dưới đây:

storagemột custom storage service cho nơi lưu trữ theo dõi throttling. Xem tại đây.
ignoreUserAgentsmột mảng các biểu thức chính quy của user-agents cần bỏ qua khi throttling request
skipIfmột function nhận vào ExecutionContext và trả về boolean để bỏ qua logic throttler
throttlersmột mảng các throttler set, được định nghĩa sử dụng bảng ở trên
errorMessagemột string HOẶC một function nhận vào ExecutionContext và ThrottlerLimitDetail rồi trả về một string để ghi đè thông báo lỗi throttler mặc định
getTrackermột function nhận vào Request và trả về một string để ghi đè logic mặc định của phương thức getTracker
generateKeymột function nhận vào ExecutionContext, tracker string và tên throttler dưới dạng string rồi trả về một string để ghi đè key cuối cùng được sử dụng để lưu trữ giá trị rate limit

Async Configuration #

Bạn có thể muốn lấy cấu hình rate-limiting một cách bất đồng bộ thay vì đồng bộ. Bạn có thể sử dụng phương thức forRootAsync().

Một cách tiếp cận là sử dụng factory function:

@Module({
  imports: [
    ThrottlerModule.forRootAsync({
      imports: [ConfigModule],
      inject: [ConfigService],
      useFactory: (config: ConfigService) => [
        {
          ttl: config.get('THROTTLE_TTL'),
          limit: config.get('THROTTLE_LIMIT'),
        },
      ],
    }),
  ],
})
export class AppModule {}

Bạn cũng có thể sử dụng cú pháp useClass:

@Module({
  imports: [
    ThrottlerModule.forRootAsync({
      imports: [ConfigModule],
      useClass: ThrottlerConfigService,
    }),
  ],
})
export class AppModule {}

Điều này hoàn toàn khả thi, miễn là ThrottlerConfigService implement interface ThrottlerOptionsFactory.

Storages #

Storage tích hợp sẵn là một in memory cache theo dõi các request đã thực hiện cho đến khi chúng vượt qua TTL được thiết lập bởi các tùy chọn toàn cục. Bạn có thể cung cấp storage option riêng cho tùy chọn storage của ThrottlerModule miễn là class đó implement interface ThrottlerStorage.

Đối với distributed servers, bạn có thể sử dụng community storage provider cho Redis để có một nguồn dữ liệu duy nhất.

Lưu ý

ThrottlerStorage có thể được import từ @nestjs/throttler.

Time Helpers #

Có một vài phương thức helper để làm cho các khoảng thời gian dễ đọc hơn nếu bạn muốn sử dụng chúng thay vì định nghĩa trực tiếp. @nestjs/throttler export năm helper khác nhau: seconds, minutes, hours, days, và weeks. Để sử dụng chúng, chỉ cần gọi seconds(5) hoặc bất kỳ helper nào khác, và số milliseconds chính xác sẽ được trả về.

Migration Guide #

Đối với hầu hết mọi người, việc bọc các options trong một mảng là đủ.

Nếu bạn đang sử dụng custom storage, bạn nên bọc ttl và limit trong một mảng và gán nó cho thuộc tính throttlers của object options.

Bất kỳ decorator @SkipThrottle() nào đều có thể được sử dụng để bỏ qua throttling cho các route hoặc method cụ thể.

Bất kỳ decorator @Throttle() nào bây giờ cũng nên nhận vào một object với các key là string, liên quan đến tên của các throttler context (một lần nữa, 'default' nếu không có tên) và giá trị là các object có key limit và ttl.

Quan trọng

ttl hiện tại tính bằng milliseconds. Nếu bạn muốn giữ ttl tính bằng giây để dễ đọc, hãy sử dụng helper seconds từ package này. Nó chỉ đơn giản nhân ttl với 1000 để chuyển sang milliseconds.
